A parser that parses its own grammar

When a standards document needs to pin down an exact text format — the shape of an email address, the header lines of an HTTP request, the syntax of a URI — it rarely does so in prose. Prose is ambiguous. Instead it writes a grammar: a compact set of rules saying which strings are valid and how they decompose. The notation most Internet standards reach for is ABNF, Augmented Backus–Naur Form, defined in RFC 5234. A rule looks like this:

DIGIT = %x30-39

read as “a DIGIT is one character whose byte value is between hexadecimal 30 and 39” — that is, 0 through 9. Rules combine: a concatenation a b means a then b, an alternation a / b means a or b, *a means zero or more copies of a, [a] means optional a, and a rule may name another rule, so grammars nest.

Here is the observation this post is about. ABNF is itself a text format with an exact syntax — so that syntax can be described by an ABNF grammar. RFC 5234 does exactly this: its Section 4 is a grammar, written in ABNF, whose subject is ABNF. A rule is a rulename, then a defined-as (=), then some elements; an alternation is a concatenation followed by zero or more /-separated concatenations; and so on down to the terminal char-val and num-val. The notation describes itself.

That self-description is only a claim on paper until something checks it. This post builds the pieces that turn it into a fact a machine verifies: a datatype for parsed ABNF, a hand-written parser that reads ABNF text into that datatype, a generic interpreter that turns any such datatype back into a working parser, and finally the round trip that feeds ABNF’s own grammar through both and confirms the result is identical to where it started.

The shape of a parsed grammar

First, the target: a Coq datatype that a piece of ABNF text parses into. It mirrors the notation piece for piece. A numeric terminal is either a list of concatenated byte values or a range, tagged with its base; a repetition carries an optional lower and upper bound:

Inductive NumBase := BinBase | DecBase | HexBase.

Inductive NumSpec :=
  | NumConcat : NumBase -> list nat -> NumSpec
  | NumRange  : NumBase -> nat -> nat -> NumSpec.

Record RepeatSpec := mkRepeat {
  repeat_min : option nat;
  repeat_max : option nat
}.

The heart is four mutually recursive types — an element is one of six things (a rule name, a quoted string, a numeric value, a prose description, a parenthesised group, or a bracketed option); an alternation is a list of concatenations; a concatenation is a list of repetitions; a repetition is an optional repeat-count wrapped around an element:

Inductive ABNFElement :=
  | ElemRuleName : string -> ABNFElement
  | ElemCharVal  : string -> ABNFElement
  | ElemNumVal   : NumSpec -> ABNFElement
  | ElemProseVal : string -> ABNFElement
  | ElemGroup    : ABNFAlternation -> ABNFElement
  | ElemOption   : ABNFAlternation -> ABNFElement

with ABNFAlternation :=
  | Alt : list ABNFConcatenation -> ABNFAlternation

with ABNFConcatenation :=
  | Concat : list ABNFRepetition -> ABNFConcatenation

with ABNFRepetition :=
  | Rep : option RepeatSpec -> ABNFElement -> ABNFRepetition.

A whole grammar is a list of rules, each a name, a marker for = versus the incremental =/, and a body alternation:

Inductive DefinedAs := DefEqual | DefIncremental.

Record ABNFRule := mkRule {
  rule_name    : string;
  rule_defined : DefinedAs;
  rule_body    : ABNFAlternation
}.

Definition ABNFRulelist := list ABNFRule.

Nothing here parses anything. It is pure data — the shape a grammar has after it has been read.

The static parser: ABNF text into that data

The first program reads ABNF text and produces one of those ABNFRulelist values. It is written by hand, once, out of small parser combinators (a satisfy that consumes one matching character, a many that repeats, an alt that tries alternatives), one function per grammar production: p_rulename, p_repeat, p_alternation, and so on, ending in a single entry point:

Definition parse_abnf (input : string) : option ABNFRulelist :=
  match p_rulelist input input with
  | Some (rules, _) => Some (List.filter (fun r => negb (String.eqb (rule_name r) "")) rules)
  | None => None
  end.

It returns option because parsing can fail: text that is not well-formed ABNF yields None. On text that is, it yields Some of the parsed rule list, with blank lines filtered out. To see it produce actual data, here is a test from the file. The input is one rule followed by a line feed:

Definition crlf_str : string := String "010"%char EmptyString.

Definition test_input :=
  ("DIGIT = %x30-39" ++ crlf_str)%string.

Example test_rule :
  run (p_rule test_input) test_input
  = Some (mkRule "DIGIT" DefEqual (Alt [Concat [Rep None (ElemNumVal (NumRange HexBase 48 57))]])).
Proof. reflexivity. Qed.

The proof is reflexivity: both sides compute to the same value, so Coq checks the claim by running the parser. Read the result inside-out. The text %x30-39 became NumRange HexBase 48 57 — a hexadecimal range, and 0x30 = 48, 0x39 = 57, the byte values of 0 and 9. That numeric value is the lone element of a single repetition (Rep None, no repeat count) inside a single concatenation inside a one-branch alternation, and the whole rule is named DIGIT and defined with = (DefEqual). The notation on the left became the tree on the right, and the machine agrees it did.

The generic interpreter: that data back into a parser

The second program goes the other way. Given any ABNFRulelist — not a fixed grammar baked in at compile time, but a rule list handed over at run time — it builds a parser for whatever language that grammar describes. It walks the AST: an alternation becomes a chain of <|> (try each branch), a concatenation becomes a sequence, a repetition applies its bounds, and a rule name is looked up in the grammar and followed. The result of a successful parse is a generic labelled tree, a ParseTree, recording what matched. The top-level builder is:

Definition make_parser (grammar : ABNFRulelist) (start_rule : string)
  (depth : nat) (input : string) : Parser string ParseTree :=
  let env := build_env grammar in
  match lookup_rule env start_rule with
  | None => fail
  | Some body =>
    ts <- interp_alternation (resolve_rule env depth) body input ;;
    ret (Node start_rule ts)
  end.

Give it a grammar, a start rule, and some input, and it returns the parse tree — or fails. Two conveniences wrap it: parse_with_abnf takes the grammar as text, parses it with the static parser above, then parses the input with the resulting dynamic parser; and parse_and_print does the same but pretty-prints the tree back to text at the end. Here is the interpreter running on a one-rule grammar:

Definition test_grammar_1 :=
  ("digit = %x30-39" ++ crlf_s)%string.

Example test_interp_digit :
  parse_with_abnf test_grammar_1 "digit" "5" 10 = Some (Node "digit" [Leaf "5"]).
Proof. reflexivity. Qed.

The grammar text digit = %x30-39 was parsed into an AST, that AST was turned into a parser, and that parser was run on the input "5", producing a tree labelled digit with the leaf "5". No parser for digits was written by hand; one was manufactured from the grammar.

The pretty-printer closes the loop. Every character the interpreter matches is kept in a Leaf, so walking the tree and concatenating the leaves recovers the exact text that was parsed:

Definition test_grammar_3 :=
  ("hi = %x68 %x69" ++ crlf_s)%string.

Example test_pretty_print :
  parse_and_print test_grammar_3 "hi" "hi" 10 = Some "hi".
Proof. reflexivity. Qed.

Parse "hi" with a grammar built from hi = %x68 %x69, then print the tree, and you get "hi" back. Parsing and printing are inverse on this input.

The self-hosting check: ABNF, parsed by a parser built from ABNF

Now put the two programs nose to tail on the one grammar that describes themselves. RFC 5234’s Section 4 — the ABNF grammar of ABNF — is transcribed into a Coq string, abnf_section4, character for character. The self-hosting check is this pipeline — the first arrow is the static parser parse_abnf, the second the grammar-driven parser-and-printer print_with_grammar:

g→parseAST→interpret, re-parse, printg′, g \xrightarrow{\ \text{parse}\ } \mathrm{AST} \xrightarrow{\ \text{interpret, re-parse, print}\ } g',

and the claim is the round-trip identity g′=gg' = g — where the second arrow does not use any hand-written ABNF parser. It uses the generic interpreter, driven by the AST that the static parser just produced. If the static parser understood ABNF correctly, and the interpreter builds a faithful parser from any grammar, then feeding ABNF’s own grammar through should reproduce ABNF’s own grammar. Any disagreement between the two programs about what “ABNF” means would surface as a mismatch here.

The file records four checks, in order. The first is a plain sanity check: the transcribed grammar is the length it should be, so nothing was dropped in transcription.

Example grammar_length :
  String.length abnf_grammar = 3388.
Proof. vm_compute. reflexivity. Qed.

The second confirms the static parser accepts the whole grammar text — it does not choke partway through:

Example parse_abnf_grammar_succeeds :
  match parse_abnf_tree abnf_grammar with
  | Some _ => true
  | None => false
  end = true.
Proof. compute. reflexivity. Qed.

The third is a round trip through the whitespace-preserving parser: parse the grammar, print the tree, and get back the original text, every byte including spaces and comments — in symbols, with tt the parse tree of the grammar text gg,

parse(g)=Somet⟹print(t)=g.\mathrm{parse}(g) = \mathrm{Some}\ t \;\implies\; \mathrm{print}(t) = g.

Example roundtrip_abnf_grammar :
  match parse_abnf_tree abnf_grammar with
  | Some t => String.eqb (pretty_print t) abnf_grammar
  | None => false
  end = true.
Proof. compute. reflexivity. Qed.

The fourth is the self-hosting check itself. Parse the Section 4 grammar with the static, typed parser (parse_abnf) to get an ABNFRulelist; hand that rule list to the generic interpreter (print_with_grammar) to build a dynamic parser and printer for the grammar it just parsed; run that dynamic parser on the same Section 4 text, starting from the rulelist rule; and check the printed result is letter-for-letter the input:

Definition abnf_section4_rules : option ABNFRulelist :=
  parse_abnf abnf_section4.

Example selfhost_abnf_section4 :
  match abnf_section4_rules with
  | None => None
  | Some rules => print_with_grammar rules "rulelist" abnf_section4
  end = Some abnf_section4.
Proof. vm_compute. reflexivity. Qed.

The proof is vm_compute followed by reflexivity: these are claims about fixed strings and a fixed grammar, so the right tool is to run the whole pipeline and compare. vm_compute evaluates it — parse the 3388-byte grammar, build the interpreter, re-parse, re-print — and reflexivity confirms the output string is syntactically identical to abnf_section4. There is no appeal to a lemma, no hand-waving; the equality is decided by computation.

Why is “parses itself, and the round trip matches exactly” a real property and not a slogan? Because each arrow is a total program with a type, and the endpoints are concrete strings. parse_abnf abnf_section4 either returns Some rules or it does not, and the check pattern-matches on that. print_with_grammar rules "rulelist" abnf_section4 either returns Some s for some definite string s, or it does not. The Example asserts that s equals the original text — a decidable equality of two byte strings, settled by reflexivity. If the static parser misread a single rule of ABNF, the AST would be wrong, the interpreter would build a different parser, and the printed string would differ; the proof would simply fail to close. That it closes is the content: the hand-written understanding of ABNF and the grammar-driven understanding of ABNF agree, on ABNF, down to the byte.
