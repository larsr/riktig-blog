Cartesian closed categories: where lambda calculus lives

A quick setting of the stage. A category is objects and arrows: for each pair of objects x, y a collection of arrows x ~> y, composing associatively with an identity at every object, and arrows compared up to a per-pair equality written == rather than Coq’s definitional equality. Two pieces of extra structure from the previous post recur here. A terminal object is one into which every object has exactly one arrow — in the category of Coq types and functions it is the one-element type unit. A binary product prod_obj a b comes with two projections pi1, pi2 and a universal pairing: to give an arrow into the product is exactly to give its two coordinates. This post adds the third piece and shows what the three together are for.

The idea it rests on is one every functional programmer already uses without naming it. A function of two arguments is the same thing as a function of one argument that returns a function. Adding two numbers can be seen as a map A × B → C, or equally as a map A → (B → C) that takes the first number and hands back “add that number to —”. No information is lost either way; the two forms are interchangeable, and turning one into the other is called currying. In Coq this is a definition you could write in your sleep:

Definition fn_curry (X A B : Type) (h : prod X A -> B) : X -> A -> B :=
  fun x a => h (x, a).

The whole content of this post is that currying is not a quirk of functions-as-Coq-values but a universal property — a statement about arrows — and that a category in which it holds has exactly enough structure to interpret a typed lambda calculus.

Making the function space an object

Here is the move. In any category the arrows a ~> b form a collection sitting outside the category — you cannot compose an arrow with a collection-of-arrows, or project from one. An exponential object exp_obj a b (written bab^a on the blackboard) puts that collection back inside the category as an object of its own, so that “a function from a to b” becomes something you can pair, project, and pass around like any other object. The universal property that makes exp_obj a b deserve the name is currying. Here is the class:

(* Exponential objects: exp_obj a b internalises the hom a ~> b, with
   eval_mor as counit and curry as the transpose across the product
   adjunction. Uniqueness is stated up to the hom-setoid. *)
Class HasExponentials {O : Type} (C : Category O) `{!HasProducts C} := {
  exp_obj : O -> O -> O;
  eval_mor : forall {a b : O}, prod_obj (exp_obj a b) a ~> b;
  curry : forall {x a b : O}, (prod_obj x a ~> b) -> (x ~> exp_obj a b);
  #[global] curry_proper :: forall x a b : O,
      Proper (equ ==> equ) (@curry x a b);
  curry_beta : forall (x a b : O) (h : prod_obj x a ~> b),
      eval_mor ∘ (curry h ×× cat_id) == h;
  curry_unique : forall (x a b : O) (h : prod_obj x a ~> b)
                        (k : x ~> exp_obj a b),
      eval_mor ∘ (k ×× cat_id) == h -> k == curry h
}.

Three ingredients carry the meaning. eval_mor is application: an arrow prod_obj (exp_obj a b) a ~> b that takes a function-object paired with an argument and produces the result — literally “apply this function to this argument.” curry transposes an arrow prod_obj x a ~> b (a two-input map) into an arrow x ~> exp_obj a b (a one-input map returning a function-object). The notation k ×× cat_id is the product acting on both coordinates at once, running k on the first and the identity on the second, so curry h ×× cat_id prepares a pair for eval_mor to consume.

The two laws are the “exactly one” of currying. curry_beta is the computation rule: apply the curried form and you get the original two- input map back, eval∘(curry h×id)=h\text{eval} \circ (\text{curry } h \times \mathrm{id}) = h. curry_unique says curry h is the only arrow with that property — anything else transposing to h already is curry h. Existence plus uniqueness, the universal-property pattern again.

In Coq’s own category the exponential is nothing but the function type, and eval_mor and curry are the obvious maps:

Definition fn_eval (A B : Type) : prod (A -> B) A -> B :=
  fun p => fst p (snd p).

Definition fn_curry (X A B : Type) (h : prod X A -> B) : X -> A -> B :=
  fun x a => h (x, a).

#[global] Instance CoqExponentials :
  @HasExponentials Type CoqCat CoqProducts :=
  @Build_HasExponentials Type CoqCat CoqProducts
    (fun A B : Type => A -> B)
    fn_eval
    fn_curry
    fn_curry_proper
    fn_curry_beta
    fn_curry_unique.

fn_eval p applies the first component of the pair to the second — fst p (snd p) — which is what “evaluate” means for real functions. fn_curry is the one-liner from the opening. The exponential object is fun A B => A -> B, the function type itself. (One subtlety the source flags: proving curry unique for real functions needs functional extensionality, since two functions agreeing at every argument must be treated as equal; that is the one standard axiom CoqCat leans on, and nothing more.)

Currying and uncurrying are inverse

The universal property says currying can be undone. Going back is uncurry: feed a one-input map to eval_mor after spreading it across the product.

(* The inverse transpose across the product adjunction. *)
Definition uncurry {x a b : O} (k : x ~> exp_obj a b) : prod_obj x a ~> b :=
  eval_mor ∘ (k ×× cat_id).

That curry and uncurry are mutually inverse is the theorem that makes “a two-argument function is a function returning a function” precise: the two hom-collections prod_obj x a ~> b and x ~> exp_obj a b are in bijection, and curry/uncurry are the bijection. Both halves are one line, because each is a law of the class pointed in the right direction.

First direction: uncurrying a curried map returns it. On the blackboard this is exactly β\beta:

uncurry(curry h)=eval∘(curry h×id){definition of uncurry}=h{β, the computation rule} \text{uncurry}(\text{curry } h) \;=\; \text{eval} \circ (\text{curry } h \times \mathrm{id}) \quad\{\text{definition of uncurry}\}\quad =\; h \quad\{\beta\text{, the computation rule}\} 

Theorem uncurry_curry : forall (x a b : O) (h : prod_obj x a ~> b),
    uncurry (curry h) == h.
Proof.
  intros x a b h.
  apply curry_beta.
Qed.

After intros, the goal uncurry (curry h) == h unfolds by the definition of uncurry to eval_mor ∘ (curry h ×× cat_id) == h — and that is the statement of curry_beta, so apply curry_beta finishes it outright. The computation rule and this direction of the inverse are the same fact wearing two names.

Second direction: currying an uncurried map returns it. Here the uniqueness clause does the work. We want curry (uncurry k) == k.

curry(uncurry k)=k,because eval∘(k×id)=uncurry k by definition. \text{curry}(\text{uncurry } k) \;=\; k, \quad\text{because } \text{eval} \circ (k \times \mathrm{id}) \;=\; \text{uncurry } k \text{ by definition.} 

Theorem curry_uncurry : forall (x a b : O) (k : x ~> exp_obj a b),
    curry (uncurry k) == k.
Proof.
  intros x a b k.
  symmetry.
  apply curry_unique.
  reflexivity.
Qed.

symmetry turns the goal into k == curry (uncurry k), the exact shape curry_unique concludes: “if k transposes to uncurry k then k is its currying.” apply curry_unique leaves the single obligation eval_mor ∘ (k ×× cat_id) == uncurry k — but the right-hand side is defined to be the left, so reflexivity closes it. uncurry k was built to be the thing k transposes to, so k is forced to be its currying. The two theorems together: curry and uncurry are inverse bijections between the two collections of arrows.

The cartesian closed bundle

Gather the three properties and you have the structure this post is named for. A cartesian closed category — CCC — is a category with a terminal object, binary products, and exponentials:

(* A cartesian closed category: terminal object, binary products, and
   exponentials. The exponential structure lives over ccc_prod. *)
Class CCC {O : Type} (C : Category O) := {
  #[global] ccc_term :: Terminal C;
  #[global] ccc_prod :: HasProducts C;
  #[global] ccc_exp :: @HasExponentials O C ccc_prod
}.

Read the three fields against a typed lambda calculus and they line up one for one. The terminal object is the unit type — the type with a single trivial value, the result of a program that returns nothing interesting. Products are pair types — forming and taking apart tuples. Exponentials are function types, with eval_mor as application and curry as lambda-abstraction. A typed lambda term with free variables of types A1,…,AnA_1, \dots, A_n and result type BB becomes an arrow A1×…×An→BA_1 \times \dots \times A_n \to B; abstracting a variable is currying; applying a function is eval_mor. Everything a simply-typed lambda calculus can say, a CCC can express as arrows. Coq’s own types and functions are the worked example — the source assembles them into CoqCCC:

#[global] Instance CoqCCC : CCC CoqCat :=
  @Build_CCC Type CoqCat
    CoqTerminal
    CoqProducts
    CoqExponentials.

And the correspondence runs the other way too. In the same source tree, the simply-typed lambda calculus is built up as syntax — intrinsically typed terms, quotiented by βη\beta\eta-conversion — and shown to form a category, Stlc_Category, whose objects are types and whose arrows A ~> B are closed terms of type A → B. That the category laws hold only up to conversion is exactly why arrows were compared with == and not definitional equality from the very first post; and the file’s examples show the arrows are real terms that compute:

Example tcomp_computes :
  tcomp (tid Base) (tid Base)
  = TAbs (TApp (TAbs (TVar here)) (TApp (TAbs (TVar here)) (TVar here))).
Proof. reflexivity. Qed.

Composing two identity terms in this category produces, on the nose, the lambda term λx.(λy.y)((λy.y)x)\lambda x.\,(\lambda y.\,y)\,((\lambda y.\,y)\,x) — identity applied to identity applied to the input — written in de Bruijn form. The arrows of Stlc_Category are lambda terms.
