riktig #7 — The evaluator leaves the building

Seventh post in the series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved, layer by layer, in Coq, so that a change only obliges you to re-establish a few local facts. Every post starts from zero. In post #4 we built the lambda calculus as a Coq datatype with a small-step evaluator and proved the evaluator correct; in post #5 we turned program transformation into setoid rewriting. But everything so far ran inside Coq. Today the evaluator becomes an ordinary executable on your shell — same code, same proofs, new habitat.

Extraction: proofs stay, code leaves

Coq’s extraction mechanism translates a Coq function into a functional language — here Haskell — erasing everything that was only there for the proving. The recipe lives in Lambda/Extract.v and is startlingly short: name the functions you want (step_fn, eval, the CPS translator cps from Lambda/CPS.v), and say how to render the primitive types:

Extract Inductive nat => "Prelude.Integer" ["0" "Prelude.succ"]
  "(\fO fS n -> if n Prelude.== 0 then fO () else fS (n Prelude.- 1))".

That line matters for more than cosmetics. Inside Coq, nat is unary — 3 is S (S (S O)) — which is wonderful for induction and terrible for arithmetic at scale. Extraction maps it to Haskell’s arbitrary-precision Integer, so de Bruijn indices cost machine words, not linked lists. The theorems were proved about the inductive structure; the running code gets the efficient representation. (A small, honest instance of post #6’s slogan: change the representation, keep the theorems.)

The binary

Around the extracted core (programs/lambda-eval/Eval.hs) sits a thin hand-written CLI (Main.hs): a parser from \x. x syntax to de Bruijn terms, and a printer back. make lambda-eval builds it; then:

$ ./programs/lambda-eval/lambda-eval '(\x. x x) (\y. y)'
\x. x
$ ./programs/lambda-eval/lambda-eval --cps '\x. x'
\x. \y. y x

The first line is the self-application classic reducing to the identity. The second runs the CPS translation from post #4 — the very Fixpoint cps we proved things about. Note what --cps does: it does not print the transformed term; it CPS-transforms the program, applies it to the identity continuation, and evaluates. What comes out is therefore the value’s CPS-converted twin from post #4: Ψ(λx.x) = λx. λk. k x, printed with fresh names as \x. \y. y x. The subtlety we could only state as a theorem back then is now observable at the shell.

What is proved, what is trusted

Honesty section. The evaluator is the verified article: post #4 proved step_fn sound and complete against the reduction relation, and eval is just iterated step_fn with fuel. The parser and printer are trusted, unverified Haskell — about eighty lines. So is GHC, and extraction itself. The trusted surface is real but small, boring, and — crucially — localized at the boundary: exactly the architecture we keep advocating, where the interesting invariants live behind a typed interface and the unproved code is a thin shell around it.

This is also the first end-to-end pass of the whole riktig pipeline:

interface→theory→instance→extraction→binary\text{interface} \longrightarrow \text{theory} \longrightarrow \text{instance} \longrightarrow \text{extraction} \longrightarrow \text{binary}

Small, yes. But the same road is the one the bigger cargo will travel: the channel codecs, the protocol state machines, eventually pieces of the kernel-to-be (working name riskabel; stability target, of course, oriskabel). Each will be a Coq structure with laws, a proof, and an extraction — and a shell command you can run.

Next: order and norms landed as interfaces, complex numbers as a field construction, and finite bases as coordinate isomorphisms — the algebra shopping list for analysis.
