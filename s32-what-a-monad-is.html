What a monad is, and why a parser needs one

Some computations might not produce an answer. Looking up a key in a table gives a value or nothing. Dividing gives a number unless the divisor is zero. Parsing the next token gives a token or fails. The trouble starts when you want to run several such computations one after another, feeding each result into the next. Every step has to first check whether the previous one failed, and if it did, stop and fail too. Written out by hand, a sequence of three lookups becomes three nested case analyses, most of the code being the same failure-check copied over and over. A monad is a pair of operations — sequence and inject — with the failure-check for sequencing written once, inside the operation itself, instead of copied at every call site.

Two operations: sequence, and inject

In Coq a computation that may fail is a value of the option type: option A is either Some x, an answer, or None, no answer. The pattern needs two operations. The first, bind, runs a computation and passes its result — if there is one — into a function that produces the next computation. The second, return, takes a plain value and makes the trivial computation that just yields it. The vocabulary is one tiny class per operation, from Interface/Monad.v:

Class HasBind (M : Type -> Type) := bind : forall {A B}, M A -> (A -> M B) -> M B.
Class HasReturn (M : Type -> Type) := ret : forall {A}, A -> M A.

The carrier is not a single type but a type constructor M : Type -> Type — option, or later a parser — because a computation of an A is a value of some wrapper type M A. bind takes an M A and a continuation A -> M B and delivers an M B; ret injects an A into M A. The infix notation >>= is attached to bind, so p >>= f reads “run p, then feed its result to f”.

For option, both operations are a few lines, from Impl/OptionMonad.v:

#[global] Instance Option_HasBind : HasBind option :=
  fun A B m f => match m with
                 | None => None
                 | Some x => f x
                 end.

#[global] Instance Option_HasReturn : HasReturn option :=
  fun A x => Some x.

That single match is the null-check, written once. Every caller that sequences option-returning steps with >>= inherits it and never writes match ... None => None again.

The three laws that make it a monad

Operations with the right types are not enough — a sort function of type list -> list need not sort. bind and ret have to behave, and the behaviour is three equations. The Monad class bundles the vocabulary with those laws, verbatim from Interface/Monad.v:

Class Monad (M : Type -> Type) `{forall A, EqRel (M A)} `{HasBind M} `{HasReturn M} := {
  (* Bundled here for the same reason Interface/Monoid.v's `mon_equiv`
     and Interface/Category.v's `hom_equiv` bundle theirs: `bind_proper`
     applied at reflexivity (both sides literally the same `m`, as in
     Theory/Monad.v's `map_compose`) needs `equ` reflexive, and generic
     rewriting under `==` needs it an Equivalence, one per index A. *)
  #[global] monad_equiv :: forall A, Equivalence (@equ (M A) _);
  (* The continuation argument of bind is compared pointwise on its
     domain A directly (`forall x, f x == g x`), not via `EqRel (A ->
     M B)`: A itself carries no EqRel in general, and quantifying over
     literally every x needs none. *)
  #[global] bind_proper :: forall A B,
    Proper (equ ==> (fun f g => forall x, f x == g x) ==> equ) (@bind M _ A B);
  bind_ret_l : forall A B (x : A) (f : A -> M B), bind (ret x) f == f x;
  bind_ret_r : forall A (m : M A), bind m ret == m;
  bind_assoc : forall A B C (m : M A) (f : A -> M B) (g : B -> M C),
    bind (bind m f) g == bind m (fun x => bind (f x) g)
}.

The equality == is a family of relations, one per index A, because M A and M B are different types and each needs its own notion of sameness. The three laws, on the blackboard — writing >>=\mathbin{>\!\!>\!\!=} for bind and η\eta for return:

η(x)>>=f=f(x),m>>=η=m,(m>>=f)>>=g=m>>=(λx.f(x)>>=g).\eta(x) \mathbin{>\!\!>\!\!=} f \;=\; f(x), \qquad m \mathbin{>\!\!>\!\!=} \eta \;=\; m, \qquad (m \mathbin{>\!\!>\!\!=} f) \mathbin{>\!\!>\!\!=} g \;=\; m \mathbin{>\!\!>\!\!=} \big(\lambda x.\ f(x) \mathbin{>\!\!>\!\!=} g\big).

  • bind_ret_l: injecting x and then binding into f is just f x — ret on the left of a bind does nothing.
  • bind_ret_r: binding a computation into ret gives the computation back — ret on the right does nothing either.
  • bind_assoc: two binds in a row can be regrouped, exactly like the associativity of a monoid, but for sequencing.

For option each law is proved by taking the value apart into Some and None and computing. For associativity: if m=Nonem = \mathrm{None}, both sides short-circuit to None\mathrm{None}; if m=Somexm = \mathrm{Some}\ x, both sides reduce by the definition of bind to the same thing —

(Somex>>=f)>>=g=f(x)>>=g=Somex>>=(λy.f(y)>>=g).(\mathrm{Some}\ x \mathbin{>\!\!>\!\!=} f) \mathbin{>\!\!>\!\!=} g \;=\; f(x) \mathbin{>\!\!>\!\!=} g \;=\; \mathrm{Some}\ x \mathbin{>\!\!>\!\!=} \big(\lambda y.\ f(y) \mathbin{>\!\!>\!\!=} g\big).

The proof is the whole thing in one line, from Impl/OptionMonad.v:

Lemma option_bind_assoc : forall A B C (m : option A) (f : A -> option B) (g : B -> option C),
  bind (bind m f) g == bind m (fun x => bind (f x) g).
Proof. intros A B C m f g. destruct m as [x |]; reflexivity. Qed.

destruct m splits on whether m is Some x or None. If it is None, both sides short-circuit to None; if it is Some x, both sides reduce to bind (f x) g. Either way the two sides are equal by computation, which is what reflexivity checks. The left-identity law is even shorter — the two sides are definitionally equal, so no case split is needed:

Lemma option_bind_ret_l : forall A B (x : A) (f : A -> option B),
  bind (ret x) f == f x.
Proof. intros A B x f. reflexivity. Qed.

With the three laws and the fact that bind respects equality (bind_proper), option is registered as a Monad, and the generic theory — for instance the derived map and its functor laws in Theory/Monad.v — applies to it with no further work.

A second choice when the first fails

Sequencing is only half of what a parser wants. The other half is choice: try one parser, and if it fails, try another. That is a separate structure, Alternative, with its own two operations — fail, the computation that always fails, and alt (written <|>), which runs its left argument and falls back to the right only if the left fails. From Interface/Alternative.v:

Class HasFail (M : Type -> Type) := fail : forall {A}, M A.
Class HasAlt (M : Type -> Type) := alt : forall {A}, M A -> M A -> M A.
Class Alternative (M : Type -> Type) `{forall A, EqRel (M A)} `{HasFail M} `{HasAlt M} := {
  (* Bundled for the same reason Interface/Monad.v's `monad_equiv` is:
     `alt_proper` at reflexivity and generic rewriting under `==` both
     need `equ` an Equivalence, one per index A. *)
  #[global] alt_equiv :: forall A, Equivalence (@equ (M A) _);
  #[global] alt_proper :: forall A, Proper (equ ==> equ ==> equ) (@alt M _ A);
  alt_assoc : forall A (p q r : M A), alt (alt p q) r == alt p (alt q r);
  fail_alt_l : forall A (q : M A), alt fail q == q;
  alt_fail_r : forall A (p : M A), alt p fail == p
}.

These three laws say (M A, <|>, fail) is a monoid: <|> is associative and fail is a two-sided identity —

(p<|>q)<|>r=p<|>(q<|>r),𝖿𝖺𝗂𝗅<|>q=q,p<|>𝖿𝖺𝗂𝗅=p(p \mathbin{<|>} q) \mathbin{<|>} r \;=\; p \mathbin{<|>} (q \mathbin{<|>} r), \qquad \mathsf{fail} \mathbin{<|>} q \;=\; q, \qquad p \mathbin{<|>} \mathsf{fail} \;=\; p

— an always-failing alternative on either side of a choice may be dropped. Note what is not in the list: commutativity. p <|> q is not q <|> p in general, because the choice is ordered — it commits to the left as soon as the left succeeds. For a parser, that is the definition of alt, from Impl/Parser.v, where Parser S T := S -> option (T * S):

#[global] Instance Parser_HasAlt (S : Type) : HasAlt (Parser S) :=
  fun A p q => fun input =>
    match p input with
    | None => q input
    | r => r
    end.

Run p on the input; if it returns None, run q; otherwise keep p’s result. First success wins.

A plausible law that is false

Now the interesting part. bind distributes over + and alt looks like an addition, so it is tempting to expect a distributivity law connecting them: binding a choice into f ought to equal choosing between the two binds. In symbols,

(p<|>q)>>=f=?(p>>=f)<|>(q>>=f).(p \mathbin{<|>} q) \mathbin{>\!\!>\!\!=} f \;\stackrel{?}{=}\; (p \mathbin{>\!\!>\!\!=} f) \mathbin{<|>} (q \mathbin{>\!\!>\!\!=} f).

This is false, and the reason is exactly the ordered, committed nature of alt. On the left, p <|> q decides first, before f is ever consulted. If p succeeds, the choice commits to p’s branch — and if f then fails on p’s result, the whole left side has failed, with no way back to q. On the right, p >>= f and q >>= f are each computed in full before the choice is made, so if p >>= f fails, the choice quietly falls back to q >>= f. The two sides disagree precisely when f turns a success of p into a failure.

The counterexample makes this concrete: let p always succeed with O, let q always succeed with S O, and let f fail exactly on O and succeed otherwise. From Impl/Parser.v:

Definition counter_p : Parser nat nat := ret O.
Definition counter_q : Parser nat nat := ret (S O).
Definition counter_f (n : nat) : Parser nat nat :=
  match n with
  | O => fail
  | S _ => ret n
  end.

Trace both sides on any input. The left side commits to p, whose result O sends f straight to fail:

(p<|>q)>>=f=p>>=f=fO=𝖿𝖺𝗂𝗅.(p \mathbin{<|>} q) \mathbin{>\!\!>\!\!=} f \;=\; p \mathbin{>\!\!>\!\!=} f \;=\; f\,O \;=\; \mathsf{fail}.

The right side lets p >>= f fail and falls back to q >>= f, which succeeds with S O:

(p>>=f)<|>(q>>=f)=𝖿𝖺𝗂𝗅<|>f(SO)=SO.(p \mathbin{>\!\!>\!\!=} f) \mathbin{<|>} (q \mathbin{>\!\!>\!\!=} f) \;=\; \mathsf{fail} \mathbin{<|>} f\,(S\,O) \;=\; S\,O.

fail on the left, a success on the right: the two are not equal. Here is the refutation in full:

Lemma bind_alt_not_distributive :
  ~ (bind (alt counter_p counter_q) counter_f
      == alt (bind counter_p counter_f) (bind counter_q counter_f)).
Proof.
  intros H.
  specialize (H O).
  unfold bind, alt, ret, fail, counter_p, counter_q, counter_f,
    Parser_HasBind, Parser_HasAlt, Parser_HasReturn, Parser_HasFail
    in H.
  simpl in H.
  discriminate H.
Qed.

Walking it:

  • intros H assumes the law holds — the goal is its negation, so we derive a contradiction from H.
  • specialize (H O) applies the assumed equality at one concrete input, the number O. (Parser equality is extensional: two parsers are equal when they agree on every input, so a single disagreeing input is enough to break it.)
  • The unfold and simpl compute both sides at that input. The left reduces to None; the right reduces to Some (S O, O).
  • discriminate H observes that None = Some (S O, O) is impossible — they are different constructors — and closes the goal.

A plausible-sounding law, refuted by a three-line construction. The lesson is that an equation that “looks like distributivity” is not true by resemblance; it has to be checked against what the operations actually do, and here the committed choice of alt is exactly what breaks it.

There is a restricted form that holds — the two sides agree whenever f cannot turn a success of p into a failure, which is the one thing the counterexample violates:

Lemma bind_alt_distr_when_no_backtrack :
  forall S T U (p q : Parser S T) (f : T -> Parser S U),
    (forall input : S, bind p f input = None -> p input = None) ->
    bind (alt p q) f == alt (bind p f) (bind q f).

The hypothesis says: whenever p >>= f fails, it was already p that failed, never f. Under that condition there is nothing for the right side to fall back to that the left side missed, and the distributivity goes through.

Sequencing and choice, together

With both structures in place, a parser is written by combining >>= and <|>. Two characters in sequence, pairing their results, is a bind inside a bind — and it computes:

Example test_bind :
  run (bind (char "a"%char) (fun x => bind (char "b"%char) (fun y => ret (x, y)))) "abc"
    = Some ("a"%char, "b"%char).
Proof. reflexivity. Qed.

And an ordered choice takes the first branch that succeeds — char "x" fails on "abc", so the fallback char "a" runs:

Example test_alt : run (alt (char "x"%char) (char "a"%char)) "abc" = Some "a"%char.
Proof. reflexivity. Qed.

Both are proved by reflexivity: the parser simply runs, and the result is what the definitions compute.
