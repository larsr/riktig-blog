riktig #6 — Choose your representation, keep your theorems

Sixth post in the series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved, layer by layer, in Coq. The goal is modifiability: a change only obliges you to re-establish a few local facts. Every post starts from zero. In post #1 we defined mathematical structures — a carrier, an equality == chosen for the job, operations, laws — and proved things once, abstractly. Today the abstractions meet concrete data: integers, rationals, functions, pairs, lists. And we cash the first big cheque: two representations of the same data are connected by an isomorphism, so you can swap one for the other with a rewrite.

Instances: the laws land on real carriers

An interface is a promise; an instance is a proof that some concrete type keeps it. Z keeps the Ring promise (Impl/Zr.v) — every law is a stdlib equality, wrapped where the orientation differs. More interesting is Q (Impl/Qf.v): rational numbers as fractions, where 1/2 and 2/4 are different data representing the same number. The instance simply declares

#[global] Instance Q_EqRel : EqRel Q := Qeq.

and every generic theorem — cancellation, 0 * a == 0, the lot — holds up to Qeq, automatically. Choosing == is a design act. Hold that thought; it returns twice below.

The workhorse is Impl/Pointwise.v: if V is a vector space, then functions into V form one — operations act at each point, laws hold because they hold at each point. One file, infinitely many spaces: nat -> Q, bool -> V, string -> (nat -> Q)… and the classic (Impl/FinVec.v):

Fn=(Finn→F)F^n \;=\; (\mathrm{Fin}\ n \to F)

n-dimensional coordinate space is the special case X := Fin n, inheriting everything, plus a dot product ⟨u,v⟩=∑iuivi\langle u , v \rangle = \sum_i u_i v_i by finite summation.

(War story, briefly: this file OOM-killed the compiler for a day. The instance search “find me a vector space structure on ?M” — carrier unknown — matches the pointwise instance forever: ?M := ?X -> ?M', recurse. The fix, Hint Mode, tells Coq an instance may only fire once the carrier is known. Infinite loops in proof search, too, are a thing one designs against.)

Yes, () is a vector space

The type unit has one inhabitant, tt. Can it be a vector space? It sounds like a joke question; it is load-bearing. Set every operation to the only thing it can be, and declare everything equal:

#[global] Instance Unit_EqRel : EqRel unit := fun _ _ => True.

Every law is trivially true (Impl/UnitVS.v). The point of the one-point space: it is the unit of the product, the empty record, the zero-column table — and in the category of vector spaces it is the zero object: exactly one linear map in, exactly one out, from anywhere. Products themselves are Impl/ProdVS.v: A * B with componentwise everything.

Lists modulo trailing zeros

Third appearance of “choosing == is a design act” (Impl/ListVS.v): growable coefficient lists, where [1; 0; 0] and [1] should be the same vector. Define lookup-with-default — nth0 xs i is 0 past the end — and let == compare lookups:

Instance LV_EqRel : EqRel (list F) := fun xs ys => forall i, nth0 xs i == nth0 ys i.

Trailing zeros vanish by definition of equality; over Q the equation [1; 0; 0] == [1] is proved by computing three lookups. The laws? Three little lemmas relate lookup to add/neg/scale, and then every vector-space law is the pointwise argument again. A list F is a finitely-supported nat -> F wearing different clothes — which is precisely an isomorphism claim. So let us make those first-class.

Isomorphisms: swap representations with a rewrite

Theory/VSIso.v packages a representation change as two structure-preserving maps that undo each other:

Record VSIso F U V ... := {
  vsi_to   : LinearMap F U V;
  vsi_from : LinearMap F V U;
  vsi_fromto : forall u, vsi_from (vsi_to u) == u;
  vsi_tofrom : forall v, vsi_to (vsi_from v) == v
}.

The starter kit: A × () ≅ A (drop the empty column), A × B ≅ B × A (reorder fields), (A × B) × C ≅ A × (B × C) (regroup nesting — so any tree of pairs normalizes to a right-nested list, by chaining). Each is two evidently-linear maps and round trips that hold by reflexivity-grade rewriting.

And the one this post exists for. A table of records is a record of columns:

(X→A×B)≅(X→A)×(X→B)(X \to A \times B) \;\cong\; (X \to A) \times (X \to B)

Definition vsiso_soa : VSIso F (X -> A * B) ((X -> A) * (X -> B)) := ...

Take X := Fin n and read it in engineering: array-of-structs ≅ struct-of-arrays — the memory-layout transformation behind columnar databases, SIMD kernels, and cache-friendly attention implementations — with a machine-checked guarantee that nothing about the data changes. Splitting is linear, merging is linear, and the round trips are laws you can rewrite with. This is the modifiability thesis in its purest form so far: the representation is a degree of freedom, and the proofs travel with the data.
