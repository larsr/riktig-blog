riktig #6 — Choose your representation, keep your theorems

Sixth post in the series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved, layer by layer, in Coq. The goal is modifiability: a change only obliges you to re-establish a few local facts. Every post starts from zero. In post #1 we defined mathematical structures — a carrier, an equality == chosen for the job, operations, laws — and proved things once, abstractly. Today one concrete payoff, worked end to end: the same data can be laid out in memory two different ways, and the two layouts are provably interchangeable — array-of-structs ≅ struct-of-arrays, as a machine-checked isomorphism. That statement is the whole post: first what it says, then what it needs, then the proof.

The claim, in engineering terms

You have a table of records — say n rows, each holding an A-value and a B-value. There are two standard ways to store it:

  • Array of structs (AoS): one array of n records. As a type: a function from row index to record, Fin n -> A * B.
  • Struct of arrays (SoA): two arrays side by side — all the A-values, then all the B-values. As a type: a pair of functions, (Fin n -> A) * (Fin n -> B).

The second layout is the transformation behind columnar databases and SIMD-friendly kernels. Everybody believes the two layouts hold the same information. The point of this post is to say precisely what “the same information, and the algebra doesn’t care” means, and to prove it once, for every field of scalars, every pair of component spaces, and every index type X (not just Fin n):

(X→A×B)≅(X→A)×(X→B)(X \to A \times B) \;\cong\; (X \to A) \times (X \to B)

Three things need content before that line is a theorem: why each side is a vector space at all, what ≅ means when a machine checks it, and the proof. In the repository the destination is vsiso_soa in Theory/VSIso.v.

Step 1: both sides are vector spaces, for free

Post #1’s structures were abstract: “any carrier with these operations and laws.” An instance is a proof that a concrete type keeps that promise — the repository has the expected ones (Z is a ring, Impl/Zr.v; Q with Qeq as its == is a field, Impl/Qf.v). The two instances the theorem needs are constructions: they build a new vector space from old ones.

Functions into a space form a space (Theory/Pointwise.v). If V is a vector space, so is X -> V, for any X: every operation acts at each point,

#[global] Instance PW_Add {X M : Type} `{HasAdd M} : HasAdd (X -> M) :=
  fun f g x => f x + g x.

equality is “equal at every point” (PW_EqRel), and every law holds because it holds at each point. A typical proof in the file, whole:

Lemma pw_idl : forall f : X -> M, 0 + f == f.
Proof. intros f x. apply mon_idl. Qed.

Read it: to prove two functions ==, prove it at an arbitrary point x — and there it is M’s own law. That single file makes nat -> Q, Fin n -> F (n-dimensional coordinate space, Impl/FinVec.v), and both sides of today’s isomorphism into vector spaces with no new proofs. (One production note, in passing: an instance whose carrier is X -> M can send Coq’s instance search into infinite recursion when the carrier is unknown; this file once ran the compiler out of memory before Hint Mode annotations confined the search.)

Pairs of spaces form a space (Theory/ProdVS.v). A * B with componentwise operations and componentwise equality — the same argument, two components instead of many points.

Now both sides of the claim are vector spaces, assembled from the two constructions: the left side is pointwise over a product, the right side is a product of two pointwise spaces. Same ingredients, different nesting order — which is exactly why one can hope the two are interchangeable.

Step 2: what “≅” means when a machine checks it

A representation change should let you convert either way, lose nothing in a round trip, and — the part that makes theorems transport — commute with the algebra: adding two tables and then converting must equal converting and then adding. The maps that commute with the algebra are the linear maps: functions f with f (u + v) == f u + f v and f (a *: u) == a *: f u, respecting == (Interface/LinearMap.v; they get a proper treatment in post #9). So the repository’s notion of isomorphism-of-representations, VSIso in Theory/VSIso.v, is: two linear maps that undo each other.

Record VSIso (F U V : Type) {FF : Field F}
    {VU : VectorSpace F U} {VV : VectorSpace F V} := {
  vsi_to : LinearMap F U V;
  vsi_from : LinearMap F V U;
  vsi_fromto : forall u : U, vsi_from (vsi_to u) == u;
  vsi_tofrom : forall v : V, vsi_to (vsi_from v) == v
}.

Four fields, each earning its keep: the two maps are the conversions, the two round-trip laws say no information is lost, and linearity says any algebra you do in one representation agrees with doing it in the other. Once you hold a VSIso, every equation about one carrier can be moved to the other by rewriting along these laws — the proofs travel with the data.

Step 3: the proof

On paper. The forward map splits a table into its two columns:

f↦(x↦fst(fx),x↦snd(fx))f \;\longmapsto\; \big(x \mapsto \mathrm{fst}(f\,x),\;\; x \mapsto \mathrm{snd}(f\,x)\big)

The backward map zips two columns back into a table:

(g,h)↦(x↦(gx,hx)).(g, h) \;\longmapsto\; \big(x \mapsto (g\,x,\, h\,x)\big).

Round trip: split then zip gives x↦(fst(fx),snd(fx))x \mapsto (\mathrm{fst}(f\,x), \mathrm{snd}(f\,x)) — at every point, a pair rebuilt from its own components, so equal componentwise. Linearity of the split: addition of tables is pointwise and addition of records is componentwise, so “first component of a sum” is “sum of first components” — the two sides are the same expression read in different orders. There is nothing deep here, and that is the point: once the definitions from steps 1 and 2 are in place, the theorem is bookkeeping, and bookkeeping is what the machine checks best.

In the repository. The two maps, with their linearity proofs attached (each proof is reflexivity at every point, in the sense above):

Definition soa_split : LinearMap F (X -> A * B) ((X -> A) * (X -> B)) :=
  @Build_LinearMap F (X -> A * B) ((X -> A) * (X -> B)) FF VXAB VCols
    (fun f : X -> A * B => (fun x : X => fst (f x), fun x : X => snd (f x)))
    soa_split_proper soa_split_add soa_split_scale.

Definition soa_merge : LinearMap F ((X -> A) * (X -> B)) (X -> A * B) :=
  @Build_LinearMap F ((X -> A) * (X -> B)) (X -> A * B) FF VCols VXAB
    (fun (p : (X -> A) * (X -> B)) (x : X) => (fst p x, snd p x))
    soa_merge_proper soa_merge_add soa_merge_scale.

one round trip, in full — equality of functions is pointwise (intros f x), equality of pairs is componentwise (split), and each component is literally the same term:

Lemma soa_fromto : forall f : X -> A * B,
  (fun x : X => (fst (f x), snd (f x))) == f.
Proof.
  intros f x.
  split.
  - reflexivity.
  - reflexivity.
Qed.

and the record assembled:

Definition vsiso_soa :
  @VSIso F (X -> A * B) ((X -> A) * (X -> B)) FF VXAB VCols :=
  @Build_VSIso F (X -> A * B) ((X -> A) * (X -> B)) FF VXAB VCols
    soa_split soa_merge soa_fromto soa_tofrom.

What was proved, plainly

  • For every field F, every index type X, and every pair of F-vector spaces A, B: splitting a table into columns and zipping columns into a table are linear maps, and they are mutually inverse (vsiso_soa, Theory/VSIso.v). With X := Fin n this is array-of-structs ≅ struct-of-arrays.
  • Consequently any equation about the vector algebra of tables holds in either layout, transported along the round-trip laws. The representation is a degree of freedom; the proofs travel with the data.
  • The same file provides the smaller representation changes the big one composes with: vsiso_unitor (A × () ≅ A — dropping an empty column; the one-point type unit is itself a vector space, Theory/UnitVS.v), vsiso_braiding (A × B ≅ B × A — field order in a record is irrelevant), vsiso_associator (nesting is irrelevant), and vsiso_sym/vsiso_trans so isomorphisms invert and chain — the file’s closing example normalizes A × (B × ()) ≅ B × A purely by chaining.
  • And what is not proved, honestly: nothing about memory or cache behaviour. The theorem says the data and its algebra are preserved exactly; the performance of a layout is precisely the thing you are now free to choose on other grounds.

(One more representation story lives nearby and gets a sentence: growable coefficient lists where [1; 0; 0] and [1] are the same vector — equality by lookup-with-default, so trailing zeros vanish by definition — are Impl/ListVS.v, with the concrete Q demonstration q_trailing_zeros in Impl/Qf.v.)
