riktig #4 — Lambda terms you can run: de Bruijn, combinators, CPS

Fourth post in the series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved, layer by layer, in the Coq proof assistant. The goal is modifiability: code where a change only obliges you to re-establish a few local facts. Every post starts from zero. Post #3 ended on a cliffhanger: cartesian closed categories are exactly the worlds where lambda calculus makes sense. Today we build the lambda calculus itself — as a datatype you can compute with, not just admire.

The lambda calculus is the smallest programming language that is still a programming language: variables, functions, application — three constructions, nothing else, Turing-complete. Formalizing it forces one famous question first.

Variables are not names — they are distances

Textbook lambda terms use names: λx. λy. x y. Names are treacherous. Is λx.x the same program as λy.y? (Yes — but now “same” needs a definition, α-equivalence.) Worse, substituting under a binder can capture a variable that just happened to share a name — the oldest bug in the book, and one that still bites real macro systems and compilers.

De Bruijn’s move: delete the names. A variable is a number — how many binders up is my λ?

λx.λy.xy⇝λ.λ.10\lambda x.\, \lambda y.\, x\, y \quad\rightsquigarrow\quad \lambda.\, \lambda.\, 1\; 0

0 means “the nearest binder”, 1 the one above it. In theories/Lambda/Syntax.v the whole language is three lines:

Inductive tm : Type :=
  | Var (n : nat)
  | Abs (b : tm)
  | App (f a : tm).

What did the numbers buy us?

  • α-equivalence becomes =. λx.x and λy.y are both Abs (Var 0) — the same term, structurally. A whole equivalence relation evaporates.
  • Capture becomes arithmetic. Substitution must adjust indices when it walks under a binder (lift — shift free variables up) — a bookkeeping function with a two-line invariant, instead of a side-condition (“provided x is not free in N…”) that every proof must drag along. The file’s subst/lift are the textbook twenty lines, and a capture-avoidance Example checks the classic trap by reflexivity.

The price is honesty about arithmetic; the prize is that every proof about terms is structural induction with no naming side-conditions. Machines love it. (Humans get used to it.)

Terms you can run

A calculus is not a programming language until it runs. Syntax.v gives evaluation twice, and proves the two agree:

  • a relation step : tm -> tm -> Prop — call-by-value β-reduction with its congruence rules; good for proofs;
  • a function step_fn : tm -> option tm — executable; good for running; with step_fn_sound and step_fn_complete tying them together, and a fuelled driver eval : nat -> tm -> tm on top (eval_sound : star t (eval fuel t)).

So inside Coq, Compute (eval 100 some_term) evaluates lambda code — and the examples in the file are discharged by reflexivity, which means the kernel itself runs the programs when it checks the file:

Example eval_id_id : eval 10 (App idt idt) = idt.
Proof. reflexivity. Qed.

(idt is the identity, Abs (Var 0).) reflexivity succeeds only because both sides compute to the same term. Evaluation is not something we argue about; it is something the kernel performs. (A standalone extracted evaluator — lambda code running as a native binary — is the queued next step, the same extraction pipeline the memory-DSL uses.)

Two side quests, briefly: combinators and CPS

Two more classic transformations live in the repository as runnable, verified artifacts. This post’s one idea is the runnable term datatype above, so each gets a paragraph here and a file you can open.

Combinators (theories/Lambda/SKI.v): three closed combinators — Sfgx=fx(gx)S\,f\,g\,x = f\,x\,(g\,x), Kxy=xK\,x\,y = x, Ix=xI\,x = x — suffice to express every lambda term, so even variables can be deleted. The file defines the combinator language (CS, CK, CI, application @@) with an executable normalizer, and implements bracket abstraction — the algorithm that eliminates a variable — with the theorem babs_correct : forall t u, cstar (babs t @@ u) (csub u t): the abstracted term, applied, reduces to the substitution — β-reduction rebuilt without binders. The classic B, C, K, W basis is derived from S and K, its specifications are Examples discharged by running the normalizer, and S is recovered from B, C, W alone. We will meet combinators again when the lambda track meets the machine-code track.

Continuation-passing style (theories/Lambda/CPS.v): the Fischer/Plotkin call-by-value CPS transform — control flow made into an explicit argument, the continuation — as one Fixpoint cps, plus the de Bruijn index accounting the textbook rules omit (the file’s header comment does that bookkeeping in public). One subtle point the proofs caught: a CPS program does not hand your continuation the original value but its CPS-converted twin (psi in the file), and the verification examples check exactly that by running both sides through the evaluator. The twin becomes observable at the shell in post #7, where cps is extracted into a binary alongside eval.

Where the two posts meet

Post #3 built categories; this post built programs. The junction is theories/Impl/StlcCat.v: simply-typed lambda terms, quotiented by a fragment of βη-conversion — exactly the instances the category laws need, form a Category — objects are types, an arrow is a program, composition is λx. g (f x), and the category laws are conversion facts. This is what the hom-setoid design was built for: “equality” of arrows is programs behave alike, and the whole categorical toolkit from post #3 now applies to syntax.

Next: conversion becomes a first-class rewriting relation — so that IF b THEN g ELSE h with b == TRUE rewrites to g in one step, partial evaluation as algebra. The lambda calculus stops being an object of study and becomes a tool.
