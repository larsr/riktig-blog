Parser combinators: associative sequencing, and repetition that terminates

A parser turns a stream of characters into structured data. The textbook way to write one is a single large function that walks the input and branches on what it sees. That function is hard to read and harder to change: the logic for “a number” is tangled up with the logic for “a comma-separated list of numbers,” and neither can be tested on its own.

The combinator style takes the opposite path. A parser is a small, self-contained value. There is a tiny parser that matches one character, another that matches any character satisfying a predicate, and a handful of ways to combine parsers into bigger ones: run this one and then that one, try this one or else that one, repeat this one as many times as it will go. A parser for a whole grammar is then assembled from these pieces, each of which does one thing and can be understood alone. This post builds that library in Coq, proves that its sequencing operation obeys the algebraic law it should, and then looks closely at the one part that is genuinely subtle: how the repetition combinators are made to terminate.

A parser is a function from input to leftover input

Everything rests on one type. A parser over input type S producing values of type T is a function that, given some input, either fails or returns a value together with the input it did not consume:

Definition Parser (S T : Type) := S -> option (T * S).

The option is the failure channel: None means “this parser did not match here.” A success is Some (v, rest) — the value v that was parsed, and rest, the part of the input still waiting to be read. The leftover input is what makes combination possible: the next parser in a sequence starts exactly where the previous one stopped.

Two parsers count as equal when they behave identically on every possible input — nothing about their internal shape matters, only their results:

#[global] Instance Parser_EqRel (S T : Type) : EqRel (Parser S T) :=
  fun p q => forall s, p s = q s.

This is the == that appears in every law below: p == q unfolds to “for all inputs s, p s = q s.”

Sequencing: the monad structure

A monad is a way to sequence computations that might fail. It is given by two operations: ret x (read return x) is the trivial computation that just produces x and consumes nothing, and bind p f — written p >>= f — runs p, and if it produced a value, hands that value to f to choose the next computation; if p failed, the whole thing fails and f never runs. Sequencing, with automatic short-circuiting on failure, is the entire idea.

For parsers, “fail” is None and “produce a value” also threads the leftover input along. Here are the two operations spelled out for the Parser type:

#[global] Instance Parser_HasBind (S : Type) : HasBind (Parser S) :=
  fun A B p f => fun input =>
    match p input with
    | None => None
    | Some (v, rest) => f v rest
    end.

#[global] Instance Parser_HasReturn (S : Type) : HasReturn (Parser S) :=
  fun A x => fun input => Some (x, input).

Read bind p f as: run p on the input; if it fails, fail; if it succeeds with value v and leftover rest, run the parser f v starting from rest. And ret x is the parser that succeeds immediately with x, touching nothing. This is precisely “do p, then do the next thing where p left off.” Two derived combinators put a convenient face on it — keep the left result, or keep the right:

Definition seqL {S T U : Type} (p : Parser S T) (q : Parser S U) : Parser S T :=
  bind p (fun x => bind q (fun _ => ret x)).

Definition seqR {S T U : Type} (p : Parser S T) (q : Parser S U) : Parser S U :=
  bind p (fun _ => q).

seqL p q runs both but returns p’s value; seqR p q runs both and returns q’s. Both are just two binds in a row.

The law: sequencing is associative

For bind to deserve the name “sequencing,” it must not matter how a chain of three steps is grouped. Bracketing the first two together, or the last two together, has to give the same parser:

(m>>=f)>>=g=m>>=(λx.fx>>=g).(m \mathbin{>\!\!>\!\!=} f) \mathbin{>\!\!>\!\!=} g \;=\; m \mathbin{>\!\!>\!\!=} (\lambda x.\; f\,x \mathbin{>\!\!>\!\!=} g).

The blackboard argument is short. Apply either side to an input. Both begin by running m. If m fails, both sides fail — nothing else runs. If m succeeds with value v and leftover rest, both sides continue in exactly the same way: run f v on rest, and whatever that produces, feed it to g. Since the two sides agree when m fails and agree when m succeeds, they agree on every input. Here is that argument as it appears in the file:

Lemma parser_bind_assoc : forall A B C
    (m : Parser S A) (f : A -> Parser S B) (g : B -> Parser S C),
  bind (bind m f) g == bind m (fun x => bind (f x) g).
Proof.
  intros A B C m f g input.
  unfold bind, Parser_HasBind.
  destruct (m input) as [[v rest] |]; reflexivity.
Qed.

Walking it:

  • intros A B C m f g input fixes the three parsers and — because == means “equal on every input” — one arbitrary input to test them on.
  • unfold bind, Parser_HasBind replaces both binds by their definition, so each side becomes a match on what m does to the input.
  • destruct (m input) as [[v rest] |] is the one case split the argument needed: either m input is Some (v, rest) or it is None. In both branches the two sides have already reduced to the very same expression — in the success branch, both are “run f v on rest, then g”; in the failure branch, both are None — so a single reflexivity closes each. That is why the whole proof is one line after the split.

The two identity laws (ret does nothing when spliced in on either side) hold the same way, and together the three make Parser S a genuine Monad instance — the combinators below are written against that abstract interface, not against the raw match.

The primitive parsers

At the bottom are the parsers that actually inspect a character. satisfy reads one item and keeps it only if a predicate accepts it:

Definition satisfy (seq : Seq C S) (pred : C -> bool) : Parser S C :=
  fun input =>
    match uncons seq input with
    | Some (c, rest) => if pred c then Some (c, rest) else None
    | None => None
    end.

The Seq argument is just a way to pull the first item off the input — for strings, uncons splits String c rest into the head character c and the tail rest, and fails on the empty string. From satisfy, matching one specific character is a one-liner:

Definition char (c : ascii) : Parser string ascii :=
  exact stringSeq Ascii.eqb c.

where exact seq eq_dec c := satisfy seq (eq_dec c) keeps the character only when it equals c. There is also a choice operator, which the repetition combinators lean on — try the first parser, and if it fails, try the second on the same input:

#[global] Instance Parser_HasAlt (S : Type) : HasAlt (Parser S) :=
  fun A p q => fun input =>
    match p input with
    | None => q input
    | r => r
    end.

alt p q (this is the alt in the code below) returns p’s result if p matched, and otherwise falls back to q.

Repetition, and why it terminates

The interesting combinators are the ones that repeat a parser an unknown number of times: many p (zero or more), many1 p (one or more), and sepBy1 p sep (one or more ps separated by sep). These recurse, and the number of iterations is not known in advance — it depends on the input. Coq will not accept a recursive function unless it can see that some argument strictly shrinks at every call; an unbounded loop is rejected outright. So how is many written?

The technique is to use the input string itself as the thing that shrinks. Look at the type: many p has type string -> Parser string (list T), which unfolds to string -> string -> option (list T * string) — two string arguments. The first is a structural bound that the recursion peels one character off at each step; the second is the actual input the parser consumes. You start the loop by seeding the bound with a copy of the input, calling many p input input. Here is the definition:

Fixpoint many {T : Type} (p : Parser string T) (input : string)
    : Parser string (list T) :=
  match input with
  | EmptyString => ret []
  | String _ input' =>
    alt (bind p (fun x => bind (many p input') (fun xs => ret (x :: xs))))
        (ret [])
  end.

Read the two cases. When the bound is EmptyString, stop: return the empty list, matching nothing more. When the bound is String _ input' — at least one character long — try to parse one p, and if that succeeds with value x, recurse with many p input' (the bound one character shorter) to gather the rest into xs, returning x :: xs; if the leading p fails, alt’s second branch returns the empty list. The recursive call passes input', which is a structural sub-part of input, so Coq sees the bound strictly shrink and accepts the definition — no separate fuel counter is introduced.

Why is bounding the loop by the input’s length correct, and not just a convenient fiction? Because each iteration that recurses does so only after p succeeded once, and a parser that makes progress consumes at least one character of the real input: writing |s||s| for the length of a string ss, a progressing parser satisfies

p(s)=Some(v,s′)⟹|s′|<|s|,p(s) = \mathrm{Some}(v, s') \;\implies\; |s'| < |s|,

so after kk iterations the remaining input has length at most |s|−k|s| - k. The bound is a true upper limit on the iteration count, read straight off the input. (It even rescues the pathological case: were p to succeed while consuming nothing, a naive many would loop forever — here it is simply cut off once the bound runs out.)

many1 is many with the first repetition made mandatory — one p, then many:

Definition many1 {T : Type} (p : Parser string T) (input : string)
    : Parser string (list T) :=
  bind p (fun x => bind (many p input) (fun xs => ret (x :: xs))).

And sepBy1 uses the same shrinking-bound recursion to parse a non-empty separated list — a p, then as many sep-then-p groups as follow:

Fixpoint sepBy1 {T U : Type} (p : Parser string T) (sep : Parser string U)
    (input : string) : Parser string (list T) :=
  match input with
  | EmptyString => bind p (fun x => ret [x])
  | String _ input' =>
    bind p (fun x =>
      alt (bind sep (fun _ => bind (sepBy1 p sep input') (fun xs => ret (x :: xs))))
          (ret [x]))
  end.

Again the recursion is structural on the bound input, peeling String _ input' down toward EmptyString, while the real parsing threads through the binds. The parsed prefix — a separator followed by another p — always consumes input, so the same length argument justifies the same bound.

Running it

To read a value back out of a finished parse, run throws away the leftover input and keeps just the value on success:

Definition run {S T : Type} (p : Parser S T) (input : S) : option T :=
  match p input with
  | Some (v, _) => Some v
  | None => None
  end.

The three smoke tests in the file exercise a primitive parser, sequencing, and choice, each on the concrete input string "abc":

Example test_char : run (char "a"%char) "abc" = Some "a"%char.
Proof. reflexivity. Qed.

Example test_bind :
  run (bind (char "a"%char) (fun x => bind (char "b"%char) (fun y => ret (x, y)))) "abc"
    = Some ("a"%char, "b"%char).
Proof. reflexivity. Qed.

Example test_alt : run (alt (char "x"%char) (char "a"%char)) "abc" = Some "a"%char.
Proof. reflexivity. Qed.

Each is proved by reflexivity: the parser and its input are concrete, so Coq computes the result and checks it equals the stated answer. test_char matches the leading a. test_bind sequences “match a, then match b,” returning the pair ("a", "b") — the second char started from the "bc" that the first one left behind. test_alt tries to match x, fails, and falls back to matching a, all without touching the input twice by hand.

Small parsers, a law that lets them be chained without surprises, and a termination measure read off the input itself: that is the whole library.
