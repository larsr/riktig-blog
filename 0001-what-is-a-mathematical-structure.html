riktig #1 — What is a mathematical structure?

This is the first post in a series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved, layer by layer, in the Coq proof assistant — sorting algorithms down to machine code, a small operating-system kernel, network protocols generated from their grammars. The goal is not proof for its own sake: it is modifiability. In ordinary code, changing anything means fearing everything, because any line may depend on an assumption made somewhere else. We want code where a change only obliges you to re-establish a few local facts, and the structure of the system guarantees the rest. Every post starts from zero; you need no Coq background, only patience for a little algebra.

Today: the bottom layer of the whole tower — mathematical structures — what they are, why a software project should care, and how we wrote the first ones.

Structures are interfaces with laws

Programmers already know half of the idea. An interface says which operations exist: this thing can be compared, that thing can be concatenated. What an interface does not say is what the operations mean. A sort function that returns its input unsorted satisfies the interface list → list perfectly.

A mathematical structure is the missing half: a vocabulary of operations plus the laws those operations must obey. Not “there is a +”, but “there is a +, and it never matters how you parenthesize it”. The laws are the meaning. Two structures with the same operations but different laws are different worlds.

The classic first example. Lots of things can be combined: numbers add, strings concatenate, program fragments sequence, rotations compose. Superficially unrelated — but in each case, combining is associative

(a⋅b)⋅c=a⋅(b⋅c)(a \cdot b) \cdot c \;=\; a \cdot (b \cdot c)

and there is a do-nothing element (zero, the empty string, the no-op, the identity rotation) that combines invisibly. That pair of laws has a name: a monoid. The payoff of naming it: anything you prove from the two laws alone is instantly true for numbers, strings, programs, and rotations — and for every combinable thing anyone invents later. Prove once, hold everywhere. That is the entire business model of algebra, and it is precisely the “don’t re-verify the world after every change” property we want for software.

Structures then stack, each adding vocabulary and laws to the last:

Structure Adds New laws (typeset the way we state them)
Monoid +, 0 (a + b) + c == a + (b + c), 0 + a == a, a + 0 == a
Group - a (- a) + a == 0, a + (- a) == 0 — every action can be undone
Abelian group — a + b == b + a — order of combination is irrelevant
Ring *, 1 * is an (associative, unital) second operation, tied to the first by distributivity: a * (b + c) == a * b + a * c
Field inv nonzero elements can be divided: inv a * a == 1 (for a =/= 0)
Vector space *: vectors form an abelian group, scalars from a field act on them: (a + b) *: u == a *: u + b *: u, 1 *: u == u
Inner product ⟨u, v⟩ a symmetric, bilinear pairing: ⟨u, v⟩ == ⟨v, u⟩, ⟨u + v, w⟩ == ⟨u, w⟩ + ⟨v, w⟩
Geometric algebra ga_vec vectors become multipliable: embed V in a ring where v * v == ⟨v, v⟩ *: 1 — a vector squared is its length²

Read the table top to bottom and you can watch capability accumulate: combine → undo → reorder → multiply → divide → scale → measure → multiply geometry itself. Each row inherits every law above it. (Topology does the same trick with a different vocabulary — instead of + and its laws, it is “open sets” and laws about how they union and intersect. Same idea: vocabulary plus laws; only the subject changes.)

How this looks in Coq

Our encoding keeps four things in four kinds of files, and the separation is the point.

1. The vocabulary (theories/Ops.v) — one tiny typeclass per symbol, no laws at all:

Class HasAdd T := add : BinOp T.
Infix "+" := add.

Class Zero T := zero : T.
Notation "'0'" := zero.

This is what lets every later law be written in notation: once a structure provides HasAdd, the literal symbol + works for it.

2. The single laws (theories/Props.v) — one class per law, so a law is a first-class, nameable thing:

Class Assoc T (e : EqRel T) (op : BinOp T) :=
  assoc : forall a b c, op (op a b) c == op a (op b c).

3. The structures (theories/Interface/) — bundles of vocabulary and laws, axioms only, no theorems. Inheritance is containment: a group literally contains a monoid.

Class Monoid T := {
  mon_equ   :: EqRel T;
  mon_zero  :: Zero T;
  mon_add   :: HasAdd T;
  mon_assoc : forall a b c, (a + b) + c == a + (b + c);
  mon_idl   : forall a, 0 + a == a;
  mon_idr   : forall a, a + 0 == a;
  (* ... *)
}.

Class Group G := {
  grp_monoid :: Monoid G;          (* a Group *is* a Monoid... *)
  grp_neg    :: HasNeg G;          (* ...plus undo... *)
  grp_invl   : forall a, (- a) + a == 0;   (* ...and its laws *)
  grp_invr   : forall a, a + (- a) == 0;
  (* ... *)
}.

The :: marks a field as visible to Coq’s inference, so anything that knows “G is a Group” automatically knows “G is a Monoid” — the inheritance in the table is these three characters.

4. The consequences (theories/Theory/) — theorems derived from the laws alone, valid for every instance forever. Here is a real one, exactly as it appears in theories/Theory/Group.v. In a group, cancelling on the left is legitimate:

Theorem cancel_l : forall a b c : G, a + b == a + c -> b == c.
Proof.
  intros a b c Heq.
  rewrite <- (idl b), <- (invl a), assoc, Heq, <- assoc, invl, idl.
  reflexivity.
Qed.

Read the rewrite line as a calculation, one comma per step:

b=0+b=(−a+a)+b=−a+(a+b)=Heq−a+(a+c)=(−a+a)+c=0+c=cb \;=\; 0+b \;=\; (-a+a)+b \;=\; -a+(a+b) \;\overset{\text{Heq}}{=}\; -a+(a+c) \;=\; (-a+a)+c \;=\; 0+c \;=\; c

That is the entire proof style of this project: every proof is a chain of rewrites by named laws — the same equational reasoning you did in school algebra, machine-checked. No proof scripts full of tactic incantations; if a step doesn’t have a name, we factor it out as a lemma and give it a name. The test we hold ourselves to: the Proof block should read like the calculation you would write on paper.

What is proved so far

So that “consequences derived from the laws alone” is not left abstract, here is the actual inventory of the first three theory files — small enough to read in one sitting. Theory/Monoid.v proves the identity element is unique (id_unique, and the specializations zero_unique_l, zero_unique_r). Theory/Group.v proves inverses are unique (neg_unique_l, neg_unique_r), that negation is an involution and injective (neg_involutive, neg_inj), - 0 == 0 (neg_zero), - (a + b) == - b + - a (neg_add_flip), both cancellation laws (cancel_l, cancel_r), and two telescoping facts (add_neg_telescope, sub_eq_zero); it closes with a section showing that any map f preserving + between groups automatically preserves 0 and - (hom_zero, hom_neg). Theory/AbGroup.v adds the reshuffling kit that commutativity makes available: moving a summand past a sum (add_swap_l, add_swap_r), the interchange law (a + b) + (c + d) == (a + c) + (b + d) (add_interchange, with add_rotate as a variant), - (a + b) == - a + - b (neg_add), and a family of cancellation-in-context lemmas (add_sub_cancel, sub_pair, add_neg_cancel_l, and friends) — each stated because some later proof needs exactly that shape of shuffle, and naming it keeps that proof a one-line rewrite.

The payoff at the top of the tower

The chain culminates (for now) in geometric algebra, where the axiom “a vector squares to its length” makes the whole preceding tower earn its keep. The interface (Interface/GeometricAlgebra.v) embeds a vector space V into a ring A by a linear map ga_vec with one law beyond linearity, the contraction axiom:

ga_contract : forall u : V, ga_vec u * ga_vec u == norm2 u *: 1

— a vector squared is its length². From just that and the bilinearity of the inner product, Theory/GeometricAlgebra.v derives the polarization identity

uv+vu=(⟨u,v⟩+⟨v,u⟩)1uv + vu \;=\; (\langle u,v\rangle + \langle v,u\rangle)\,1

— the fact that the symmetric part of the geometric product is the inner product, which is the seed of everything geometric algebra is good for (rotations, reflections, and eventually the matrix-reordering proofs we have planned). Since this is where the tower pays off, the proof deserves the same slow walk as cancel_l.

On paper. Square the sum u+vu + v two ways. Expanding the product, (u+v)2=u2+uv+vu+v2(u+v)^2 = u^2 + uv + vu + v^2, and the contraction axiom turns the two squares into (∥u∥2+∥v∥2)1(\|u\|^2 + \|v\|^2)\,1. Contracting directly, (u+v)2=∥u+v∥21(u+v)^2 = \|u+v\|^2\,1, and bilinearity expands ∥u+v∥2=∥u∥2+∥v∥2+⟨u,v⟩+⟨v,u⟩\|u+v\|^2 = \|u\|^2 + \|v\|^2 + \langle u,v\rangle + \langle v,u\rangle. Equate the two computations and cancel (∥u∥2+∥v∥2)1(\|u\|^2 + \|v\|^2)\,1 from both sides — with exactly the cancel_l proved above.

In the repository, exactly as it appears (a helper lemma contract_add packages the first computation):

Theorem polarization : forall u v : V,
  ga_vec u * ga_vec v + ga_vec v * ga_vec u
  == (⟨ u , v ⟩ + ⟨ v , u ⟩) *: (1 : A).
Proof.
  intros u v.
  apply (cancel_l (norm2 u *: (1 : A) + norm2 v *: 1)).
  rewrite contract_add, norm2_add,
    (sc_add_scale (norm2 u + norm2 v)), (sc_add_scale (norm2 u) (norm2 v)).
  reflexivity.
Qed.

Read it step by step. apply (cancel_l …) is the “cancel from both sides” move — this post’s own first theorem, already earning reuse. The goal is now the two squarings equated, with (∥u∥2+∥v∥2)1(\|u\|^2 + \|v\|^2)\,1 added on both sides. contract_add is the helper: it contracts both squares and collects the cross terms (itself a rewrite chain through add_rotate and distributivity — the abelian-group inventory above at work). norm2_add is the bilinear expansion of ∥u+v∥2\|u+v\|^2, proved one file down in Theory/InnerProduct.v. The two sc_add_scale rewrites split (x+y)*:1(x+y) \mathbin{*:} 1 into x*:1+y*:1x \mathbin{*:} 1 + y \mathbin{*:} 1 so both sides have the same shape, and reflexivity closes. One axiom in the interface, a dozen lines of named rewrites in the theory file, true in every model anyone ever builds.

Why a software project starts here

Because everything upstream is this pattern. “A kernel scheduler never runs one process on two CPUs” is a law over a vocabulary (processes, CPUs, schedule). “Encoding then decoding a protocol message is the identity” is a law over a vocabulary (encode, decode). The discipline we practiced on monoids — vocabulary in one file, laws in another, consequences proved once from laws alone, implementations plugged in underneath — is exactly how we will state what a syscall, a network layer, or a memory allocator means. The algebra files are the training ground where the style is cheap to get right.
