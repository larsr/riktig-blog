riktig #29 — decidable if-then-else

Twenty-ninth post documenting the riktig project. A change of register: instead of the algebra tower’s setoid-heavy proof style, four small, recognizable programs written in a style meant to read like ordinary code.

What we are doing, and why. Verified software in Coq, built for modifiability. Every post so far in this series has come from the algebra/analysis tower — Interface/Theory/Impl, == instead of =, a deliberately narrow tactic set. docs/DESIGN.md names a second, separate style contract for actual programs — “top level reads in the domain’s language,” errors as values, no boilerplate — that hadn’t yet been realized in any Coq code in this repository. Today’s post introduces the concrete pattern that fills that gap: if dec (P) then .. else .., plus Fix/Fix_eq for recursion that doesn’t fit a structural mold, applied to four well-known algorithms.

The one idea underneath everything below

Ordinary Coq forces a choice whenever a condition isn’t a plain bool: either work with bool and lose the connection to the proposition you actually care about, or destruct/case on a Prop and accept that every branch needs its own tactic block. The Decidable typeclass sidesteps that trade-off entirely:

Class Decidable (P : Prop) := dec : {P} + {~ P}.

An instance of Decidable P is a computable proof that P holds or doesn’t — and Coq’s if already knows how to branch on a {P} + {~ P} value. Write if dec (a < b) then .. else .., and each branch receives the actual proposition (a < b, or its negation) in the local context, ready to feed straight into a later termination argument or correctness lemma. Programs/Decidable.v collects the instances needed below: decidable equality on nat/bool/list/ products, decidable order on nat, and the expected logic combinators (and_dec, or_dec, not_dec, …) so compound conditions compose the same way compound booleans would.

Recursion that doesn’t fit a structural mold

Structural recursion — recurse on a literal sub-part of the input — is what plain Fixpoint checks for automatically. Plenty of natural recursions aren’t structural: quicksort recurses on the result of partitioning, not a sub-list of the input; binary search recurses on a shrinking numeric range, not a shrinking list. Coq’s standard-library Fix combinator handles this general case, given a well-founded order and a proof that each recursive call’s argument is smaller. The practical friction is that Fix-defined functions don’t unfold the way a Fixpoint does — reasoning about them needs one lemma, proved once per function, that states the one-step unfolding explicitly:

Lemma gcd_step a b :
  gcd a b = if dec (b = 0) then a else gcd b (a mod b).
Proof. unfold gcd. apply gcd_aux_step. Qed.

Once that lemma exists, later reasoning treats gcd exactly like an ordinary recursive function — the Fix machinery itself never needs touching again.

Four examples

Euclid’s algorithm (Programs/Gcd.v) is the smallest complete demonstration: one decidable condition, one recursive call, well-founded on the shrinking divisor.

Definition gcd_aux : nat -> nat -> nat.
refine (Fix lt_wf (fun _ => nat -> nat)
  (fun b rec a =>
     if dec (b = 0)
     then a
     else rec (a mod b) _ b)).
abstract (apply Nat.mod_upper_bound; assumption).
Defined.

Binary search (Programs/BinarySearch.v) generalizes over an abstract read interface rather than committing to list, the same move quicksort below makes, and needs one supporting fact lia can’t discharge unaided: a bound on integer division ((n-1)/2 <= n-1), needed to show the left half of a search range is always strictly smaller than the whole.

Merge sort (Programs/MergeSort.v) is the clearest illustration of where Fix is and isn’t needed. split and merge both recurse on a literal sub-list, so plain Fixpoint is the right tool for them — no well-founded machinery, no unfolding lemma required. msort is different: it recurses on split’s output, which Coq’s structural checker cannot see as smaller than the input, so msort alone needs Fix, well-founded on list length.

Fixpoint merge (l1 l2 : list nat) : list nat :=
  match l1 with
  | [] => l2
  | x :: xs =>
      let fix merge_r (l2 : list nat) : list nat :=
        match l2 with
        | [] => l1
        | y :: ys =>
            if dec (Nat.leb x y = true)
            then x :: merge xs l2
            else y :: merge_r ys
        end
      in merge_r l2
  end.

Quicksort (Programs/Quicksort.v) is the most substantial of the four, ported closely from ~/coq-memprog’s reference implementation — the source this whole style is drawn from. It works over an abstract memory interface (read/swap on an opaque mem type, rather than a concrete array), with the Lomuto partition scheme as the inner Fix and the recursive sort itself as the outer one:

Definition part (m0 : mem) (store0 i0 hi : nat) : mem * nat.
refine (Fix (well_founded_ltof _ (fun i => hi - i)) (fun _ => mem -> nat -> mem * nat)
  (fun i rec m store =>
    if dec (i < hi)
    then if dec (leb (read m i) (read m hi) = true)
         then rec (S i) _ (swap m i store) (S store)
         else rec (S i) _ m store
    else (swap m store hi, store))
  i0 m0 store0).
all: unfold ltof; abstract lia.
Defined.

What these four programs are, and aren’t, verified to do

Every function above is checked to actually compute the right answer — each file ends with a short block of Check Logic.eq_refl : .. = .. statements that Coq accepts only if both sides genuinely reduce to the same value:

Check Logic.eq_refl : gcd 48 18 = 6.
Check Logic.eq_refl : msort [5;3;8;1;9;2;7] = [1;2;3;5;7;8;9].

That’s real evidence the definitions typecheck and reduce correctly — but it is not a correctness proof. None of these four files show that msort produces a sorted permutation of its input, or that bsearch returns None only when the target is genuinely absent. Matching the scope of the reference implementation exactly, what’s here is the recursion pattern and its unfolding lemma — the infrastructure a correctness proof would build on, not the proof itself. Each file names this explicitly as the natural next step, rather than leaving it an unstated gap.

Why this belongs in riktig

docs/DESIGN.md’s “Program style” section has existed since before this determinant-and-derivative-heavy stretch of posts began, inherited from a sibling project, describing how riktig’s actual verified programs (as opposed to its verified mathematics) should read. It had no code behind it yet. theories/Programs/ is a deliberately separate track from the Interface/Theory/Impl trichotomy that governs the rest of the tree — plain = instead of ==, no restriction on which tactics are in play — matching how Channel/ and Lambda/ already keep their own internal conventions rather than forcing every file in the repository into one mold.
