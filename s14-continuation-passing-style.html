Continuation-passing style, concretely

A lambda term is a variable, an abstraction λ. body, or an application f a. Variables are de Bruijn indices — a number counting the binders out to the λ that bound the variable — so λ.0 is the identity function. We evaluate call-by-value and left-to-right: to run f a, reduce f to a function, reduce a to a value, then substitute the value into the function’s body. A value is anything that is not a pending application: an abstraction, or a variable that is stuck for lack of an argument.

What CPS makes explicit

In ordinary evaluation a function returns: it computes a result and hands it back to whoever called it, and “whoever called it” is implicit machinery, not something the program can name. Continuation-passing style makes that machinery explicit. Every function takes one extra argument — the continuation, conventionally called k — which is a function describing “what happens next.” Instead of returning its result, a CPS function calls the continuation with the result. Nothing ever returns in the ordinary sense; a program is one long chain of tail calls threading a continuation through.

Why go to this trouble: once “what happens next” is an ordinary value, you can do ordinary things with it. You can hold onto a continuation and call it later, call it more than once, or call it from somewhere else entirely. Control-flow features that otherwise each need their own mechanism — early return, exceptions, generators, coroutines — all become the same move: capture a continuation, invoke it when you please. That uniformity is why CPS is a standard intermediate form inside compilers.

The transform

The call-by-value CPS transform (due to Fischer and Plotkin) rewrites each of the three term shapes so that it takes a continuation k and ends by calling k:

[[x]]=λk.kx[\![x]\!] = \lambda k.\,k\,x

[[λ.E]]=λk.k(λx.[[E]])[\![\lambda.E]\!] = \lambda k.\,k\,(\lambda x.\,[\![E]\!])

[[FE]]=λk.[[F]](λf.[[E]](λe.fek))[\![F\,E]\!] = \lambda k.\,[\![F]\!]\,(\lambda f.\,[\![E]\!]\,(\lambda e.\,f\,e\,k))

A variable hands itself to the continuation. An abstraction hands the continuation a CPS-converted function. An application is the interesting case: run [[F]] with a continuation that names its result f, then run [[E]] with a continuation that names its result e, then apply f to e — passing along the outer continuation k so the call knows what to do next. The left-to-right order of evaluation is now literally the nesting order of the continuations.

In de Bruijn form the binders introduced by each rule shift the payloads so their free variables keep pointing at the right place, and that lifting is all the arithmetic in the definition:

Fixpoint cps (t : tm) : tm :=
  match t with
  | Var n => Abs (App (Var 0) (Var (S n)))
  | Abs b => Abs (App (Var 0) (Abs (lift 1 1 (cps b))))
  | App f a =>
      Abs (App (lift 1 0 (cps f))
               (Abs (App (lift 2 0 (cps a))
                         (Abs (App (App (Var 1) (Var 0)) (Var 2))))))
  end.

In the Var rule, x now sits under the new k binder, so index n becomes S n. In the App rule, inside the innermost abstraction the three names in scope are e = 0, f = 1, k = 2, which is why the body is App (App (Var 1) (Var 0)) (Var 2) — f e k.

There is a companion, the value translation Ψ, which converts a value without wrapping it in a continuation-taker: a variable is left alone, and an abstraction has its body CPS-converted.

Definition psi (v : tm) : tm :=
  match v with
  | Abs b => Abs (cps b)
  | other => other
  end.

What the transform guarantees

The heart of the correctness argument is one small step. Take a value v and a continuation c that is itself a value. Then the CPS translation of v, applied to c, takes a single beta step to c applied to Ψ v:

[[v]]c→c(Ψv)[\![v]\!]\,c \;\longrightarrow\; c\,(\Psi\,v)

The subtlety worth pausing on is that the result is c (Ψ v), not c v. A CPS program cannot hand back the syntactically identical value; it hands back the value’s CPS-converted twin. For a variable the twin is the variable itself, but for a function the twin has its body rewritten. Verbatim:

Lemma cps_value_step : forall v c, value v -> value c ->
  step (App (cps v) c) (App c (psi v)).
Proof.
  intros v c Hv Hc. destruct Hv as [n | b]; simpl.
  - rewrite <- (cps_var_beta c n). apply step_beta. exact Hc.
  - rewrite <- (cps_abs_beta c b). apply step_beta. exact Hc.
Qed.

The proof splits on which kind of value v is. In both cases cps v is an abstraction λk. …, so applying it to the value c is a single beta step (step_beta, which fires only when the argument is a value — here c). What remains is to compute the substitution that the beta step performs, and that is exactly what the two helper lemmas say:

Lemma cps_var_beta : forall c n,
  subst 0 c (App (Var 0) (Var (S n))) = App c (Var n).

Lemma cps_abs_beta : forall c b,
  subst 0 c (App (Var 0) (Abs (lift 1 1 (cps b)))) = App c (Abs (cps b)).

For a variable, substituting c for the continuation k turns the body k x into c x. For an abstraction, the same substitution turns k (λ. cps b ↑) into c (λ. cps b) — the outward lift lift 1 1 that the transform inserted is undone precisely by the substitution, which is why the twin psi (Abs b) = Abs (cps b) comes out clean rather than carrying stray index shifts.

Run against the identity continuation specifically, the answer is Ψ v on the nose:

Lemma cps_value_id : forall v, value v ->
  star (App (cps v) idt) (psi v).

One term, translated and run

Take the smallest application there is: the identity applied to itself, (λ.0)(λ.0). Evaluated directly it reduces to the identity λ.0. Now CPS-translate the whole term, apply it to the identity continuation λ.0, and run it. The claim is that it delivers Ψ of the direct result — the CPS twin of the identity, not the identity itself:

Example cps_id_id :
  eval 100 (App (cps (App idt idt)) idt) = psi (eval 10 (App idt idt)).
Proof. reflexivity. Qed.

Because both sides are closed terms, reflexivity settles it: Coq runs the fuelled evaluator on the left and the value translation on the right and finds them identical. Reading off the concrete shape of that common value:

Example cps_id_id_shape :
  eval 100 (App (cps (App idt idt)) idt) = Abs (Abs (App (Var 0) (Var 1))).
Proof. reflexivity. Qed.

The result is λx.λk. k x (in de Bruijn form Abs (Abs (App (Var 0) (Var 1))), where inside the two binders Var 0 is k and Var 1 is x). That is precisely Ψ(λ.0) — the CPS-converted identity, a function that takes its argument and a continuation and hands the argument to the continuation. Direct evaluation returned λ.0; the CPS run returned that function’s twin, which is exactly what cps_value_step predicted, played out through a two-step computation rather than a single beta step.
