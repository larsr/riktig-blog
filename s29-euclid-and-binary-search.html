Euclid’s algorithm and binary search, proved directly

Euclid’s algorithm computes a greatest common divisor by repeatedly replacing the pair (a,b)(a, b) with (b,amod⁡b)(b, a \bmod b); binary search finds a target in a sorted range by repeatedly halving the range. Neither recursion calls itself on a piece of its own input — the arguments of the recursive call are freshly computed numbers — so the reason either one stops is not visible in the shape of the data. What stops them is a quantity that shrinks at every call: the divisor in one case, the width of the range in the other. A proof assistant that accepts only recursions it can see terminating has to be told about that quantity. This post shows the two programs written so that the shrinking quantity, and the proof that it shrinks, are part of the program text itself — and then proves both programs correct.

There are two ways to arrange that in Coq. One is to reach for a special elaboration front-end — Program Fixpoint, or the Equations plugin — that rewrites your definition into something involving well-founded recursion, hides the rewriting, and hands you obligations to discharge afterwards. The other is to write the program as an ordinary lambda term yourself: apply the well-founded recursion combinator Fix by hand, supply the measure that decreases, and prove each recursive call’s side-condition inline. The second way is more typing, but in return the definition you read is the program that runs — no elaboration layer between the source and the term. Both programs below are written that way.

Both programs branch on a decision. A decidable proposition is one for which you can compute a definite yes-or-no answer; the library packs that answer as a value dec P that reduces either to a proof that P holds or to a proof that it does not. The construct if dec P then u else v is an ordinary two-way branch, except that inside the then branch a proof of P is in scope and inside the else branch a proof of its negation is — the same test that steers the computation also hands the correctness proof the fact it needs. That is the only piece of vocabulary these two programs share.

Euclid’s algorithm as one term

The mathematical rule is one line:

gcd⁡(a,b)={aif b=0,gcd⁡(b,amod⁡b)otherwise.\gcd(a, b) \;=\; \begin{cases} a & \text{if } b = 0,\\ \gcd(b,\ a \bmod b) & \text{otherwise.} \end{cases}

The recursion terminates because the second argument strictly shrinks — a mod b < b whenever b is nonzero — even though neither argument is a structural sub-term of the other. Here is the whole definition, from Programs/Gcd.v:

Definition gcd : nat -> nat -> nat.
 apply (uncurry ∘ (Fix (well_founded_ltof (nat * nat) snd) (fun _ => nat))).
  refine
  ( fun p gcd =>
      match p as x return p=x -> nat with
      | (a, b) =>
          fun e =>
          if dec (b = 0)
            then a
            else gcd (b, a mod b) _
      end eq_refl );
  (subst p; unfold ltof; simpl; abstract (now apply Nat.mod_upper_bound)).
 Defined.

Reading it top to bottom: Fix takes a proof that a relation is well-founded and builds a recursor for it. The relation here is ltof (nat * nat) snd — compare two pairs by the second component, snd — and well_founded_ltof supplies the proof that this ordering has no infinite descent. So the recursion is on a single tupled argument p : nat * nat; the leading uncurry ∘ turns the tupled function back into the ordinary two-argument gcd a b we wanted to write. Inside, gcd is the recursive call the recursor provides. The if dec (b = 0) is the base test. In the recursive branch, gcd (b, a mod b) _ leaves a hole _ for the proof that the new pair is smaller; the final line fills every such hole with Nat.mod_upper_bound, the fact that a mod b < b. The abstract seals that proof into a separate opaque lemma so it never bloats the term that runs.

Applying Fix by hand has one consequence: gcd a b does not reduce by simpl the way a plain Fixpoint would, because Fix is guarded by the well-foundedness proof. So the file derives the defining equation — the displayed case rule above, verbatim — as a lemma, gcd_step, once:

Lemma gcd_step: forall a b : nat,
    gcd a b = if dec (b = 0) then a else gcd b (a mod b).
  intros; unfold gcd at 1, compose, uncurry.
  rewrite Fix_eq.
  - destruct dec; reflexivity.
  - intros [x y] f g H.
    destruct dec; [reflexivity | apply H].
 Qed.

Fix_eq is the general lemma that a Fix equals its body applied to itself, provided the body does not inspect which recursive function it was handed — the second bullet discharges exactly that, showing the two branches use the recursive argument the same way under any f and g. From gcd_step on, every proof rewrites with this equation and never touches the Fix again. And the program computes; these are checked in the file by eq_refl, which forces both sides to the same normal form:

Check Logic.eq_refl : gcd 48 18 = 6.
Check Logic.eq_refl : gcd 17 5 = 1.
Check Logic.eq_refl : gcd 0 5 = 5.
Check Logic.eq_refl : gcd 5 0 = 5.
Check Logic.eq_refl : gcd 270 192 = 6.

What “correct” means, and the bridge that proves it

The specification of a greatest common divisor is two facts: gcd a b divides both a and b, and any c that divides both also divides gcd a b (it is greatest in the divisibility ordering). Writing d=gcd⁡(a,b)d = \gcd(a,b):

d∣a,d∣b,∀c,c∣a and c∣b⟹c∣d.d \mid a, \qquad d \mid b, \qquad \forall c,\ \ c \mid a \ \text{ and }\ c \mid b \ \implies\ c \mid d.

Coq’s standard library already proves these for its own Nat.gcd. So the cleanest route is a bridge: show the program computes exactly Nat.gcd, then inherit the specification. The bridge carries the substantial work.

The identity to prove is that the program and the library agree — gcd a b = Nat.gcd a b — by induction on the argument that shrinks. Write GG for the library’s Nat.gcd. On the blackboard, the base case b=0b = 0 is gcd⁡(a,0)=a=G(a,0)\gcd(a,0) = a = G(a,0), and the nonzero case is a chain of three equalities:

gcd⁡(a,b)=gcd⁡(b,amod⁡b){the program’s step, since b≠0}=G(b,amod⁡b){induction hypothesis at the smaller pair: amod⁡b<b}=G(a,b){G(b,amod⁡b)=G(a,b), the library’s own Euclid step.} \begin{aligned} \gcd(a,b) &= \gcd(b,\ a \bmod b) && \{\text{the program's step, since } b \neq 0\}\\ &= G(b,\ a \bmod b) && \{\text{induction hypothesis at the smaller pair: } a \bmod b < b\}\\ &= G(a,b) && \{G(b,\ a \bmod b) = G(a,b)\text{, the library's own Euclid step.}\} \end{aligned}

Here is the proof from Theory/Gcd.v:

  Lemma gcd_eq_stdlib : forall a b, gcd a b = Nat.gcd a b.
  Proof.
    apply (measure_ind2 (fun (_ b : nat) => b)
             (fun a b => gcd a b = Nat.gcd a b)).
    intros a b IH.
    rewrite gcd_step. destruct (dec (b = 0)) as [->|Hb].
    - now rewrite Nat.gcd_0_r.
    - rewrite (IH b (a mod b)) by (apply Nat.mod_upper_bound; exact Hb).
      rewrite (Nat.gcd_comm b (a mod b)), Nat.Lcm0.gcd_mod.
      apply Nat.gcd_comm.
  Qed.

Walking it:

  • measure_ind2 is strong induction along a measure of two arguments; here the measure is fun (_ b) => b, the second argument, so the induction hypothesis IH may be used at any pair whose second component is smaller.
  • rewrite gcd_step replaces gcd a b by its defining if, and destruct (dec (b = 0)) as [->|Hb] splits on the test — the -> substitutes b := 0 in the base case, and Hb : b <> 0 names the hypothesis in the step.
  • In the base case now rewrite Nat.gcd_0_r closes a = Nat.gcd a 0, since Nat.gcd a 0 is a.
  • In the step, rewrite (IH b (a mod b)) is the induction hypothesis — its side-condition a mod b < b is discharged right there by Nat.mod_upper_bound using Hb. That leaves Nat.gcd b (a mod b) = Nat.gcd a b, which the last two rewrites and apply Nat.gcd_comm settle through the standard gcd_comm and gcd_mod lemmas — this is the outer two equalities of the blackboard chain.

With the bridge in hand the specification is three one-line corollaries, each a rewrite followed by the matching standard-library fact:

  Lemma gcd_divides_l : forall a b, Nat.divide (gcd a b) a.
  Proof. intros a b. rewrite gcd_eq_stdlib. apply Nat.gcd_divide_l. Qed.

  Lemma gcd_divides_r : forall a b, Nat.divide (gcd a b) b.
  Proof. intros a b. rewrite gcd_eq_stdlib. apply Nat.gcd_divide_r. Qed.

  Lemma gcd_greatest : forall a b c,
      Nat.divide c a -> Nat.divide c b -> Nat.divide c (gcd a b).
  Proof.
    intros a b c Ha Hb. rewrite gcd_eq_stdlib.
    now apply Nat.gcd_greatest.
  Qed.

Each one rewrites gcd to Nat.gcd and then quotes the library: gcd_divide_l/gcd_divide_r for the two divisibility facts, gcd_greatest for maximality. The program is a common divisor of its arguments, and it is the greatest one.

Binary search, generic in the element type

The second program searches a sorted range for a target. It is written against an abstract read interface rather than a concrete list: a memory type mem, a read : mem -> nat -> V that fetches the element at an index, and a single comparison leb : V -> V -> bool. Nothing else is assumed of the element type V — not even an equality test, so “the element here equals the target” is expressed the textbook antisymmetric way, leb x target = true and leb target x = true both holding.

Like gcd above, the search recurses on a tupled pair — here the two bounds (lo, hi) themselves, not a derived size — well-founded on the measure S hi - lo, the range size. There is one genuine subtlety absent from gcd: descending into the left half needs an upper bound mid - 1, and nat’s truncated subtraction cannot represent “the empty range below index 0” — at mid = lo (which happens once the remaining range holds one or two elements), mid - 1 would silently reinterpret “nothing left to search” as “search index mid - 1 again”. Since lo only ever stays put or grows across the recursion, mid = lo unambiguously means the left half is empty at this call, regardless of lo’s value, so that case is answered directly (None) without ever computing mid - 1. From Programs/BinarySearch.v:

  Definition bsearch : nat -> nat -> option nat.
  apply (uncurry ∘ (Fix (well_founded_ltof (nat * nat) (fun p => S (snd p) - fst p))
           (fun _ => option nat))).
  refine
  ( fun p bsearch =>
      match p as x return p = x -> option nat with
      | (lo, hi) =>
          fun e =>
          if dec (hi < lo)
          then None
          else
            let mid := lo + (hi - lo) / 2 in
            if dec (leb (read m mid) target = true)
            then if dec (leb target (read m mid) = true)
                 then Some mid
                 else bsearch (S mid, hi) _
            else if dec (mid = lo)
                 then None
                 else bsearch (lo, mid - 1) _
      end eq_refl
  );
  (subst p; unfold ltof; cbn [fst snd]; pose proof (half_le (hi - lo)); abstract lia).
  Defined.

Reading it the same way as gcd: the well-founded relation is now on pairs, fun p => S (snd p) - fst p — the range size — via well_founded_ltof, and uncurry ∘ again turns the tupled recursor back into the two-argument bsearch lo hi this file wants. if dec (hi < lo) is the base case, an empty range. Otherwise mid is the midpoint; the program reads it once and asks the same two questions as before: read m mid <= target and target <= read m mid. If both hold, mid is the answer. If only the first holds, the target lies above the midpoint, and the recursion continues into the strictly smaller pair (S mid, hi). Otherwise dec (mid = lo) decides whether the left half is empty (None, no mid - 1 ever formed) or continues into (lo, mid - 1). The two remaining holes are proofs that the new pair is smaller under the measure; both close the same way, pose proof (half_le (hi - lo)) supplying the genuine integer-division bound d / 2 <= d that lia cannot derive on its own, followed by abstract lia. As before, a bsearch_step lemma (in the same file) derives the defining equation through Fix_eq, unfolding one layer of the tupled recursion into exactly this four-way branch.

Instantiated at nat with Nat.leb over a list read through nth_default, it runs — a found case and a not-found case among the checks in the file:

Check Logic.eq_refl : bsearch nat Nat.leb (list nat) (nth_default 0) l 7 0 6 = Some 3.
Check Logic.eq_refl : bsearch nat Nat.leb (list nat) (nth_default 0) l 4 0 6 = None.

with l := [1;3;5;7;9;11;13]: the target 7 sits at index 3, and 4 is absent, so the search reports Some 3 and None respectively.

Soundness needs no assumptions about the order

Here is the pleasant asymmetry of binary search. To find an element in a sorted range you need the range to be sorted and the comparison to be transitive — that is the completeness direction. But to trust an answer the search does return, you need nothing at all about the order: the program only replies Some mid after it has itself checked both comparisons, so a returned index provably holds an element equal to the target, no hypotheses required. That is the cleaner theorem to walk.

The theorem is bsearch_sound: if bsearch lo hi returns Some i, then i lies in the range lo <= i <= hi and its element compares equal to the target on both sides. In symbols — writing m[i]m[i] for the element at index ii, tt for the target, and x≼yx \preceq y for the comparison leb x y = true:

search(lo,hi)=Somei⟹lo≤i≤hi∧m[i]≼t∧t≼m[i].\mathrm{search}(lo, hi) = \mathrm{Some}\ i \;\implies\; lo \le i \le hi \;\wedge\; m[i] \preceq t \;\wedge\; t \preceq m[i].

It goes by well-founded induction on the pair (lo, hi), via the measure S hi - lo (measure_ind2), following the four branches of bsearch_step:

  Theorem bsearch_sound :
    forall lo hi i,
      bsearch lo hi = Some i ->
      lo <= i <= hi
      /\ leb (read m i) target = true
      /\ leb target (read m i) = true.
  Proof.
    apply (measure_ind2 (fun lo hi => S hi - lo)
             (fun lo hi => forall i,
                  bsearch lo hi = Some i ->
                  lo <= i <= hi
                  /\ leb (read m i) target = true
                  /\ leb target (read m i) = true)).
    intros lo hi IH i.
    rewrite bsearch_step.
    destruct (dec (hi < lo)) as [L|L]; [intros E; discriminate E|].
    cbv zeta.
    pose proof (half_le (hi - lo)) as Hq.
    destruct (dec (leb (read m (lo + (hi - lo) / 2)) target = true)) as [L1|L1].
    - destruct (dec (leb target (read m (lo + (hi - lo) / 2)) = true)) as [L2|L2].
      + (* found at the midpoint.  (congruence, not injection: the
           latter normalizes (hi - lo) / 2 into its divmod unfolding,
           which lia then no longer matches with half_le.) *)
        intros E.
        assert (Ei : i = lo + (hi - lo) / 2) by congruence.
        subst i.
        split; [lia|split; assumption].
      + (* right half: indices S mid .. hi *)
        intros E.
        assert (Hd : S hi - S (lo + (hi - lo) / 2) < S hi - lo) by lia.
        destruct (IH _ hi Hd _ E) as (Hr & H1 & H2).
        split; [lia|split; assumption].
    - destruct (dec (lo + (hi - lo) / 2 = lo)) as [Eeq|Eeq];
        [intros E; discriminate E|].
      (* left half: indices lo .. mid - 1 *)
      intros E.
      assert (Hd : S (lo + (hi - lo) / 2 - 1) - lo < S hi - lo) by lia.
      destruct (IH lo _ Hd _ E) as (Hr & H1 & H2).
      split; [lia|split; assumption].
  Qed.

Walking the argument:

  • apply (measure_ind2 (fun lo hi => S hi - lo) ...) sets up well-founded induction on the pair: IH applies at every pair whose measure is strictly smaller. rewrite bsearch_step exposes the program’s four-way branch, and hi < lo is dismissed because None = Some i is impossible (discriminate). cbv zeta unfolds the let for mid, and pose proof (half_le (hi - lo)) puts the division bound in scope for lia.
  • The two destruct (dec ...) split on the same two comparisons the program tests. In the found case, Some (lo + (hi-lo)/2) = Some i forces i = lo + (hi-lo)/2 by congruence; then the range membership is arithmetic (lia) and the two comparison facts are exactly L1 and L2, in scope from the branch.
  • In the right-half case the answer came from the smaller pair (S mid, hi): assert (Hd : ... < S hi - lo) supplies the measure decrease, IH _ hi Hd _ E applies the hypothesis there, and its conclusion is re-assembled — the range bound widens back out to lo <= i <= hi by lia.
  • In the left-half case, dec (mid = lo) is dismissed first (None = Some i is impossible there too — this is the empty-left-half short-circuit, dead whenever the program actually answers Some), then IH lo _ Hd _ E applies at the smaller pair (lo, mid - 1).

The companion result is bsearch_complete, which does need the two order assumptions: if ≼\preceq is transitive and the range is pairwise sorted (m[i]≼m[j]m[i] \preceq m[j] whenever lo≤i≤j≤hilo \le i \le j \le hi), then a present element is always found —

∃j,lo≤j≤hi∧m[j]≼t∧t≼m[j]⟹∃i,search(lo,hi)=Somei.\exists j,\ lo \le j \le hi \,\wedge\, m[j] \preceq t \,\wedge\, t \preceq m[j] \;\implies\; \exists i,\ \mathrm{search}(lo, hi) = \mathrm{Some}\ i.

Its proof, in Theory/BinarySearch.v, runs the same well-founded induction; in each failed-comparison branch, transitivity plus sortedness turn the witness position j into a contradiction with the branch condition, so the witness always survives into the half that is searched. Soundness says the answer you get is right; completeness says you get an answer when one exists.
