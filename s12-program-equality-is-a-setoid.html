Program equality is a setoid, and optimizations are rewrites

The untyped lambda calculus has three terms: a variable Var n, a function Abs b, and an application App f a, with variables written as de Bruijn indices — Var n names the variable bound by the n-th enclosing lambda, counting from 0 inward. A call-by-value step relation step says how a term reduces: App (Abs b) v with v a value substitutes v for index 0 in the body b, and two congruence rules let a step happen in the function or argument position of an application. This post takes that relation and grows it into an equality — one under which simplifying a program is a rewrite.

From one step to an equality

step is a good notion of “reduces to” but a poor notion of “equal to”. It only ever moves forward, and only at the one place call-by-value allows. What an optimizer wants is different: it wants to say two programs are the same program, and to replace either by the other wherever it appears — including inside a function body it has not called and cannot call, because the argument is not known yet.

A setoid is a type paired with an equality relation that need not be syntactic identity — two distinct terms can be declared equal, as long as the relation is reflexive, symmetric, and transitive. The relation conv makes tm a setoid: it takes a single step as one instance of equality and closes it up.

Inductive conv : tm -> tm -> Prop :=
  | conv_step : forall t t', step t t' -> conv t t'
  | conv_abs : forall b b', conv b b' -> conv (Abs b) (Abs b')
  | conv_appl : forall f f' a, conv f f' -> conv (App f a) (App f' a)
  | conv_appr : forall f a a', conv a a' -> conv (App f a) (App f a')
  | conv_refl : forall t, conv t t
  | conv_sym : forall t u, conv t u -> conv u t
  | conv_trans : forall t u v, conv t u -> conv u v -> conv t v.

The last three constructors are exactly reflexivity, symmetry, and transitivity — they make conv an equivalence. The middle three are congruence: conv_abs lets a conversion happen underneath an Abs, in the body of a function; conv_appl and conv_appr let it happen on either side of an application. This is the capability plain evaluation lacks. step will never reduce under an Abs (a lambda is already a value, it stops there), but conv_abs says that if two bodies are convertible then the two functions are convertible. An optimizer can simplify a function body without ever supplying an argument.

Registering conv as the type’s equality lets Coq’s rewriting machinery use it. In this codebase t == u is notation for equ t u, where equ is whatever relation a type’s EqRel instance provides; for tm, that instance is conv:

#[global] Instance Tm_EqRel : EqRel tm := conv.

#[global] Instance Tm_Equivalence : Equivalence conv.
Proof.
  split.
  - exact conv_refl.
  - exact conv_sym.
  - exact conv_trans.
Qed.

For rewrite to fire inside a term, Coq needs to know the surrounding operation respects == — a Proper instance, the formal statement that a function maps equal inputs to equal outputs. conv_abs and the two application rules are precisely those facts, packaged:

#[global] Instance Abs_Proper : Proper (equ ==> equ) Abs.
Proof.
  intros b b' Hb. apply conv_abs. exact Hb.
Qed.

#[global] Instance App_Proper : Proper (equ ==> equ ==> equ) App.
Proof.
  intros f f' Hf a a' Ha.
  eapply conv_trans.
  - apply conv_appl. exact Hf.
  - apply conv_appr. exact Ha.
Qed.

Abs_Proper reads “if b == b' then Abs b == Abs b'” — rewriting under a binder is allowed. App_Proper handles both positions at once, rewriting the function then the argument and chaining the two with conv_trans. With these in place, rewrite on a tm behaves exactly as it does on a group element in the algebra files: replace a subterm by an equal one, anywhere it sits.

A boolean is the choice it would make

The lambda calculus has no built-in booleans, but it does not need any. A boolean’s entire job is to choose between two alternatives, so encode it as that choice: a function of two arguments that returns one of them. True returns the first, false returns the second.

Definition TRUE : tm := Abs (Abs (Var 1)).           (* λ.λ.1 *)
Definition FALSE : tm := Abs (Abs (Var 0)).          (* λ.λ.0 *)

Read TRUE as \t. \e. t: given a then-branch t and an else-branch e, hand back t. In de Bruijn terms the then-branch is index 1 (one lambda further out than e), so TRUE is Abs (Abs (Var 1)) and FALSE is Abs (Abs (Var 0)). A conditional is then nothing but applying the boolean to its two branches:

Definition IFB (b t e : tm) : tm := App (App b t) e.

IFB b t e feeds t and e to b and lets b pick. There is no if primitive and no case analysis — the boolean carries out the selection by being applied. And because IFB is built from App, which we proved Proper, IFB respects == too (the source registers this as IFB_Proper): a conversion in any of b, t, e is a conversion of the whole conditional.

Branch selection is a rewrite

Here is the payoff, and it is the same move as cancelling in a group. The theorem: if a boolean-valued subprogram b is convertible to TRUE, then the conditional built on it collapses to its then-branch. The optimizer knows only b == TRUE — not what b is — and simplifies anyway.

IFB bgh==b==TRUEIFB TRUE gh==g\text{IFB } b\; g\; h \;\overset{b\,==\,\text{TRUE}}{=\!=}\; \text{IFB TRUE } g\; h \;=\!=\; g

Lemma if_of_true : forall b g h,
  value g -> value h -> b == TRUE -> IFB b g h == g.
Proof.
  intros b g h Hg Hh Hb. rewrite Hb. apply IF_TRUE; assumption.
Qed.

Two steps. rewrite Hb uses the hypothesis b == TRUE to replace b by TRUE inside IFB b g h — legitimate only because IFB is Proper for ==, so the rewrite reaches into the subterm. The goal is now IFB TRUE g h == g, closed by the lemma IF_TRUE. (value g and value h are call-by-value’s requirement that the branches be evaluated before selection.)

This is the identical discipline as the group proof cancel_l, where a + b == a + c becomes b == c by rewriting through a chain of named laws and closing with reflexivity. Both proofs work because the surrounding operations are congruences for == — there, + and -; here, IFB — so a rewrite of one piece is a rewrite of the whole. Rewrite by a known equality, then close: the same two beats.

The lemma if_of_true leans on does the actual computing, and it is worth seeing, because it shows the setoid earning the reduction one step at a time:

Lemma IF_TRUE : forall g h, value g -> value h -> IFB TRUE g h == g.
Proof.
  intros g h Hg Hh.
  eapply conv_trans.
  { apply conv_appl. apply conv_beta. exact Hg. }
  cbn.
  eapply conv_trans.
  { apply conv_beta. exact Hh. }
  rewrite subst_lift. reflexivity.
Qed.

IFB TRUE g h is App (App TRUE g) h. The first conv_trans names an intermediate term and conv_appl aims the work at the function position App TRUE g; there conv_beta — a single CBV step as a conversion — fires TRUE on g, leaving App (Abs (lift 1 0 g)) h after cbn computes the substitution. The second conv_trans with conv_beta fires that abstraction on h, producing subst 0 h (lift 1 0 g). Finally rewrite subst_lift applies the law subst k u (lift 1 k t) = t: substituting into a freshly-lifted term changes nothing, because g never mentions the variable h would fill. So g comes back untouched, and reflexivity closes. The discarded else-branch is discarded precisely because the lift it would have landed in is undone by the substitution.

The same file proves IF_FALSE, FST_PAIR, and SND_PAIR the same way, and for fully concrete programs it needs no lemmas at all — running the evaluator on both sides and comparing normal forms suffices:

Example fst_pair_closed : FST (App (App PAIR idt) k) == idt.
Proof. apply (conv_by_eval 20 10). reflexivity. Qed.

conv_by_eval evaluates each side to a normal form (here with fuel 20 and 10) and, finding them identical, concludes the two programs are convertible. Where the branches are symbolic, congruence and the subst_lift law do the reasoning; where everything is closed, plain computation does. Either way, program equality is conv, and simplifying a program is a rewrite under it.
