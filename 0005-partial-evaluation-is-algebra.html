riktig #5 — Partial evaluation is algebra

Fifth post in the series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved, layer by layer, in Coq. The goal is modifiability: a change only obliges you to re-establish a few local facts. Every post starts from zero. In post #1 we proved group equations by chaining rewrites; in post #4 we built the lambda calculus as a runnable datatype. Today the two collide: programs become one more algebraic structure, and compiler transformations — unfolding, inlining, simplifying an if — become the same rewrite chains we used on groups.

Every compiler you have ever used performs equational reasoning it never writes down: this call can be inlined; this branch is dead; this expression is that constant. Each of those is a claim that two programs are interchangeable. Today we give that claim a symbol, laws, and machine-checked proofs — for the lambda calculus of post #4.

Making == mean “behaves alike”

Post #1’s structures all began the same way: a carrier, an equality == chosen for the job, laws as equations. The move (theories/Lambda/Conv.v) is to do exactly that for the type tm of lambda terms:

Inductive conv : tm -> tm -> Prop :=
  | conv_step  : forall t t', step t t' -> conv t t'   (* running is equal    *)
  | conv_abs   : ...   (* equal bodies, equal functions                       *)
  | conv_appl  | conv_appr : ...  (* equals applied to equals stay equal      *)
  | conv_refl  | conv_sym | conv_trans : ...            (* it's an equivalence *)

#[global] Instance Tm_EqRel : EqRel tm := conv.

That last line is the whole point: tm now has a ==, the same == as monoids and vector spaces, with the same consequences — the Proper instances for Abs and App mean rewrite descends into programs like it descends into sums. Program syntax is now a setoid, and program transformation is setoid rewriting.

Two design notes. First, conv deliberately contains more than execution: step only reduces in call-by-value positions, but conv_abs lets us rewrite under a binder — an optimizer transforms function bodies without calling the function. Second, equality-by- running comes with a bridge that turns the evaluator into a proof engine:

conv_by_eval : forall n m s t, eval n s = eval m t -> s == t

If two terms compute to the same normal form, they are convertible. Proofs of program equations become apply (conv_by_eval 10 10). reflexivity. — the kernel runs both programs and compares. We prove by executing.

Booleans without booleans

The lambda calculus has no if, no true, no false — and needs none. Church’s encoding: a boolean is the choice it would make.

TRUE=λt.λe.tFALSE=λt.λe.e\mathrm{TRUE} = \lambda t.\lambda e.\, t \qquad \mathrm{FALSE} = \lambda t.\lambda e.\, e

TRUE applied to two arguments keeps the first; FALSE keeps the second — your if is just application: IFB b g h = (b\; g)\; h. In theories/Lambda/Church.v the branch-simplification laws every compiler applies are theorems:

IF_TRUE  : forall g h, value g -> value h -> IFB TRUE  g h == g
IF_FALSE : forall g h, value g -> value h -> IFB FALSE g h == h

(The value hypotheses are call-by-value’s honesty: you may only discard a branch you were allowed not to evaluate.) Pairs come from the same trick — a pair is a function awaiting a selector, and the projection laws FST_PAIR, SND_PAIR literally reduce to the boolean laws in their proofs, because a projection is an applied boolean. The encoding’s elegance shows up as proof reuse.

The punchline: dead-branch elimination is a two-line proof

Here is the theorem this post exists for, exactly as it appears in the repository:

Lemma if_of_true : forall b g h,
  value g -> value h -> b == TRUE -> IFB b g h == g.
Proof.
  intros b g h Hg Hh Hb.
  rewrite Hb.
  apply IF_TRUE; assumption.
Qed.

Read it slowly: we know the condition is (equivalent to) TRUE — perhaps an analysis proved it, perhaps it came from a constant — so we rewrite the condition inside the program (rewrite Hb, which works because IFB respects ==), and then the branch law finishes. Compare with cancel_l from post #1: same shape, same tactic vocabulary, same reading experience. Group elements, vectors, programs — one discipline.

This is the bridge to where the project began. The memprog vision was a rewrite pipeline: a program refined step by step — inline this, specialize that, lower arithmetic, introduce a stack — with every step meaning-preserving. What was missing was the substrate: what is a meaning-preserving step, formally? Answer, now on disk: an == on programs, with Proper instances so steps compose and localize. Unfold, inline, partially evaluate — each is a named lemma, each application is one rewrite, and the chain of them is a verified compilation.
