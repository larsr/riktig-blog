riktig #18 — The shape of the library

Eighteenth post documenting the riktig project. Not a new proof this time — a look at how the proofs are organized, and an honest account of where that organization has held and where it has slipped.

What we are doing, and why. Verified software in Coq, built for modifiability. Every post so far has been about a theorem. This one is about the filing system the theorems live in — because a proof assistant repository isn’t just a pile of true statements; it’s supposed to be organized so that a change in one place only obliges you to revisit a few others. That promise depends entirely on keeping three different kinds of content apart. This post explains the separation, shows where it earns its keep, and — because pretending everything is tidy would defeat the point of the project — reports honestly on where seventeen posts’ worth of fast, parallel work let it slip.

Three questions, three directories

Every piece of mathematics in this library answers one of three questions, and the directory it lives in is the answer:

  • Interface/ — what must be true? A typeclass or record stating operations and axioms. Nothing here is proved; it’s promised. Class Monoid T := { mon_assoc : ...; mon_idl : ...; ... } is a contract, not a theorem.
  • Theory/ — what follows, for anything that keeps the promise? Lemmas proved generically — Context {G}{Group G}` — using only the Interface’s axioms. A Theory file never invents a new concrete type; it only ever says “given any group, here is a fact about it.”
  • Impl/ — does this specific thing keep the promise? A concrete carrier, concrete operations, and a proof that the Interface’s axioms hold for them. Z is a Ring; Q is a Field; Fin n -> F is a VectorSpace.

docs/DESIGN.md states the rule in one line: a file does exactly one of these jobs. The payoff is real and it’s the whole reason to bother: prove cancel_l once, generically, for Group, and it is available — unconditionally, no re-proving — for integers, rationals, complex numbers, linear maps, matrices, and everything else that ever becomes a group. Post #1 built this idea from nothing; by post #9 it was letting whole theorems fall out “by pure instance search.”

Where it holds — cleanly

Go back to the beginning and it’s exactly as advertised. Interface/Monoid.v through Interface/VectorSpace.v contain nothing but classes, axioms, and Hint Mode declarations. Theory/Group.v’s cancel_l, Theory/InnerProduct.v’s bilinear-expansion lemmas, Theory/Order.v’s whole ε-argument toolbox — all of it proved generically, none of it tied to a concrete carrier. Impl/Zr.v, Impl/Qf.v, Impl/Complex.v’s C F instance — concrete types, discharging axioms, nothing more. The core algebra tower — the part of the library doing the most foundational work — keeps the discipline all the way through.

Where it slipped

Here is the honest part. A sweep through every file in Interface/, Theory/, and Impl/ — 78 files, checked one by one against what its directory promises — turned up real drift, concentrated almost entirely in the faster-moving linear-algebra wave (posts #9 through #17), exactly where you’d expect discipline to erode under the pressure of many parallel changes landing quickly.

The “one accepted exception” quietly became several unacknowledged ones. Interface/LinearMap.v’s header states, in so many words: “the Theory.Group import is this file’s one accepted exception.” That sentence excuses an import. It says nothing about the fact that the file goes on to prove lm_zero, lm_neg, and to define lm_id, lm_comp, lm_const0 — real Theory-shaped content, sitting in Interface/. And it’s not alone: Interface/Basis.v proves four lincomb lemmas; Interface/Bilinear.v proves the zero-bilinear-map laws and — more than that — builds a concrete instance (bm_zero_map) inside a file that is supposed to contain nothing but promises. Interface/Integral.v proves int_empty, int_rev, int_zero_fn — and doesn’t even have a header disclaimer. Four files, one drift pattern, only one of them honest about it.

A missing spec forced generality into the wrong directory. There is no Interface/Determinant.v. detM (the Laplace-expansion formula in Impl/Det.v) was never given an abstract contract to satisfy — it was simply defined, and then proved to have properties. That’s fine in isolation, but it meant Impl/Det.v had nowhere to put general facts like “the determinant is linear in column 1” except inside itself, and Impl/Elim.v inherited the same problem at larger scale: more than half of that 1,266-line file — unskip, the row-surgery lemmas, “swapping any two rows negates the determinant” — is genuinely general reasoning about any matrix, sitting in Impl/ only because there was no Theory/ file naturally waiting to receive it.

Two files import in the wrong direction, and cite each other. docs/DESIGN.md states dependencies point one way: implementations import theories, theories import structures, never backwards. Theory/Hom.v and Theory/VSIso.v both import Impl/ modules directly — and each one’s comment justifies it by pointing at the other file doing the same thing. Two files normalizing an exception between themselves is a quieter, more insidious kind of drift than one undisclosed lemma: it looks like precedent, which is exactly what makes it easy to not notice.

A few files are honestly ambiguous, not really wrong. Theory/BigOp.v defines the bigop/bigopF summation algorithm — a Fixpoint, not a derived theorem. So does Theory/GramSchmidt.v’s orthogonalization procedure. Strictly, “Theory only derives consequences” says these don’t belong there. But there’s no natural Interface for “summation” to satisfy, and no natural Impl carrier either — the algorithm is the mathematical content being studied, generic over any Monoid or PosInner space. This might not be a bug so much as a fourth pattern the original three-way split never named: a Theory file may define the canonical generic construction for the thing it studies. Worth writing down as a rule rather than quietly treating as an exception.

What isn’t drift: Channel/ and Lambda/

One more finding, and it’s reassuring rather than concerning: theories/Channel/ and theories/Lambda/ don’t follow the Interface/Theory/Impl split at all — and that’s on purpose, correctly. Channel/ABP.v doesn’t have a “spec” for the alternating-bit protocol that some other Impl might satisfy differently; there is exactly one alternating-bit protocol, and the file studies it directly, mixing its data (Record Packet), its algorithm (Fixpoint deliver), and its theorems in one place. The three-way split earns its keep precisely when many different things can satisfy one abstract contract — vector spaces, protocols-in-general. It has nothing to offer when the object of study is a single concrete thing you’re exploring for its own sake. The audit checked: every Channel and Lambda file keeps its own spec-like, algorithm-like, and theorem-like content in clearly ordered — usually explicitly numbered — sections internally. Different discipline, applied consistently, not absence of discipline.

What happens next

None of this is urgent — every file still compiles, there isn’t a single Admitted or Axiom anywhere in the tree, and nothing here is a correctness bug. It’s quality debt: harder-to-navigate code, not wrong code. It’s now an itemized wave on the roadmap (LINALG-ROADMAP.md, Wave L8) rather than a vague unease — the biggest single fix being the missing Interface/Determinant.v, which alone would let Impl/Det.v and Impl/Elim.v shed most of their misplaced generality in one move.

The instinct that prompted this post — “I sense there’s been a slide, or is there a well-thought-out organization I haven’t understood” — was the right question to ask, and the honest answer is some of both: the organization is real and it is understood, and it has also genuinely slipped in places, and both of those things are now written down instead of living only as a feeling.

Postscript (2026-07-13)

Wave L8 landed, and the drift catalogued above is gone. Interface/Basis.v, Interface/Bilinear.v, and Interface/Integral.v were slimmed to specification only, their derived lemmas — and the bm_zero_map instance — moved to matching Theory/ files. The Theory/Hom.v/Theory/VSIso.v import direction was resolved by decision rather than by moving code: their theorems are about the named Impl/ constructions they import, and docs/DESIGN.md now states that pattern explicitly instead of two files citing each other. One residue remains real: Interface/LinearMap.v still proves lm_zero and lm_neg inside Interface/. And the missing Interface/Determinant.v — the biggest single item — is still open on the roadmap.
