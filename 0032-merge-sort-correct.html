riktig #32 — merge sort, correct

Thirty-second post in the series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved in Coq. Post #29 introduced the Programs/ track: algorithms written with if dec (..) then .. else .. for decidable conditions and the standard library’s Fix for recursion that isn’t structural, each Fix-defined function coming with one unfolding lemma proved once via Fix_eq. That post was honest about its scope: the four programs were checked to compute correct answers on examples, but none carried a correctness proof — “the infrastructure a correctness proof would build on, not the proof itself.” Today the first such proof: merge sort sorts. The output of msort is sorted and is a permutation of the input — and the proof never touches Fix at all, which is the point of this post: the unfolding lemma is the whole interface.

What was proved

Theorem msort_perm : forall l : list nat, Permutation l (msort l).

Theorem msort_sorted : forall l : list nat, Sorted le (msort l).

Corollary msort_correct : forall l : list nat,
  Permutation l (msort l) /\ Sorted le (msort l).

Permutation and Sorted are the standard library’s own predicates (Coq.Sorting), so these statements mean exactly what any Coq user expects: no element gained or lost, and every adjacent pair in the output in order. Together they pin down sorting completely — a permutation of the input that is sorted is the sorted rearrangement.

The one idea: reason through the step lemma, never through Fix

Recall the situation from post #29: msort recurses on the outputs of split, which Coq’s structural checker cannot accept, so it is defined with Fix over well-founded induction on list length. A Fix-defined function does not unfold by simpl — but it has msort_step, proved once:

msort l = match l with
          | [] => l
          | [_] => l
          | x :: y :: rest =>
              merge (msort (fst (split (x :: y :: rest))))
                    (msort (snd (split (x :: y :: rest))))
          end

The correctness proof uses this and only this. Three small “match-free” corollaries first — msort_nil, msort_single, msort_cons2, one per branch, so later rewriting never leaves a match in the goal — and then both main theorems have the same five-line shape: well-founded induction on length l (the very measure msort recurses on), the three-way case split matching msort_step’s branches, and the file’s existing split_length_lt to discharge the induction hypotheses on the two halves. From msort_sorted, the whole inductive step is:

- pose proof (split_length_lt x y rest) as [H1 H2].
  rewrite msort_cons2.
  apply merge_sorted; apply IH; [exact H1 | exact H2].

Fix, Fix_eq, functional extensionality — none of it appears. Once the step lemma exists, a Fix-defined function is reasoned about exactly like a Fixpoint.

The supporting lemmas, briefly

The real content sits in three places, each matching the shape of the definition it is about.

split loses nothing. split_perm : Permutation l (fst (split l) ++ snd (split l)). split consumes two elements at a time, so plain structural induction doesn’t fit; the proof reuses the file’s existing bounded-induction pattern (induction on a bound n with length l <= n), and the inductive step is perm_skip twice plus Permutation_middle to move y past the first half.

merge loses nothing. merge_perm : Permutation (l1 ++ l2) (merge l1 l2), by the induction that mirrors merge’s own nested structure — outer on l1, inner on l2. Two computation-only unfolding equations (merge_nil_r, merge_cons_cons) are named first so the nested inner fix never appears raw in a goal.

merge preserves sortedness. The pivot is a lemma about heads:

Lemma merge_hdrel (a : nat) (l1 l2 : list nat) :
  HdRel le a l1 -> HdRel le a l2 -> HdRel le a (merge l1 l2).

— the head of a merge is the head of one of its arguments, so a bound dominating both heads dominates the merge; case analysis only, no induction. merge_sorted then follows the same two-level induction as merge_perm, and this is where the dec style pays its way: the guard dec (Nat.leb x y = true) puts the actual boolean fact in the context, translated to the order by Nat.leb_le in the true branch and Nat.leb_gt (giving y < x, hence y <= x) in the false one.

What was proved, plainly — and what wasn’t

  • msort returns a sorted permutation of its input, for every list (msort_correct), with Sorted and Permutation from the standard library. The computation tests from post #29 still stand and are now corollaries of something stronger.
  • The proof pattern — step lemma, match-free branch corollaries, well-founded induction on the function’s own measure — used nothing specific to sorting, and is the template for the track’s remaining debts.
  • Not proved, honestly: the other three programs from post #29. gcd correctness (divides both, greatest), bsearch soundness and completeness on sorted ranges, and quicksort’s version of today’s theorem are all still open, and quicksort’s will be harder — its recursion goes through an abstract mutable memory rather than lists.

Next: the determinant’s last open ingredient — multiplicativity via triangular decomposition — or another entry from the Programs/ debt list.
