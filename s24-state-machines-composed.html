State machines and composing them through a channel

A protocol has two ends. A sender reads a stream of application messages and decides what to put on the wire; a receiver reads what comes off the wire and decides what to hand back to the application. Both ends are small state machines: they remember something (a sequence number, whether they are waiting for an acknowledgement) and react to each input by emitting an output and moving to a new state. This post models such an endpoint as a Mealy machine, runs one, and — the point of the exercise — composes two of them through a channel and proves exactly what relation the composite induces between the messages that went in and the messages that came out.

A Mealy machine is the automaton whose output depends on both the current state and the current input. (Its cousin, the Moore machine, lets the output depend on the state alone; a Mealy machine is the one you want when the reply must be a function of what just arrived, not only of where you were.) That is the whole distinction, and it is why a single step produces one output per input.

This post leans on one idea from its companion: a channel is not a probability but a relation between the finite list of messages that was sent and the finite list that could be received. A perfect channel relates each sent trace only to itself. A lossy-and-duplicating channel, written lossy_dup, relates a sent trace to any trace you get by dropping some messages and repeating others. Both are just two-argument predicates on traces, and either can be slotted into the composition below.

A Mealy machine, and running it

An endpoint is a start state and a step function that turns a state and an input into a new state paired with an output:

Record Mealy (S I O : Type) := {
  m_init : S;
  m_step : S -> I -> S * O;
}.

To run a machine on a whole list of inputs, thread the state through and collect one output per input, in order:

Fixpoint run {S I O : Type} (m : Mealy S I O) (s : S) (is : list I)
    : S * list O :=
  match is with
  | [] => (s, [])
  | i :: is' =>
      let (s1, o) := m_step m s i in
      let (s2, os) := run m s1 is' in
      (s2, o :: os)
  end.

Definition run_from_init {S I O : Type} (m : Mealy S I O) (is : list I)
    : S * list O :=
  run m (m_init m) is.

run on the empty input returns the state unchanged and no output. On i :: is' it takes one step to get an output o and a next state s1, runs the rest from s1, and puts o at the front of whatever came out. run_from_init just starts from the machine’s own initial state and keeps only the output list — the output trace the endpoint produces.

Composing two endpoints through a channel

Now glue a sender and a receiver together. The sender turns an application trace into a wire trace; the channel relates that sent wire trace to a possibly-different received wire trace; the receiver turns the received wire trace back into an application trace. The composite is the relation on application traces that holds when there is some pair of wire traces making all three steps line up:

Definition composed_trace (app_in app_out : list App) : Prop :=
  exists wire_sent wire_recv,
    snd (run_from_init snd_m app_in) = wire_sent /\
    chan wire_sent wire_recv /\
    snd (run_from_init rcv_m wire_recv) = app_out.

Here snd_m is the sender machine, rcv_m the receiver, and chan any channel relation on wire traces — deliberately abstract, so a perfect channel and a faulty one plug into the same definition. The two endpoints never share memory; they communicate only through chan, exactly as two real protocol ends communicate only over the wire.

The worked example: two echo endpoints

The simplest genuine endpoint is a pass-through — an echo machine that outputs whatever it reads and keeps no state (its state type is unit):

Definition relay_m : Mealy unit App App :=
  {| m_init := tt; m_step := fun _ a => (tt, a) |}.

Its step fun _ a => (tt, a) ignores the state and returns the input as the output — a Mealy machine whose output is a function of the input. Running it reproduces its input list exactly: for every input list 𝑖𝑠\mathit{is} and (trivial) state ss,

runrelays𝑖𝑠=(*,𝑖𝑠),\mathrm{run}\;\mathrm{relay}\;s\;\mathit{is} \;=\; (\ast,\ \mathit{is}),

where *\ast is the unique value of the one-element state type. The proof is a two-line induction:

Lemma run_relay : forall (is : list App) (s : unit),
  run relay_m s is = (tt, is).
Proof.
  intros is.
  induction is as [| a is IH]; intros s.
  - destruct s. reflexivity.
  - simpl. rewrite IH. reflexivity.
Qed.

The empty case: run on [] gives (s, []), and the only value of type unit is tt, so destruct s turns s into tt and both sides are (tt, []). The cons case: one step of the echo machine emits a and stays in state tt, the inductive hypothesis IH rewrites the recursive call to (tt, is), and the result (tt, a :: is) is what we wanted.

Now put an echo sender and an echo receiver on the two ends of a perfect channel. Nothing anywhere alters a message, so the trace that comes out equals the trace that went in. On the blackboard, reading the three conjuncts of composed_trace left to right:

𝑎𝑝𝑝_𝑜𝑢𝑡=𝑤𝑖𝑟𝑒_𝑟𝑒𝑐𝑣{receiver echoes}=𝑤𝑖𝑟𝑒_𝑠𝑒𝑛𝑡{perfect channel}=𝑎𝑝𝑝_𝑖𝑛{sender echoes}.\mathit{app\_out} \;=\; \mathit{wire\_recv} \quad\{\text{receiver echoes}\} \;=\; \mathit{wire\_sent} \quad\{\text{perfect channel}\} \;=\; \mathit{app\_in} \quad\{\text{sender echoes}\}.

Here is the theorem and its proof, verbatim:

Theorem composed_trace_perfect_relay : forall app_in app_out : list App,
  composed_trace relay_m relay_m perfect app_in app_out ->
  app_out = app_in.
Proof.
  intros app_in app_out [wire_sent [wire_recv [Hsent [Hperfect Hrecv]]]].
  unfold run_from_init in Hsent, Hrecv.
  rewrite run_relay in Hsent. simpl in Hsent.
  rewrite run_relay in Hrecv. simpl in Hrecv.
  unfold perfect in Hperfect.
  (* app_out and app_in are the two ends of one equality chain through
     the wire: received = sent because the channel is perfect, and each
     relay copies its input verbatim. *)
  calc app_out
    == wire_recv { Hrecv }
    == wire_sent { Hperfect }
    == app_in    { Hsent }.
Qed.

Walking it against the blackboard chain. The intros pattern unpacks the existential: it names the two wire traces and the three facts Hsent : snd (run_from_init relay_m app_in) = wire_sent, Hperfect : perfect wire_sent wire_recv, and Hrecv : snd (run_from_init relay_m wire_recv) = app_out. Rewriting with run_relay (after unfolding run_from_init) computes each relay’s output: Hsent becomes wire_sent = app_in and Hrecv becomes wire_recv = app_out. unfold perfect in Hperfect turns the channel fact into wire_recv = wire_sent. The calc block is the blackboard chain, machine-checked: it reads app_out == wire_recv == wire_sent == app_in, and each { ... } names the single fact justifying that step — Hrecv for app_out = wire_recv, Hperfect for wire_recv = wire_sent, Hsent for wire_sent = app_in. The checker confirms the two endpoints of the chain really are the goal (app_out = app_in) and fills each step by the named fact. Two echo endpoints over a perfect channel are the identity on traces — nothing was added, lost, or moved.

Swapping in a faulty channel

The composition is transparent: the echo endpoints add nothing, so whatever relation the channel imposes is exactly the relation the composite imposes. Replace perfect with the faulty lossy_dup and the same three-step reading gives, on paper,

composed(𝑎𝑝𝑝_𝑖𝑛,𝑎𝑝𝑝_𝑜𝑢𝑡)⇒lossy_dup(𝑎𝑝𝑝_𝑖𝑛,𝑎𝑝𝑝_𝑜𝑢𝑡):\mathrm{composed}(\mathit{app\_in}, \mathit{app\_out}) \;\Longrightarrow\; \mathrm{lossy\_dup}(\mathit{app\_in}, \mathit{app\_out}):

the sender relay forces 𝑤𝑖𝑟𝑒_𝑠𝑒𝑛𝑡=𝑎𝑝𝑝_𝑖𝑛\mathit{wire\_sent} = \mathit{app\_in} and the receiver relay forces 𝑤𝑖𝑟𝑒_𝑟𝑒𝑐𝑣=𝑎𝑝𝑝_𝑜𝑢𝑡\mathit{wire\_recv} = \mathit{app\_out} (both by run_relay, exactly as in the perfect case), and substituting those two equalities into the channel fact lossy_dup wire_sent wire_recv leaves lossy_dup app_in app_out. The source file proves the perfect-channel theorem shown above and stops there; this faulty-channel counterpart is the same substitution argument with the channel fact carried through instead of collapsed to an equality, and is stated here on paper only. The conclusion it points at: the fault class of the wire is exactly the fault class the application sees, because the endpoints in between are faithful.

That is the ground a real protocol has to improve on: with echo endpoints the application inherits every fault of the wire. A sender and receiver that instead track sequence numbers and acknowledgements are Mealy machines with genuine state, and the point of their state is to make the composite relation better than the channel they run over.
