What is a mathematical structure?

Programmers already know half of what a mathematical structure is. An interface says which operations exist: this thing can be compared, that thing can be concatenated. What an interface does not say is what the operations mean. A sort function that returns its input untouched satisfies the interface list -> list perfectly; nothing in the type list -> list says the output must be ordered. The meaning lives outside the interface.

A mathematical structure is the missing half: the vocabulary of operations together with the laws those operations must obey. Not “there is a +”, but “there is a +, and it never matters how you parenthesise a sum”. The laws are the meaning. Two structures with the same operations but different laws are different worlds — and anything you prove from the laws alone holds in every world that satisfies them, at once. This post builds the smallest such structure, the monoid, and then proves one small theorem about every monoid there will ever be.

The monoid: combine, with a do-nothing element

Many different things can be combined. Numbers add. Strings and lists concatenate. Program fragments run one after another. In each case the combining operation shares two properties. First, it is associative: regrouping a chain of combinations changes nothing,

(a+b)+c=a+(b+c).(a + b) + c \;=\; a + (b + c).

Second, there is a do-nothing element — zero for addition, the empty string for concatenation, the empty program for sequencing — that combines invisibly, on the left and on the right:

0+a=a,a+0=a.0 + a \;=\; a, \qquad a + 0 \;=\; a.

A set with a combining operation +, a do-nothing element 0, and those laws is a monoid. The integers under addition are a monoid. Lists under concatenation are a monoid (a different one: concatenation is not commutative, [1] + [2] is not [2] + [1], but it is still associative with the empty list as its 0). Naming the shared pattern is what lets a single proof serve all of them.

Vocabulary and laws, kept in separate files

The encoding in this library keeps the vocabulary and the laws apart on purpose, so each can be reused. The vocabulary is one tiny typeclass per symbol, stating that an operation exists and saying nothing about how it behaves. Here are the two the monoid needs, from Ops.v:

Class Zero T := zero : T.
Notation "'0'" := zero.

Class HasAdd T := add : BinOp T.
Infix "+" := add.

That is the entire content of Zero and HasAdd: a designated element and a binary operation, with the familiar notation attached. Once a type provides HasAdd, the literal symbol + works for it.

A law is also a first-class, nameable thing — one class per law, in Props.v, each parameterised by the equality, the operation, and whatever else it mentions:

Class Assoc T (e : EqRel T) (op : BinOp T) :=
  assoc : forall a b c, op (op a b) c == op a (op b c).

Class Idl T (e : EqRel T) (op : BinOp T) unit :=
  idl : forall a, op unit a == a.

Class Idr T (e : EqRel T) (op : BinOp T) unit :=
  idr : forall a, op a unit == a.

Assoc is associativity, Idl is “unit is a left identity”, Idr the right-hand version. Because these are named classes, a proof can write rewrite assoc no matter which structure supplied the associativity — the law is looked up by its name, not by where it came from. (== here is the structure’s own equality, itself just another operation, EqRel, so the same machinery works for structures whose equality is not Coq’s built-in =.)

The structure itself bundles the vocabulary and the laws into one class. Here is the whole monoid, from Interface/Monoid.v:

Class Monoid T := {
  #[global] mon_equ        :: EqRel T;
  #[global] mon_equiv      :: Equivalence mon_equ;
  #[global] mon_zero       :: Zero T;
  #[global] mon_add        :: HasAdd T;
  mon_assoc      : forall a b c : T, (a + b) + c == a + (b + c);
  mon_idl        : forall a : T, 0 + a == a;
  mon_idr        : forall a : T, a + 0 == a;
  #[global] mon_add_proper :: Proper (equ ==> equ ==> equ) (@add T _)
}.

Read it top to bottom: an equality mon_equ that is an equivalence relation (mon_equiv), a zero, an addition, and then the three laws we wrote on the blackboard — mon_assoc, mon_idl, mon_idr. The last field, mon_add_proper, says + respects the equality: replacing either argument by an equal one gives an equal result. The :: marks a field as visible to Coq’s instance search, so once a type is known to be a monoid, its zero and its addition are found automatically.

The payoff: the identity element is unique

Now a theorem that holds in every monoid, proved once. Suppose you have two candidate do-nothing elements: an e that is a left identity (e + a == a for all a) and an f that is a right identity (a + f == a for all a). They must be equal. The argument is a single trick — look at the combination e + f from both sides:

e=e+f{f is a right identity}=f{e is a left identity}e \;=\; e + f \quad\{\text{$f$ is a right identity}\}\quad =\; f \quad\{\text{$e$ is a left identity}\}

The first equality holds because f does nothing on the right; the second because e does nothing on the left. Here is the proof exactly as it appears in Theory/Monoid.v:

Theorem id_unique : forall e f : M,
  (forall a, e + a == a) -> (forall a, a + f == a) -> e == f.
Proof.
  intros e f He Hf.
  calc e
    == (e + f) { (Hf e) }
    == f       { (He f) }.
Qed.

The calc block is the blackboard chain, machine-checked. It reads e == (e + f) == f, and each { ... } names the one fact that justifies that step; the checker confirms the written endpoints really are the goal (e == f) and fills each step by that fact. Walking it:

  • intros e f He Hf names the two elements and their identity hypotheses — He : forall a, e + a == a and Hf : forall a, a + f == a.
  • == (e + f) { (Hf e) } is the step e = e + f. The justification Hf e is the fact e + f == e; the checker uses it right-to-left.
  • == f { (He f) } is the step e + f = f, justified by He f, which is exactly e + f == f.

Two identities, sandwiched around one product, forced to coincide.

The theorem the rest of the library actually calls has one identity named up front — the structure’s own 0 — and a candidate compared against it. That is a one-step corollary. A left identity must be the zero:

e=e+0{right identity law}=0{hypothesis e+a=a at a=0}e \;=\; e + 0 \quad\{\text{right identity law}\}\quad =\; 0 \quad\{\text{hypothesis } e + a = a \text{ at } a = 0\}

Theorem zero_unique_l : forall e : M, (forall a, e + a == a) -> e == 0.
Proof.
  intros e He.
  calc e
    == (e + 0) { idr    }
    == (0 : M) { (He 0) }.
Qed.

The first step, e = e + 0, is the monoid’s own right-identity law idr (used right-to-left). The second, e + 0 = 0, is the hypothesis He at the argument 0. Its mirror image is zero_unique_r, with the statement forall e : M, (forall a, a + e == a) -> e == 0 — a right identity is the zero too, proved the same way through e = 0 + e = 0.

A monoid you can compute with

None of this is abstract for its own sake: the moment a concrete type is shown to satisfy the Monoid class, all of the above becomes true of it with no further work. The integers are the first instance. Here is the whole declaration from Impl/Zr.v — each law field is discharged by the corresponding fact from Coq’s standard library:

#[global] Instance Z_Monoid : Monoid Z := {|
  mon_equ        := Z_EqRel;
  mon_zero       := Z_Zero;
  mon_add        := Z_Add;
  mon_equiv      := eq_equivalence;
  mon_assoc      := fun a b c => eq_sym (Z.add_assoc a b c);
  mon_idl        := Z.add_0_l;
  mon_idr        := Z.add_0_r;
  mon_add_proper := ltac:(solve_proper)
|}.

Z.add_0_l is exactly the left-identity law, Z.add_0_r the right one. The one wrinkle is mon_assoc: the standard library states associativity as a + (b + c) = (a + b) + c, the mirror image of the class’s field, so eq_sym flips it around. The list monoid from Impl/ListMonoid.v gives the same laws for concatenation, where 0 is the empty list. First the vocabulary — equality, zero, addition — then the bundle:

#[global] Instance List_EqRel A : EqRel (list A) := eq.
#[global] Instance List_Zero  A : Zero (list A) := nil.
#[global] Instance List_Add   A : HasAdd (list A) := @app A.
#[global] Instance List_Monoid A : Monoid (list A) := {|
  mon_equiv      := eq_equivalence;
  mon_assoc      := fun l m n => eq_sym (app_assoc l m n);
  mon_idl        := @app_nil_l A;
  mon_idr        := @app_nil_r A;
  mon_add_proper := app_proper A
|}.

The laws here are app_nil_l (prepending the empty list changes nothing) and app_nil_r (appending it changes nothing), again with eq_sym reorienting associativity. Both instances inherit id_unique and zero_unique_l for free: whatever left identity you can find in Z under +, or in list A under concatenation, is provably the one already named 0.
