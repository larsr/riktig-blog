Categories, in the way a programmer already half-knows them

Two functions compose. If f sends A to B and g sends B to C, then g ∘ f sends A to C: run f, then g. There is a do-nothing function on each type, the identity, that hands back its argument untouched, and composing anything with it changes nothing. And composition does not care how you group it — h ∘ (g ∘ f) and (h ∘ g) ∘ f are the same function. Anyone who has written one compose helper already knows these three facts by heart. A category is precisely the structure those three facts describe, taken on its own terms and studied directly.

The earlier posts in this series looked at structures built on a single carrier: a monoid is one set with an associative + and a neutral element 0; a group adds an inverse for every element; a vector space adds scaling. A category is a different but closely related idea. Instead of one set with operations on its elements, a category has many objects, and arrows between them, with composition of arrows as the operation, and two laws — a do-nothing identity arrow, and associativity — as the algebra. Where a monoid asks how elements combine, a category asks how arrows chain.

The Category class

Here is the whole definition, from theories/Interface/Category.v:

Class Category (obj : Type) := {
  hom : obj -> obj -> Type;
  #[global] hom_equ :: forall x y, EqRel (hom x y);
  #[global] hom_equiv :: forall x y, Equivalence (hom_equ x y);
  cat_id : forall {x : obj}, hom x x;
  comp : forall {x y z : obj}, hom y z -> hom x y -> hom x z;
  #[global] comp_proper :: forall x y z, Proper (equ ==> equ ==> equ) (@comp x y z);
  cat_idl : forall (x y : obj) (f : hom x y), comp cat_id f == f;
  cat_idr : forall (x y : obj) (f : hom x y), comp f cat_id == f;
  cat_assoc : forall (w x y z : obj) (f : hom y z) (g : hom x y) (h : hom w x),
      comp f (comp g h) == comp (comp f g) h
}.

Notation "x ~> y" := (hom x y) (at level 70, right associativity).
Infix "∘" := comp (at level 40, left associativity).

Read it field by field.

  • obj is the collection of objects; hom x y is the type of arrows from x to y, written x ~> y once the notation is in scope. An arrow is also called a morphism.
  • hom_equ and hom_equiv say that each pair of objects carries its own equality on arrows, and that this equality is an equivalence relation (reflexive, symmetric, transitive). This per-pair equality is what “==” means for arrows, and it is called a hom-setoid. Why not use plain, definitional equality of arrows? Because categories of syntax want to identify arrows that compute to the same thing — lambda terms up to conversion, for instance — and definitional equality is too fine for that. Making the equality a field lets each category choose its own.
  • cat_id is the identity arrow on each object; comp is composition, with the same “second argument runs first” reading as g ∘ f. The braces {x : obj} mean the object indices are inferred, so you write g ∘ f and Coq fills in the objects.
  • comp_proper says composition respects the hom equality: replacing an arrow by an equal one, on either side, gives an equal composite. This is what lets a proof rewrite one arrow inside a composite.
  • The last three fields are the laws. cat_idl and cat_idr: the identity does nothing on the left and on the right. cat_assoc: composition is associative.

A category you already use: types and functions

None of that is abstract nonsense — it has concrete instances. The first one, theories/Impl/CoqCat.v, is the category whose objects are types and whose arrows are functions. It starts by fixing what an arrow is, and when two arrows count as equal:

Definition fn_equ (X Y : Type) (f g : X -> Y) : Prop :=
  forall x : X, f x = g x.

Definition fn_id (X : Type) : X -> X := fun x => x.

Definition fn_comp (X Y Z : Type) (g : Y -> Z) (f : X -> Y) : X -> Z :=
  fun x => g (f x).

Arrows X ~> Y are ordinary functions X -> Y; two functions are equal when they agree on every input — pointwise equality, the hom-setoid for this category. The identity arrow is the identity function; composition is function composition. The three category laws then hold, and each proof is a single reflexivity:

Lemma fn_idl (X Y : Type) (f : X -> Y) :
  fn_equ X Y (fn_comp X Y Y (fn_id Y) f) f.
Proof.
  intros x. reflexivity.
Qed.

Lemma fn_idr (X Y : Type) (f : X -> Y) :
  fn_equ X Y (fn_comp X X Y f (fn_id X)) f.
Proof.
  intros x. reflexivity.
Qed.

Lemma fn_assoc (W X Y Z : Type) (f : Y -> Z) (g : X -> Y) (h : W -> X) :
  fn_equ W Z (fn_comp W Y Z f (fn_comp W X Y g h))
             (fn_comp W X Z (fn_comp X Y Z f g) h).
Proof.
  intros x. reflexivity.
Qed.

intros x picks an input; reflexivity closes the goal because both sides compute to the same result on that input. In the category of functions, the category laws are not deep facts to be argued — they hold by computation. Assembling the pieces gives the instance:

#[global] Instance CoqCat : Category Type := {|
  hom := fun X Y : Type => X -> Y;
  hom_equ := fn_equ;
  hom_equiv := fn_equ_equivalence;
  cat_id := fn_id;
  comp := fn_comp;
  comp_proper := fn_comp_proper;
  cat_idl := fn_idl;
  cat_idr := fn_idr;
  cat_assoc := fn_assoc
|}.

Every field of the class is filled by one of the definitions or lemmas above. From here on, CoqCat is a genuine category, and any theorem proved for categories in general holds for it.

One theorem: an identity is the only arrow that behaves like one

Here is such a theorem, from theories/Theory/Category.v. In any category, if an arrow e : x ~> x leaves every arrow unchanged when composed first — that is, f ∘ e == f for all f out of x — then e is the identity. The identity is not merely an arrow that does nothing; it is the only one.

On the blackboard the argument is two steps:

e=id∘e=ide \;=\; \mathrm{id} \circ e \;=\; \mathrm{id}

The first equality is the left-identity law read backwards; the second uses the hypothesis about e, taking the arrow f to be the identity itself. In Coq, verbatim:

Theorem id_unique_l :
  forall (x : O) (e : x ~> x), (forall y (f : x ~> y), f ∘ e == f) -> e == cat_id.
Proof.
  intros x e Hu.
  rewrite <- (cat_idl x x e).
  exact (Hu x cat_id).
Qed.

Step by step:

  • intros x e Hu names the object x, the arrow e, and the hypothesis Hu that e leaves every arrow unchanged.
  • rewrite <- (cat_idl x x e) uses the left-identity law cat_id ∘ e == e backwards, rewriting the e in the goal into cat_id ∘ e. The goal e == cat_id becomes cat_id ∘ e == cat_id — this is the first blackboard step.
  • exact (Hu x cat_id) closes it: the hypothesis Hu, applied to the identity arrow, says exactly cat_id ∘ e == cat_id — the second blackboard step.

Read back in CoqCat, the theorem says something concrete about functions: the only function on a type that leaves every function unchanged when you run it first is the identity function. We proved it once, from the category laws alone, so it holds not just for functions but in every category built on those laws — lambda terms, linear maps, and the rest.
