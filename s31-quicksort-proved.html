Quicksort, proved: a sorted permutation of the range, in place

Quicksort sorts a range of memory in place. It picks one element as a pivot, rearranges the range so that everything smaller sits to the pivot’s left and everything larger to its right, and then sorts those two sides the same way. The rearranging step is called partition, and it is where all the interesting reasoning lives: the rest is two recursive calls on strictly smaller ranges. This post takes the Lomuto partition scheme, states the invariant that makes it correct, and proves that the whole algorithm produces a sorted permutation of its input while leaving everything outside the range untouched.

A recursive lambda term with a termination measure

As in the two preceding posts, the algorithm is written as a plain recursive term rather than through any elaboration machinery. Two pieces of vocabulary carry over. A decidable proposition is one whose truth can be computed, so that if dec P then .. else .. is an ordinary if whose chosen branch also hands the proof the fact it just decided. Quicksort here is generic in a bare boolean comparison leb : V -> V -> bool; the correctness file will later add exactly two laws about it — totality and transitivity — that turn leb into a total preorder.

A function that calls itself on smaller arguments is not automatically accepted by Coq: the checker insists on a reason the recursion stops. The reason is supplied explicitly with Fix over a well-founded measure. Here is the Lomuto partition, verbatim from Programs/Quicksort.v. It works over an abstract memory — a type mem with read : mem -> nat -> V and swap : mem -> nat -> nat -> mem — so nothing below depends on how the memory is actually represented:

  (* Lomuto partition via Fix on i with measure hi - i.
     pivot = m[hi], scan i from lo to hi-1. *)
  Definition part (m0 : mem) (store0 i0 hi : nat) : mem * nat.
  refine (Fix (well_founded_ltof _ (fun i => hi - i)) (fun _ => mem -> nat -> mem * nat)
    (fun i rec m store =>
      if dec (i < hi)
      then if dec (leb (read m i) (read m hi) = true)
           then rec (S i) _ (swap m i store) (S store)
           else rec (S i) _ m store
      else (swap m store hi, store))
    i0 m0 store0).
  all: unfold ltof; abstract lia.
  Defined.

The pivot is the value at the top index hi. A cursor i scans upward from lo; the measure hi - i shrinks on every recursive call, and the two holes _ are the proofs that it does, discharged in one line by abstract lia (kept abstract so the generated proof term stays small). A second pointer, store, marks the boundary of the “small” region. When read m i is at most the pivot, the element is swapped down to store and store advances; otherwise the cursor moves on alone. When the scan reaches hi, the pivot is swapped into place at store, and store is returned as its final index.

Because Fix is opaque to computation, a companion lemma unfolds it one step. This is the only place the fixpoint is ever taken apart; every proof afterwards rewrites with this equation instead:

  Lemma part_step m store i hi :
    part m store i hi =
      if dec (i < hi)
      then if dec (leb (read m i) (read m hi) = true)
           then part (swap m i store) (S store) (S i) hi
           else part m store (S i) hi
      else (swap m store hi, store).
  Proof.
    unfold part at 1. rewrite Fix_eq.
    - reflexivity.
    - intros x f g Hfg.
      do 2 (apply functional_extensionality; intro).
      destruct (dec (x < hi)); [|reflexivity].
      destruct (dec (_ = true)); rewrite Hfg; reflexivity.
  Qed.

rewrite Fix_eq replaces one layer of the fixpoint by its body; the second bullet is the obligation that the body uses its recursive argument in the same way whichever function is plugged in, which is what Fix_eq demands before it will unfold.

Why partition needs an invariant at all

Partition promises something precise: when it finishes, the pivot sits at some index p, everything to the left of p is at most the pivot, and everything to the right is at least the pivot. That promise is what lets the two recursive sorts be glued back together — sort the left, sort the right, and the pivot already separates them, so the concatenation is sorted with no comparison across the seam.

Nothing about swapping elements around guarantees that promise on its own. Suppose the scan moved elements without keeping track of what it had already seen. When it finally drops the pivot at store, why should everything below store be small and everything above be large? If a large element had been left below store, the pivot would land in the wrong spot, the left half would contain something bigger than the pivot, and gluing two sorted halves would produce an unsorted whole — silently, with no error. The invariant is the running promise that this cannot happen. It says: the region [lo, store) already holds elements at most the pivot, and the region [store, i) already holds elements strictly greater. Writing m[k]m[k] for the value at index kk, ss for the store boundary, and v=m[hi]v = m[hi] for the pivot value:

lo≤s≤i≤hi,∀k∈[lo,s):m[k]≤v,∀k∈[s,i):m[k]>v.lo \le s \le i \le hi, \qquad \forall k \in [lo, s):\ m[k] \le v, \qquad \forall k \in [s, i):\ m[k] > v.

Here it is, verbatim from Theory/Quicksort.v:

  Definition scan_inv (m : mem) (lo store i hi : nat) : Prop :=
    lo <= store /\ store <= i /\ i <= hi /\
    (forall k, lo <= k < store -> leb (read m k) (read m hi) = true) /\
    (forall k, store <= k < i -> leb (read m k) (read m hi) = false).

Every scan step preserves it: when read m i is at most the pivot the swap extends the small region by one, and when it is greater the cursor advances and the large region grows by one. Because the invariant holds all the way to i = hi, the final swap plants the pivot exactly where small ends and large begins. That conclusion is packaged as the partition’s postcondition — pivot bounds, a frame saying cells outside the range are untouched, a permutation of the range, and the two pivot-ordering facts. In symbols, if partitioning turns memory mm into m′m' with the pivot at index pp, and ∼\sim means “is a permutation of”:

s≤p≤hi,{the pivot lands in the range}∀k,k<s∨hi<k⟹m′[k]=m[k],{frame: outside cells untouched}m[s..hi]∼m′[s..hi],{no element lost or invented}∀k∈[lo,p):m′[k]≤m′[p],{left of the pivot: at most it}∀k∈(p,hi]:m′[p]≤m′[k].{right of the pivot: at least it} \begin{aligned} & s \le p \le hi, && \{\text{the pivot lands in the range}\}\\ & \forall k,\ k < s \ \vee\ hi < k \implies m'[k] = m[k], && \{\text{frame: outside cells untouched}\}\\ & m[s..hi] \sim m'[s..hi], && \{\text{no element lost or invented}\}\\ & \forall k \in [lo, p):\ m'[k] \le m'[p], && \{\text{left of the pivot: at most it}\}\\ & \forall k \in (p, hi]:\ m'[p] \le m'[k]. && \{\text{right of the pivot: at least it}\} \end{aligned} 

  Definition part_out (m : mem) (lo store hi : nat) (m' : mem) (p : nat) : Prop :=
    store <= p <= hi /\
    (forall k, k < store \/ hi < k -> read m' k = read m k) /\
    Permutation (slice m store (S hi - store)) (slice m' store (S hi - store)) /\
    (forall k, lo <= k < p -> leb (read m' k) (read m' p) = true) /\
    (forall k, p < k <= hi -> leb (read m' p) (read m' k) = true).

Here slice m lo n := map (read m) (seq lo n) is the list of values the range actually holds — the observable content of memory that the permutation and sortedness statements talk about.

The frame theorem, in full

The three main theorems all go by well-founded induction on the pair (lo, hi), via the measure S hi - lo, and each rewrites the recursion open with qsort_step_part (the analogue of part_step for the outer recursion) and then appeals to the partition postcondition. The shortest of the three is the frame: sorting the range [lo, hi] changes no cell outside it —

k<lo∨hi<k⟹(qsortmlohi)[k]=m[k].k < lo \ \vee\ hi < k \;\implies\; (\mathrm{qsort}\ m\ lo\ hi)[k] = m[k].

The argument is short because every step is a named fact rather than fresh work — the two recursive calls stay within their sub-ranges, so a cell outside the whole range is outside both, and the partition’s own frame covers the rest.

  Theorem qsort_frame : forall lo hi m k,
    k < lo \/ hi < k -> read (qsort m lo hi) k = read m k.
  Proof.
    apply (measure_ind2 (fun lo hi => S hi - lo)
             (fun lo hi => forall m k,
                  k < lo \/ hi < k -> read (qsort m lo hi) k = read m k)).
    intros lo hi IH m k Hk.
    destruct (dec (S hi - lo < 2)) as [L|L]; [now rewrite qsort_small|].
    destruct (part m lo lo hi) as [m' piv] eqn:Ep.
    destruct (part_correct m lo hi m' piv ltac:(lia) Ep) as ((Hp1 & Hp2) & Hfr & _).
    rewrite (qsort_step_part m lo hi m' piv L Ep ltac:(lia)).
    rewrite (IH (S piv) hi ltac:(lia) _ k ltac:(lia)).
    rewrite (IH lo (piv - 1) ltac:(lia) _ k ltac:(lia)).
    apply Hfr; lia.
  Qed.

Walking it:

  • apply (measure_ind2 (fun lo hi => S hi - lo) ...) starts the strong induction on the pair, giving an induction hypothesis IH for every pair whose measure is strictly smaller.
  • destruct (dec (S hi - lo < 2)) ...; [now rewrite qsort_small|] handles the base case: a range of fewer than two elements is returned unchanged, so the cell is trivially untouched.
  • destruct (part ...) as [m' piv] eqn:Ep names the partitioned memory m' and the pivot index piv.
  • destruct (part_correct ...) as ((Hp1 & Hp2) & Hfr & _) reads off the partition postcondition — the pivot bounds Hp1, Hp2 and the frame Hfr; the permutation and ordering facts are not needed here.
  • rewrite (qsort_step_part ... ltac:(lia)) unfolds one step of the recursion into “partition, then sort the left sub-range, then the right”, the side condition (the pivot lands in bounds) closed inline.
  • The two rewrite (IH ...) lines apply the induction hypothesis to the pairs (S piv, hi) and (lo, piv - 1): cell k lies outside both sub-ranges (the pivot bounds make that an arithmetic fact), so neither recursive sort moves it.
  • apply Hfr; lia finishes with the partition’s own frame: partition did not touch k either.

The proof itself is barely a dozen lines, and no step is arithmetic dressed up as reasoning — the lia calls only discharge index bookkeeping. The two harder theorems, the permutation and the sortedness, follow the same skeleton but do more gluing work, so they run longer; each of their steps is again a single named lemma (slice_split_pivot, perm_transpose, sorted_glue), which is what keeps them readable rather than short.

The two halves, and the whole

The two facts that make “sorted” mean something are stated directly for qsort m lo hi, about the observable content of the range — the list m[lo..hi]m[lo..hi] of values it holds. Writing m′=qsortmlohim' = \mathrm{qsort}\ m\ lo\ hi for the sorted memory, they say

m[lo..hi]∼m′[lo..hi]andm′[lo]≤m′[lo+1]≤⋯≤m′[hi].m[lo..hi] \;\sim\; m'[lo..hi] \qquad\text{and}\qquad m'[lo] \le m'[lo+1] \le \cdots \le m'[hi].

The range ends up a permutation of what it started as:

  Theorem qsort_perm : forall lo hi m,
    Permutation (slice m lo (S hi - lo)) (slice (qsort m lo hi) lo (S hi - lo)).

and it ends up sorted, in the adjacent-pair sense of Coq’s standard Sorted:

  Theorem qsort_sorted : forall lo hi m,
    Sorted leP (slice (qsort m lo hi) lo (S hi - lo)).

where leP x y := leb x y = true. With transitivity of leb in hand, adjacent-pair sortedness strengthens for free to the every-pair form StronglySorted, by the standard-library bridge:

  Corollary qsort_strongly_sorted : forall lo hi m,
    StronglySorted leP (slice (qsort m lo hi) lo (S hi - lo)).
  Proof.
    intros. apply Sorted_StronglySorted;
      [exact leb_trans | apply qsort_sorted].
  Qed.

The two halves are then bundled into a single correctness statement — a permutation that is sorted:

  Corollary qsort_correct : forall m lo hi,
    Permutation (slice m lo (S hi - lo)) (slice (qsort m lo hi) lo (S hi - lo))
    /\ Sorted leP (slice (qsort m lo hi) lo (S hi - lo)).
  Proof. intros. split; [apply qsort_perm | apply qsort_sorted]. Qed.

Running it, and instantiating the memory

All of the above is generic in the memory, and the program file checks that the algorithm actually computes on a concrete one — a list, with nth_default for reading and a list swap for writing:

Check Logic.eq_refl : qsort nat Nat.leb (list nat) (nth_default 0) (list_swap 0) [3;4;2;0] 0 3 = [0;2;3;4].

[3;4;2;0] sorted over indices 0 through 3 is [0;2;3;4], and Coq accepts the reflexivity proof, so the two sides really are equal by computation. The correctness theorem needs a memory whose read/swap laws hold at every index; a list only satisfies them when both indices are in range, so the correctness file instantiates the abstract section at a total function memory nat -> nat instead, where the laws hold unconditionally:

  Example fun_qsort_correct (m : nat -> nat) (lo hi : nat) :
    let m' := qsort nat Nat.leb (nat -> nat) fread fswap m lo hi in
    Permutation (slice nat (nat -> nat) fread m lo (S hi - lo))
                (slice nat (nat -> nat) fread m' lo (S hi - lo))
    /\ Sorted (fun x y => Nat.leb x y = true)
              (slice nat (nat -> nat) fread m' lo (S hi - lo)).
  Proof.
    apply qsort_correct;
      auto using fswap_l, fswap_r, fswap_other, nat_leb_total.
  Qed.

The proof is one line: supply the four memory-and-comparison laws, and qsort_correct does the rest. Every guarantee proved in the abstract becomes a fact about this particular sorting routine for free.
