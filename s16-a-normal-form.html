A-normal form: naming every intermediate result

Look at the expression f (g x) (h y). A compiler that wants to generate code for it has to answer questions the surface syntax leaves implicit: in what order do g x and h y run? Where does each intermediate result live while the outer call to f waits for both? Almost every later stage — assigning registers, spilling to the stack, converting to continuation-passing style — needs those answers, and reconstructs them from the nesting. A-normal form (ANF), introduced by Cormac Flanagan, Amr Sabry, Bruce Duba and Matthias Felleisen (The Essence of Compiling with Continuations, 1993), computes the answers once and writes them into the syntax. It rewrites every program so that each non-trivial subexpression is given a name by a let before it is used:

let a = g x in let b = h y in f a b

Now the order is on the page (g x, then h y, then f a b), every intermediate value has a name, and every function is applied only to atoms — variables or lambdas, never a nested computation. This post defines that transform on the lambda calculus, proves its output really is in this shape, and proves that a let is semantically transparent when the thing it names has a value.

The lambda calculus, and a let we have to encode

The lambda calculus has three terms: a variable, a function \x. body, and an application f a. Variables are de Bruijn indices — a number counting the enclosing \s out to the binder — so \x. x is Abs (Var 0):

Inductive tm : Type :=
  | Var : nat -> tm
  | Abs : tm -> tm
  | App : tm -> tm -> tm.

There is no let in this language, so we encode one as an immediate application:

let x := c in e   is   App (Abs e) c

Under call-by-value this reduces exactly as a let should: once c has run to a value, that value is substituted for x (de Bruijn index 0) in e. So “name c, then continue with e” is written App (Abs e) c, and that is the only building block the transform needs.

What “in ANF” means, precisely

With the encoding fixed, an ANF term is generated by a small grammar: an atom is a variable or a lambda whose body is itself in ANF; a computation is an application of two atoms; and an ANF term is an atom, a computation, or a let binding a computation. In Coq this is two mutually inductive predicates:

Inductive Atom : tm -> Prop :=
  | at_var : forall n, Atom (Var n)
  | at_abs : forall b, Anf b -> Atom (Abs b)
with Anf : tm -> Prop :=
  | anf_atom : forall a, Atom a -> Anf a
  | anf_comp : forall a1 a2, Atom a1 -> Atom a2 -> Anf (App a1 a2)
  | anf_let  : forall a1 a2 body,
      Atom a1 -> Atom a2 -> Anf body -> Anf (App (Abs body) (App a1 a2)).

Read anf_let: App (Abs body) (App a1 a2) is the encoding of let x := (a1 a2) in body, and it is in ANF when the bound computation a1 a2 is two atoms and the body is in ANF. The defining property of ANF is that no application has a nested computation as operator or operand — every argument is already a name.

The transform

The transform flattens a term into a telescope of lets ending in an atom. assemble builds the telescope, folding a binding list into nested immediate-application lets:

Definition assemble (binds : list tm) (atom : tm) : tm :=
  fold_right (fun c acc => App (Abs acc) c) atom binds.

assemble [c0; c1] atom is let x0 := c0 in let x1 := c1 in atom. The worker flat t returns the pair (binds, atom) — the computations to let-bind, and the atom that names t’s value. The delicate case is App f a: flatten f and a separately, then concatenate their telescopes, with de Bruijn indices adjusted so each part still points at the right binder once the parts sit in sequence.

Fixpoint flat (t : tm) : list tm * tm :=
  match t with
  | Var n => ([], Var n)
  | Abs b => let (bb, ab) := flat b in ([], Abs (assemble bb ab))
  | App f a =>
      let (bf, af) := flat f in
      let (ba, aa) := flat a in
      let m := length bf in
      let p := length ba in
      let ba' := mapi_from 0 (fun j c => lift m j c) ba in
      let c := App (lift p 0 af) (lift m p aa) in
      (bf ++ ba' ++ [c], Var 0)
  end.

Definition anf (t : tm) : tm := let (b, a) := flat t in assemble b a.

On App f a the bindings are f’s telescope bf, then a’s telescope ba shifted past f’s m binders (ba'), then one final binding c that applies f’s atom to a’s atom — each lifted past the other’s binders. The atom returned is Var 0, the name of that last binding. Under a lambda, the body is flattened and assembled in place (Abs (assemble bb ab)), so no binding escapes the binder.

The output really is in ANF

The syntactic correctness theorem: the transform’s output is in ANF for every input term. Writing 𝖠𝖭𝖥\mathsf{ANF} for the set of terms generated by the grammar above,

∀t.anf(t)∈𝖠𝖭𝖥.\forall t.\quad \mathrm{anf}(t) \in \mathsf{ANF}.

Theorem anf_is_anf : forall t, Anf (anf t).
Proof.
  intros t. unfold anf.
  destruct (flat_inv t) as [Hatom Hbinds].
  destruct (flat t) as [b a]. simpl in Hatom, Hbinds.
  apply assemble_Anf; assumption.
Qed.

It rests on two facts about the flattener, both proved by induction. The invariant flat_inv says flat t’s returned atom is an Atom and every binding it returns is a computation (Forall Comp):

Lemma flat_inv : forall t,
  Atom (snd (flat t)) /\ Forall Comp (fst (flat t)).

And assemble_Anf says a telescope of computations ending in an atom is in ANF:

Lemma assemble_Anf : forall binds atom,
  Atom atom -> Forall Comp binds -> Anf (assemble binds atom).

So anf_is_anf reads off directly: flat_inv supplies the two hypotheses assemble_Anf needs, and anf t is assemble of flat t. The one subtlety the proofs handle is that the de Bruijn lifts in flat must not spoil ANF-ness — but Atom and Anf look only at a term’s shape, never at index values, so lifting preserves both (lift_pred, proved by mutual induction on the derivation).

A let is transparent when it names a value

Now the semantic side. Naming a subexpression is only sound if using the name gives the same answer as using the subexpression. The exact statement: a let whose bound computation runs to a value v is convertible to its body with v substituted in.

Here == is conv, the least equivalence on terms containing call-by-value reduction and closed under the term constructors — program equality, on which optimizations are rewrites. Blackboard version: writing bb for the body, cc for the bound expression, and vv for the value it reduces to,

(λ.b)c≡(λ.b)v{c→*v,congruence in argument position} (\lambda.\,b)\;c \;\equiv\; (\lambda.\,b)\;v \quad\{c \to^{*} v,\ \text{congruence in argument position}\} ≡b[0:=v]{call-by-value β, since v is a value} \;\equiv\; b[0 := v] \quad\{\text{call-by-value } \beta\text{, since } v \text{ is a value}\}

First run the bound expression to its value; then fire the let by beta. In Coq:

Lemma let_correct : forall body rhs v,
  value v -> star rhs v -> App (Abs body) rhs == subst 0 v body.
Proof.
  intros body rhs v Hv Hstar.
  calc (App (Abs body) rhs)
    == (App (Abs body) v)
    == (subst 0 v body).
  - apply conv_appr. apply conv_of_star. exact Hstar.
  - apply conv_beta. exact Hv.
Qed.

The calc chain writes the two blackboard objects and leaves one obligation per link. The first link, App (Abs body) rhs == App (Abs body) v: conv_of_star turns the reduction rhs →* v into the conversion rhs == v, and conv_appr lifts it under the application’s argument position (a congruence rule of conv). The second link, App (Abs body) v == subst 0 v body: this is conv_beta, the call-by-value beta law, which needs v to be a value — exactly the hypothesis Hv. Each administrative let the transform inserts binds a computation that, when reached under call-by-value, has been run to a value, so this lemma is precisely what makes it disappear.

One caveat, and the reason for the conditional statement: the unconditional anf t == t is false for call-by-value. If a subterm sitting in argument position diverges, naming it cannot be undone by any finite conversion — both the source and its ANF loop, but no finite chain of steps connects them. So the semantic content is stated where it holds, on terminating computations, and demonstrated end-to-end on closed terms by running the evaluator on both sides.

One term, transformed and run

Take (\x.x) (\x.x) — the identity applied to itself. Its ANF names the redex with a let: on the blackboard,

anf((λ.0)(λ.0))=𝐥𝐞𝐭r:=(λ.0)(λ.0)𝐢𝐧r,\mathrm{anf}\bigl((\lambda.\,0)\,(\lambda.\,0)\bigr) \;=\; \mathbf{let}\ r := (\lambda.\,0)\,(\lambda.\,0)\ \mathbf{in}\ r,

which in the immediate-application encoding is:

Example anf_id_id_shape :
  anf (App idt idt)
  = App (Abs (Var 0)) (App (Abs (Var 0)) (Abs (Var 0))).
Proof. reflexivity. Qed.

Reading the encoding, that is let r := (\.0)(\.0) in r: the application (\.0)(\.0) is bound to a name, and the body just returns the name. Both its operator and operand are now atoms. And the transform preserves the value — source and ANF reach the same normal form when run by the evaluator:

Example anf_preserves_id_id :
  eval 50 (anf (App idt idt)) = eval 50 (App idt idt).
Proof. reflexivity. Qed.

The same holds for genuinely nested terms — (\x.x) ((\x.x)(\x.x)), ((\x.x)(\x.x)) (\x.x), and K (\x.x) (\x.\y.y) — each checked by running the evaluator on the term and on its ANF and confirming the results agree.
