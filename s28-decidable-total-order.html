A decidable total order, and four comparisons for the price of one

A proposition is decidable when it comes with a decision procedure: a value of type {P} + {~ P} that either proves P, tagged on the left, or proves its negation ~ P, tagged on the right — and computes which. Writing if dec P then .. else .. branches on that tag like an ordinary if, and in a proof, destruct (dec P) hands you P in one branch and ~ P in the other. This post applies that idea to ordering: a single decision procedure for one comparison, from which <, <=, >, and >= and all their laws follow.

One strict order, decided; everything else derived

The specification is a typeclass. It fixes a single strict order olt (think “less than”), asks that it be decidable and well-behaved, and defines the weak order ole (“less than or equal”) as its mirror. From Interface/DecTotalOrder.v:

Class DecTotalOrder (A : Type) := {
  olt : A -> A -> Prop;
  ole : A -> A -> Prop;
  #[global] olt_dec :: forall x y, Decidable (olt x y);
  #[global] olt_strict :: StrictOrder olt;
  olt_total : forall x y, olt x y \/ x = y \/ olt y x;
  ole_nlt   : forall x y, ole x y <-> ~ olt y x;
}.

Reading the fields: olt and ole are the two order relations. Then three demands on olt. olt_dec is the single decision procedure — for every pair x y, olt x y is decidable, so the program can branch on it. olt_strict asks that olt be a StrictOrder, standard-library shorthand for irreflexive (x < x never holds) and transitive (x < y and y < z give x < z). olt_total is trichotomy: any two elements compare one of exactly three ways — x < y, or x = y, or y < x. The last field, ole_nlt, is the definition tying the two relations together: x <= y means precisely not y < x.

The other two comparisons need no new content — they are the same two relations with their arguments swapped:

Definition ogt {A} `{DecTotalOrder A} (x y : A) : Prop := olt y x.
Definition oge {A} `{DecTotalOrder A} (x y : A) : Prop := ole y x.

x > y is just y < x; x >= y is just y <= x. So the four comparisons <, <=, >, >= all trace back to one relation olt, and its one decider olt_dec settles every one of them — decide olt y x and you have decided x > y too. Four comparisons for the price of one.

The order laws follow: antisymmetry

The point of stating the axioms so sparely is that the rest of the order’s behaviour is a consequence, proved once, for every type that satisfies the class. Take antisymmetry: if x <= y and y <= x, then x and y are equal. The whole argument runs through trichotomy — of the three ways x and y can compare, the two strict ones are each ruled out by one of the hypotheses, leaving only equality:

x≤y∧y≤x;trichotomy gives x<y∨x=y∨y<x;x<y contradicts y≤x,y<x contradicts x≤y.x \le y \ \wedge\ y \le x; \quad \text{trichotomy gives } x<y \,\vee\, x=y \,\vee\, y<x; \quad x<y \text{ contradicts } y \le x, \quad y<x \text{ contradicts } x \le y.

Here is the proof exactly as it stands in Theory/DecTotalOrder.v:

Lemma ole_antisym : forall x y : A, ole x y -> ole y x -> x = y.
  intros x y X Y; destruct (olt_total x y) as [XY|[E|YX]].
  - now destruct (proj1 (ole_nlt y x) Y).
  - exact E.
  - now destruct (proj1 (ole_nlt x y) X).
Qed.

Walking it:

  • intros x y X Y names the two elements and the two hypotheses, X : ole x y and Y : ole y x.
  • destruct (olt_total x y) invokes trichotomy, splitting into the three cases XY : olt x y, E : x = y, and YX : olt y x.
  • First case, olt x y. Reading Y : ole y x through ole_nlt (its forward direction, proj1 (ole_nlt y x)) yields ~ olt x y — which contradicts XY, and now destruct closes it.
  • Middle case, E : x = y, is the goal itself: exact E.
  • Third case, olt y x, is symmetric: reading X : ole x y through ole_nlt yields ~ olt y x, contradicting YX.

The companion laws — reflexivity of ole, its transitivity, the bridge olt x y -> ole x y — sit beside this one in the same file, each a short run through the same axioms.

Decidable equality, for free, from the order

A total order carries enough information to decide equality as well — which is worth having, because equality of two elements is itself a proposition, not something you can test directly. The idea: to decide x = y, compare in both directions. If x < y, they differ (equal things cannot compare strictly — that is irreflexivity). If y < x, likewise. If neither strict comparison holds, trichotomy has only one option left, x = y. As a decision by cases:

dec⁡(x=y)={x≠yif x<y{else x=y contradicts irreflexivity}x≠yif y<x{same argument}x=yotherwise{trichotomy, both strict cases excluded}. \operatorname{dec}(x = y) \;=\; \begin{cases} x \neq y & \text{if } x < y \quad \{\text{else } x = y \text{ contradicts irreflexivity}\} \\ x \neq y & \text{if } y < x \quad \{\text{same argument}\} \\ x = y & \text{otherwise} \quad \{\text{trichotomy, both strict cases excluded}\}. \end{cases}

Since olt is decidable, both comparisons are decisions the machine can make, so this whole procedure computes. From Theory/DecTotalOrder.v:

#[global] Instance ord_eq_dec : EqDec A.
Proof.
  intros x y.
  destruct (dec (olt x y)) as [XY|NXY].
  - right; intros E; subst y; exact (StrictOrder_Irreflexive x XY).
  - destruct (dec (olt y x)) as [YX|NYX].
    + right; intros E; subst y; exact (StrictOrder_Irreflexive x YX).
    + left; destruct (olt_total x y) as [XY|[E|YX]];
        [contradiction | exact E | contradiction].
Defined.

EqDec A is exactly “equality on A is decidable.” The proof decides it by two uses of the order’s own decider:

  • destruct (dec (olt x y)) asks whether x < y. If yes (XY), then right — they are unequal — and to prove x <> y, assume E : x = y, substitute, and StrictOrder_Irreflexive x XY refutes x < x.
  • If x < y is false (NXY), destruct (dec (olt y x)) asks the other direction. If y < x (YX), the same irreflexivity argument gives right again.
  • If both strict comparisons fail, left — they are equal — and trichotomy (olt_total x y) is invoked: its two strict alternatives are already excluded (contradiction), so only exact E, the equality, remains.

The instance is Defined, keeping it transparent, so dec (x = y) genuinely computes on a concrete type.

A concrete instance: strings in dictionary order

All of the above is stated for an abstract A. The moment a real type is shown to satisfy DecTotalOrder, every one of those results — the four comparisons, antisymmetry, decidable equality — holds for it. Strings are one such type. They are ordered by comparing their sequences of characters left to right, the ordinary dictionary order, which Impl/Orders.v builds by pullback: a string is mapped to its list of characters (an injective map), and lists are compared lexicographically, so the string order is the character-list order transported back.

#[global] Instance String_DecTotalOrder : DecTotalOrder string :=
  DecTotalOrder_pullback list_ascii_of_string list_ascii_of_string_inj.

With that instance in scope, dec (olt s t) decides string ordering, and because every part is transparent it reduces on closed strings. The file closes with computation checks — each a claim that a comparison reduces to a definite true or false, verified by Logic.eq_refl, meaning Coq confirms the two sides are equal purely by computing:

Check Logic.eq_refl : (if dec (olt "abc"%string "abd"%string) then true else false) = true.
Check Logic.eq_refl : (if dec (olt "b"%string "abc"%string) then true else false) = false.

The first computes to true: "abc" precedes "abd" because they agree until the third character, where c precedes d. The second computes to false: "b" does not precede "abc", since b comes after a at the very first character. Decidable equality — the ord_eq_dec instance from the previous section, now specialized to strings — computes on the same strings:

Check Logic.eq_refl : (if dec ("ab"%string = "ab"%string) then true else false) = true.

The same construction, applied to different embeddings, gives orders on ascii and byte; nat, N, Z, and the canonical rationals Qc are instances directly. Every one of them inherits antisymmetry and decidable equality without another line of proof.
