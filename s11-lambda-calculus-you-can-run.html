The lambda calculus you can run

A cartesian closed category is a category with just enough structure to treat functions as first-class objects: it has products (so you can pair two values) and exponentials (an object BAB^A that stands for “the functions from AA to BB”), together with currying, the one-to-one correspondence between a two-argument function and a function that returns a function. The lambda calculus is the syntax for that structure — a tiny language whose only ideas are “make a function” and “apply a function”. This post is about writing that language down in Coq as a datatype, giving it an evaluator that actually computes, and proving that the evaluator agrees exactly with the mathematical relation that says what a computation step is.

Names are the hard part, so we remove them

The lambda calculus has three kinds of term: a variable, a function \x. body (an abstraction), and an application f a. That is the whole language. The awkward part is not the functions — it is the variables. Consider substitution, the operation at the heart of evaluation: to apply (\x. body) a, you replace every free x in body by a. If you do this naively with names, you can capture a variable by accident. Substituting y into \y. x for x would give \y. y, silently turning a function that ignores its argument into the identity — the inner y was a different y, and the substitution was supposed to leave it alone. Getting this right with names means renaming bound variables on the fly, and that bookkeeping is entirely beside the point.

De Bruijn indices remove the names. Instead of writing a variable’s name, you write a number: the count of enclosing \ you have to cross, from the inside out, to reach the one that binds this variable. Index 0 is the nearest enclosing lambda, 1 is the next one out, and so on.

Take the term \x. \y. x — a function of two arguments that returns the first and discards the second. Stand at the occurrence of x in the body. The nearest enclosing lambda is \y, and this variable is not bound by it. The next enclosing lambda is \x, and that is the one that binds it — one lambda crossed, so the index is 1. The names x and y now carry no information and can be dropped. The term becomes:

λ.λ.1_\lambda.\,\lambda.\,\underline{1}

Two anonymous binders and a single index. With names gone, substitution becomes pure arithmetic on those indices — no name is ever compared, so no name can ever be captured. That is the whole reason to pay the up-front cost of counting.

Here is the datatype, exactly as it stands in the source:

Inductive tm : Type :=
  | Var : nat -> tm
  | Abs : tm -> tm
  | App : tm -> tm -> tm.

Var n is the de Bruijn index n; Abs b is \-of-body-b with no name to record; App f a is application. Our example \x. \y. x is Abs (Abs (Var 1)), and the identity function \x. x is Abs (Var 0).

Substitution is arithmetic

Because variables are numbers counting binders, moving a term underneath a binder means shifting its free indices up by one, so they keep pointing at the same outer lambdas. That shift is lift:

Fixpoint lift (d k : nat) (t : tm) : tm :=
  match t with
  | Var n => Var (if Nat.leb k n then d + n else n)
  | Abs b => Abs (lift d (S k) b)
  | App f a => App (lift d k f) (lift d k a)
  end.

lift d k t adds d to every variable of t that is at least k. The number k counts binders crossed so far: variables below k are bound inside t and must not move; crossing an Abs raises the threshold to S k. Substitution then replaces one index and closes the gap the consumed binder leaves behind:

Fixpoint subst (k : nat) (u : tm) (t : tm) : tm :=
  match t with
  | Var n =>
      if Nat.ltb n k then Var n
      else if Nat.eqb n k then lift k 0 u
      else Var (pred n)
  | Abs b => Abs (subst (S k) u b)
  | App f a => App (subst k u f) (subst k u a)
  end.

Reading the Var case: an index below k is bound locally and stays; an index equal to k is the variable being replaced, so it becomes u lifted to survive the k binders it now sits under; an index above k was pointing past the binder that is being consumed, so it drops by one. Under an Abs, the target index grows to S k. No names, no renaming — just comparisons on nat.

The source states and proves this is capture-free on a concrete term. The following Example is checked by Coq (reflexivity, so both sides compute to the same normal form):

Example subst_avoids_capture :
  subst 0 (Abs (App (Var 0) (Var 1))) (Abs (Var 1))
  = Abs (Abs (App (Var 0) (Var 2))).
Proof. reflexivity. Qed.

The substituted term Abs (App (Var 0) (Var 1)) has a free index 1; pushing it under the binder of Abs (Var 1) bumps that free index to 2, exactly so it keeps pointing where it did before. The free variable escapes capture, mechanically.

What a computation step is

Evaluation is a relation between a term and its immediate successor. This is call-by-value, left to right: the argument of an application must be fully evaluated (a value) before the function fires. A value here is a variable or an abstraction — anything but an application:

Inductive value : tm -> Prop :=
  | value_var : forall n, value (Var n)
  | value_abs : forall b, value (Abs b).

The single-step relation has one rule that does real work and two that say where to look for the next step:

Inductive step : tm -> tm -> Prop :=
  | step_beta : forall b v,
      value v ->
      step (App (Abs b) v) (subst 0 v b)
  | step_app1 : forall f f' a,
      step f f' ->
      step (App f a) (App f' a)
  | step_app2 : forall v a a',
      value v ->
      step a a' ->
      step (App v a) (App v a').

On the blackboard the three rules are written as inference rules — what is above the line must hold for the step below it to fire, with t→t′t \to t' for “term tt steps to t′t'” and b[0↦v]b[0 \mapsto v] for substitution of vv at index 00:

vvalue(λ.b)v→b[0↦v]f→f′fa→f′avvaluea→a′va→va′\frac{v\ \text{value}}{(\lambda.\,b)\;v \;\to\; b[0 \mapsto v]} \qquad \frac{f \to f'}{f\;a \;\to\; f'\;a} \qquad \frac{v\ \text{value}\quad a \to a'}{v\;a \;\to\; v\;a'}

step_beta is the actual reduction: an abstraction applied to a value substitutes that value for index 0 in the body. step_app1 reduces the function position while it is still not a value; step_app2 waits until the function is a value, then reduces the argument. Together they pin down a unique place to reduce at each stage.

The evaluator, and why it is trustworthy

The relation says what a step is, but it does not run — it is a Prop, a statement, not a program. The executable counterpart is a function that, given a term, returns its next term if one exists:

Fixpoint step_fn (t : tm) : option tm :=
  match t with
  | App f a =>
      if is_value f then
        if is_value a then
          match f with
          | Abs b => Some (subst 0 a b)
          | _ => None
          end
        else option_map (App f) (step_fn a)
      else option_map (fun f' => App f' a) (step_fn f)
  | _ => None
  end.

Now there are two descriptions of a step: the relation step and the function step_fn. A separate evaluator would be worthless if it disagreed with the definition, so the source proves the two agree in both directions. Soundness: every step the function takes is a genuine step according to the relation. Completeness: every step the relation permits is the one the function finds. Writing ss for the function and t→t′t \to t' for the relation, the two lemmas say

s(t)=Somet′⟹t→t′andt→t′⟹s(t)=Somet′.s(t) = \mathrm{Some}\;t' \;\implies\; t \to t' \qquad\text{and}\qquad t \to t' \;\implies\; s(t) = \mathrm{Some}\;t'.

Between them, the executable function and the mathematical relation are interchangeable.

Soundness, in full:

Lemma step_fn_sound : forall t t', step_fn t = Some t' -> step t t'.
Proof.
  induction t as [n | b _ | f IHf a IHa]; intros t' H; simpl in H.
  - discriminate.
  - discriminate.
  - destruct (is_value f) eqn:Hf.
    + destruct (is_value a) eqn:Ha.
      * destruct f; try discriminate.
        injection H as <-. constructor. apply is_value_value; exact Ha.
      * destruct (step_fn a) as [a'|]; [|discriminate].
        injection H as <-. constructor.
        { apply is_value_value; exact Hf. }
        { apply IHa; reflexivity. }
    + destruct (step_fn f) as [f'|]; [|discriminate].
      injection H as <-. constructor. apply IHf; reflexivity.
Qed.

The proof is an induction on the term, following step_fn’s own shape. A Var or an Abs makes step_fn return None, so the hypothesis step_fn t = Some t' is contradictory and discriminate closes both. For App f a, split on whether f is a value: if it is and a is too, f must be an Abs for a step to exist, and the result is a step_beta by constructor. If f is a value but a is not, the function recursed into a, and the induction hypothesis IHa turns that recursive step into a step_app2. If f is not a value, it recursed into f, and IHf gives a step_app1. Every branch that could return Some is matched to the constructor that justifies it.

Completeness is the shorter of the two — it inducts on the derivation instead of the term:

Lemma step_fn_complete : forall t t', step t t' -> step_fn t = Some t'.
Proof.
  intros t t' H. induction H; simpl.
  - rewrite (value_is_value _ H). reflexivity.
  - rewrite (step_not_value _ _ H). rewrite IHstep. reflexivity.
  - rewrite (value_is_value _ H).
    rewrite (step_not_value _ _ H0). rewrite IHstep. reflexivity.
Qed.

One case per rule of step. For step_beta, the argument is a value, so value_is_value rewrites the inner test to true and the function returns the same substitution subst 0 v b. For step_app1, the function position is mid-reduction, hence not a value; step_not_value rewrites that test to false, and the induction hypothesis IHstep supplies the recursive answer. For step_app2, both facts are needed — the function is a value, the argument is not — and again IHstep finishes. The two directions together license using step_fn wherever step was meant.

Running a real term

Iterating step_fn with a fuel bound gives a driver, eval fuel t, that takes up to fuel steps or stops early at a value. Because it is an ordinary Coq function, terms reduce inside the proof assistant. Two Examples in the source compute and are closed by reflexivity:

Example eval_id_id : eval 10 (App idt idt) = idt.
Proof. reflexivity. Qed.

Example eval_k : eval 10 (App (App k idt) (Abs (Abs (Var 0)))) = idt.
Proof. reflexivity. Qed.

Here idt is Abs (Var 0), the identity \x. x, and k is Abs (Abs (Var 1)), our \x. \y. x from the start. The first applies the identity to itself and gets the identity back. The second applies k to idt and a throwaway argument and recovers idt — k keeps its first argument and drops its second, exactly as its de Bruijn index 1 said it would.

The same evaluator, extracted to Haskell, drives a command-line tool that parses ordinary named syntax, converts to de Bruijn form, runs eval, and prints the result back with names. From the tool’s README, a real transcript:

$ ./programs/lambda-eval/lambda-eval "(\x. x x) (\y. y)"
\x. x
$ ./programs/lambda-eval/lambda-eval "(\x. \y. x) a b"
a

The first applies \x. x x to the identity, which duplicates the identity, applies it to itself, and settles at \x. x. The second is k again with concrete arguments a and b: it returns a. The evaluator you just watched agree with its defining relation is the same one producing this output — line for line the extraction of the Coq eval.
