Decidable propositions: branching with the proof attached

An ordinary if needs a boolean. You write if b then .. else .., the machine looks at b, and one of the two branches runs. But most of the facts a program actually wants to branch on are not booleans. In Coq, a < b is a proposition — a member of the sort Prop, the sort of things that can be proved — not a member of bool. So is x = y. A proposition has no runtime value to test; it is a statement, not a bit. You cannot feed it to if.

The usual escape is to keep a separate boolean function around — Nat.ltb : nat -> nat -> bool next to the proposition a < b — and branch on the boolean. That works, but now the program has forgotten why it took a branch. Inside then, all it knows is that some boolean came out true; the fact a < b that the boolean stood for is nowhere in the proof context. This post is about a small typeclass that keeps the two together: a decision procedure that both chooses the branch, so the term still computes, and hands you the corresponding fact for free, so a later proof can use it.

A decision is a boolean carrying its reason

The type Coq already has for “a boolean that carries its reason” is the standard library’s sumbool, written {P} + {~ P}: a value that is either a proof of P, tagged on the left, or a proof of ~ P (“not P”), tagged on the right. The tag is what an if looks at — it is a two-way choice, exactly like a boolean — but riding along with the tag is the actual evidence. A proposition that has such a value is called decidable, and the whole vocabulary is two lines, from Interface/Decidable.v:

Class Decidable (P : Prop) := dec : {P} + {~ P}.
Arguments dec _ {_}.

Class EqDec (A : Type) := eq_dec : forall (x y : A), Decidable (x = y).
#[global] Instance EqDec_Decidable {A} `{EqDec A} : forall x y : A, Decidable (x = y) := eq_dec.

Decidable P says: there is a decision procedure for P, namely a value dec of type {P} + {~ P}. The Arguments dec _ {_} line makes the proposition explicit and the instance implicit, so you write dec P and instance resolution finds the procedure. EqDec A is the special case that comes up most: a type whose equality is decidable, x = y decidable for every pair x y : A. The last line registers the reverse direction — knowing A has decidable equality, Coq can supply Decidable (x = y) on demand.

Now the payoff of the informative type. Write if dec P then a1 else a2. At the term level this is an ordinary if: dec P reduces to a left or a right, and the matching branch is taken, so the whole expression computes and simpl sees straight through it. But in a proof about this term, destruct (dec P) splits into two cases and, in each, adds the evidence to the context: P itself in the then case, ~ P in the else case. The branch and its justification arrive together.

Decidability is closed under “and”

Before decidable propositions are useful you want to build big ones from small ones, and the logical connectives all preserve decidability. Take conjunction: if you can decide P and you can decide Q, you can decide P /\ Q. The reasoning is the obvious case analysis — decide P; if it is false, the conjunction is false; if it is true, decide Q, and the conjunction holds exactly when Q does:

decide P⇝{no⇒P∧Q falseyes⇒decide Q⇝(P∧Q iff Q)\text{decide } P \;\rightsquigarrow\; \begin{cases} \text{no} & \Rightarrow P \wedge Q \text{ false} \\ \text{yes} & \Rightarrow \text{decide } Q \rightsquigarrow (P \wedge Q \text{ iff } Q) \end{cases}

Here is the instance exactly as it appears in Theory/Decidable.v:

#[global] Instance and_dec (P Q : Prop) `{Decidable P} `{Decidable Q} : Decidable (P /\ Q).
destruct H; [destruct H0; [left; auto | right; tauto] | right; tauto].
Defined.

The two backtick-brace arguments `{Decidable P} and `{Decidable Q} bind the two incoming decision procedures under the automatic names H and H0. Walking the one-line tactic:

  • destruct H splits on the decision for P. Left branch: P holds. Right branch: ~ P holds.
  • In the P-holds branch, destruct H0 splits on the decision for Q. If Q holds too, left; auto builds the proof of P /\ Q. If ~ Q, right; tauto builds ~ (P /\ Q) — a conjunction with a false half is false.
  • Back in the ~ P branch, right; tauto again builds ~ (P /\ Q), since a conjunction with a false half is false.

The instance is closed with Defined, not Qed, on purpose: Defined leaves the decision procedure transparent, so a decision built from and_dec still reduces by computation. The disjunction, negation, and implication cases (or_dec, not_dec, impl_dec) sit right next to it, each the same shape of case analysis.

Reading a decision through a plain boolean

Sometimes you have both: a decision procedure dec P and a separate boolean b that is known to reflect the same proposition — P holds exactly when b is true. Then the two kinds of if agree. This is the bridge a program crosses when it wants to restate a clause written with dec in terms of a stdlib boolean comparison such as Nat.ltb. On the blackboard:

(P↔b=𝗍𝗋𝗎𝖾)⇒(if dec⁡P then a1 else a2)=(if b then a1 else a2),\bigl(P \leftrightarrow b = \mathsf{true}\bigr) \;\Longrightarrow\; \Bigl(\text{if } \operatorname{dec} P \text{ then } a_1 \text{ else } a_2\Bigr) \;=\; \Bigl(\text{if } b \text{ then } a_1 \text{ else } a_2\Bigr),

because in each of the four tag/value combinations either both sides pick the same branch, or the reflection hypothesis is contradicted. The lemma, in full, from Theory/Decidable.v:

Lemma dec_if_bool (P : Prop) {DP : Decidable P} (b : bool) (A : Type) (a1 a2 : A) :
  (P <-> b = true) ->
  (if dec P then a1 else a2) = (if b then a1 else a2).
Proof.
  intros Hb.
  destruct (dec P) as [p | np]; destruct b; try reflexivity.
  - apply Hb in p; discriminate.
  - exfalso; apply np, Hb; reflexivity.
Qed.

The hypothesis Hb : P <-> b = true is the reflection assumption. The proof splits every combination and discards the consistent ones:

  • destruct (dec P) as [p | np]; destruct b produces four cases, one for each pairing of the decision’s tag with the boolean’s value.
  • try reflexivity closes the two agreeing cases outright — decision says yes and b is true, or decision says no and b is false — because both sides then pick the same branch.
  • The remaining two are contradictory. apply Hb in p; discriminate: p : P gives b = true through Hb, clashing with the b = false of this case. exfalso; apply np, Hb; reflexivity: here b is true, so Hb yields P, which np : ~ P refutes.

Two cases survive, two cases contradict, and the two forms of if are proved equal.

The payoff: the standard basis vector

Here is the pattern doing real work. In coordinate space — functions from a finite index type Fin.t n (the type with exactly n distinct values 0, 1, ..., n-1) into a field F — the standard basis vector e i is the function that is 1 at coordinate i and 0 everywhere else:

ei(j)={1if i=j0if i≠j.e_i(j) \;=\; \begin{cases} 1 & \text{if } i = j \\ 0 & \text{if } i \neq j. \end{cases}

Writing it down needs a branch on “is this coordinate the special one?”, and that question, i = j, is an equality of indices — a proposition. From Impl/FinVec.v:

Definition e {n : nat} (i : Fin.t n) : Fin.t n -> F :=
  fun j => if dec (i = j) then 1 else 0.

This reads exactly like the mathematics: e i at input j is if i = j then 1 else 0. It type-checks because Fin.t n has decidable equality — the instance registered in Impl/Decidable.v:

#[global] Instance fin_eq_dec {n : nat} : EqDec (Fin.t n) := Fin.eq_dec.

With that instance in scope, dec (i = j) resolves: instance search finds fin_eq_dec, which supplies the standard library’s Fin.eq_dec as the decision procedure. Because every piece — fin_eq_dec, the sumbool it returns — is transparent, e i j computes to 1 or 0 on concrete indices, so e is a genuine function you can evaluate. And when the time comes to prove something about e i j — that the basis is orthonormal, that a coordinate expansion collapses to a single term — a proof does destruct (dec (i = j)) and finds i = j waiting in one branch and i <> j in the other, the two facts the definition itself was branching on, handed over for free.
