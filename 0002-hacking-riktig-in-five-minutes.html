riktig #2 — Hacking on riktig in five minutes

Second post in the series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved, layer by layer, in the Coq proof assistant — algebra and category theory at the bottom, sorting algorithms, protocol parsers, and eventually a small OS kernel on top. The goal is modifiability: code where a change only obliges you to re-establish a few local facts, and the structure of the system guarantees the rest. Every post starts from zero. Today’s post is the most from-zero of all: how to get a working development environment without installing anything but Docker and VS Code.

A proof project has a bootstrapping problem: Coq, the right Coq version, coq-lsp matched to that version, GHC for the Haskell side, and an editor that speaks to all of them. Ask a colleague to “just install” that stack and you have lost them for an afternoon — and their build still won’t match yours.

Our answer is a dev container: the repository carries a complete description of its own development environment, and VS Code builds and enters it for you. The recipe:

git clone git@git.ri.se:lars.rasmusson/riktig.git
code riktig            # open the folder in VS Code

VS Code notices .devcontainer/ and offers “Reopen in Container”. Click it. The first launch takes a few minutes (it is building the environment); every launch after that is seconds. That is the whole tutorial. You now have:

  • Coq 8.20 with coq-lsp, so opening any theories/*.v file gives you live proof goals — put the cursor after a rewrite and see the goal change (the ejgallego.coq-lsp extension is preinstalled; it is the Coq plugin we prefer of the two that exist, because it checks the file continuously instead of stepping a fragile focus point).
  • The Haskell extension, ready for the Haskell programs as they move into programs/.
  • direnv + nix pre-wired: any terminal you open inside the container is already in the project’s Nix shell — make builds all the theories, coqc is the right version, no activation commands.

What’s in the box, and why it “just works”

Three small files in .devcontainer/ do all the work, and each one solves one specific failure mode of shared dev environments:

Dockerfile — the proof environment is pinned. A Debian base with a non-root vscode user, plus the Determinate Nix installer. The Coq toolchain — Coq 8.20 and its matching coq-lsp — is fixed by the repository’s flake.nix and flake.lock, so your container and mine resolve to bit-identical proof toolchains: if the theories check here, they check there. “Works on my machine” becomes a theorem with a proof. (One deliberate exception: GHC is not pinned — the Makefile builds the one Haskell demo with a plain nix shell nixpkgs#ghc. Haskell only runs extracted code; no proof depends on it, so it doesn’t get the reproducibility ceremony.)

A named volume for /nix — rebuilds don’t recompile the world. Coq and GHC toolchains are gigabytes. The container mounts a Docker volume (riktig-nix-store) at /nix, so everything Nix ever downloads or builds survives container rebuilds. Delete the container, change the Dockerfile, reopen — the cache is still there. This is the single most important line in the setup; without it, every rebuild costs you the afternoon we were trying to save.

start-nix-daemon.sh + lifecycle hooks — no ceremony. The Nix daemon starts when the container starts (postStartCommand), VS Code waits for it (waitFor), and the first-ever launch pre-warms the development shell (onCreateCommand) so that by the time you see a terminal, coqc answers. direnv is pre-trusted for /workspaces, so there is no direnv allow ritual either.

Try it: your first proof interaction

Open theories/Theory/Group.v. Find cancel_l:

Theorem cancel_l : forall a b c : G, a + b == a + c -> b == c.
Proof.
  intros a b c Heq.
  rewrite <- (idl b), <- (invl a), assoc, Heq, <- assoc, invl, idl.
  reflexivity.
Qed.

Click anywhere inside the rewrite chain. The coq-lsp panel shows the goal at that point — delete the last two rewrites and watch the goal become 0 + b == 0 + c… wait, no: try it and see what it actually becomes. That one-click what-if loop is the entire reason we insist on proofs being rewrite chains: every comma is a checkpoint you can inspect, and the proof state between any two steps is a legible algebra goal, not a tactic soup.

If it builds in the container, it builds for everyone. That property — the environment as a checked, versioned artifact — is the same philosophy as the proofs themselves, applied to the toolchain: make the implicit explicit, and let a machine verify it.
