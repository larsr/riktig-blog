riktig #30 — a matrix is an operator

Thirtieth post in the series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved in Coq, structure by structure — carriers, a task-appropriate equality ==, operations, laws. the post on matrices as a representation proved the abstract fact that a matrix represents a linear map between finite-dimensional spaces, and that under this representation composition of maps is matrix multiplication (mat_of_comp, Theory/MatRep.v). Today the same idea in its concrete, computable form: a matrix applied to a coordinate vector is a linear operator, and multiplying matrices is composing those operators. One definition, one packaging, one theorem — and a practical consequence the next post builds on: to apply a product of matrices you never need to form the product.

The definition

A Matrix m n F in this repository is just a rectangle of scalars, Fin.t m -> Fin.t n -> F (Impl/Matrix.v). A coordinate vector is a function Fin.t n -> F — already a vector space, for free, by the pointwise construction of post #6. What a matrix does is one line (Impl/Attention.v, where today’s whole story lives):

Definition mat_vec {m n : nat} (A : Matrix m n F) (v : Fin.t n -> F)
    : Fin.t m -> F :=
  fun i => bigopF (fun j => A i j * v j).

Entry i of A·v is the dot product of row i with v — the textbook definition, as a sum bigopF over the shared index.

The operator is linear

“Operator” is earned, not assumed: mat_vec A must commute with the vector algebra. Additivity is distributivity pushed under the sum, homogeneity is a commutativity swap under the sum:

Lemma mat_vec_add : forall (m n : nat) (A : Matrix m n F)
    (v w : Fin.t n -> F),
  mat_vec A (v + w) == mat_vec A v + mat_vec A w.

Lemma mat_vec_scale : forall (m n : nat) (A : Matrix m n F) (a : F)
    (v : Fin.t n -> F),
  mat_vec A (a *: v) == a *: mat_vec A v.

With properness for ==, that is exactly the content of the LinearMap record (Interface/LinearMap.v), so the file packages it: mat_vec_lm A : LinearMap F (Fin.t n -> F) (Fin.t m -> F). A matrix denotes a linear map between coordinate spaces — as a value you can hand to every Hom-space theorem the library already has.

The theorem: multiplication is composition

Theorem mat_vec_mul : forall (m n p : nat) (A : Matrix m n F)
    (B : Matrix n p F) (v : Fin.t p -> F),
  mat_vec (mat_mul A B) v == mat_vec A (mat_vec B v).

Applying the product is applying the factors in turn. The proof is a small pleasure: view v as a one-column matrix fun k _ => v k : Matrix p 1 F. Then both sides of the goal are, definitionally, one entry of the two associations of a triple matrix product — and mat_mul_assoc (the Fubini argument proved once in Impl/Matrix.v) closes it. No sum is manipulated twice. The same one-column view gives the identity law from mat_mul_1l:

Theorem mat_vec_id : forall (n : nat) (v : Fin.t n -> F),
  mat_vec mat_id v == v.

And at the LinearMap level, mat_vec_lm (mat_mul A B) == lm_comp (mat_vec_lm A) (mat_vec_lm B) (mat_vec_lm_mul): the packaging sends mat_mul to composition, which is the concrete coordinate-space face of post #10’s representation theorem.

What this buys

Read mat_vec_mul right to left and it is an algorithm choice: to compute (A·B)·v you may instead compute A·(B·v) — two matrix-vector applications, no matrix product, no m × p intermediate. For one vector this drops the multiplication count from m·n·p + m·p to n·p + m·n. The equality is a theorem, so the cheap order is not an approximation; it is the same value, with a machine-checked proof.

What was proved, plainly

  • mat_vec A is a linear operator on coordinate spaces, packaged as a LinearMap (mat_vec_lm), for every field F and every shape.
  • The identity matrix acts as the identity operator (mat_vec_id), and matrix multiplication is operator composition (mat_vec_mul, mat_vec_lm_mul).
  • Not proved here, honestly: anything about which linear maps arise this way (that every linear map between finite-dimensional spaces has a matrix is post #10’s Theory/MatRep.v, over abstract spaces with chosen bases; today’s file works directly on Fin.t n -> F), and anything about evaluation cost — that is the next post, where this theorem does real work.
