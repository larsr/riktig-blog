Safety isn’t enough: prefix delivery needs an acknowledgement

Two programs exchange a stream of messages over a wire that can drop packets and duplicate them, in any amount and any mixture. The alternating-bit protocol (ABP, from here on) makes that survivable with one extra bit attached to every packet: the sender tags all copies of the same message with one bit and flips the bit only when it moves on to the next message, and the receiver accepts a packet only when its bit matches the one it is currently expecting, delivers the message, and flips its own expectation. One bit suffices because at any instant only two logical messages are ever in play — the one just delivered and the one coming next — and they always carry opposite bits.

The property already established for that protocol is that whatever list of messages the receiver hands to the application uses only messages that were genuinely sent, in the order they were sent, possibly with some of them missing — nothing invented, corrupted, or reordered. That property holds no matter how much the wire loses or duplicates, with no assumption that it ever cooperates.

Is that already the guarantee you actually want, though? Picture asking the receiver, at some arbitrary moment, for the list of everything it has handed the application so far. There are two very different things that list could be. It could be a clean initial segment — the first several messages, in order, with nothing after the point where it stops, exactly the picture you would want if you were resuming an interrupted file transfer or replaying a log from where it left off. Or it could have a hole: some message skipped in the middle, with later content still arriving afterward and quietly sliding into the earlier message’s place. Both pictures satisfy “nothing invented, corrupted, or reordered” equally well — the second one only reuses positions, it never breaks the relative order of what does survive. So the question this post answers is: does ruling out corruption and reordering already rule out the hole in the middle, or can the two come apart? They come apart, and a small, concrete example below shows exactly how — a case where a later message is delivered under an earlier message’s identity, in a way that fully satisfies the guarantee already established and would still mislead anyone relying on it. Fixing that turns out to be free for one design and to cost an acknowledgement channel for the other.

Two readings of “nothing wrong survives,” made precise

Call the first picture above — everything delivered really was sent, in order, but perhaps with gaps — a list’s subsequence relationship to what was sent. Call the second, stronger picture — a clean initial segment, nothing after the point where it stops — its prefix relationship. Coq’s name for the first is subseq, defined in an earlier post and reused unchanged here; its name for the second, introduced for the question this post is about, is prefix. A list p is a prefix of a list s when s is literally p followed by some remaining tail:

p is a prefix of s⇔∃s2.s=p++s2p \text{ is a prefix of } s \iff \exists\, s_2.\ s = p \mathbin{+\!+} s_2

Here is that definition, from Theory/Subseq.v:

Definition prefix (p s : list A) : Prop := exists s2, s = p ++ s2.

And here is the fact that makes “prefix is stronger” a precise claim rather than an intuition — every prefix is in particular a subsequence. On the blackboard, writing ⊑\sqsubseteq for “is a subsequence of”, the claim reduces to p⊑p++s2p \sqsubseteq p \mathbin{+\!+} s_2, which is an induction on pp:

[]⊑s2andp′⊑p′++s2⇒x::p′⊑x::(p′++s2),[\,] \sqsubseteq s_2 \qquad\text{and}\qquad p' \sqsubseteq p' \mathbin{+\!+} s_2 \;\Longrightarrow\; x :: p' \,\sqsubseteq\, x :: (p' \mathbin{+\!+} s_2),

the base case because the empty list is a subsequence of anything, the step by keeping the shared head xx on both sides.

Lemma prefix_subseq : forall p s, prefix p s -> subseq p s.
Proof.
  intros p s [s2 Heq]. subst s.
  induction p as [| x p' IH]; simpl.
  - apply subseq_nil_l.
  - apply subseq_keep. exact IH.
Qed.

Walking it: destructuring the hypothesis prefix p s immediately hands over the witness s2 with s = p ++ s2, so subst s rewrites the goal to subseq p (p ++ s2). Induction on p then does the rest: an empty p is a subsequence of anything, by subseq_nil_l; and if p' is (by the induction hypothesis) a subsequence of p' ++ s2, then x :: p' is a subsequence of x :: (p' ++ s2) by keeping the shared head, subseq_keep. So the relationship between the two guarantees proved so far is exactly this one lemma: prefix implies subsequence, never the other way around. Everything below is about that gap.

A subsequence that is not a prefix

Here is a case where the alternating-bit protocol’s safety theorem holds and its conclusion is still, in plain terms, wrong. Send four messages, [0; 1; 2; 3], starting with bit false, with a retransmission budget of k = 1 — one copy per message, no redundancy. The sender’s packets, in order, are:

bit message
false 0
true 1
false 2
true 3

Now let the channel do the most ordinary thing a lossy channel can do: drop exactly one packet, the one carrying message 1, and deliver everything else untouched. The wire the receiver actually sees is

bit message
false 0
false 2
true 3

The receiver starts out expecting false. The first packet matches: it delivers 0 and now expects true. The second packet carries false, which does not match true, so it is silently discarded — exactly the behaviour designed to reject stale retransmissions. The receiver is still expecting true. The third packet carries true, which matches, so it is delivered — but it is message 3, not the message-1 retransmission the bit true was originally reserved for. The receiver has no way to know that; a bit that happens to match is a bit that happens to match. The application ends up receiving [0; 3].

That is a genuine subsequence of [0; 1; 2; 3] — 0 and 3 really were sent, in that relative order, nothing was invented or duplicated, so abp_safety holds exactly as stated. But it is not a prefix: position 1 of a prefix of [0;1;2;3] starting with 0 would have to be 1, and instead it is 3. Message 3 has been delivered in message 1’s slot. This is the two-value alphabet’s specific failure mode: losing one whole burst leaves the receiver’s expected bit stuck, and the very next burst carries the opposite bit and is correctly skipped too — but the burst after that reuses the stuck bit, and the expectation “resynchronizes” purely by coincidence, not because anything was actually recovered.

Here is the witness that this wire is a legitimate outcome of the lossy-and-duplicating channel — no duplication needed at all, just one ordinary drop — from Channel/ABP.v:

Example abp_gap_wire :
  lossy_dup (sender_stream false [0; 1; 2; 3] 1)
    [{| pkt_bit := false; pkt_msg := 0 |};
     {| pkt_bit := false; pkt_msg := 2 |};
     {| pkt_bit := true; pkt_msg := 3 |}].
Proof.
  simpl.
  apply lossy_dup_keep.
  apply lossy_dup_drop.
  apply lossy_dup_keep.
  apply lossy_dup_keep.
  apply lossy_dup_nil.
Qed.

Walking it: lossy_dup_keep keeps message 0’s packet, lossy_dup_drop drops message 1’s, and the two remaining lossy_dup_keep steps keep messages 2 and 3 untouched, ending with lossy_dup_nil once the sent list is exhausted — the derivation is literally “keep, drop, keep, keep,” matching the table above constructor for constructor.

And here is the refutation that the resulting delivery is a prefix. On the blackboard it is one forced equation and one mismatch:

[0;1;2;3]=[0;3]++s2⇒1=3,[0;1;2;3] = [0;3] \mathbin{+\!+} s_2 \;\Longrightarrow\; 1 = 3,

which is false. In Coq:

Example abp_gap_not_prefix :
  ~ prefix (deliver false
    [{| pkt_bit := false; pkt_msg := 0 |};
     {| pkt_bit := false; pkt_msg := 2 |};
     {| pkt_bit := true; pkt_msg := 3 |}])
    [0; 1; 2; 3].
Proof.
  intros [s2 Heq].
  simpl in Heq.
  discriminate Heq.
Qed.

Walking it: assuming prefix (deliver ...) [0;1;2;3] hands over a witness s2 with [0;1;2;3] = deliver false wire ++ s2. Simplifying the left side computes deliver false wire down to [0; 3], so the equation being assumed is [0;1;2;3] = [0; 3] ++ s2, i.e. 1 = 3 at the second position — discriminate closes it immediately, since 1 and 3 are different constructors of nat.

A sharper boundary: reordering breaks even subsequence

The gap example shows prefix failing while subsequence still holds. It is worth being precise about where the boundary actually sits, because it is not “prefix is fragile, subsequence is robust.” Restrict attention to only three messages, [0; 1; 2], so the sent wire is [(false,0); (true,1); (false,2)] — note that messages 0 and 2 share a bit, since the bit cycles with period two while the messages do not repeat. Reverse that wire outright: a completely faithful reorder of the channel — nothing lost, nothing duplicated, purely shuffled. The receiver, still starting at false, now meets message 2’s packet first (bit false, matches), then message 1’s (bit true, matches), then message 0’s (bit false, matches): it delivers [2; 1; 0]. That is not merely lacking a prefix property — it is not even a subsequence of [0;1;2], because a subsequence of a list can never be a distinct permutation of the same length:

[2;1;0]⊑[0;1;2]⇒[2;1;0]=[0;1;2]{equal lengths force equality},[2;1;0] \sqsubseteq [0;1;2] \;\Longrightarrow\; [2;1;0] = [0;1;2] \quad \{\text{equal lengths force equality}\},

which is false. From Channel/ABP.v:

Example abp_reorder_scrambles_wire :
  reorder (sender_stream false [0; 1; 2] 1) (rev (sender_stream false [0; 1; 2] 1)).
Proof.
  apply Permutation_rev.
Qed.

Example abp_reorder_scrambles_not_subseq :
  ~ subseq (deliver false (rev (sender_stream false [0; 1; 2] 1))) [0; 1; 2].
Proof.
  intros H.
  assert (Heq : deliver false (rev (sender_stream false [0; 1; 2] 1)) = [0; 1; 2]).
  { apply subseq_length_eq.
    - exact H.
    - reflexivity. }
  simpl in Heq.
  discriminate Heq.
Qed.

Walking the second proof: assuming a subseq witness H, and knowing the delivered list and [0;1;2] have equal length (both length 3), subseq_length_eq (a subsequence as long as its parent must equal its parent) forces the delivered list to equal [0;1;2] outright. But computing the left side gives [2;1;0], and discriminate finds the contradiction at once. So reorder is excluded from every theorem in this family for a stronger reason than “it would break prefix” — it breaks the weaker guarantee too. Everything from here on stays, as it always has, scoped to channels that only lose and duplicate, never reorder.

Sliding windows get prefix for free

The obvious question after seeing the gap example is whether it is a fact about packet loss in general, or specifically about ABP’s two-value bit. It is the latter. A different protocol, Go-Back-N, uses the same “tag every packet, accept on a match, silently drop everything else” design as ABP, but tags each message with an ordinary natural number instead of a bit — 0, 1, 2, 3, ..., incrementing once per message and never wrapping around. Nothing else changes: the sender still emits a burst of copies per message, the receiver still accepts only a packet whose tag equals what it currently expects, delivers it, and advances the expectation.

That one change — an unbounded tag instead of a two-value one — is enough to get the stronger guarantee with no new mechanism at all. Here is why, informally, before the Coq: suppose a whole burst is lost, so the receiver’s expected tag, say 5, never gets matched. Every later burst’s tag is strictly greater than 5 — sequence numbers only go up, one at a time, and never repeat — so no packet arriving afterward can ever equal 5 again. The receiver’s expectation is not merely behind; it is stuck forever, and nothing more is delivered, ever. Contrast this with the bit example above, where the stuck expectation true was bound to recur two messages later purely because the tag alphabet has only two values. With unbounded tags there is no “two messages later” — the stuck value never comes back.

Here is the core fact this rests on, from Channel/GoBackN.v: once the receiver’s expectation falls behind the sender’s running tag, it can never catch up, because every remaining burst carries a tag at or above the sender’s current position — always beyond what is being waited for. In symbols,

𝑒𝑥𝑝𝑒𝑐𝑡<𝑠𝑒𝑞𝑛≤tag of every later packet⇒no later packet matches 𝑒𝑥𝑝𝑒𝑐𝑡⇒nothing more is delivered.\mathit{expect} < \mathit{seqn} \le \text{tag of every later packet} \;\Longrightarrow\; \text{no later packet matches } \mathit{expect} \;\Longrightarrow\; \text{nothing more is delivered.}

Lemma gbn_deliver_nomatch : forall (k : nat) (ms : list M) (seqn expect : nat) (wire : list Packet),
  expect < seqn -> lossy_dup (gbn_stream_from seqn ms k) wire -> gbn_deliver expect wire = [].

And here is the theorem that fact drives, and its proof, verbatim:

Theorem gbn_prefix : forall (k : nat) (ms : list M) (wire : list Packet),
  lossy_dup (gbn_stream ms k) wire -> prefix (gbn_deliver 0 wire) ms.
Proof.
  intros k ms wire H.
  apply (gbn_deliver_prefix_gen k ms 0 wire H).
Qed.

The one-line proof is a corollary of the general lemma gbn_deliver_prefix_gen, which runs the induction on ms with the sender’s tag and the receiver’s expectation kept equal to each other at every step — the invariant that makes the “stuck forever” argument apply the instant it is ever broken. Its two cases mirror the table above exactly: either the current burst is entirely lost, in which case gbn_deliver_nomatch shows the rest of the run delivers nothing more at all (so the delivered list so far is trivially a prefix, with the untouched remainder of ms as the witness tail), or the burst’s message is delivered and the tag/expectation both advance together, and the induction hypothesis handles the rest. There is no case where a later message slips through under an earlier one’s identity, because there is no way for a later tag to ever equal an earlier, already-lost one. The stronger guarantee falls out of a design choice already made for an unrelated reason — sequence numbers big enough not to need wraparound within one run — not from any additional proof effort spent specifically on it.

Recovering prefix for two tags: halting on silence

Go-Back-N gets prefix delivery as a side effect of having room for more than two tags. The alternating-bit protocol does not have that room — by design, it only ever needs one bit — so getting the same guarantee for it needs an actual change to the protocol, not just a sharper look at the existing one. The fix is a new file, Channel/Ack.v, and a new discipline: stop-and-wait with an acknowledgement. The sender does not begin retransmitting message i+1 until it has seen an acknowledgement for message i come back over a return channel. If no acknowledgement ever arrives — because the data packet was lost, or because the acknowledgement itself was lost — the sender simply never sends anything more, for any message, ever. The run halts on silence rather than guessing and moving on.

That single rule is the whole mechanism. It costs an acknowledgement channel (modeled with the same fault-class machinery, lossy_dup, as the forward data channel, just carrying bits instead of packets), and in exchange it closes exactly the gap the counterexample above exploited: nothing can ever be delivered under the wrong message’s identity, because the sender never moves past a message until it knows the message got through.

Here is the run relation itself, from Channel/Ack.v, one round of recursion per application message:

Fixpoint abp_ack_run (b : bit) (ms : list M) (fwire : list Packet) (awire : list bit) : Prop :=
  match ms with
  | [] => fwire = [] /\ awire = []
  | m :: ms' =>
      exists (k : nat) (w1 : list Packet),
        lossy_dup (sender_burst b m k) w1 /\
        ( (deliver b w1 = [] /\ fwire = w1 /\ awire = [])
        \/ (exists a1 : list bit,
              deliver b w1 = [m] /\ lossy_dup [b] a1 /\
              ( (a1 = [] /\ fwire = w1 /\ awire = a1)
              \/ (a1 <> [] /\
                  exists fwire2 awire2,
                    fwire = w1 ++ fwire2 /\ awire = a1 ++ awire2 /\
                    abp_ack_run (negb b) ms' fwire2 awire2) ) ) )
  end.

Read one round at a time. Each message gets its own retransmission budget k, related to a received fragment w1 by the ordinary data channel. Three things can then happen: (1) nothing from w1 was delivered — the data round itself failed — and the whole run stops, with fwire/awire ending right there; (2) the message was delivered, its bit is echoed back as an acknowledgement over the ack channel producing a fragment a1, but a1 came back empty — the ack never made it — and the run stops there too, one round later than case (1); or (3) the acknowledgement genuinely came back (a1 <> []), and the run continues into the next round with the bit flipped. Every path where the run continues has confirmed, via a real acknowledgement, that the previous message actually arrived.

The theorem: whatever deliver computes over the whole forward wire of such a run — the exact same deliver function the plain, unacknowledged protocol already uses — is a genuine prefix of the messages, not merely a subsequence. The blackboard argument is an induction on the message list with one case per branch of the relation, each case producing an explicit prefix witness:

delivered={[]{data round lost; [] is a prefix of anything}[m]{ack lost; m::ms′=[m]++ms′}m::d′{run continues; d′ a prefix of ms′ by induction}\text{delivered} \;=\; \begin{cases} [\,] & \{\text{data round lost; } [\,] \text{ is a prefix of anything}\}\\ [m] & \{\text{ack lost; } m :: ms' = [m] \mathbin{+\!+} ms'\}\\ m :: d' & \{\text{run continues; } d' \text{ a prefix of } ms' \text{ by induction}\} \end{cases}

Theorem abp_ack_prefix : forall (b : bit) (ms : list M) (fwire : list Packet) (awire : list bit),
  abp_ack_run b ms fwire awire -> prefix (deliver b fwire) ms.
Proof.
  intros b ms.
  revert b.
  induction ms as [| m ms' IH]; intros b fwire awire H.
  - simpl in H. destruct H as [Hf Ha]. rewrite Hf. simpl. apply prefix_refl.
  - simpl in H.
    destruct H as [k [w1 [Hburst Hcase]]].
    destruct Hcase as [[Hd [Hf Ha]] | [a1 [Hd [Hack Hcase2]]]].
    + (* halt: data round failed, nothing delivered, run stops *)
      rewrite Hf. rewrite Hd. apply prefix_nil_l.
    + destruct Hcase2 as [[Ha1 [Hf Ha]] | [Ha1 [fwire2 [awire2 [Hf [Ha Hrun]]]]]].
      * (* halt: message m WAS delivered, but its ack never came back *)
        rewrite Hf. rewrite Hd. unfold prefix. exists ms'. reflexivity.
      * (* continue: m delivered, ack came back, recurse *)
        rewrite Hf.
        rewrite (deliver_app w1 fwire2 b).
        rewrite Hd.
        assert (Hnext : next_bit b w1 = negb b).
        { destruct (burst_delivery b m k w1 b Hburst) as [[Hd0 Hn0] | [Hd0 Hn0]].
          - rewrite Hd0 in Hd. discriminate Hd.
          - exact Hn0. }
        rewrite Hnext. simpl.
        unfold prefix.
        destruct (IH (negb b) fwire2 awire2 Hrun) as [ms2 Heq2].
        exists ms2. rewrite Heq2. reflexivity.
Qed.

Walking it, case by case, in the same order the relation offers them:

  • Base case, ms = []. Nothing was ever sent, so fwire = [], deliver of the empty wire is [], and prefix_refl closes prefix [] [].
  • Halt on a failed data round. Hd : deliver b w1 = [] and fwire = w1, so the delivered list is deliver b w1 = [], and prefix_nil_l — the empty list is a prefix of anything — finishes it. This is the same “vacuously safe, nothing delivered” case that already appeared for plain ABP; nothing new here.
  • Halt on a missing acknowledgement. This is the case that did not exist before this file. Here Hd : deliver b w1 = [m] — the message really was delivered — but a1 = [] and fwire = w1, so the delivered list is exactly [m]. Choosing the witness ms' for the existential in prefix gives m :: ms' = [m] ++ ms', closed by reflexivity. The message that got through stays at the end of what is delivered, with the rest of ms as the untouched witness tail; nothing after it is ever claimed to have arrived.
  • Continue. The acknowledgement came back, so the run recurses. deliver_app splits delivery across the two wire fragments w1 and fwire2; burst_delivery (the same crux fact the plain protocol’s safety proof used) pins down that w1 really does flip the expected bit to negb b, since Hd already says w1 delivered [m] and the other disjunct of burst_delivery would contradict that by discriminate. What remains is m :: deliver (negb b) fwire2, and the induction hypothesis on the recursive run gives a prefix witness ms2 for the tail, which extends to a prefix witness for the whole thing.

Every branch either finishes with the empty prefix, finishes with exactly the messages actually delivered and nothing claimed beyond them, or hands off cleanly to the next round. There is no branch, and no way to construct one, in which a later message is ever delivered under an earlier message’s slot — the failure mode the counterexample exhibited cannot arise here, because the sender that produced it (moving on regardless of whether the previous message survived) no longer exists in this model.

A concrete run: everything gets through

The relation abp_ack_run is not vacuous — it is worth checking that an ordinary, fully successful exchange is one of its members, and that on such a run delivery is not just a prefix but the entire message list. Here is the witness, using one copy per round (k := 1, no retransmission needed) and an acknowledgement stream that simply echoes back each accepted bit:

Fixpoint ack_witness (b : bit) (ms : list M) : list bit :=
  match ms with
  | [] => []
  | m :: ms' => b :: ack_witness (negb b) ms'
  end.

Lemma abp_ack_run_witness : forall (b : bit) (ms : list M),
  abp_ack_run b ms (sender_stream b ms 1) (ack_witness b ms).

and the corollary that ties it back to full delivery:

Corollary abp_ack_run_delivers_all : forall (b : bit) (ms : list M),
  exists fwire awire, abp_ack_run b ms fwire awire /\ deliver b fwire = ms.
Proof.
  intros b ms.
  exists (sender_stream b ms 1), (ack_witness b ms).
  split.
  - apply abp_ack_run_witness.
  - apply (abp_collapse_perfect_gen 1 ms b). lia.
Qed.

Walking the corollary: the two exists supply the untouched wires — the sender’s own stream and the echoed acknowledgements — as witnesses; abp_ack_run_witness shows they really form a run of the relation; and abp_collapse_perfect_gen (the fact, proved earlier for the plain protocol, that delivery over an untouched stream returns the message list exactly, for any positive per-message budget) computes deliver b fwire = ms, with lia discharging its k > 0 side condition at k = 1.

So the mechanism does not merely fail to break the good case — on a run where every packet and every acknowledgement gets through, the protocol still delivers the whole list, exactly as the un-acknowledged version did.

The scope this result does and does not cover

Channel/Ack.v applies the data channel and the acknowledgement channel per round, independently: each message gets its own fresh existential wire fragment and its own fresh existential acknowledgement fragment, related to what was sent for that round alone, rather than one single relation governing the whole concatenated run. This is a real simplification, not an oversight — it models “the sender is stopped and waiting for exactly one round trip,” during which no other round’s traffic can possibly be in flight, which is a faithful picture of what stop-and-wait actually does. It is also exactly the simplification the earlier fairness post’s channel model already made for the data side alone; this file extends the same choice to the acknowledgement side.

One consequence worth stating plainly: because each round’s acknowledgement is a single bit or nothing, and each round’s data burst is a list of copies of one packet, there is no way, at per-round granularity, to even pose the question “what if the channel reorders traffic across rounds?” — there is nothing to reorder within one homogeneous list, and the per-round split by construction never gives the channel a chance to interleave one round’s packets with another’s. That is not evidence that reordering across rounds would be safe; it simply is not a question this model can currently ask. Extending abp_ack_run to a single global relation over an entire run — the way the fairness hypothesis’s own sanity check ties back to the plain, ungrouped loss-and-duplication relation — is left for later. What is proved here is exactly what the per-round model supports, no more: prefix delivery whenever both the data channel and the acknowledgement channel lose and duplicate packets, never reorder them.
