riktig #3 — Structures grow arrows: category theory up to CCC

Third post in the series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved, layer by layer, in the Coq proof assistant — algebra at the bottom, then category theory, and on top of those: sorting algorithms, protocol parsers, a small OS kernel. The goal is modifiability: code where a change only obliges you to re-establish a few local facts, and the structure of the system guarantees the rest. Every post starts from zero. Post #1 introduced mathematical structures — interfaces with laws. Today the structures grow arrows.

In post #1 we studied structures from the inside: a group is a set with +, 0, - obeying laws. Category theory makes one radical move: it forgets the inside. You are not allowed to ask what an object contains — only how it connects. Everything must be said with arrows.

The surprise, and the reason a software project cares: this austerity is not a loss. Products, function types, truth values, lambda calculus — all of it can be reconstructed from arrows alone, and anything reconstructed this way transports to every world that has the right arrows. It is the “prove once, hold everywhere” trick from post #1, applied one level up: not laws about elements, but laws about composition itself. (Our tour follows the classic route of Bartosz Milewski’s Category Theory for Programmers, which is the recommended companion reading.)

A category: the algebra of composition

A category has objects and arrows (morphisms) between them. Arrows compose associatively, and every object has a do-nothing arrow. That’s all. In theories/Interface/Category.v:

Class Category (obj : Type) := {
  hom : obj -> obj -> Type;
  cat_id : forall {x : obj}, hom x x;
  comp : forall {x y z : obj}, hom y z -> hom x y -> hom x z;
  cat_idl : forall x y (f : hom x y), comp cat_id f == f;
  cat_idr : forall x y (f : hom x y), comp f cat_id == f;
  cat_assoc : forall w x y z (f : hom y z) (g : hom x y) (h : hom w x),
      comp f (comp g h) == comp (comp f g) h
  (* ... hom-equality is a setoid, composition respects it ... *)
}.

Notation "x ~> y" := (hom x y).
Infix "∘" := comp.

Squint and you’ll recognize post #1’s monoid laws — associativity and two identities — except ∘ is only defined when the arrows fit end-to-end. A category is a typed monoid; a monoid is a category with one object. The two posts are studying the same phenomenon at different resolutions.

One Coq detail is load-bearing: equality of arrows (==) is a setoid per hom — chosen equality, not built-in equality. That choice is why, later in this post’s companion, programs modulo equivalence can be the arrows of a category. Wire equality in too tightly and syntax categories become impossible.

Two categories we now own, machine-checked:

  • Coq’s own types and functions (Impl/CoqCat.v): objects are types, arrows are functions, == is pointwise equality. This is Milewski’s Set, the category programmers secretly live in.
  • Lambda terms modulo conversion (Impl/StlcCat.v): objects are simple types, an arrow A ~> B is a program of type A → B, and two arrows are equal when β/η-conversion says the programs behave alike. Composition is λx. g (f x). The category laws are conversion facts. More on this in post #4.

“The” needs a theorem: initial and terminal objects

With arrows only, how do you point at a specific object? By a property of its arrows. A terminal object has exactly one arrow coming in from everywhere; an initial object exactly one going out to everywhere (Interface/Terminal.v, Interface/Initial.v). In Coq-types: unit is terminal (every type has exactly one function into it), the empty type is initial.

But saying “the terminal object” is an obligation: what if there are two? The first real theorem of the category track (Theory/InitialTerminal.v) discharges it: any two terminal objects are isomorphic — there are arrows both ways composing to identities — and the isomorphism is itself unique. Uniqueness-up-to-iso is category theory’s notion of “the”, and from here on the project earns every definite article it uses.

Products and exponentials: data and functions, from arrows alone

A product of a and b is an object with projections pi1, pi2 such that any object mapping into both a and b factors through it, uniquely (Interface/Product.v):

pair_pi1 : pi1 ∘ pair_mor f g == f
pair_pi2 : pi2 ∘ pair_mor f g == g
pair_unique : pi1 ∘ h == f -> pi2 ∘ h == g -> h == pair_mor f g

No mention of “pairs of elements” — yet in Coq-types this is exactly prod/fst/snd, and the theory file derives the working algebra (pair_eta, pair_comp, commutativity of × up to iso — Theory/Product.v).

Exponentials are the same game played on function types (Interface/Exponential.v): exp_obj a b internalizes the arrows a ~> b as an object, with

eval_mor : prod_obj (exp_obj a b) a ~> b          (* apply *)
curry    : (prod_obj x a ~> b) -> (x ~> exp_obj a b)
curry_beta : eval_mor ∘ (curry h ×× cat_id) == h

Read curry_beta aloud: currying then applying is the original function. Every functional programmer has used this fact; here it is an axiom about arrows, and Theory/Exponential.v proves the inverse direction (curry_uncurry, uncurry_curry) in the usual few rewrite lines.

Cartesian closed categories

A category with a terminal object, all products, and all exponentials is cartesian closed (Interface/CCC.v) — three capabilities bundled:

𝐂𝐂𝐂=nothing+pairing+function types\textbf{CCC} \;=\; \text{nothing} \;+\; \text{pairing} \;+\; \text{function types}

And the tower’s capstone instance, at the bottom of Impl/CoqCat.v:

#[global] Instance CoqCCC : CCC CoqCat := ...

Coq’s own types form a cartesian closed category, machine-checked, no axioms added. Why this is the right stopping point for today: CCCs are exactly the worlds where lambda calculus makes sense. The curry/eval algebra is the typing of λ and application; Impl/StlcCat.v is the syntactic side of that same coin. Which is the cliffhanger for post #4: lambda terms as a formal, executable object — with variables done right.
