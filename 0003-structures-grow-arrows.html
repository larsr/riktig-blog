riktig #3 — Structures grow arrows: category theory up to CCC

Third post in the series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved, layer by layer, in the Coq proof assistant — algebra at the bottom, then category theory, and on top of those: sorting algorithms, protocol parsers, a small OS kernel. The goal is modifiability: code where a change only obliges you to re-establish a few local facts, and the structure of the system guarantees the rest. Every post starts from zero. the post on mathematical structures introduced mathematical structures — interfaces with laws. Today the structures grow arrows.

In post #1 we studied structures from the inside: a group is a set with +, 0, - obeying laws. Category theory makes one radical move: it forgets the inside. You are not allowed to ask what an object contains — only how it connects. Everything must be said with arrows.

The surprise, and the reason a software project cares — the one idea of this post: products, function types, and lambda calculus can be reconstructed from arrows alone, and anything reconstructed this way transports to every world that has the right arrows. It is the “prove once, hold everywhere” trick from post #1, applied one level up: not laws about elements, but laws about composition itself. (Our tour follows the classic route of Bartosz Milewski’s Category Theory for Programmers, which is the recommended companion reading.)

A category: the algebra of composition

A category has objects and arrows (morphisms) between them. Arrows compose associatively, and every object has a do-nothing arrow. That’s all. In theories/Interface/Category.v:

Class Category (obj : Type) := {
  hom : obj -> obj -> Type;
  cat_id : forall {x : obj}, hom x x;
  comp : forall {x y z : obj}, hom y z -> hom x y -> hom x z;
  cat_idl : forall x y (f : hom x y), comp cat_id f == f;
  cat_idr : forall x y (f : hom x y), comp f cat_id == f;
  cat_assoc : forall w x y z (f : hom y z) (g : hom x y) (h : hom w x),
      comp f (comp g h) == comp (comp f g) h
  (* ... hom-equality is a setoid, composition respects it ... *)
}.

Notation "x ~> y" := (hom x y).
Infix "∘" := comp.

Squint and you’ll recognize post #1’s monoid laws — associativity and two identities — except ∘ is only defined when the arrows fit end-to-end. A category is a typed monoid; a monoid is a category with one object. The two posts are studying the same phenomenon at different resolutions.

One Coq detail is load-bearing: equality of arrows (==) is a setoid per hom — chosen equality, not built-in equality. That choice is why, later in this post’s companion, programs modulo equivalence can be the arrows of a category. Wire equality in too tightly and syntax categories become impossible.

Two categories we now own, machine-checked:

  • Coq’s own types and functions (Impl/CoqCat.v): objects are types, arrows are functions, == is pointwise equality. This is Milewski’s Set, the category programmers secretly live in.
  • Lambda terms modulo conversion (Impl/StlcCat.v): objects are simple types, an arrow A ~> B is a program of type A → B, and two arrows are equal when a conversion relation says the programs behave alike. The file’s conv is deliberately a small fragment of full βη-conversion — three generators (conv_beta_id, conv_beta_comp, conv_eta) plus congruence and weakening, exactly the instances the category laws need; the file discloses the scope cut in its header. Composition is λx. g (f x), and the category laws are conversion facts. More on this in post #4.

“The” needs a theorem: initial and terminal objects

With arrows only, how do you point at a specific object? By a property of its arrows. A terminal object has exactly one arrow coming in from everywhere; an initial object exactly one going out to everywhere. The whole interface (Interface/Category.v) is three fields:

Class Terminal {O : Type} (C : Category O) := {
  term : O;
  bang : forall x : O, x ~> term;
  bang_unique : forall (x : O) (f : x ~> term), f == bang x
}.

In Coq-types: unit is terminal (every type has exactly one function into it), the empty type is initial (Interface/Category.v is the mirror image, with init and emit).

But saying “the terminal object” is an obligation: what if there are two? The first real theorem of the category track (Theory/InitialTerminal.v) discharges it: any two terminal objects are isomorphic. It deserves the same slow walk that cancel_l got in post #1, because it is the first proof in the project conducted entirely with arrows — no elements anywhere.

First, what is claimed. An isomorphism between objects x and y is a pair of arrows that undo each other (Theory/Category.v):

Record Iso (x y : O) := {
  iso_to : x ~> y;
  iso_from : y ~> x;
  iso_fromto : iso_from ∘ iso_to == cat_id;
  iso_tofrom : iso_to ∘ iso_from == cat_id
}.

Next, the only lemma the proof needs: every endomorphism of a terminal object is the identity. Why? Both f and cat_id are arrows into a terminal object, and bang_unique says there is only one of those:

Lemma term_endo_id (T : Terminal C) :
  forall f : term (Terminal:=T) ~> term (Terminal:=T), f == cat_id.
Proof.
  intros f.
  rewrite (bang_unique _ f).
  symmetry.
  apply bang_unique.
Qed.

Read it: f == bang term (uniqueness, applied to f), and cat_id == bang term (uniqueness, applied to cat_id), hence f == cat_id. The proof never opens an object; it plays two uses of the same law against each other.

Finally the theorem. Given two terminal objects T1, T2, where do the two arrows of the isomorphism come from? There is no choice to make: T2’s bang at T1 is an arrow T1 ~> T2, and vice versa. Their composites are endomorphisms of terminal objects — identities, by the lemma. The construction is the record filled in with the only inhabitants the types allow:

Definition terminal_iso (T1 T2 : Terminal C) :
  Iso (term (Terminal:=T1)) (term (Terminal:=T2)) := {|
  iso_to := bang (Terminal:=T2) (term (Terminal:=T1));
  iso_from := bang (Terminal:=T1) (term (Terminal:=T2));
  iso_fromto := term_endo_id T1 _;
  iso_tofrom := term_endo_id T2 _
|}.

(initial_iso is the same proof with all arrows reversed. And since any arrow into a terminal object is the bang, the isomorphism could not have been anything else — that uniqueness is immediate from bang_unique, though the file does not state it as a separate theorem.) Uniqueness-up-to-iso is category theory’s notion of “the”, and from here on the project earns every definite article it uses.

Products, exponentials, CCC: the rest of the reconstruction

The remaining structures play the same game the terminal object just played — pick out an object by a universal property of its arrows — so we state them at a glance and let the files speak.

A product of a and b (Interface/Category.v) is an object with projections pi1, pi2 through which any object mapping into both a and b factors, uniquely (pair_pi1, pair_pi2, pair_unique). No mention of “pairs of elements” — yet in Coq-types this is exactly prod/fst/snd, and Theory/Product.v derives the working algebra (pair_eta, pair_comp, commutativity of × up to iso). An exponential (Interface/Category.v) internalizes the arrows a ~> b as an object exp_obj a b, with an application arrow eval_mor and a currying operation whose central law is

curry_beta : eval_mor ∘ (curry h ×× cat_id) == h

— currying then applying is the original function; Theory/Exponential.v proves the inverse round trip (curry_uncurry, uncurry_curry) in the usual few rewrite lines.

A category with a terminal object, all products, and all exponentials is cartesian closed (Interface/Category.v) — three capabilities bundled:

𝐂𝐂𝐂=nothing+pairing+function types\textbf{CCC} \;=\; \text{nothing} \;+\; \text{pairing} \;+\; \text{function types}

And the tower’s capstone instance, at the bottom of Impl/CoqCat.v:

#[global] Instance CoqCCC : CCC CoqCat := ...

Coq’s own types form a cartesian closed category, machine-checked, no axioms added. Why this is the right stopping point for today: CCCs are exactly the worlds where lambda calculus makes sense. The curry/eval algebra is the typing of λ and application; Impl/StlcCat.v is the syntactic side of that same coin. Which is the cliffhanger for post #4: lambda terms as a formal, executable object — with variables done right.
