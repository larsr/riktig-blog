Why autorewrite works on an abstract group but ring and field need a concrete carrier

A group is a set equipped with a way to combine two elements and, for every element, an undo: combining an element with its own undo, in either order, gets back to a single fixed neutral element. The one fact this buys that matters here is cancellation — if combining a fixed element with two different things gives the same result, those two things must already have been equal — and the reasoning behind it uses nothing but the combining rule and the undo rule, never anything about what the elements actually are.

What this post is about is not that fact itself, but two more general questions about getting a machine to reproduce reasoning like it without a person picking each step by hand.

The first question is about automation nobody has to supervise. Simplifying an expression by hand often means repeatedly applying a short list of moves — drop a “plus nothing,” collapse a term sitting next to its own undo, cancel a double negation — in whatever order they happen to apply. Handing that list of moves to a machine and letting it run unattended is only safe if the process is guaranteed to stop: every move has to make the expression strictly simpler, in some sense that no other move in the list can ever undo, or the machine could shuffle the expression back and forth forever.

The second question is less obvious: why would a general-purpose “solve this algebra automatically” tool ever refuse to work on a structure known only by its own defining laws — combine, distribute, has a unit — when the very same tool handles ordinary whole numbers without complaint? Two separate reasons turn out to be involved. One is about what the laws actually promise: a tool that rearranges a calculation into one fixed normal form usually needs the order of combining two things not to matter — three eggs and four eggs is the same total as four eggs and three — and some structures that obey every other algebra law still do not have that property (composing “turn ninety degrees, then flip” is not the same operation as “flip, then turn ninety degrees”). A structure described only by its abstract laws, with no such promise built in, cannot be handed to a tool that depends on it. The other reason has nothing to do with mathematics and everything to do with bookkeeping: telling such a tool about a structure usually means registering, once, the exact operations it should look for — and if that registration is made against a placeholder standing in for “any structure of this shape,” and the placeholder later disappears, there is nothing left for the tool to match against when a genuinely different, concrete structure shows up afterward. Each real structure has to be registered again, by name.

Coq answers the first question with a rewrite database, and the second with the ring/field tactics — and the two answers sit at opposite ends of a hierarchy of algebraic structures: one works on the fully abstract group, the other, for the two reasons above, only works once a concrete structure is in front of it.

Automatic rewriting, formalized

A rewrite database is Coq’s answer to the first question: a named set of equations that a single tactic, autorewrite, applies on repeat — left-hand side found in the goal, replace it with the right-hand side, scan again — stopping only when no equation in the set matches anywhere. Theory/Group.v registers six equations under the name grp:

#[global] Hint Rewrite @mon_idl @mon_idr @grp_invl @grp_invr
                       @neg_involutive @neg_zero : grp.

mon_idl/mon_idr are the two identity laws (combining with the neutral element on the left or right does nothing), grp_invl/ grp_invr are the two undo laws, and neg_involutive/neg_zero say that undoing an undo returns the original element and that undoing the neutral element gives the neutral element back. This is exactly the kind of move list the first question above was about, and each of the six rules answers the termination requirement individually: read left to right, every one makes the right-hand side strictly smaller than the left — an identity element disappears entirely, an undo pair collapses to the single symbol for the neutral element, a double undo collapses to its argument. No rule’s right-hand side can ever be rewritten back by another rule in the set, so applying all six on repeat has nowhere to loop, and is bound to stop.

Closing goals with autorewrite

Here is the whole GroupRewriteExamples section, verbatim:

Section GroupRewriteExamples.
  Context {G : Type} `{Group G}.

  (* The payoff case from the TODO: a double negation next to a right
     unit, closed by the database alone instead of a two-step calc. *)
  Example autorewrite_double_neg_idr : forall g : G, - (- g) + 0 == g.
  Proof. intros g. autorewrite with grp. reflexivity. Qed.

  (* A left unit, a double negation, and a right unit collapse in one
     `autorewrite`; written out as a `calc` chain this is three
     justified steps (idl, neg_involutive, idr already folded into
     idl's rewrite of the outer sum) — here it is one line. *)
  Example autorewrite_multi : forall g : G, (0 + - (- g)) + 0 == g.
  Proof. intros g. autorewrite with grp. reflexivity. Qed.

  (* Inverse pairs collapsing to 0, then 0 vanishing against 0 + 0. *)
  Example autorewrite_invl_invr : forall g : G,
    ((- g) + g) + (g + - g) == 0.
  Proof. intros g. autorewrite with grp. reflexivity. Qed.

End GroupRewriteExamples.

Take autorewrite_multi as the one to walk in full. In English: for any group element g, combining the neutral element with the undo of the undo of g, and then combining that with the neutral element again, gives back g. As a blackboard chain, using the left identity law, the right identity law, and double-undo cancellation:

(0+−(−g))+0=−(−g)+0{left identity}=−(−g){right identity}=g{double undo cancels}\begin{aligned} (0 + -(-g)) + 0 &= -(-g) + 0 && \{\text{left identity}\}\\ &= -(-g) && \{\text{right identity}\}\\ &= g && \{\text{double undo cancels}\} \end{aligned}

autorewrite with grp runs exactly this kind of pass: it scans (0 + - (- g)) + 0 for a left-hand side from the grp database, rewrites, and scans again — three of the six rules fire, in whichever order their patterns happen to match — until what is left is the bare variable g. The goal becomes g == g, and reflexivity closes it. autorewrite_double_neg_idr and autorewrite_invl_invr run the same mechanism on two different inputs; autorewrite_invl_invr is worth a second look because two collapses fire independently in the same pass — (-g) + g and g + (-g) each become the neutral element on their own, and the resulting “neutral element plus neutral element” then collapses again.

In practice, this is the tool to reach for whenever a goal is pure shrinking with no hypothesis that has to be placed at a particular point in the chain — exactly what none of the six rules above can do for you. Cancellation itself still needs a hand-written proof, several calc lines long, because it has to invoke a hypothesis at exactly the right spot in the middle of the reasoning; a goal like autorewrite_invl_invr, with nothing to invoke, needs no such attention and closes in one tactic call instead of a chain written out by hand.

ring and field, formalized

A ring adds a second combining operation, multiplication, to a group’s addition: multiplication is associative, has its own neutral element, and distributes over addition on both sides. A field is a ring where every element except the additive neutral element has a multiplicative undo, letting division make sense. Coq ships two tactics for these — ring for rings, field for fields — that answer the second question from the opening: both normalize an algebraic expression into one fixed form and compare, so both need to know, for a given carrier, exactly which named operations play the role of +, *, 0, 1. Registering a carrier with either tactic means handing it a value of a stdlib bundle type — ring_theory for ring, field_theory for field — packaging proofs of the laws each tactic relies on.

The tempting move is to build that bundle once, for the fully abstract ring and field structures, and have every concrete instance inherit it for free. What the bundle has to contain is the standard axiom list, on the blackboard:

0+x=x,x+y=y+x,(x+y)+z=x+(y+z),1⋅x=x,xy=yx,0 + x = x, \quad x + y = y + x, \quad (x+y)+z = x+(y+z), \quad 1 \cdot x = x, \quad xy = yx, (xy)z=x(yz),(x+y)z=xz+yz,x−y=x+(−y),x+(−x)=0.(xy)z = x(yz), \quad (x+y)\,z = xz + yz, \quad x - y = x + (-y), \quad x + (-x) = 0.

Theory/Ring.v gets partway there — each exact line below discharges one axiom from that list with the corresponding law the abstract Ring class already carries (the two symmetry; apply assoc lines only flip an associativity stated in the other direction):

  Definition ring_theory_of_Ring (Hcomm : forall a b : R, a * b == b * a) :
    ring_theory (0 : R) (1 : R) add mult (fun x y => x + - y) neg equ.
  Proof.
    constructor.
    - exact idl.
    - exact comm.
    - intros x y z; symmetry; apply assoc.
    - exact idl.
    - exact Hcomm.
    - intros x y z; symmetry; apply assoc.
    - exact distr.
    - reflexivity.
    - exact invr.
  Qed.

The argument Hcomm : forall a b : R, a * b == b * a is the first reason from the opening, made concrete: ring’s normal form is a sum of monomials with no fixed order to the factors inside a monomial, which is only meaningful if multiplication commutes. The codebase’s Ring class deliberately does not assume that in general — the comment at the top of Interface/Ring.v says, in its own words, that Ring’s

mult is not commutative in general (e.g. End V, Impl/HomVS.v), so there is deliberately no Commutes R equ mult instance here.

and there is a genuine example already in the library that needs this left open: the endomorphisms End V of a vector space (LinearMap F V V in Theory/HomVS.v), a ring whose multiplication is composition, (f * g) u = f (g u). Composing linear maps does not commute in general — applying f then g is a different map from applying g then f — so ring_theory_of_Ring genuinely cannot be discharged for End V without a hypothesis that is false there. This is exactly the “turn, then flip” case from the opening, occurring for real inside the library rather than as an illustration.

A field avoids that particular obstruction — commutative multiplication is one of the field laws itself, field_mul_comm — so Theory/Field.v builds its bundle with no extra hypothesis at all. On top of the ring axioms above, the field bundle adds three:

1≠0,x/y=x⋅y−1,y≠0⇒y−1⋅y=1,1 \neq 0, \qquad x / y = x \cdot y^{-1}, \qquad y \neq 0 \Rightarrow y^{-1} \cdot y = 1,

and the proof discharges the four obligations in that order — ring axioms first, then the three extras:

  Definition field_theory_of_Field :
    field_theory (0 : F) (1 : F) add mult (fun x y => x + - y) neg
                 (fun x y => x * inv y) inv equ.
  Proof.
    constructor.
    - exact (ring_theory_of_Ring field_mul_comm).
    - exact field_nontrivial.
    - reflexivity.
    - exact field_invl.
  Qed.

Yet even here, Add Field — the vernacular command that actually makes the field tactic usable — is not run at this point in the file. This is the second reason from the opening, the bookkeeping one, stated directly in the comment above ring_theory_of_Ring in Theory/Ring.v:

This is deliberately NOT registered with Add Ring here: that vernacular command keys its lookup table on the exact operation terms in scope at the point it runs, so a Section-local R would not survive Section End for use at a later, unrelated carrier — Add Ring/Add Field only work when run again at each concrete instance (Impl/Zr.v’s Z, Impl/Qf.v’s Q), reusing this record.

Add Ring/Add Field are not ordinary definitions exported from a section the way a lemma is. They register a lookup-table entry keyed on the literal operation symbols in scope at the moment the command runs — here, the abstract add/mult/neg/equ bound to a section-local placeholder R : Type. Once the section closes, that placeholder and its operations go out of scope, leaving nothing for the registration to key on even if the command were issued. There is no single command, run once inside the abstract theory file, that leaves ring/field usable afterward for whatever concrete carrier shows up later — the command has to be reissued at each one, against that carrier’s own concrete operations.

ring and field, at Z and Q

Impl/Zr.v reissues the registration for the integers:

Definition Z_ring_theory :
  ring_theory (0 : Z) (1 : Z) add mult (fun x y => x + - y) neg equ :=
  ring_theory_of_Ring (R := Z) Z.mul_comm.

Add Ring Z_ring : Z_ring_theory.

(* Payoff: a square expansion `ring` closes in one line, stated with
   the tower's own notation, where the manual route is a `distl`/`distr`
   calc chain of several steps (`Theory/Ring.v`'s `mul_add_expand`). *)
Example z_ring_tac_square : forall a b : Z,
  (a + b) * (a + b) == (a * a + a * b) + (b * a + b * b).
Proof. intros a b. ring. Qed.

In English: for any two integers, a + b squared equals (a·a + a·b) + (b·a + b·b) — an ordinary expansion of a product of sums:

(a+b)(a+b)=(a⋅a+a⋅b)+(b⋅a+b⋅b).(a+b)(a+b) \;=\; (a{\cdot}a + a{\cdot}b) + (b{\cdot}a + b{\cdot}b).

Z_ring_theory reuses ring_theory_of_Ring, supplying Z.mul_comm for the commutativity premise the abstract bundle demanded; Add Ring Z_ring runs the registration against Z’s own add/ mult/equ, at the concrete carrier, where the command has something to key on that stays in scope afterward; and with that done, ring alone — no distl, no distr, no named lemma — proves the expansion.

Impl/Qf.v does the same for the rationals, with field:

Definition Q_field_theory :
  field_theory (0 : Q) (1 : Q) add mult (fun x y => x + - y) neg
               (fun x y => x * inv y) inv equ :=
  field_theory_of_Field (F := Q).

Add Field Q_field : Q_field_theory.

(* Payoff: cancelling a division `field` closes in one line, stated
   with the tower's own notation, where the manual route is the
   `field_mul_div_cancel` calc chain (`Theory/Field.v`). *)
Example q_field_tac_cancel : forall a b : Q,
  b =/= 0 -> b * (a * inv b) == a.
Proof. intros a b Hb. field. exact Hb. Qed.

In English: for rationals a and b with b nonzero, multiplying by b and then dividing by b cancel:

b⋅(a⋅b−1)=a.b \cdot \left(a \cdot b^{-1}\right) \;=\; a.

Q_field_theory needs no extra hypothesis, since field_theory_of_Field was already unconditional for any field; Add Field Q_field registers it against Q’s own operations; and field proves the cancellation in one line, discharging the side condition b =/= 0 (which the hand-written route, field_mul_div_cancel in Theory/Field.v, also needs) with exact Hb.

So the grp database and the ring/field tactics end up at opposite ends of the same hierarchy. autorewrite with grp is registered once, on the fully abstract group, and every group inherits it — the six rules need no commutativity, and Hint Rewrite is not scoped to a section’s local placeholders. ring and field need a bundle that can, in principle, be built abstractly (with a commutativity hypothesis for ring, unconditionally for field), but the vernacular commands that make the tactics fire are keyed to concrete operation symbols and have to be reissued, by hand, at every carrier that wants to use them.
