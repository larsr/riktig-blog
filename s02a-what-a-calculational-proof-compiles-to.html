What a calculational proof compiles to: transitivity plus one law per step

A chain of expressions, glued by transitivity

There is a style of proof that every algebra student learns at the blackboard. To show that one expression equals another, you write a column of expressions, each obtained from the one above it by a single cited law, and you read the conclusion off the ends. Here is one, in a system with an addition, a zero, and a negation:

(a+b)+(−b)=a+(b+(−b))(associativity)=a+0(inverse)=a(identity) \begin{aligned} (a + b) + (-b) &= a + (b + (-b)) && \text{(associativity)} \\ &= a + 0 && \text{(inverse)} \\ &= a && \text{(identity)} \end{aligned}

Two kinds of reasoning hold this together. Each adjacent pair of lines is one step, justified by one law. And the steps are glued into a single claim about the two ends — (a+b)+(−b)=a(a+b)+(-b) = a — by transitivity: from x=yx = y and y=zy = z, conclude x=zx = z. A chain of three steps is two applications of transitivity; a chain of ten steps is nine. The format is so familiar that the gluing is invisible.

A careful reader should ask two questions before trusting this format.

First: what does “equal” mean? On the blackboard the symbol == is silently overloaded. When the objects are representations of mathematical values, distinct representations can stand for the same value: the fractions 12\tfrac{1}{2} and 24\tfrac{2}{4} are different pairs of integers, yet the same rational number. A chain of “equalities” between fractions is really a chain in some coarser relation — “represents the same number” — and the gluing is only valid because that relation is itself transitive. Nobody checks this at the blackboard. Somebody should.

Second: if a machine is to check such a chain, what exactly does it check? Presumably each step, against its cited law. But also: that the first and last lines really are the two sides of the claimed theorem, and not merely something that looks similar; that the steps have been glued with an actual transitivity principle for the actual relation in use; and that a law cited for a step really does close that step, in whichever direction it was meant. A proof format is only as good as the least-checked thing in it.

This post answers both questions concretely, for the chain notation used throughout this series. The answer to the second is pleasantly short: a chain is notation — a small, ordinary definition — and what the machine checks is transitivity over a declared equivalence relation, plus one rewrite or lemma application per step. The rest of the post shows exactly that, ending with the same theorem proved twice — once as a chain, once with the notation expanded away.

Equality is chosen, not built in

In Coq, the equality written = is syntactic: two expressions are = when they compute to identical values. That is too fine for representations. A rational number in Coq’s standard library is stored as a pair — numerator and denominator, written 1 # 2 for 12\tfrac{1}{2} — and the pairs 1 # 2 and 2 # 4 are simply different pairs. 1 # 2 = 2 # 4 is not provable, and should not be.

The remedy is to choose, for each type, the relation that deserves to be called its equality. For rationals the standard library chooses cross-multiplication: two fractions are equivalent when numerator of one times denominator of the other matches the opposite product. It names this relation Qeq and gives it the notation ==. Under that relation the two representations of one half really are equal, and the proof is a computation:

From Coq Require Import QArith.

Example half_eq_two_fourths : (1 # 2 == 2 # 4)%Q.
Proof. reflexivity. Qed.

The goal unfolds to 1⋅4=2⋅21 \cdot 4 = 2 \cdot 2; both sides compute to 44; reflexivity closes it. (This snippet, like every Coq snippet in this post, is compiled, not typeset by hand.)

A type packaged together with a chosen equivalence relation is called a setoid. What entitles a relation to be written like equality — to appear in a chain at all — is exactly the three properties of an equivalence relation: reflexivity (x == x), symmetry (from x == y, conclude y == x), and transitivity. Transitivity is the license to glue; without it the chain format is unsound for that relation. This is the precise answer to the first blackboard question: “equal” means the equivalence relation declared for this type, and the machine will demand to see that declaration.

The algebraic structures in this series make the same move. A recap of the two structures the earlier posts built: a monoid is a type — its carrier — with an addition +, a zero 0, an equality ==, and three laws (associativity and the two identity laws); a group is a monoid with a negation - and two inverse laws, from which cancellation on either side was proved. Here is the monoid class itself:

Class Monoid T := {
  #[global] mon_equ        :: EqRel T;
  #[global] mon_equiv      :: Equivalence mon_equ;
  #[global] mon_zero       :: Zero T;
  #[global] mon_add        :: HasAdd T;
  mon_assoc      : forall a b c : T, (a + b) + c == a + (b + c);
  mon_idl        : forall a : T, 0 + a == a;
  mon_idr        : forall a : T, a + 0 == a;
  #[global] mon_add_proper :: Proper (equ ==> equ ==> equ) (@add T _)
}.

The first four fields supply the vocabulary: an equality relation (EqRel), a proof that it is an equivalence (Equivalence), a zero, an addition — each a tiny class of its own. The field mon_equiv is the point of this section: every monoid carries, as part of its definition, the proof that its == is an equivalence relation. You cannot declare a monoid over a relation that isn’t one. The last field, mon_add_proper, says addition respects == — equal arguments give equal sums — which is what permits a law to be applied inside a larger expression: rewriting 00 to g+(−g)g + (-g) under an outer h + _ is replacement of equals by equals in context, and that needs the context to preserve the relation.

Where transitivity comes from

When a proof reaches a goal of the form R x z and wants to go through a midpoint y, Coq’s tactic transitivity y splits the goal into R x y and R y z. To do that it needs a proof that R is transitive — and it finds one by typeclass resolution.

A typeclass is a named shape of fact — Transitive R is one — and an instance is a registered fact of that shape, stored in a database. When some proof step needs “a proof of Transitive R, for this particular R”, Coq searches the database for an instance whose type matches, filling in unknowns as it goes. The programmer never names the instance; the type of the thing needed is the query.

The :: marks in the class above are registrations: each such field of every declared monoid instance goes into the database. Declaring a monoid therefore automatically registers Equivalence for its ==, and the standard library derives Transitive (and Reflexive, and Symmetric) from Equivalence. So by the time any chain over a monoid’s == is written, the transitivity that glues it is already on file, and transitivity y finds it without being told.

The same database search supplies the laws cited in the steps. The name idr in the proofs below is not a lemma about one particular type; it is the sole field of a two-line class of law-shaped facts:

Class Idr T (e : EqRel T) (op : BinOp T) unit :=
  idr : forall a, op a unit == a.

and every monoid registers its own right-identity law under it:

#[global] Instance Monoid_Idr T (M : Monoid T) : Idr T equ add zero := mon_idr.

So rewrite idr means: search the database for a right-identity law matching the operation and unit in the current goal, and rewrite with it. One mechanism — search the fact database by type — resolves both the glue of the chain and the justification of each step.

Why the chain names nothing but the terms

One thing will be conspicuous in the chains below: the carrier and the relation are never written. The chain mentions only the expressions themselves. This is the work of implicit arguments: arguments that Coq fills in by unification — matching two expressions against each other and solving for the unknowns — instead of requiring them to be written.

The transitivity principle behind transitivity y is, spelled out, a statement with five arguments: a carrier A, a relation R, a proof that R is transitive, and the three points x, y, z. When the goal is h == - g, four of those are read straight off the goal: the carrier is the type of h, the relation is == at that type, and x and z are the two sides. The transitivity proof comes from the database, as above. The only thing the goal cannot determine is the midpoint y — and that is precisely the one thing the chain writes down. A calculational chain is, in this exact sense, the minimal information content of the proof: the intermediate expressions and the laws, nothing else.

The whole implementation

Now the notation itself. It is defined in one ordinary source file, and its two working parts are each a few lines of Ltac, Coq’s language for proof-script procedures (a tactic is such a procedure; a goal is a statement it is asked to prove). The first part checks the ends of the chain:

Ltac calc_ends x y :=
  lazymatch goal with
  | |- _ ?l ?r => unify l x; unify r y
  end.

lazymatch goal inspects the current goal; the pattern |- _ ?l ?r matches any goal that is some relation applied to two arguments, binding the left one to l and the right one to r. Then unify l x; unify r y demands that the written first term of the chain is the goal’s left side and the written last term is its right side. This is the answer to part of the second blackboard question: the endpoints you write are not trusted, and they are not decoration — they are checked against the theorem statement, and a chain whose ends have drifted from the goal fails on its first token.

The second part discharges one justified step:

Ltac calc_by r :=
  first [ now rewrite r
        | now rewrite <- r
        | now apply r
        | now (symmetry; apply r) ].

first [ ... ] tries the alternatives in order and keeps the first that succeeds; now t runs t and then requires the goal to be completely closed by trivial reasoning (reflexivity, an assumption), undoing everything otherwise. The four alternatives are the four ways one law can justify one step: rewrite with it left to right somewhere in the step; rewrite with it right to left (<-); the step is the law, up to instantiation (apply); the step is the law read backwards (symmetry; apply). This is why the chains never annotate a direction: a law such as idr states a + 0 == a, and whether a given step uses it to erase a + 0 or to introduce one is discovered here, by trying.

Everything else in the file is surface syntax. Tactic Notation declares a new spelling for a tactic, one declaration per chain length; here is the three-step justified form, in full:

Tactic Notation "calc" constr(x0)
    "==" constr(x1) "{" uconstr(r1) "}"
    "==" constr(x2) "{" uconstr(r2) "}"
    "==" constr(x3) "{" uconstr(r3) "}" :=
  calc_ends x0 x3;
  transitivity x1; [ calc_by r1 |
  transitivity x2; [ calc_by r2 | calc_by r3 ] ].

Read the header first: constr(x0) declares a slot parsed as a term — the chain’s expressions — and uconstr(r1) declares a slot parsed as an untyped term, whose elaboration (resolving notations, types, and implicit arguments) is delayed until the moment it is used. The delay matters because a bare law name like idr does not determine its own carrier; only the goal of its step does, and by the time calc_by r1 runs, that goal exists.

Now the body, which is the entire semantics of a chain: check the written endpoints against the goal (calc_ends x0 x3); split at the first midpoint (transitivity x1), producing two goals dispatched by the bracket [ ... | ... ]; close the first step with its law (calc_by r1); split the remainder at the second midpoint; close the two remaining steps. The file continues the same pattern up to chains of eight terms — a page of near-identical declarations, no recursion, no cleverness. There is no dedicated proof language here and nothing resembling one: a chain of nn steps compiles to n−1n-1 uses of transitivity and nn uses of calc_by.

The same group theorem, with and without the notation

Here is a real chain from the group-theory file of this series, chosen because every law it cites is one the earlier posts introduced. The theorem: in any group, an element that cancels gg on the left is the inverse of gg — if h+g=0h + g = 0 then h=−gh = -g. This is a uniqueness statement: nothing besides −g-g can do −g-g’s job.

On the blackboard, writing == for the group’s equality:

h=h+0(right identity)=h+(g+(−g))(right inverse)=(h+g)+(−g)(associativity)=0+(−g)(the hypothesis h+g=0)=−g(left identity) \begin{aligned} h &= h + 0 && \text{(right identity)} \\ &= h + (g + (-g)) && \text{(right inverse)} \\ &= (h + g) + (-g) && \text{(associativity)} \\ &= 0 + (-g) && \text{(the hypothesis } h + g = 0\text{)} \\ &= -g && \text{(left identity)} \end{aligned}

In Coq, inside a section that fixes a group — Context {G : Type} `{Group G}. — the proof is that chain, transcribed:

  Theorem neg_unique_l : forall g h : G, h + g == 0 -> h == - g.
  Proof.
    intros g h Heq.
    calc h
      == (h + 0)          { idr  }
      == (h + (g + - g))  { invr }
      == ((h + g) + - g)  { assoc }
      == (0 + - g)        { Heq  }
      == (- g)            { idl  }.
  Qed.

Walking it: intros brings the group elements and the hypothesis Heq : h + g == 0 into scope. calc first unifies the written endpoints h and (- g) against the goal h == - g — they match — then splits the goal into five steps, one per line, and closes each with calc_by on the cited law. Step by step, naming which of calc_by’s four attempts succeeds:

  • h == h + 0: rewriting with idr (a + 0 == a) erases the + 0 on the right; both sides are now h; first attempt.
  • h + 0 == h + (g + - g): rewriting with invr (a + - a == 0) collapses g + - g to 0 on the right — inside the sum, which is what mon_add_proper licenses; first attempt again. Note the difference in reading: the blackboard introduced g+(−g)g + (-g), using the inverse law from right to left, while the machine used the law forward, erasing the pair until both sides matched. The writer said neither; calc_by tried and found what works.
  • h + (g + - g) == (h + g) + - g: rewriting with assoc reassociates the right side to match the left.
  • (h + g) + - g == 0 + - g: rewriting with Heq — a step’s brace can cite a local hypothesis just as well as a registered law — replaces h + g by 0.
  • 0 + - g == - g: rewriting with idl erases the 0 +.

And now the distinctive exercise: the same theorem with the notation expanded away. This is not a sketch of what the tactic “morally” does — it is a proof that compiles as written, in the same section, and its shape is exactly the body of the five-step Tactic Notation:

  Theorem neg_unique_l_expanded : forall g h : G, h + g == 0 -> h == - g.
  Proof.
    intros g h Heq.
    transitivity (h + 0);           [ now rewrite idr  |
    transitivity (h + (g + - g));   [ now rewrite invr |
    transitivity ((h + g) + - g);   [ now rewrite assoc |
    transitivity (0 + - g);         [ now rewrite Heq  |
                                      now rewrite idl  ] ] ] ].
  Qed.

Each transitivity names one midpoint and leaves two goals; each bracket sends the left goal to its one-law rewrite and the right goal into the next split. Two things changed in the expansion, and both are instructive. The rewrite directions are now written by hand — in this chain all five laws happened to close their steps with a forward rewrite, but had one needed the other direction, this proof would have to say rewrite <- where the chain said nothing. And the endpoints have disappeared: transitivity only ever mentions midpoints, so h and - g occur nowhere in the expanded script. The chain form restates them purely for the reader — and calc_ends is what keeps that restatement from silently drifting away from the theorem it decorates.

That is the entire content of the notation: what the machine checked, in both proofs, is five setoid-equality steps and four transitivities, with the equivalence relation and its transitivity instance found by searching the typeclass database.

The three forms, and how to write a chain that parses

The chain notation comes in three forms, uniform within one chain.

Justified — the form used above: each step carries, in braces, the one law that proves it, direction found automatically. This is the default, and the one to reach for whenever every step really is one law. The one-law restriction is deliberate: a step that needs two laws is two steps — write the intermediate expression. A step that needs a compound tactic does not belong in this form at all.

Plain — the chain lists only the expressions, and the step obligations follow as bullets, in order, each proved by any tactic whatever:

  Example chain_plain : forall a : G, 0 + (a + 0) == a.
  Proof.
    intros a.
    calc (0 + (a + 0)) == (a + 0) == a.
    - now rewrite idl.
    - now rewrite idr.
  Qed.

This form fits when some step needs more than a one-law rewrite: the chain still documents the route, and each bullet may be as long as it must be.

Named — each step is given a name in brackets, becoming a named goal proved afterwards with the standard [name]: selector, in any order:

  Example chain_named : forall a : G, 0 + (a + 0) == a.
  Proof.
    intros a.
    calc (0 + (a + 0)) ==[left_unit] (a + 0) ==[right_unit] a.
    [right_unit]: now rewrite idr.
    [left_unit]:  now rewrite idl.
  Qed.

This form fits long chains whose steps deserve documentation, or proofs where it reads better to dispose of the routine steps first and save the interesting one for last — the names decouple the order of proving from the order of the chain.

Three practical notes on writing a chain that parses, from the implementation file’s own header. First, each compound expression in a chain needs its own parentheses: every slot is parsed as a full term, so an unparenthesized == would be swallowed into the expression before it. Second, an expression that nothing else pins down needs a type annotation — 0 alone could be the zero of any type, so a chain step consisting of just the zero is written (0 : G). Third, as above: one law per brace, one brace per step; if a step wants two laws, it is two steps, and if it wants a compound tactic, use the plain or named form. Chains run from three to eight terms; a longer calculation is two chains through an intermediate lemma, which is usually a sign the intermediate fact deserved a name anyway.
