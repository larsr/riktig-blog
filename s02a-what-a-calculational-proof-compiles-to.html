What a calculational proof compiles to: transitivity plus one law per step

A chain of expressions, glued by transitivity

There is a style of proof every algebra student learns at the blackboard: a column of expressions, each obtained from the one above it by one cited law, with the conclusion read off the two ends. Here is one, in a system with an addition, a zero, and a negation:

(a+b)+(−b)=a+(b+(−b))(associativity)=a+0(inverse)=a(identity) \begin{aligned} (a + b) + (-b) &= a + (b + (-b)) && \text{(associativity)} \\ &= a + 0 && \text{(inverse)} \\ &= a && \text{(identity)} \end{aligned}

Each adjacent pair of lines is one step, justified by one law. The steps are glued into a single claim about the two ends, (a+b)+(−b)=a(a+b)+(-b)=a, by transitivity: from x=yx=y and y=zy=z, conclude x=zx=z — nn lines is n−1n-1 uses of transitivity, so familiar the gluing is invisible.

Two things go unchecked at the blackboard: what “==” means — distinct representations of a value can be equal without being identical, as 12\tfrac12 and 24\tfrac24 are different pairs of integers but the same rational number, so a chain of “equalities” between representations is really a chain in some coarser relation, glued validly only because that relation is itself transitive — and what checking the chain requires: that each step follows from its cited law, that the endpoints really are the theorem’s two sides, and that the gluing uses an actual transitivity proof for the relation in play.

This post shows exactly what gets checked, for the chain notation used throughout this series: a chain is an ordinary piece of notation, and what it expands to is a sequence of transitivity calls plus one proof obligation per step.

Choosing what “equal” means

Coq’s built-in = is syntactic: two expressions are = when they compute to identical values, too fine for representations. Coq’s rationals are pairs — numerator and denominator, 1 # 2 for 12\tfrac12 — and 1 # 2 and 2 # 4 are different pairs, so 1 # 2 = 2 # 4 does not hold. The remedy is to pick, for each type, the relation that deserves to be called its equality: rationals use cross-multiplication (Qeq, notation ==), under which the two representations of one half are equal.

A type paired with a chosen equivalence relation is a setoid. What licenses a relation to be chained like equality is exactly the three properties of an equivalence relation — reflexivity, symmetry, and transitivity, the last one licensing the gluing.

A monoid, recapping the earlier post, is a carrier type with an addition, a zero, an equality, and three laws: associativity and the two identity laws:

Class Monoid T := {
  #[global] mon_equ        :: EqRel T;
  #[global] mon_equiv      :: Equivalence mon_equ;
  #[global] mon_zero       :: Zero T;
  #[global] mon_add        :: HasAdd T;
  mon_assoc      : forall a b c : T, (a + b) + c == a + (b + c);
  mon_idl        : forall a : T, 0 + a == a;
  mon_idr        : forall a : T, a + 0 == a;
  #[global] mon_add_proper :: Proper (equ ==> equ ==> equ) (@add T _)
}.

mon_equiv is the field that matters: every monoid carries, as part of its definition, a proof that its == is an equivalence relation — a monoid cannot be declared over a relation that isn’t one. mon_add_proper says addition respects ==, which is what lets a law rewrite an expression buried inside a larger one, such as replacing g + (-g) by 0 under an outer h + _.

Coq puts mon_equiv to work without being asked. A typeclass declares a family of facts indexed by a type (or a relation); an instance registers one such fact in a database that Coq searches whenever a proof step needs a fact of that family — nothing is ever addressed by name. The :: marks above are such registrations, so every monoid instance registers Equivalence for its ==, and the standard library derives Transitive from Equivalence automatically. That is why transitivity y — which splits a goal R x z into R x y and R y z — already finds its proof unassisted: it searches the database for Transitive at the relation in the goal. The same search resolves each step’s cited law: idr, used below, is the sole field of a two-line class shaped op a unit == a, and every monoid registers its own right-identity law under that name, so rewrite idr finds whichever instance matches the goal’s operation and unit.

The chain notation, in full

The notation lives in one file; its two working parts are each a few lines of Ltac, Coq’s language for proof-script procedures. The first checks the chain’s endpoints against the goal:

Ltac calc_ends x y :=
  lazymatch goal with
  | |- _ ?l ?r => unify l x; unify r y
  end.

|- _ ?l ?r matches a goal that is some relation applied to two arguments; unify demands that the written first and last terms of the chain equal the goal’s own two sides — a chain whose ends drifted from the theorem fails on its first token.

The second discharges one step:

Ltac calc_by r :=
  first [ now rewrite r
        | now rewrite <- r
        | now apply r
        | now (symmetry; apply r) ].

first [...] tries each alternative and keeps the one that closes the goal completely (now): rewrite the law forward, rewrite it backward, apply it, or apply it symmetrically — a chain never states a direction because whichever of the four works is however the step is justified.

Everything else is surface syntax, one declaration per chain length. Here is the three-step form, in full:

Tactic Notation "calc" constr(x0)
    "==" constr(x1) "{" uconstr(r1) "}"
    "==" constr(x2) "{" uconstr(r2) "}"
    "==" constr(x3) "{" uconstr(r3) "}" :=
  calc_ends x0 x3;
  transitivity x1; [ calc_by r1 |
  transitivity x2; [ calc_by r2 | calc_by r3 ] ].

Read the body in order: check the endpoints against the goal, split at the first midpoint, close that step with its law, split the rest at the second midpoint, close the two remaining steps. That is the entire semantics of a chain — nn terms compile to n−1n-1 uses of transitivity and calc_by apiece. The file repeats the identical pattern up to eight terms: no recursion, nothing clever, one declaration per length.

A real group theorem, blackboard and Coq

Here is a theorem from the group-theory file, chosen because every law it cites was introduced earlier in this series: in any group, an element that cancels gg on the left is the inverse of gg — if h+g=0h + g = 0 then h=−gh = -g.

h=h+0(right identity)=h+(g+(−g))(right inverse)=(h+g)+(−g)(associativity)=0+(−g)(the hypothesis h+g=0)=−g(left identity) \begin{aligned} h &= h + 0 && \text{(right identity)} \\ &= h + (g + (-g)) && \text{(right inverse)} \\ &= (h + g) + (-g) && \text{(associativity)} \\ &= 0 + (-g) && \text{(the hypothesis } h + g = 0\text{)} \\ &= -g && \text{(left identity)} \end{aligned} 

  Theorem neg_unique_l : forall g h : G, h + g == 0 -> h == - g.
  Proof.
    intros g h Heq.
    calc h
      == (h + 0)          { idr  }
      == (h + (g + - g))  { invr }
      == ((h + g) + - g)  { assoc }
      == (0 + - g)        { Heq  }
      == (- g)            { idl  }.
  Qed.

intros brings in the group elements and Heq : h + g == 0. calc checks h and (- g) against the goal’s sides, then closes each step with calc_by on its cited law. idr erases the + 0. invr collapses g + - g to 0, applied backward — the blackboard introduces the pair, calc_by finds the same law reads the other way. assoc reassociates. Heq replaces h + g by 0, a brace citing a local hypothesis exactly like a registered law. idl erases the leading 0 +.

This six-term chain compiles the same way, one nesting level deeper: five transitivity splits and five calc_by calls, an ordinary Coq proof term that Qed accepts — the chain notation only spells it more legibly.

Three ways to write a chain, and where else it applies

neg_unique_l used the justified form: one law per step, in braces, direction found automatically. Two siblings share the skeleton. The plain form drops the braces and proves each step afterward as its own bullet, in order — for a step needing more than a one-law rewrite. The named form gives each step a bracketed name instead, proved afterward with Coq’s [name]: selector in any order. All three compile to the same transitivity/calc_by skeleton; only the bookkeeping around each step differs.

The mechanism works over any type with a registered equivalence relation, not only the abstract carrier G above — rationals are one instance, 1#1 and 2#2 being different pairs equal only under Qeq:

Example q_dec_eq_check : {(1#1)%Q == (2#2)%Q} + {(1#1)%Q =/= (2#2)%Q} :=
  dec_eq (1#1)%Q (2#2)%Q.
