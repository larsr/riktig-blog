Merge and merge sort: a sorted permutation of the input

To sort is to do two things at once: rearrange a list so it comes out in order, and rearrange only — losing no element and inventing none. Miss the first and you have not sorted; miss the second and you have not even kept the data. A sorting proof therefore has two halves, and merge sort lets us prove each half where it belongs: the “in order” half lives in the merge step, which combines two already-sorted lists into one, and the “no element lost” half is a permutation fact that holds regardless of any order at all. This post builds the merge as a single lambda term, proves both halves for it, and then assembles merge sort on top.

The order the elements are compared against is kept abstract. A decidable total order on a type A is a strict comparison olt (written <) together with its reflexive companion ole (written <=) such that any two elements are comparable — one way or the other always holds — and, crucially, the comparison is decidable: for any x and y you can compute whether x < y. That last property is what lets a program branch on the order. The merge here is written once, over any type carrying this structure, and specialises to nat (with < and <=) or to anything else that provides it.

Merge as one term

merge walks two lists, at each step emitting whichever head is smaller and recursing on the rest:

merge(x::a,y::b)={x::merge(a,y::b)if x<y,y::merge(x::a,b)otherwise,\mathrm{merge}(x{::}a,\ y{::}b) \;=\; \begin{cases} x :: \mathrm{merge}(a,\ y{::}b) & \text{if } x < y,\\ y :: \mathrm{merge}(x{::}a,\ b) & \text{otherwise,} \end{cases}

and merge(a,[])=a\mathrm{merge}(a, [\,]) = a, merge([],b)=b\mathrm{merge}([\,], b) = b. Neither list is a structural sub-term across the recursive call — when it keeps the left head it recurses on (a', y :: b'), rebuilding the right list — so, as with Euclid’s algorithm, the recursion is justified by a measure: the sum of the two lengths, which drops by one at every call. From Programs/Merge.v:

Section Merge.
  Context {A : Type} `{O : DecTotalOrder A}.
  Local Open Scope order_scope.

  Definition merge : list A -> list A -> list A.
   apply (uncurry ∘ (Fix (well_founded_ltof (list A * list A)
                (fun p => length (fst p) + length (snd p)))
             (fun _ => list A))).
    refine
    (
      fun p merge =>
        match p as x return p=x -> list A with
        | (a, []) => fun _ => a
        | ([], b) => fun _ => b
        | (a0::a', b0::b') =>
            fun e =>
            if dec (a0 < b0)
              then a0 :: merge (a', b0 :: b') _
              else b0 :: merge (a0 :: a', b') _
        end eq_refl
    );
    (subst p; unfold ltof; simpl; abstract lia).
   Defined.

The opening Context line, with its DecTotalOrder A binder, is what makes the program generic: everything below is parameterised by an arbitrary carrier A with a decidable total order O, and Local Open Scope order_scope makes < mean that order’s olt. The Fix recurses on a tupled argument p : list A * list A under the measure length (fst p) + length (snd p); uncurry ∘ restores the two-argument shape. If either list is empty the answer is the other. Otherwise if dec (a0 < b0) decides the comparison — this is the only place the order touches the program, and it uses just decidability, no order laws — and the smaller head is emitted. The two holes are the measure-decrease proofs, closed uniformly by abstract lia.

The defining equations — the displayed case rule above, one lemma per clause — follow through Fix_eq:

  Lemma merge_nil_l: forall l : list A, merge [] l = l.
    now destruct l. Qed.
  Lemma merge_nil_r: forall l : list A, merge l [] = l.
    now destruct l. Qed.
  Lemma merge_cons: forall (a : A) (b : list A) (c : A) (d : list A),
      merge (a::b) (c::d) = if dec (a < c) then a::merge b (c::d) else c::merge (a::b) d.
    intros; unfold merge at 1, compose, uncurry.
    rewrite Fix_eq.
    - destruct dec; f_equal.
    - intros [[ | x x'] [ | y y']] f g H; auto.
      destruct dec; rewrite !H; f_equal.
   Qed.

Every later proof rewrites with merge_nil_l, merge_nil_r, and merge_cons, never with the Fix itself. And the program computes, at nat:

Check Logic.eq_refl : merge [1;3;5] [2;4;6] = [1;2;3;4;5;6].
Check Logic.eq_refl : merge [1;2] [] = [1;2].
Check Logic.eq_refl : merge [2;4;6;8] [1;3] = [1;2;3;4;6;8].
Check Logic.eq_refl : merge [1;1;2] [1;3] = [1;1;1;2;3].

Half one: merging loses nothing

The permutation fact says a ++ b — the two inputs simply concatenated — is a rearrangement of merge a b. It needs no order laws whatever: merge only ever moves an element from an input to the output, so whichever way each comparison falls, the multiset of elements is preserved. On the blackboard, writing ∼\sim for “is a permutation of”, the case where the left head wins is a short chain:

(x::a)++(y::b)=x::(a++(y::b)){definition of ++}∼x::merge(a,y::b){induction hypothesis, under the shared head x}=merge(x::a,y::b){the clause equation, in the branch x<y.} \begin{aligned} (x{::}a) \mathbin{+\!\!+} (y{::}b) &= x :: (a \mathbin{+\!\!+} (y{::}b)) && \{\text{definition of } \mathbin{+\!\!+}\}\\ &\sim x :: \mathrm{merge}(a,\ y{::}b) && \{\text{induction hypothesis, under the shared head } x\}\\ &= \mathrm{merge}(x{::}a,\ y{::}b) && \{\text{the clause equation, in the branch } x < y.\} \end{aligned}

Here is the proof from Theory/Merge.v:

  Lemma merge_perm: forall a b : list A, Permutation (a ++ b) (merge a b).
    apply (measure_ind2 (fun (a b : list A) => length a + length b)).
    intros [|x a] [|y b] IH.
    - reflexivity.
    - reflexivity.
    - now rewrite app_nil_r.
    - destruct (dec (olt x y)) as [L|L].
      + calc ((x::a) ++ (y::b))
          == (x :: (a ++ (y::b)))
          == (x :: merge a (y::b))
          == (merge (x::a) (y::b)).
        * reflexivity.
        * apply perm_skip, IH; simpl; lia.
        * rewrite merge_cons; destruct (dec (olt x y)); [reflexivity | contradiction].
      + calc ((x::a) ++ (y::b))
          == (y :: ((x::a) ++ b))
          == (y :: merge (x::a) b)
          == (merge (x::a) (y::b)).
        * apply Permutation_sym, Permutation_middle.
        * apply perm_skip, IH; simpl; lia.
        * rewrite merge_cons; destruct (dec (olt x y)); [contradiction | reflexivity].
   Qed.

Walking it:

  • measure_ind2 gives strong induction on the length sum, so IH applies at any pair of lists with strictly smaller total length. The intros [|x a] [|y b] splits both lists into empty/non-empty; the first three bullets are the cases where at least one list is empty, closed by reflexivity or app_nil_r since merge returns the other list untouched.
  • The last bullet is the interesting one, destruct (dec (olt x y)) splitting on the comparison exactly as the program does. Each branch is a calc chain — the machine-checked blackboard — of three permutation steps. In the x < y branch: the first step is app’s definition (reflexivity); the second is perm_skip (prepend the same x) applied to IH, whose length side-condition falls to lia; the third rewrites merge_cons and picks the matching branch, where the leftover dec agrees with L so the wrong side is closed by contradiction.
  • The y <= x branch is the mirror image, opening instead with Permutation_middle, the fact that y :: (l ++ b) is a permutation of l ++ (y :: b).

Half two: merging sorted inputs stays sorted

Now the order laws are needed. A list is Sorted ole when each element is <= the next; the standard-library Sorted predicate is built from a head-relation HdRel ole x l, “x is <= the head of l” (vacuously true for the empty list). The one fact merge needs is that a bound below both inputs bounds the merge, because the head of merge a b is a head of a or of b. Writing x⊑lx \sqsubseteq l for “xx is below the head of ll, or ll is empty”:

x⊑a∧x⊑b⟹x⊑merge(a,b).x \sqsubseteq a \;\wedge\; x \sqsubseteq b \;\implies\; x \sqsubseteq \mathrm{merge}(a, b).

  Lemma merge_hdrel: forall (x : A) a b,
      HdRel ole x a -> HdRel ole x b -> HdRel ole x (merge a b).
    intros x [|p a] [|q b] HA HB;
      rewrite ?merge_nil_l, ?merge_nil_r, ?merge_cons; auto.
    destruct (dec (olt p q)); constructor; [now inversion HA | now inversion HB].
   Qed.

The empty cases rewrite merge away and finish by auto; when both lists are non-empty, whichever head merge_cons selects, x bounds it by inverting the corresponding hypothesis. With that, sortedness —

sorted(a)∧sorted(b)⟹sorted(merge(a,b))\mathrm{sorted}(a) \;\wedge\; \mathrm{sorted}(b) \;\implies\; \mathrm{sorted}(\mathrm{merge}(a, b))

— is an induction on the length sum:

  Lemma merge_sorted: forall a b : list A,
      Sorted ole a -> Sorted ole b -> Sorted ole (merge a b).
   Proof.
    apply (measure_ind2 (fun (a b : list A) => length a + length b)
             (fun a b => Sorted ole a -> Sorted ole b -> Sorted ole (merge a b))).
    intros [|x a] [|y b] IH SA SB.
    - constructor.
    - now rewrite merge_nil_l.
    - now rewrite merge_nil_r.
    - rewrite merge_cons; inversion_clear SA; inversion_clear SB;
        destruct (dec (olt x y)); constructor;
        eauto using merge_hdrel, HdRel_cons, olt_ole, nolt_ole, Sorted_cons;
        apply IH; simpl; eauto using Sorted_cons; lia.
     Qed.

Walking it:

  • Same measure_ind2 on the length sum, now carrying the two sortedness hypotheses SA, SB. The three degenerate cases are constructor (the empty list is sorted) or a rewrite of merge to the other list.
  • The last case rewrites merge_cons, then inversion_clear peels each input into “the tail is sorted” and “the head bounds the tail.” The destruct (dec (olt x y)) follows the program’s branch, and constructor reduces “merge is sorted” to two obligations: the tail is sorted (closed by apply IH at the strictly shorter arguments, lia for the measure) and the emitted head bounds that tail (closed by merge_hdrel). The lemmas olt_ole and nolt_ole bridge the strict comparison the program tested and the <= that Sorted wants: x < y gives x <= y, and not (x < y) gives y <= x by totality.

Merge does not commute on ties

You might expect merge a b = merge b a. It is not true as a plain equation. On a tie — two equal elements, one from each list — both orderings take the head of their own second list first, and the two outputs differ. The file records the counterexample by computation:

Check Logic.eq_refl : merge [1] [1;0] = [1;0;1].
Check Logic.eq_refl : merge [1;0] [1] = [1;1;0].

Swapping the arguments genuinely changes the result. What is true is that on already-sorted inputs merge commutes —

sorted(a)∧sorted(b)⟹merge(a,b)=merge(b,a)\mathrm{sorted}(a) \;\wedge\; \mathrm{sorted}(b) \;\implies\; \mathrm{merge}(a, b) = \mathrm{merge}(b, a)

— and the reason is exactly the two halves proved above: both merge a b and merge b a are sorted (half two) and both are permutations of the same elements (half one), and a sorted list is determined by its multiset, so they must be equal. That is merge_comm:

  Lemma merge_comm: forall a b : list A,
      Sorted ole a -> Sorted ole b -> merge a b = merge b a.
    intros; apply sorted_perm_unique;
      eauto using merge_sorted, merge_comm_perm.
   Qed.

sorted_perm_unique is the “sorted list determined by its multiset” fact; the two eauto hints supply the sortedness and the permutation. The tie is why the hypothesis Sorted cannot be dropped.

Merge sort

Merge sort splits a list into two halves, sorts each, and merges the results. In Programs/MergeSort.v, which works over list nat, the split recurses structurally (it consumes two elements at a time), so it is a plain Fixpoint; the merge is not redefined at all but is the generic merge above instantiated at nat’s order, with its two arguments swapped so that ties keep the original left-biased convention (Definition merge (l1 l2 : list nat) : list nat := Riktig.Programs.Merge.merge (O := Nat_DecTotalOrder) l2 l1.). The recursion that needs a measure is msort itself, because it recurses on the outputs of split, which Coq cannot see are shorter than the input. The measure is list length:

Definition msort : list nat -> list nat.
apply (Fix (well_founded_ltof (list nat) (@length nat)) (fun _ => list nat)).
intros l rec.
(* Term-mode `match l with ...` (as used by `part`/`qsort_aux` in
   Quicksort.v) does not propagate the pattern `x :: y :: rest` back
   into `rec`'s type, since the Fix body here doesn't depend on `l`
   pointwise; `destruct` (tactic-mode) does perform that
   substitution, so the two termination holes below see the goal in
   exactly `split_length_lt`'s shape. *)
destruct l as [|x [|y rest]].
- exact [].
- exact [x].
- refine (merge (rec (fst (split (x :: y :: rest))) _)
                (rec (snd (split (x :: y :: rest))) _));
  abstract (unfold ltof; apply (split_length_lt x y rest)).
Defined.

Lists of length zero or one are already sorted and returned as-is. For two or more elements, split produces two halves, rec sorts each — the two holes being the proofs that each half is strictly shorter, closed by split_length_lt — and merge combines them. As usual a msort_step lemma derives the defining equation through Fix_eq, and Theory/MergeSort.v refines it into one match-free corollary per branch (msort_nil, msort_single, msort_cons2) so the proofs never expose a residual match. And it computes:

Check Logic.eq_refl : msort [5;3;8;1;9;2;7] = [1;2;3;5;7;8;9].
Check Logic.eq_refl : msort [2;1] = [1;2].

Correctness is both halves together — for every list ll, the output is a rearrangement of the input and it is in order:

l∼msort(l)andsorted(msort(l)).l \;\sim\; \mathrm{msort}(l) \qquad\text{and}\qquad \mathrm{sorted}(\mathrm{msort}(l)).

Corollary msort_correct : forall l : list nat,
  Permutation l (msort l) /\ Sorted le (msort l).
Proof. intros l. split; [apply msort_perm | apply msort_sorted]. Qed.

The corollary is a one-liner because the substance is in its two conjuncts. I will walk the sortedness half in full, msort_sorted, and state the permutation half; both are strong inductions on length with the same shape.

Theorem msort_sorted : forall l : list nat, Sorted le (msort l).
Proof.
  induction l as [l IH]
    using (well_founded_ind (well_founded_ltof (list nat) (@length nat))).
  destruct l as [|x [|y rest]].
  - rewrite msort_nil. constructor.
  - rewrite msort_single. constructor; constructor.
  - pose proof (split_length_lt x y rest) as [H1 H2].
    rewrite msort_cons2.
    apply merge_sorted; apply IH; [exact H1 | exact H2].
Qed.

Walking it:

  • well_founded_ind on length gives an induction hypothesis IH usable at any strictly shorter list. destruct l splits into the three shapes msort itself distinguishes.
  • Empty: rewrite msort_nil then constructor — the empty list is Sorted. Singleton: rewrite msort_single, and two constructors build Sorted le [x] from “the tail is sorted” and “the head bounds the empty tail.”
  • The main case rewrites msort_cons2, exposing merge of the two recursively sorted halves. split_length_lt gives the two “shorter” facts H1, H2; apply merge_sorted reduces the goal to “each half is sorted,” and apply IH closes both, using H1/H2 for the measure. This is where the earlier merge_sorted is spent: merging two sorted halves is what keeps the whole result sorted.

The permutation half is msort_perm : forall l, Permutation l (msort l), proved on the same skeleton: in the two-or-more case it chains split_perm (the input is a permutation of its two halves concatenated) with the induction hypothesis on each half and finally merge_perm (merging is a permutation of the concatenation).
