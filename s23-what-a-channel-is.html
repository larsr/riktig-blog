What a channel is: relations, not probabilities

When a message travels from a sender to a receiver, things can go wrong. A message can be lost. It can arrive twice. A batch of messages can turn up out of order. The usual way to describe such a channel attaches a probability to each mishap: this link drops one message in a thousand, that one reorders with some small chance. Probabilities are the right tool when you want to predict how often a channel misbehaves.

But there is an earlier, coarser question that has nothing to do with frequencies: can this channel lose a message at all? Can it duplicate? Can it reorder? A channel that never reorders is a genuinely different object from one that sometimes does, however rarely — and most things you want to prove about a protocol (that it never delivers a corrupted result, that an acknowledgement scheme is safe) depend only on which mishaps are possible, not on how likely they are. This is the possibilistic view: forget the probabilities and keep only the set of behaviours a channel is allowed to exhibit.

That set is a relation. Write down the finite list of messages the sender put in — the sent trace — and the finite list the receiver got out — the received trace. A channel is exactly the relation that says which received traces are possible given a sent trace. A perfect wire relates each sent trace to itself and nothing else. A lossy wire relates a sent trace to every trace you could get by dropping some of its messages. No numbers anywhere; just sent and received, and a yes-or-no verdict on whether that pair could happen. This post builds five such relations, arranges them by how permissive they are, and then proves something concrete that follows purely from knowing which one you are dealing with.

Traces are plain lists

A trace is a finite list of messages. The messages come from some type M with no structure at all: a message is only ever compared for being the same message or a different one, so ordinary list equality is all we need — no algebra, no notion of “close enough.” Every channel relation below takes its two arguments in the order (sent, received).

Five fault classes

The simplest channel delivers exactly what was sent:

Definition perfect s r : Prop := r = s.

A lossy channel may drop messages but keeps the survivors in order. “Received is what is left after dropping some sent messages” is precisely the subsequence relation, written subseq r s (“r is a subsequence of s”):

Definition lossy s r : Prop := subseq r s.

A duplicating channel never drops anything but may repeat a message it just delivered. It is given inductively — three ways to build a valid (sent, received) pair:

Inductive dup : list M -> list M -> Prop :=
  | dup_nil : dup [] []
  | dup_keep : forall x s r, dup s r -> dup (x :: s) (x :: r)
  | dup_stutter : forall x s r, dup s (x :: r) -> dup s (x :: x :: r).

dup_nil relates the empty trace to itself. dup_keep delivers the next sent message once. dup_stutter re-emits the message currently at the head of the received trace — turning a received x :: r into x :: x :: r — without consuming anything new from the sender. Because only dup_keep ever draws from the sent list and it always delivers what it drew, this relation can grow a trace but never shrink it.

A channel that both drops and duplicates combines the two constructor sets:

Inductive lossy_dup : list M -> list M -> Prop :=
  | lossy_dup_nil : lossy_dup [] []
  | lossy_dup_keep : forall x s r,
      lossy_dup s r -> lossy_dup (x :: s) (x :: r)
  | lossy_dup_drop : forall x s r,
      lossy_dup s r -> lossy_dup (x :: s) r
  | lossy_dup_stutter : forall x s r,
      lossy_dup s (x :: r) -> lossy_dup s (x :: x :: r).

Finally, a reordering channel delivers exactly the messages that were sent, each the right number of times, but possibly in a different order — which is what the standard library’s Permutation already means:

Definition reorder s r : Prop := Permutation s r.

The lattice: better channels are smaller relations

A channel that can do fewer things is a better channel — its relation is a subset of the worse one’s. A perfect wire can do strictly less than a lossy wire, so perfect sits inside lossy; the same wire viewed as possibly-reordering is inside reorder; loss alone and duplication alone are both inside the combined lossy_dup. Each of these containments is a short lemma. Here is the first, proved in full.

The claim: any behaviour a perfect channel allows, a lossy channel also allows. On the blackboard it is one step. perfect s r says r = s, and the subsequence relation is reflexive (subseq s s always holds), so

r=s⇒subseqrs.r = s \;\Longrightarrow\; \mathrm{subseq}\ r\ s.

The Coq proof, verbatim:

Lemma perfect_lossy : forall s r, perfect s r -> lossy s r.
Proof.
  intros s r H. destruct H. exact (subseq_refl r).
Qed.

Walking it: intros s r H names the two traces and the hypothesis H : perfect s r, which unfolds to r = s. destruct H uses that equation to replace s by r throughout, so the goal lossy s r — that is, subseq r s — becomes subseq r r. That is exactly subseq_refl r, the fact that every list is a subsequence of itself, so exact closes the goal. The other containments have the same shape: perfect_reorder (perfect is inside reorder, via reflexivity of Permutation), lossy_lossy_dup (loss is inside lossy+duplication), and dup_lossy_dup (duplication is inside lossy+duplication).

The containments are strict — the bigger classes really can do more. A reordering channel can swap two messages, and no amount of dropping and duplicating reproduces a swap, because once a message is dropped it is gone and duplication never moves a message past another. One such gap is pinned down by a concrete counterexample over the naturals — no derivation of the dropped-message pair exists in the duplication class:

¬dup([0;1],[0]).\neg\,\mathrm{dup}\bigl([0;1],\,[0]\bigr).

Example lossy_not_dup : ~ dup [0; 1] [0].
Proof.
  intros H.
  repeat match goal with
  | Hc : dup _ _ |- False => inversion Hc; clear Hc; subst
  | Heq : _ = _ |- False => discriminate Heq
  end.
Qed.

Reading [0; 1] [0] as (sent, received): a lossy channel is perfectly happy to deliver [0; 1] as [0] — it dropped the 1. A duplicating channel cannot, because delivering [0] from [0; 1] would require losing the 1, and dup has no constructor that loses anything. The proof repeatedly inverts the assumed derivation dup [0;1] [0]: each inversion either exposes a mismatch that discriminate rejects outright, or peels one constructor and continues, and after a few layers every branch has hit a contradiction. So the classes are genuinely distinct, not just formally different names.

The payoff: what a fault class buys you

The point of naming these relations is that assuming one of them, as a hypothesis, lets you prove things about the received trace that would be false for a worse channel. Here is the cleanest example, straight from the file: a duplicating channel can only make a trace longer.

On the blackboard the argument is an induction on the derivation of dup s r. In the base case both traces are empty, 0 ≤ 0. Each dup_keep step adds one message to both the sent and the received trace, so an inequality m ≤ n becomes m+1 ≤ n+1. Each dup_stutter step adds a message to the received trace only, so m ≤ n becomes the even weaker m ≤ n+1. Sent length therefore never overtakes received length:

dupsr⇒|s|≤|r|.\mathrm{dup}\ s\ r \;\Longrightarrow\; |s| \le |r|.

The Coq proof, verbatim:

Lemma dup_length : forall s r, dup s r -> length s <= length r.
Proof.
  intros s r H.
  induction H as [| x s r H IH | x s r H IH]; simpl.
  - apply le_n.
  - apply le_n_S. exact IH.
  - apply le_S. exact IH.
Qed.

induction H splits on the three constructors, and simpl computes the two lengths in each case. The dup_nil case leaves 0 <= 0, closed by le_n. The dup_keep case has IH : length s <= length r and goal S (length s) <= S (length r), which is le_n_S applied to the inductive hypothesis — the “add one to both sides” step. The dup_stutter case has the same IH but goal length s <= S (length r), one weaker, which is le_S IH — the “add one to the received side only” step. Three constructors, three arithmetic moves, done.

Contrast that with what a lossy channel guarantees, which is the mirror image:

lossysr⇒|r|≤|s|.\mathrm{lossy}\ s\ r \;\Longrightarrow\; |r| \le |s|.

Because lossy s r is by definition subseq r s, this is not a new theorem at all — it is the subsequence length lemma, subseq_length r s, applied directly. Here is that lemma, verbatim from Theory/Subseq.v:

  Lemma subseq_length : forall r s, subseq r s -> length r <= length s.
  Proof.
    intros r s H.
    induction H as [| x r s H IH | x r s H IH]; simpl.
    - apply le_n.
    - apply le_n_S. exact IH.
    - apply le_S. exact IH.
  Qed.

The induction has exactly the same three-case shape as dup_length above, with the roles of the two sides exchanged: keeping a message adds one to both lengths (le_n_S), dropping one adds only to the parent’s side (le_S).

So the fault class you assume is not bookkeeping — it decides the theorem you get. Told only that the channel is reorder, you would know the received trace has the same length as the sent one but nothing about its order. Told it is dup, you know the received trace is at least as long as the sent one. Told it is lossy, you know it is at most as long. Each of these is a real, checkable fact that follows from the relation alone, with no probability ever mentioned — which is exactly what the possibilistic view promised.
