exp’s derivative, log, and the kink in ReLU

An earlier post pinned down the exponential function without ever writing a formula for it, using four laws: exp 0 is 1; exp turns sums into products, exp(a+b) = exp(a) * exp(b); its values are never negative; and it is strictly increasing. Those four facts alone already force exp to be strictly positive and injective — different inputs always give different outputs. This post looks at two places where the four laws run out before delivering something you’d also want from exp — inverting it, and finding its rate of change. The file that works out the second of those also answers a third, independent question, about a completely different function: at a sharp corner in a function’s graph, what exactly can be said about its rate of change on each side, and what has to be left unsaid?

Can exp always be inverted?

Suppose all you know about a function is the four facts above: it sends 0 to 1, it turns sums into products, it never goes negative, and it is strictly increasing. Those facts alone already guarantee that no two different inputs produce the same output — a strictly increasing function can’t repeat a value. That answers half of “can this function be inverted”: for a given output, there is at most one input that produced it. The other half is a separate question: for every positive number you might want as an output, is there at least one input that produces it? A function can be injective and still miss values — a strictly increasing function on the whole number line could, in principle, only ever output numbers bigger than 5, and no input would ever produce 3.

Whether every positive value gets hit turns out to depend on more than addition, multiplication, and order. Classically, you’d find the input that produces a target value y by taking the supremum of all inputs whose output stays below y — a search that only works if every bounded set of candidates actually has a least upper bound. That “no gaps” property is not implied by being an ordered field. The rational numbers are the standard counterexample: the set of rationals x with x*x < 2 is bounded above, yet no rational number is its least upper bound — the gap sits exactly where √2 should be. Nothing about the four properties above — sends 0 to 1, turns sums into products, stays nonnegative, is increasing — rules out a function living on a number system riddled with gaps like that one, so nothing about them forces every positive output to actually get hit.

Coq’s Theory/Log.v records this as a separate, explicit assumption, rather than something the four properties already give:

Context (exp_surj : forall y : F, 0 < y -> {x : F | exp_fn x == y}).

Definition log_fn (y : F) (Hy : 0 < y) : F := proj1_sig (exp_surj y Hy).

exp_surj is exactly the “every positive y gets hit” property from above, stated for a positive y and an unspecified field F. The type {x : F | exp_fn x == y} — a sigma type — is a pair: a witness input x, together with a proof that exp_fn x really does equal y. That choice of type, rather than the more familiar exists x, exp_fn x == y, is what lets log_fn be an actual definition instead of a bare claim: an ordinary Coq exists only asserts that a witness is out there, and pulling a concrete value back out of it needs a choice-style axiom, which nothing in this codebase relies on. A sigma type carries the witness directly, so proj1_sig — “take the first component” — reads log_fn straight off it.

With exp_surj granted, exp (log y) == y — called exp_log — is immediate: it’s proj2_sig, the proof half of the pair, read off directly. A less immediate consequence is that log inherits monotonicity from exp. Here is the theorem and its proof, exactly as they stand in the file:

Lemma log_mono : forall (a b : F) (Ha : 0 < a) (Hb : 0 < b),
  a < b -> log_fn a Ha < log_fn b Hb.
Proof.
  intros a b Ha Hb Hab.
  apply exp_lt_reflect.
  rewrite (exp_log a Ha), (exp_log b Hb).
  exact Hab.
Qed.

On the blackboard, the argument runs backwards from the goal:

log⁡a<log⁡b⇐exp⁡(log⁡a)<exp⁡(log⁡b)⇔a<b{exp⁡ reflects <;exp⁡(log⁡y)=y}.\log a < \log b \;\Longleftarrow\; \exp(\log a) < \exp(\log b) \;\Longleftrightarrow\; a < b \qquad \{\exp \text{ reflects } <;\ \exp(\log y) = y\}.

exp_lt_reflect is a small lemma one page earlier — exp a < exp b -> a < b, “exp reflects the order,” essentially exp_mono run in reverse — and applying it turns the goal log a < log b into the seemingly harder exp (log a) < exp (log b). It only looks harder: rewrite (exp_log a Ha) and rewrite (exp_log b Hb) replace exp (log a) by a and exp (log b) by b, since that is exactly what exp_log says, leaving the goal a < b — which is Hab, the hypothesis just given. The proof does no new order-theoretic work; it reduces “log is increasing” to “exp is increasing,” a fact already in hand.

The same reduce-to-an-exp-equality-and-let-injectivity-finish-it pattern proves the rest of the file’s small law list — log turns products into sums, the mirror image of exp_add. Since exp is injective, it is enough to show both sides have the same exponential:

exp⁡(log⁡(ab))=ab{exp⁡(log⁡y)=y}=exp⁡(log⁡a)⋅exp⁡(log⁡b){exp⁡(log⁡y)=y, twice}=exp⁡(log⁡a+log⁡b){sums-to-products, read backwards}\begin{aligned} \exp(\log(ab)) &= ab && \{\exp(\log y) = y\}\\ &= \exp(\log a)\cdot\exp(\log b) && \{\exp(\log y) = y \text{, twice}\}\\ &= \exp(\log a + \log b) && \{\text{sums-to-products, read backwards}\} \end{aligned}

Lemma log_mul : forall (a b : F) (Ha : 0 < a) (Hb : 0 < b),
  log_fn (a * b) (mul_pos a b Ha Hb) == log_fn a Ha + log_fn b Hb.
Proof.
  intros a b Ha Hb.
  apply exp_injective.
  rewrite (exp_log (a * b) (mul_pos a b Ha Hb)), exp_add,
    (exp_log a Ha), (exp_log b Hb).
  reflexivity.
Qed.

exp_injective reduces an equality of logs to an equality of exps; three rewrites by exp_log and one by exp_add turn both sides into a * b, and reflexivity closes it.

None of this — log_fn, exp_log, log_mono, log_mul — is specific to any particular exp. Every lemma is proved for any F and any exp satisfying the four original laws plus exp_surj. That means whoever eventually builds a concrete exponential function — a power series over the reals, say, with surjectivity established from completeness — gets every lemma above for free the moment they supply one proof of exp_surj, with none of this reasoning to redo. No such construction exists in this codebase yet, so there is no concrete log to compute a number with today; what exists is the guarantee that whenever one shows up, its logarithm comes for free.

What’s the slope of a function with a corner?

Consider the simplest function with a corner in its graph: zero for every negative input, equal to the input itself for every non-negative input, the two pieces meeting at a sharp bend at 0. Away from that bend, the function is exactly as well-behaved as whichever of its two pieces it currently is — the identity line has a definite slope of 1 everywhere, the flat zero line has a definite slope of 0 everywhere, and the function coincides with one or the other on an entire stretch around any point that isn’t 0 itself. At the bend, something different happens: approaching from the right the slope is heading toward 1, approaching from the left it’s heading toward 0 — two different numbers, both converging on the same point. Does the function have a single, well-defined instantaneous rate of change at 0, or does it not?

This is the rectified linear function, ReLU, and away from its corner the answer is exactly what the picture suggests: a definite rate of change on each side, the two rates different. Making that precise needs the notion of derivative from earlier posts: the derivative of a function f at a point a, written is_deriv f a L, packages the rate of change not as a bare number but as a linear map L — the best linear approximation of f near a, in the sense that the gap f (a + h) - (f a + L h) shrinks faster than h itself as h shrinks toward 0. Size here is measured by a norm ‖ · ‖, which for a field acting on itself is just absolute value; Theory/MLDerivatives.v opens by building exactly that instance — F normed over itself via abs — because nothing in the tree supplied it before, and every derivative statement in the rest of the file, ReLU’s included, rests on it.

The file never writes down a single total function relu : F -> F. Deciding whether a given field element is >= 0 is exactly the kind of case split a bare ordered field doesn’t license without extra assumptions — the same reason abs and inv are taken as given operations here rather than built by case analysis. Instead of proving a fact about one specific relu, the file proves a fact about any function that happens to behave like ReLU on one side, which is both more general and reusable by whatever concrete relu eventually gets built. The tool that makes this possible is a lemma saying that a derivative only depends on a function’s behavior in a neighborhood of the point in question:

Lemma is_deriv_of_local_eq : forall (f g : F -> F),
    Proper (equ ==> equ) f -> Proper (equ ==> equ) g ->
    forall (a : F) (L : LinearMap F F F) (d : F),
    0 < d -> (forall h : F, ‖ h ‖ < d -> f (a + h) == g (a + h)) ->
    is_deriv g a L -> is_deriv f a L.

If f and g agree everywhere within distance d of a, and g is known to have derivative L at a, then f has that same derivative at a — is_deriv cannot distinguish the two functions, since it only ever asks about points near a.

In one sentence, using that tool: any function f that agrees with x itself at every strictly positive x has, at every strictly positive point a, the identity map lm_id (the linear map sending h to h) as its derivative. On the blackboard, the argument is “swap f for the identity function on a neighborhood of a, then read off the identity’s own derivative”:

f(x)=x for all x with ∥x−a∥<a⇒Df(a)=D(id)(a)=lm_id.f(x) = x \text{ for all } x \text{ with } \|x - a\| < a \quad\Longrightarrow\quad Df(a) = D(\mathrm{id})(a) = \mathrm{lm\_id}.

Here is the theorem and its proof exactly as they stand in the file:

Theorem is_deriv_relu_pos : forall (f : F -> F),
  Proper (equ ==> equ) f ->
  forall a : F, 0 < a -> (forall x : F, 0 < x -> f x == x) ->
  is_deriv f a lm_id.
Proof.
  intros f Pf a Ha Hf.
  apply (is_deriv_of_local_eq f (fun x : F => x) Pf (fun x y Hxy => Hxy)
    a lm_id a Ha).
  - intros h Hh.
    apply Hf.
    exact (pos_add_of_norm_lt a h Ha Hh).
  - exact (deriv_lm lm_id a lm_id_bounded_F).
Qed.

Walking it against that blackboard step: apply (is_deriv_of_local_eq ...) chooses g := fun x => x, the plain identity function, and radius d := a — a deliberate choice, because a is exactly how far a sits from 0, the one point f must stay clear of. That choice creates two obligations. The first, discharged by intros h Hh; apply Hf; exact (pos_add_of_norm_lt a h Ha Hh), is that f and the identity really do agree throughout the radius-a neighborhood: pos_add_of_norm_lt is the fact that a + h stays positive whenever ‖h‖ < a and a itself is positive, which is precisely what lets Hf — “f agrees with the identity on all positive inputs” — apply at a + h. The second, exact (deriv_lm lm_id a lm_id_bounded_F), supplies the anchor fact: the identity function’s own derivative at any point is the identity map itself, since a linear map is its own exact linear approximation everywhere, not merely an approximate one.

The left branch is the mirror image, with the constantly-zero function in place of the identity and the zero linear map lm_const0 in place of lm_id:

Theorem is_deriv_relu_neg : forall (f : F -> F),
    Proper (equ ==> equ) f ->
    forall a : F, a < 0 -> (forall x : F, x < 0 -> f x == 0) ->
    is_deriv f a lm_const0.

Neither theorem says anything about a == 0. That’s the answer to the question posed at the top of this section: ReLU’s right-derivative at the corner is 1 (lm_id), its left-derivative is 0 (lm_const0), and a genuine two-sided derivative would force those equal — which they are not. There is no radius around 0 small enough for any single linear map to approximate both branches at once, so the file proves no theorem there. Once a concrete decidable order lets someone actually write relu x := if x >= 0 then x else 0, applying is_deriv_relu_pos or is_deriv_relu_neg at any nonzero point produces its derivative immediately, with only the case split itself left to supply; at 0, no amount of applying either lemma will produce one, because none exists.

Does exp’s rate of change match its own value?

A separate question about exp: is its instantaneous rate of change at any point exactly equal to its value there — the property usually summarized as “exp is its own derivative”? And, the same question this whole post keeps returning to: does that follow from the four original properties alone, or does it need something else?

It needs something else, and for a structural reason rather than a missing calculation: rate of change is a limiting notion — it talks about what happens to a ratio as a gap shrinks toward zero — while the four properties (sends 0 to 1, turns sums into products, stays nonnegative, is increasing) are purely algebraic statements about single inputs and outputs, with no mention of limits or shrinking gaps at all. No amount of algebraic manipulation of the four properties produces a limiting statement; something limiting has to be supplied from outside them.

Stating the claim at all first needs two pieces of scaffolding the four laws don’t supply either. exp_fn : F -> F needs a way to measure distance in F itself — the same NormedSpace F F instance built for ReLU above — and the rate of change, to be stated as a derivative in this codebase’s sense, needs to be packaged as a linear map, “multiply by exp a,” rather than the bare scalar exp a. Theory/MLDerivatives.v builds that scaling linear map generically, for any constant c:

Definition scale_map (h : F) : F := c * h.

together with proofs that it is linear, bundled as lm_scale_map c. With that scaffolding in place, “exp is its own derivative” is recorded as an explicit assumption, the same move as exp_surj for log:

Context (exp_is_deriv : forall a : F, is_deriv exp_fn a (lm_scale_map (exp_fn a))).

Granting it, ordinary chain-rule reasoning goes through mechanically. For instance, the function x↦exp⁡(−x)x \mapsto \exp(-x) has, at any point aa, the derivative “multiply by −exp⁡(−a)-\exp(-a)” — on the blackboard, the chain rule applied to the composite of negation and exp\exp:

D(exp⁡∘neg)(a)=Dexp⁡(−a)∘Dneg(a)=(⋅exp⁡(−a))∘(⋅(−1))=⋅(−exp⁡(−a)).D(\exp \circ\, \mathrm{neg})(a) = D\exp(-a) \circ D\,\mathrm{neg}(a) = (\,\cdot\,\exp(-a)) \circ (\,\cdot\,(-1)) = \,\cdot\,(-\exp(-a)).

Theorem is_deriv_exp_neg : forall a : F,
  is_deriv (fun x : F => exp_fn (- x)) a (lm_scale_map (- exp_fn (- a))).
Proof.
  intros a.
  apply (is_deriv_L_equ (fun x : F => exp_fn (- x)) a
    (lm_comp (lm_scale_map (exp_fn (- a))) lm_neg_map)).
  - exact (lm_comp_scale_neg (exp_fn (- a))).
  - exact (deriv_comp neg_map exp_fn a lm_neg_map
      (lm_scale_map (exp_fn (- a))) exp_fn_proper
      (is_deriv_neg_map a) (exp_is_deriv (- a))).
Qed.

Walking it: deriv_comp is the chain rule — fed the derivative of negation (is_deriv_neg_map, the middle factor of the blackboard chain) and exp_is_deriv at the point - a (the left factor), it produces the derivative as the composite linear map lm_comp (lm_scale_map (exp_fn (- a))) lm_neg_map; is_deriv_L_equ then replaces that composite by the equal single scaling map lm_scale_map (- exp_fn (- a)) (the last blackboard step, proved as lm_comp_scale_neg). No further assumptions beyond exp_is_deriv are needed. This is the pattern any future caller follows: state the target composite function, name which already-known derivatives it’s built from, and hand them to deriv_comp or one of its specializations. Nothing here computes a number, because exp_is_deriv has no proof supplied for any concrete exp in this codebase yet — the theorem is ready the moment one is.

Two things this file doesn’t attempt

Softmax and layer normalization are the natural next targets — both are built from exp and division — but neither is proved differentiable here. Both are functions on a many-coordinate space, and stating a derivative for them at all needs a norm on that whole space, which doesn’t exist anywhere in the tree yet (only a norm for matrices has been built, not for bare coordinate vectors); both also divide by something (a sum of exponentials, a standard deviation), and no rule for differentiating 1/x itself exists yet either. Both are separate pieces of work with their own scaffolding to build, not results this post glosses over.
