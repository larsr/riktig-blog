When the obvious compiler is wrong: SKI, revisited

Here is a question that sounds like it should have an obvious answer. Take a program written the ordinary way, out of function definitions that take a parameter and a body mentioning it, and mechanically rewrite it into an equivalent program built from nothing but three fixed, closed building blocks, combined only by writing them next to each other — no parameters anywhere, nothing ever filled in by matching a name. Classically the three building blocks are called SS, KK, and II, and their behavior is pinned down once and for all by three rewrite rules:

Ix→xKxy→xSfgx→fx(gx)I\,x \to x \qquad K\,x\,y \to x \qquad S\,f\,g\,x \to f\,x\,(g\,x)

II just returns its argument; KK returns its first argument and throws the second away; SS hands a shared argument xx to both ff and gg and combines the two results. There is a standard, purely mechanical procedure for turning any function definition into an expression built only from SS, KK, and II: eliminate one parameter at a time, working from the innermost definition outward, by a case analysis on the shape of the body — a bare occurrence of the parameter becomes II, anything not mentioning it gets wrapped in KK so a future argument is discarded, and an application gets split with SS so a future argument reaches both halves. It looks exactly the way a compiler ought to look: syntax-directed, no guesswork, nothing that could plausibly go wrong.

Here is the question that procedure raises, and it is not as obviously “yes” as it looks. Run the original program one step — substitute an argument into a function body, the ordinary way. Separately, run the rewritten combinator version by its own three fixed rules, as far as it goes. Do the two sides still describe the same computation, step for step? Does simplifying the combinator version always land on (or simplify further into) the combinator translation of wherever the original program’s one step landed — or could the translation merely happen to end up looking equivalent once everything is fully run, while disagreeing about the intermediate stops, or even disagreeing permanently?

There is a genuine reason to worry, and it has nothing to do with any particular program — it is a fact about the SS/KK/II rewriting system by itself. Two already-fully-simplified SS/KK/II expressions can compute exactly the same function on every future argument and still be different, unrelated pieces of syntax, with no rewrite connecting one to the other. S (K f) (K g) and K (f g) are the standard instance: give either one more argument xx, and both simplify to f g — but with no argument yet supplied, neither simplifies into the other. S (K f) (K g) needs three arguments before its rule can fire and only has two; K (f g) needs two and only has one; both are already stuck. A rewriting system built purely from SS, KK, II has no rule for identifying two expressions just because they will always behave alike on some future input — it only ever simplifies a term when an argument is actually supplied and drives a step forward. If the translation procedure above ever produces one of these two shapes at one point in a computation and the other shape at a different point, the mechanical rewriting rules will not recognize them as the same, and a step-for-step correspondence can fail without either side doing anything unreasonable.

Here is a concrete case built to test exactly that. Take the function of two arguments that applies the first to the second, (λx.λy.xy)(\lambda x.\lambda y.\; x\,y), and apply it to a small function of its own, (λz.zw)(\lambda z.\; z\,w), for some other fixed value ww. Running this one step substitutes the small function in for xx: the result is λy.(λz.zw)y\lambda y.\; (\lambda z.\; z\,w)\,y — the same computation with one level of application unfolded, ww renamed to keep pointing at the same thing because it now sits one binder deeper than it used to, which is the ordinary bookkeeping every substitution performs when it crosses a fresh binder. Translate both the two-step program and its one-step-reduced version separately by the mechanical elimination procedure above, and simplify each as far as it goes. Do the two results land on the same combinator expression — or on a pair connected by further simplification — or do they diverge the way S(Kf)(Kg) and K(fg) can diverge? That is a checkable question, not a matter of opinion, and it is exactly what the rest of this post checks.

Formalizing the rewrite

Lambda/Syntax.v represents a function-definition-style program (a lambda term) with three constructors — a variable, an abstraction that binds one, or an application — using de Bruijn indices, where a variable is a number counting the binders crossed on the way out to whichever one bound it. The combinator side gets its own type, comb, in Lambda/SKI.v: the three combinators, named CS, CK, CI (bare S/I would collide with other library names in scope), an inert placeholder CVar n standing in for an opaque, untouched argument, and application written f @@ x. The three rewrite rules from the opening section become three constructors of a one-step relation on comb, plus two rules letting a step happen inside either side of an application; its reflexive-transitive closure is written cstar.

The mechanical elimination procedure — case-split on the shape of the body, I for the bare parameter, K for anything that doesn’t mention it, S to split every application — is the function babs (“bracket abstraction”), read straight from Lambda/SKI.v:

Fixpoint babs (t : comb) : comb :=
  match t with
  | CVar O => CI
  | CVar (S n) => CK @@ CVar n
  | CApp f a => CS @@ babs f @@ babs a
  | c => CK @@ c
  end.

translate compiles a whole lambda term by calling babs once per abstraction:

Fixpoint translate (t : tm) : comb :=
  match t with
  | Var n => CVar n
  | Abs b => babs (translate b)
  | App f a => translate f @@ translate a
  end.

A one-redex correctness fact already holds for this pair: applying a bracket-abstracted term to one argument reduces to substituting that argument in. What that single-redex fact does not say — and what the opening question is really asking — is whether the whole, recursively-applied translation keeps tracking substitution once a substitution happens underneath a second abstraction, which is exactly the shape of the concrete case above (the parameter x sits inside the body of the inner λy). The Coq development states the opening section’s worry precisely, as a proved theorem named translate_not_step_simulation: translating the two-step program and translating its one-step reduct, then normalizing both, gives two different comb values. The next section proves it.

The remedy is a one-line change to the case analysis: before deciding to split an application with S, check whether the parameter being eliminated actually occurs anywhere inside it, using a boolean occurrence test:

Fixpoint occurs0 (t : comb) : bool :=
  match t with
  | CVar O => true
  | CApp f a => occurs0 f || occurs0 a
  | _ => false
  end.

Only fall back to a structural S-split when the parameter genuinely occurs; otherwise collapse the whole subterm behind a single K, with every remaining index shifted down by one (cdown1) rather than splitting it apart and re-wrapping each half in its own K:

Fixpoint babs2 (t : comb) : comb :=
  match t with
  | CVar O => CI
  | CApp f a =>
      if occurs0 (CApp f a) then CS @@ babs2 f @@ babs2 a
      else CK @@ cdown1 (CApp f a)
  | c => CK @@ cdown1 c
  end.
Fixpoint translate2 (t : tm) : comb :=
  match t with
  | Var n => CVar n
  | Abs b => babs2 (translate2 b)
  | App f a => translate2 f @@ translate2 a
  end.

This directly addresses the failure mode from the opening section: the naive procedure always splits an application with S, regardless of whether the parameter occurs in it, so it can produce an S (K f) (K g)-shaped tree in a place where a fresh translation of a substituted term would instead produce a K (f g)-shaped tree — two shapes the rewriting rules alone can never identify. Checking occurrence first means the choice of S versus K depends only on where the parameter actually appears, which is exactly the information substitution changes, so translating-then-substituting and substituting-then-translating end up making the same choice at every step. The Coq development proves this directly, as translate2_step_star: every one-step reduction of the original program is matched by zero or more rewriting steps of its translate2 translation. The rest of this post proves both theorems.

Proving the naive rewrite really does disagree

State the concrete case from the opening section in the de Bruijn notation Lambda/Syntax.v uses: xx, yy, zz become indices 1, 0, 0 at their respective depths, so (λx.λy.xy)(λz.zw)(\lambda x.\lambda y.\, x\,y)\,(\lambda z.\, z\,w) is App (Abs (Abs (App (Var 1) (Var 0)))) (Abs (App (Var 0) (Var 1))), and its one-step reduct is subst 0 (Abs (App (Var 0) (Var 1))) (Abs (App (Var 1) (Var 0))) — substituting the argument for x in the body of the outer abstraction, using the same subst function the source language’s own step relation uses. The theorem: translating and normalizing both sides gives different comb values.

nf200(translatet)≠nf200(translatet′),t→t′\mathrm{nf}_{200}\big(\mathrm{translate}\;t\big) \;\neq\; \mathrm{nf}_{200}\big(\mathrm{translate}\;t'\big), \qquad t \longrightarrow t'

Verbatim from Lambda/SKI.v:

Example translate_not_step_simulation :
  nf 200 (translate (App (Abs (Abs (App (Var 1) (Var 0))))
                          (Abs (App (Var 0) (Var 1)))))
  <> nf 200 (translate
       (subst 0 (Abs (App (Var 0) (Var 1))) (Abs (App (Var 1) (Var 0))))).
Proof. discriminate. Qed.

Walking it: nf 200 runs the fuelled combinator normalizer for up to two hundred rewrite steps on each side, far more than either side needs to reach a point where no rewrite rule applies any longer, so both arguments to <> are concrete, fully-simplified comb values by the time discriminate runs. discriminate then closes the goal by Coq’s structural equality check on those two values, which finds a mismatch immediately — no search needed, because the values are already computed.

Printing what the normalizer actually lands on (checked directly against these definitions, with Compute (nf 200 (translate t)). and the matching line for the reduct) makes the mismatch visible rather than merely asserted:

nf 200 (translate t)  = CS @@ (CK @@ (CS @@ CI @@ (CK @@ CVar 0))) @@ CI

nf 200 (translate t') = CS @@ (CS @@ (CS @@ (CK @@ CS) @@ (CK @@ CI))
                                    @@ (CS @@ (CK @@ CK) @@ (CK @@ CVar 0))) @@ CI

The left tree comes from plugging the compiled argument straight into the one CVar 0 slot the already-compiled body left open for it. The right tree comes from taking the fully substituted program at face value and bracket-abstracting it again from nothing, where the slot that used to hold a bare variable now holds a whole application and gets split with S. Both trees, given enough further opaque arguments, behave the same way — that is what it means for the two lambda terms to be one reduction step apart — but with no argument yet applied, they are different, and translate produces one or the other depending on whether the substitution happened before or after translation. Substitution and translation do not commute for babs/translate, and a step-for-step simulation theorem for them, as stated, is false.

Proving the fix simulates every step

babs2 satisfies the same one-redex law babs did: applying a bracket-abstracted term to one argument rewrites, in zero or more steps, to the result of substituting that argument in —

babs2(t)u→*t[0:=u]\mathrm{babs2}(t)\;u \;\longrightarrow^{*}\; t[0 := u]

— proved the same way as before, with one extra case for the branch where the parameter does not occur:

Lemma babs2_correct : forall t u, cstar (babs2 t @@ u) (csub u t).
Proof.
  induction t as [ | | | n | f IHf a IHa]; intros u; simpl.
  - eapply cstar_step. { apply cstep_K. } apply cstar_refl.
  - eapply cstar_step. { apply cstep_K. } apply cstar_refl.
  - eapply cstar_step. { apply cstep_K. } apply cstar_refl.
  - destruct n as [| n]; simpl.
    + eapply cstar_step. { apply cstep_I. } apply cstar_refl.
    + eapply cstar_step. { apply cstep_K. } apply cstar_refl.
  - destruct (occurs0 f || occurs0 a) eqn:Hocc.
    + eapply cstar_step. { apply cstep_S. }
      eapply cstar_trans. { apply cstar_app1. apply IHf. }
      apply cstar_app2. apply IHa.
    + eapply cstar_step. { apply cstep_K. }
      apply orb_false_iff in Hocc. destruct Hocc as [H1 H2].
      rewrite (csub_no_occurs0 f u H1), (csub_no_occurs0 a u H2).
      apply cstar_refl.
Qed.

The first four branches match babs_correct exactly (CVar 0 steps by I, everything else atomic steps by K). The fifth branch splits on whether the parameter occurs in the application: if it does, one S-step followed by the two induction hypotheses closes it, the same as before; if it does not, a single K-step lands directly on csub u f @@ csub u a, and the helper lemma csub_no_occurs0 — substituting into a term that never mentions the parameter is the same as just shifting its other indices down — shows that is exactly cdown1 f @@ cdown1 a, the same destination reached by a shorter route.

A one-redex law is not enough by itself, though — proving that a step deep inside a term is simulated needs substitution to agree with translation at every abstraction depth, not only the outermost one. csubst2 k w t is the combinator-level mirror of substitution at depth k. The theorem: translating a substituted lambda term gives the same combinator term, not merely one that further simplifies to match, as translating the pieces separately and then performing the matching combinator-level substitution.

translate2(substkut)=csubst2k(translate2u)(translate2t)\mathrm{translate2}\big(\mathrm{subst}\;k\;u\;t\big) \;=\; \mathrm{csubst2}\;k\;\big(\mathrm{translate2}\;u\big)\;\big(\mathrm{translate2}\;t\big)

Verbatim from Lambda/SKI.v:

Theorem translate2_subst_gen : forall t k u,
  translate2 (subst k u t) = csubst2 k (translate2 u) (translate2 t).
Proof.
  induction t as [n | b IH | f IHf a IHa]; intros k u; simpl.
  - destruct (Nat.ltb n k) eqn:E1.
    + reflexivity.
    + destruct (Nat.eqb n k) eqn:E2.
      * apply Nat.eqb_eq in E2. subst n. apply translate2_lift.
      * reflexivity.
  - rewrite IH. apply babs2_csubst_comm.
  - rewrite IHf, IHa. reflexivity.
Qed.

Walking it, one case per shape of t:

  • t = Var n. subst and csubst2 make the identical three-way split on how n compares to k — untouched below k, replaced at k, shifted down above k — so the first and third branches close by reflexivity outright. The middle branch, where n equals k, needs translate2_lift, a companion lemma stating that translation commutes with re-indexing the same way it needs to commute with substitution.
  • t = Abs b. The induction hypothesis IH handles the body, and babs2_csubst_comm shows that pushing a substitution through one more babs2 at a deeper index matches bracket-abstracting a term that was already substituted into. This is exactly the step that fails for babs: it holds for babs2 because its choice between S and K depends on occurs0, which shifts and substitutes predictably, in a way the unconditional structural recursion of babs does not.
  • t = App f a. Both sides are congruent in the two subterms; rewrite IHf, IHa finishes it.

The shape actually needed downstream — substitution at depth 0, the shape a single reduction step produces, i.e. the displayed equation at k=0k = 0 — follows as a corollary:

Corollary translate2_subst0 : forall t u,
  translate2 (subst 0 u t) = csub (translate2 u) (translate2 t).
Proof.
  intros t u. rewrite (translate2_subst_gen t 0 u). apply csubst2_0.
Qed.

With that in hand, the simulation theorem for a single step is three lines:

t→t′⇒translate2t→*translate2t′t \longrightarrow t' \quad\Longrightarrow\quad \mathrm{translate2}\;t \;\longrightarrow^{*}\; \mathrm{translate2}\;t'

Theorem translate2_step_star : forall t t',
  step t t' -> cstar (translate2 t) (translate2 t').
Proof.
  intros t t' H. induction H as [b v Hv | f f' a Hs IH | v a a' Hv Hs IH].
  - simpl. eapply cstar_trans. { apply babs2_correct. }
    rewrite (translate2_subst0 b v). apply cstar_refl.
  - simpl. apply cstar_app1. exact IH.
  - simpl. apply cstar_app2. exact IH.
Qed.

Walking it, one case per constructor of the step relation:

  • A beta step. The redex App (Abs b) v translates to babs2 (translate2 b) @@ translate2 v, which babs2_correct reduces to csub (translate2 v) (translate2 b); rewriting with translate2_subst0 turns that into translate2 (subst 0 v b), which is exactly the translation of the reduct, so cstar_refl closes the goal — the two sides meet exactly, with no further rewriting needed to reconcile them.
  • A step inside the function, or inside the argument, of an application. Pure congruence: the induction hypothesis is wrapped in cstar_app1 or cstar_app2, matching the corresponding congruence rule of the source language’s own step relation.

Extending from one step to a whole sequence of steps,

t→*t′⇒translate2t→*translate2t′,t \longrightarrow^{*} t' \quad\Longrightarrow\quad \mathrm{translate2}\;t \;\longrightarrow^{*}\; \mathrm{translate2}\;t',

is ordinary induction, chaining cstar_trans along the way:

Theorem translate2_star_sim : forall t t',
  star t t' -> cstar (translate2 t) (translate2 t').
Proof.
  intros t t' H. induction H as [t0 | t0 u v Hstep _ IH].
  - apply cstar_refl.
  - eapply cstar_trans. { apply translate2_step_star. exact Hstep. } exact IH.
Qed.

An empty sequence of steps needs nothing (cstar_refl); a sequence that starts with one step followed by more steps applies translate2_step_star to the first step and chains the rest on by transitivity.

Running it on the case that broke the naive translation

The concrete case from the opening section, run back through translate2 instead of translate, is now a positive instance of the theorem rather than a counterexample to it:

Example translate2_simulates_the_counterexample :
  cstar (translate2 (App (Abs (Abs (App (Var 1) (Var 0))))
                          (Abs (App (Var 0) (Var 1)))))
        (translate2
           (subst 0 (Abs (App (Var 0) (Var 1))) (Abs (App (Var 1) (Var 0))))).
Proof.
  apply translate2_step_star. apply step_beta. constructor.
Qed.

The proof is translate2_step_star applied to the same step_beta fact that produced the disagreement earlier. Checked directly, Compute (nf 200 (translate2 t)). and the matching line for the reduct both come out as CS @@ (CK @@ (CS @@ CI @@ (CK @@ CVar 0))) @@ CI — the shorter of the two trees from the earlier section. The occurrence-checking translation of the reduct lands exactly where the occurrence-checking translation of the original redex already was; substituting first and translating second, or translating first and substituting second, now reach the same place.

What is proved, and what is not

translate2_step_star and translate2_star_sim are a forward simulation: every reduction step of the source language, taken one at a time, has a matching sequence of zero or more combinator rewrites underneath the translation. That is soundness of the compilation — a step in the original program is never lost or misrepresented once compiled to combinators. It is not a correspondence in the other direction. Nothing proved here shows that every combinator rewrite available on translate2 t corresponds to some step of t in the source language (the combinator rewriting rules can take shortcuts a source-language step never would), and nothing shows that reaching a combinator normal form implies the original term reaches one, or the other way around. That converse direction — completeness of the compilation, rather than soundness — is a different theorem and was not attempted here.
