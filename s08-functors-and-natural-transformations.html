Functors and natural transformations

A category is a collection of objects with arrows between them (written x ~> y), a way to compose two arrows into one (∘), an identity arrow on each object, and two laws: the identity does nothing, and composition is associative. The running example was CoqCat, where the objects are types, an arrow X ~> Y is a function X -> Y, and composition is ordinary function composition.

Once there are two categories, the next question asks itself: what is a map between them? Not a map of objects alone, nor of arrows alone, but one that carries the whole structure across — objects to objects, arrows to arrows, respecting identities and composition. That is a functor. And a programmer writes functors all the time without naming them: List.map is one, and it will be our worked example.

The Functor class

From theories/Interface/Category.v:

Class Functor {C D : Type} (cC : Category C) (cD : Category D) := {
  fobj : C -> D;
  fmap : forall {x y : C}, (x ~> y) -> (fobj x ~> fobj y);
  #[global] fmap_proper :: forall x y : C, Proper (equ ==> equ) (@fmap x y);
  fmap_id : forall x : C, fmap (@cat_id C cC x) == cat_id;
  fmap_comp : forall (x y z : C) (f : y ~> z) (g : x ~> y),
      fmap (f ∘ g) == fmap f ∘ fmap g
}.

A functor from category C to category D has:

  • fobj, the action on objects: every object of C is sent to an object of D.
  • fmap, the action on arrows: an arrow x ~> y in C is sent to an arrow fobj x ~> fobj y in D. The functor moves the endpoints by fobj and the arrow between them by fmap.
  • fmap_proper: fmap respects the hom equality — equal arrows go to equal arrows.
  • Two laws. fmap_id: the identity arrow is sent to the identity arrow. fmap_comp: the image of a composite is the composite of the images, fmap (f ∘ g) == fmap f ∘ fmap g. Structure in, same structure out.

The list functor: map

Take C and D both to be CoqCat. For each type X, the type list X is another type — that is the action on objects. For each function f : X -> Y, map f : list X -> list Y applies f to every element — that is the action on arrows. This is a functor from CoqCat to itself, defined in theories/Impl/CatFunctors.v. Its two laws are exactly the two facts a programmer relies on when reasoning about map: mapping the identity changes nothing, and mapping a composite is composing the maps,

mapid=id,map(f∘g)=mapf∘mapg.\mathrm{map}\,\mathrm{id} \;=\; \mathrm{id}, \qquad \mathrm{map}\,(f \circ g) \;=\; \mathrm{map}\,f \circ \mathrm{map}\,g.

In Coq, with pointwise equality of functions spelled fn_equ:

Lemma list_fmap_id (X : Type) :
  fn_equ (list X) (list X) (map (fun x => x)) (fun l => l).
Proof.
  intros l. induction l as [| a l IH]; simpl.
  - reflexivity.
  - rewrite IH. reflexivity.
Qed.

Lemma list_fmap_comp (X Y Z : Type) (f : Y -> Z) (g : X -> Y) :
  fn_equ (list X) (list Z)
    (map (fun x => f (g x))) (fun l => map f (map g l)).
Proof.
  intros l. symmetry. apply map_map.
Qed.

list_fmap_id is fmap_id for this functor: mapping the identity function over a list gives back the same list. Its proof is an induction on the list — empty list closes by reflexivity, and the cons case rewrites by the induction hypothesis IH.

list_fmap_comp is fmap_comp: mapping a composed function fun x => f (g x) is the same as mapping g and then mapping f. This is the standard-library fact map_map, and it is the law a compiler uses to fuse two passes over a list into one. A third small lemma, list_fmap_proper, gives fmap_proper from the standard map_ext (equal functions map to equal results). The instance collects the pieces:

#[global] Instance ListFunctor : Functor CoqCat CoqCat :=
  @Build_Functor Type Type CoqCat CoqCat
    list
    (fun (X Y : Type) (f : X -> Y) => map f)
    list_fmap_proper
    list_fmap_id
    list_fmap_comp.

The smallest functor of all is the identity functor, which leaves every object and every arrow where it is; the same file defines it, IdFunctor, on any category, and both its laws are reflexivity.

Natural transformations

Given two functors F and G from C to D, a natural transformation is a way to turn F into G: a family of arrows in D, one for each object of C, that commutes with everything the functors do to arrows. Same file, same source:

Class NaturalTransformation {C D : Type} {cC : Category C} {cD : Category D}
    (F G : Functor cC cD) := {
  eta : forall x : C, fobj (Functor:=F) x ~> fobj (Functor:=G) x;
  naturality : forall (x y : C) (f : x ~> y),
      fmap (Functor:=G) f ∘ eta x == eta y ∘ fmap (Functor:=F) f
}.
  • eta is the family: for each object x, one arrow from F’s version of x to G’s version, fobj F x ~> fobj G x. (eta is the traditional name, the Greek letter η.)
  • naturality is the single law that makes the family coherent. Take any arrow f : x ~> y in C. There are two ways to get from F’s version of x to G’s version of y: apply the family at x and then move along G f, or move along F f and then apply the family at y. Naturality says these two are equal: fmap_G f ∘ eta x == eta y ∘ fmap_F f. This equation is called the naturality square, because it says two sides of a square of arrows compose to the same thing.

Reverse is a natural transformation

rev reverses a list. As a family — one arrow list X -> list X for each type X — it is a natural transformation from the list functor to itself:

Lemma rev_naturality (X Y : Type) (f : X -> Y) :
  fn_equ (list X) (list Y)
    (fun l => map f (rev l)) (fun l => rev (map f l)).
Proof.
  intros l. apply map_rev.
Qed.

#[global] Instance RevNat :
  NaturalTransformation ListFunctor ListFunctor :=
  @Build_NaturalTransformation Type Type CoqCat CoqCat
    ListFunctor ListFunctor
    (fun X => @rev X)
    rev_naturality.

Here both functors are ListFunctor, so fmap on each side is map, and the family eta is rev. Writing the naturality square out with those substitutions, it says, for every f:

mapf∘rev=rev∘mapf\mathrm{map}\,f \;\circ\; \mathrm{rev} \;=\; \mathrm{rev} \;\circ\; \mathrm{map}\,f

Reverse a list and then map f over it, or map f over it and then reverse — the same list either way. Spelled out on an element l, that is map f (rev l) = rev (map f l), which is the standard-library fact map_rev. So the entire proof is one step: intros l; apply map_rev. The content is not the proof effort — it is the recognition that naturality of rev is exactly this familiar identity about reversing and mapping.

It runs. Mapping the successor function S over the reversed list [0; 1; 2] computes to [3; 2; 1], and both routes round the square agree on it:

Example rev_map_value :
  map S (rev (O :: S O :: S (S O) :: nil))
  = S (S (S O)) :: S (S O) :: S O :: nil.
Proof. reflexivity. Qed.

Example rev_naturality_value :
  map S (rev (O :: S O :: S (S O) :: nil))
  = rev (map S (O :: S O :: S (S O) :: nil)).
Proof. reflexivity. Qed.

Both close by reflexivity: the two sides are not just provably equal, they compute to the very same list, [3; 2; 1].
