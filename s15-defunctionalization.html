Defunctionalization: closures without closures

Suppose your target is a machine, or a language, with no first-class functions: it can branch, it can call a fixed set of named routines, it can build and take apart data, but it cannot make a new function at run time and pass it around. Plenty of real targets look like this — flat bytecode, a state machine, C without function pointers, a hardware description. And yet the program you want to run is higher-order: it builds functions on the fly, stores them, applies them later. How do you run a higher-order program on a first-order machine?

The answer, due to John Reynolds (Definitional Interpreters for Higher-Order Programming Languages, 1972) and given its modern form by Olivier Danvy and Lasse Nielsen (Defunctionalization at Work, 2001), is a change of representation called defunctionalization. The observation is simple: in any one program, all the functions that ever get created come from a finite list of lambda-abstractions written in the source. A function value at run time is always one of those abstractions, plus the values it captured from its surroundings. So represent each abstraction as a constructor of an ordinary datatype, let that constructor carry the captured values as fields, and replace every call f x with a call to one dispatcher — traditionally named apply — that looks at which constructor f is and does what that abstraction would have done. The function space becomes a datatype; application becomes a match. No functions survive.

This post builds a small higher-order program in the lambda calculus, defunctionalizes it, and proves the two versions compute the very same term.

The lambda calculus, in three sentences

The lambda calculus has three kinds of term: a variable, a function \x. body (an abstraction), and an application f a. We write variables as de Bruijn indices — a variable is a number counting how many enclosing \ you cross, from the inside out, to reach the one that binds it — so \x. \y. x is Abs (Abs (Var 1)) and the identity \x. x is Abs (Var 0). The datatype is exactly that:

Inductive tm : Type :=
  | Var : nat -> tm
  | Abs : tm -> tm
  | App : tm -> tm -> tm.

For this post tm is just the data our higher-order program builds; we will not need reduction, only that two terms come out equal.

A higher-order program: a term-builder

Here is a small program that is genuinely higher-order — its state is a function. It walks a list of instructions and, at each step, extends a continuation k : tm -> tm that records “the term context assembled so far”. Two instructions: IArg t says “apply the running result to t”, and ILam says “wrap the running result in a lambda”. At the end it feeds a base term to the continuation.

Inductive instr : Type :=
  | IArg : tm -> instr
  | ILam : instr.

Fixpoint build (base : tm) (is : list instr) (k : tm -> tm) : tm :=
  match is with
  | []             => k base
  | IArg t :: is'  => build base is' (fun r => k (App r t))
  | ILam   :: is'  => build base is' (fun r => k (Abs r))
  end.

The continuation k is a real function, created fresh at every step and handed to the next call. That is the higher-order feature we want to remove. And notice — this is the observation defunctionalization rests on — that the functions ever stored in k come from exactly three source abstractions:

  • the initial identity fun x => x, whatever the caller passes at the top;
  • fun r => k (App r t), built at an IArg t step, capturing t and the enclosing k;
  • fun r => k (Abs r), built at an ILam step, capturing the enclosing k.

Three shapes. That finiteness is what makes the technique work.

The datatype and the apply dispatcher

Turn each of the three shapes into a constructor of a first-order type cont, carrying exactly what that shape captured:

Inductive cont : Type :=
  | Kid  : cont
  | Kapp : tm -> cont -> cont
  | Kabs : cont -> cont.

Kid is the identity; Kapp t k is fun r => k (App r t) with its captured argument t and enclosing continuation k; Kabs k is fun r => k (Abs r) with its enclosing k. Now the one dispatcher. “Apply the continuation k to v” becomes a match on k doing, in each case, precisely what the original abstraction did:

Fixpoint apply_clo (k : cont) (v : tm) : tm :=
  match k with
  | Kid       => v
  | Kapp t k' => apply_clo k' (App v t)
  | Kabs k'   => apply_clo k' (Abs v)
  end.

Kid returns its argument, as fun x => x did. Kapp t k' forms App v t and hands it to the enclosing k', as fun r => k (App r t) did. Kabs k' forms Abs v and hands it on. This function is structurally recursive on the closure, so it is total — no fuel, no well-founded recursion. The defunctionalized program is the original walk with the function space replaced by cont and the final call replaced by apply_clo:

Fixpoint build_d (base : tm) (is : list instr) (k : cont) : tm :=
  match is with
  | []             => apply_clo k base
  | IArg t :: is'  => build_d base is' (Kapp t k)
  | ILam   :: is'  => build_d base is' (Kabs k)
  end.

Compare it line for line with build: k base became apply_clo k base, the lambda fun r => k (App r t) became the constructor Kapp t k, and fun r => k (Abs r) became Kabs k. There is not a single function value left in build_d — cont is plain data.

The two programs compute the same term

The claim to prove: build_d with the identity closure and build with the identity function return the same term.

Blackboard version. Write ⌈k⌉\lceil k \rceil for the function a closure stands for — ⌈𝖪𝗂𝖽⌉=λv.v\lceil \mathsf{Kid} \rceil = \lambda v.\,v, and so on. Then for every instruction list, running the defunctionalized walk under a closure kk equals running the higher-order walk under ⌈k⌉\lceil k \rceil:

build_d𝑏𝑎𝑠𝑒𝑖𝑠k=build𝑏𝑎𝑠𝑒𝑖𝑠⌈k⌉.\mathrm{build\_d}\;\mathit{base}\;\mathit{is}\;k \;=\; \mathrm{build}\;\mathit{base}\;\mathit{is}\;\lceil k \rceil.

That is the general fact; the top-level statement is the case k=𝖪𝗂𝖽k = \mathsf{Kid}, where ⌈𝖪𝗂𝖽⌉\lceil \mathsf{Kid} \rceil is the identity.

The bridge is a function denote that reads a closure back as the function it stands for — used only in the proof, never in the program:

Fixpoint denote (k : cont) : tm -> tm :=
  match k with
  | Kid       => fun v => v
  | Kapp t k' => fun r => denote k' (App r t)
  | Kabs k'   => fun r => denote k' (Abs r)
  end.

First, dispatching on a closure computes exactly what applying its denotation would — apply_clo is the first-order image of application. On the blackboard, for every closure kk and term vv,

apply_clokv=⌈k⌉v,\mathrm{apply\_clo}\;k\;v \;=\; \lceil k \rceil\,v,

proved by induction on the structure of kk: the 𝖪𝗂𝖽\mathsf{Kid} case is the definition of the identity, and each of the other two constructors unfolds, on both sides, to the same call on the enclosing closure.

Lemma apply_denote : forall k v, apply_clo k v = denote k v.
Proof.
  induction k as [| t k' IH | k' IH]; intros v; simpl.
  - reflexivity.
  - rewrite IH. reflexivity.
  - rewrite IH. reflexivity.
Qed.

Induction on the closure k. For Kid, both sides are v — reflexivity. For Kapp t k', both sides step to the k' case on App v t, and the induction hypothesis IH closes it; Kabs k' is the same on Abs v.

Now the general correctness, by induction on the instruction list:

Lemma build_defunc_correct : forall is base k,
  build_d base is k = build base is (denote k).
Proof.
  induction is as [| i is' IH]; intros base k; simpl.
  - apply apply_denote.
  - destruct i as [t |].
    + apply IH.
    + apply IH.
Qed.

The empty list: build_d base [] k is apply_clo k base and build base [] (denote k) is denote k base, so the goal is exactly apply_denote. For a non-empty list, split on the instruction. On IArg t, the defunctionalized side recurses with Kapp t k and the higher-order side with denote (Kapp t k) — but that is definitionally fun r => denote k (App r t), the very lambda build builds, so the two recursive calls line up and the induction hypothesis IH finishes. ILam is the same with Kabs k. The whole proof is two apply IHs because the datatype’s denote was chosen to match the source abstractions exactly.

The top-level equality is the identity-closure instance:

Corollary build_defunc_correct_id : forall base is,
  build_d base is Kid = build base is (fun x => x).
Proof.
  intros base is. exact (build_defunc_correct is base Kid).
Qed.

denote Kid is fun x => x, so this is build_defunc_correct at k := Kid, handed over directly.

One term, both ways

Take the instruction list [IArg (Var 1); ILam] over the base Var 0. The higher-order build threads: IArg (Var 1) makes fun r => App r (Var 1), then ILam makes fun r => App (Abs r) (Var 1), then base is fed in, giving App (Abs (Var 0)) (Var 1). The defunctionalized build_d instead accumulates the closure Kabs (Kapp (Var 1) Kid) and lets apply_clo unwind it to the same term. As a chain of equalities, with each step’s justification alongside:

build_dv0[𝖨𝖠𝗋𝗀v1;𝖨𝖫𝖺𝗆]𝖪𝗂𝖽=buildv0[𝖨𝖠𝗋𝗀v1;𝖨𝖫𝖺𝗆](λx.x){correctness at k=𝖪𝗂𝖽} \mathrm{build\_d}\;v_0\;[\mathsf{IArg}\,v_1;\ \mathsf{ILam}]\;\mathsf{Kid} \;=\; \mathrm{build}\;v_0\;[\mathsf{IArg}\,v_1;\ \mathsf{ILam}]\;(\lambda x.\,x) \quad\{\text{correctness at } k = \mathsf{Kid}\} =(λx.x)v1{computation} \;=\; (\lambda x.\,x)\,v_1 \quad\{\text{computation}\}

where v0,v1v_0, v_1 abbreviate the de Bruijn variables 00 and 11, and (λx.x)v1(\lambda x.\,x)\,v_1 is the term the run assembles, not a redex it reduces. The file states the same equality as a calc chain — defunctionalized run, then higher-order run, then the explicit term — each link checked, not asserted:

Example build_worked :
  build_d (Var 0) [IArg (Var 1); ILam] Kid = App (Abs (Var 0)) (Var 1).
Proof.
  calc (build_d (Var 0) [IArg (Var 1); ILam] Kid)
    == (build (Var 0) [IArg (Var 1); ILam] (fun x => x))
    == (App (Abs (Var 0)) (Var 1)).
  - apply build_defunc_correct_id.
  - reflexivity.
Qed.

The first link is the correctness corollary; the second is plain computation. The result App (Abs (Var 0)) (Var 1) is the identity lambda applied to Var 1 — a term the first-order build_d produced without ever holding a function.
