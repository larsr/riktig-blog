riktig #15 — Introducing channels: possibilistic, not probabilistic

Fifteenth post documenting the riktig project, and the proper opening of a new thread: communication channels, the telecom leg of the work. Every post starts from zero, so does this one.

What we are doing, and why. Verified software in Coq, built for modifiability. So far the project has proved things about numbers, vector spaces, and programs. Today it starts proving things about wires — the physical or logical links that carry messages between machines, and that lose them, duplicate them, and reorder them, because that is simply what real networks do. The goal is the same as everywhere else in riktig: state the honest general fact, prove it, and let real protocols (TCP’s retransmission logic, QUIC’s per-stream reliability) inherit it instead of re-deriving it by hand each time.

The modeling decision that shapes everything downstream

Ask an engineer to describe a lossy channel and they’ll usually reach for a number: “packet loss rate is 2%.” That’s the right tool for capacity planning — but it is the wrong tool for a correctness proof. A probability doesn’t tell you what your protocol must do when the 2% case actually happens; it just tells you how often to worry about it. Worse, reasoning about probabilities forces you to pick a probability space — independence assumptions, distributions — baggage a correctness argument shouldn’t need to carry.

riktig makes a different choice, and it’s the choice this whole thread is built on: a channel is a relation between what was sent and what arrived. Not a number, not a distribution — a set of possible outcomes. “This channel might deliver the messages untouched, or drop some, or duplicate some, or all three” is captured completely by which pairs (sent, received) the relation contains — no probabilities anywhere.

This buys two things, and they’re the two things a communications engineer actually wants:

  • Safety proofs become unconditional. “The application never sees a corrupted or duplicated message” can be proved for every possible behavior in the relation — you don’t need to know how likely bad behavior is, only that your protocol handles it when it occurs.
  • Liveness proofs get an honest hypothesis. “Everything sent is eventually delivered” genuinely cannot be unconditional — a channel that eats every packet forever really does deliver nothing. So liveness proofs state exactly the assumption they need (a fairness condition — future posts), and probability, if it ever shows up again, only enters to justify that the fairness assumption holds in practice. It never touches the correctness argument itself.

The fault classes, as relations on lists

A conversation is a list of messages. In Channel/Trace.v every fault class is an inductively-defined relation between the sent list and the received list — literally a specification of which transformations the channel is allowed to perform.

Perfect — the trivial case, everything arrives exactly as sent:

Definition perfect s r : Prop := r = s.

Lossy — the received list is a subsequence of the sent one (elements kept in order, some dropped):

Inductive subseq : list M -> list M -> Prop :=
  | subseq_nil  : subseq [] []
  | subseq_keep : forall x r s, subseq r s -> subseq (x :: r) (x :: s)
  | subseq_drop : forall x r s, subseq r s -> subseq r (x :: s).

Definition lossy s r : Prop := subseq r s.

Read the three constructors as the three things a lossy wire can do at each step: deliver nothing more (nil), deliver the next message and continue (keep), or silently swallow the next message and continue (drop). That’s a complete description of “arbitrary packet loss, order preserved” — no counting, no probability, just which sequences of keep/drop are possible.

Duplicating — every received element traces back, in order, to a sent element, but a sent element may show up more than once:

Inductive dup : list M -> list M -> Prop :=
  | dup_nil     : dup [] []
  | dup_keep    : forall x s r, dup s r -> dup (x :: s) (x :: r)
  | dup_stutter : forall x s r, dup s (x :: r) -> dup s (x :: x :: r).

Lossy and duplicating together — the wire a real retransmission protocol actually faces, combining drop and stutter:

Inductive lossy_dup : list M -> list M -> Prop :=
  | lossy_dup_nil     : lossy_dup [] []
  | lossy_dup_keep    : forall x s r, lossy_dup s r -> lossy_dup (x :: s) (x :: r)
  | lossy_dup_drop    : forall x s r, lossy_dup s r -> lossy_dup (x :: s) r
  | lossy_dup_stutter : forall x s r, lossy_dup s (x :: r) -> lossy_dup s (x :: x :: r).

Reordering — no need to invent anything; this is exactly the standard library’s Permutation, wearing the domain’s name:

Definition reorder s r : Prop := Permutation s r.

A lattice, and the arrows that are genuinely missing

These classes aren’t independent — they nest. Every perfect run is also a lossy run (drop nothing); every lossy run and every duplicating run is also a lossy-and-duplicating run. The file proves the whole inclusion lattice as small lemmas — perfect_lossy, lossy_lossy_dup, dup_lossy_dup — each a short induction, each doing exactly what its name says.

The more interesting proofs are the ones showing an arrow is genuinely absent — that the lattice isn’t secretly a total order in disguise. Reordering is not a special case of losing-and-duplicating:

Example reorder_not_lossy_dup : ~ lossy_dup [0; 1] [1; 0].

Sending [0; 1] and receiving [1; 0] swaps their order — and a wire that can only drop and duplicate, never reorder, cannot produce a swap: once 0 is gone (dropped), it’s gone for good; stutter can only repeat something already delivered, never conjure 0 back after 1. The proof is a finite, mechanical case analysis (every branch of lossy_dup’s definition either doesn’t apply or leads to a contradiction), but the statement is the valuable part: it confirms lossy_dup really is a distinct, weaker notion than reorder, not an accident of how the lemmas happened to get proved.

What this sets up

Every one of these relations is now available as a hypothesis. A protocol’s safety theorem will read almost exactly like “for any wire such that lossy_dup (sender's packets) wire, the receiver never delivers anything corrupted” — a single forall over every possible bad thing the wire could do, proved once, holding forever. That is genuinely the whole plan for the rest of this thread: define what a channel is allowed to do, and let every protocol’s correctness proof be a statement about all the things in that set at once.

Next: protocol endpoints as state machines, composed through a channel relation into one end-to-end trace — and then the first real payoff, the alternating-bit protocol’s safety theorem.
