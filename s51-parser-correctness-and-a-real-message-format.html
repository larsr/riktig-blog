riktig — correct, real, propersource

What it means for a parser combinator to be correct, and using one on a real message format

A parser combinator library gives you small parser values — match one character, try this parser or that one, repeat a parser — and ways to glue them into bigger parsers. The appeal is that each piece is small enough to read on its own, and a parser for a whole grammar is just an assembly of pieces, not one large hand-written function that branches on everything at once. A separate piece of machinery then goes one step further: given a grammar written down as data (parsed from ABNF text, the notation RFC 5234 uses to describe text formats), a generic interpreter walks that data and builds a parser for it automatically — make_parser does the building, and parse_with_abnf wraps the whole pipeline so it can be handed the grammar as plain text.

Earlier posts built both of these — the combinator library and the grammar interpreter — and each was shown working: run on an input, output checked by hand. Neither asked the harder question: how do you know the combinators behave correctly on inputs nobody wrote a test for, or that a grammar-driven parser is guaranteed to work on formats nobody built by hand? This post has two parts, each opening with a question before any code appears: first, what would it even mean for a piece of parsing code to be correct, and can that be proved once instead of checked example by example; second, can a text format’s syntax be handed over as data, with a program building a working parser from it and none of the format’s own logic written down anywhere.

Part 1 — proving the combinators correct

What does it mean for parsing code to be correct?

Any piece of code that reads text and decides what it means can go wrong in two directions. It can claim success on text that does not actually have the shape being asked for — read a line like "not a date" and hand back a calendar date anyway, inventing a match where none exists. Or it can give up on text that does have the right shape — refuse a perfectly well-formed line because some case was missed when the code was written. A property ruling out the first kind of mistake — never inventing a match — is usually called soundness. A property ruling out the second — never refusing input that fits — is usually called completeness. A hand-written parser, built as one large function with a case for everything the author thought of, leaves both of these as hopes: the author believes every case was covered, but nothing checks it beyond whatever tests get written.

The question worth asking about a combinator library is whether these two properties can be proved differently — not per grammar, checked example by example, but once, about the small handful of building blocks, in a way that then holds automatically for every larger parser assembled out of them. That is what the rest of Part 1 does.

From the question to the actual proofs

The earlier post’s combinator library is proved sound and, where it makes sense, complete, one building block at a time, in Theory/Parser.v. Its opening comment states the coverage precisely, and it is worth being explicit about that scope before looking at a single proof: what gets proved is a property of the combinators — character-matching, sequencing, choice, repetition — not of any one grammar built from them. Proving that a parser assembled for some particular message format accepts exactly the strings a human reader would say that format allows is a different, larger claim about that specific parser, and it is not what is proved here. What is proved here is narrower and reusable precisely because of that: soundness is established once for the primitives and once for each way of combining them (sequencing, choice, repetition), so that any parser built by composing those pieces inherits soundness automatically, with no separate argument required for the parser itself.

A warm-up: exactly when the single-character combinator succeeds

The most basic combinator reads one item off the input and keeps it only if a predicate accepts it. “Correct” for something this small has an exact meaning: it should succeed in precisely the cases where there is a next item and the predicate holds of it, no more and no less. The file proves the two directions separately and then packages them as an if-and-only-if:

𝚜𝚊𝚝𝚒𝚜𝚏𝚢(seq,pred,s)=𝚂𝚘𝚖𝚎(c,rest)⇔𝚞𝚗𝚌𝚘𝚗𝚜(seq,s)=𝚂𝚘𝚖𝚎(c,rest)∧pred(c)=𝚝𝚛𝚞𝚎.\texttt{satisfy}(seq, pred, s) = \texttt{Some}(c, rest) \iff \texttt{uncons}(seq, s) = \texttt{Some}(c, rest) \land pred(c) = \texttt{true}.

  (* Soundness: a `satisfy` success really did read one `uncons` step
     matching the predicate. *)
  Lemma satisfy_sound : forall seq pred input c rest,
    satisfy seq pred input = Some (c, rest) ->
    uncons seq input = Some (c, rest) /\ pred c = true.
  Proof.
    intros seq pred input c rest H.
    unfold satisfy in H.
    destruct (uncons seq input) as [[c0 rest0] |] eqn:Hun; [ | discriminate].
    destruct (pred c0) eqn:Hpred; [ | discriminate].
    injection H as -> ->.
    split; [reflexivity | exact Hpred].
  Qed.

  (* Completeness: whenever `uncons` agrees with the predicate,
     `satisfy` does not spuriously fail. *)
  Lemma satisfy_complete : forall seq pred input c rest,
    uncons seq input = Some (c, rest) -> pred c = true ->
    satisfy seq pred input = Some (c, rest).
  Proof.
    intros seq pred input c rest Hun Hpred.
    unfold satisfy. rewrite Hun, Hpred. reflexivity.
  Qed.

  Lemma satisfy_spec : forall seq pred input c rest,
    satisfy seq pred input = Some (c, rest) <->
    uncons seq input = Some (c, rest) /\ pred c = true.
  Proof.
    intros seq pred input c rest. split.
    - apply satisfy_sound.
    - intros [Hun Hpred]. apply satisfy_complete; assumption.
  Qed.

Walking it: satisfy_sound unfolds satisfy to the match it actually is, and destructs that match — either reading the next item produced nothing (the proof stops there, discriminate closes it because the hypothesis claimed a success) or it produced a character that either failed or passed the predicate (again, failing closes by discriminate). Only one branch survives, and in it injection reads the value and leftover straight out of the equality — the success could only have arisen because a character really was read and really did pass the predicate. satisfy_complete is the converse, a single rewrite: if reading the next item and the predicate already agree, unfolding satisfy and rewriting both facts in immediately produces the claimed result. satisfy_spec is neither proof; it is the two glued together with split — soundness and completeness stated as one fact. The parser that accepts any character at all gets the same treatment for free, with the predicate hypothesis simply discarded.

This is the soundness-and-completeness question from the previous section, answered exactly, for the smallest possible piece of parsing code. The rest of Part 1 is about whether that answer survives being built into something bigger.

Soundness composes: prove it once for sequencing, get it for everything built from it

Call the soundness property this way, precisely: whatever a parser does, it never invents input out of nowhere, and whatever is left over when it succeeds is genuinely a suffix of what it started with — recoverable by prepending the exact prefix that was consumed. The file states this once, as a property of parsers in general, rather than a proof about any one of them:

(* "Whenever this parser succeeds, the leftover really is a suffix of
   the input" — witnessed concretely by the consumed prefix. This is
   the general form of "satisfy/many/etc. succeed, the leftover
   suffix really is a suffix of the original input" from the task
   description, stated once and composed through bind/ret/alt instead
   of re-derived combinator by combinator. *)
Definition ParserSound {T : Type} (p : Parser string T) : Prop :=
  forall s v s', p s = Some (v, s') -> exists prefix, s = prefix ++ s'.

This is the formal counterpart of “never claims a match on input it shouldn’t”: the definition itself demands a witness, a concrete prefix that was actually consumed, rather than merely asserting the leftover is somehow shorter. The gain from stating it this way is that it can be proved for sequencing once, as a fact about any two sound parsers glued together, rather than re-derived for every parser that happens to sequence two steps (which, since sequencing is how every combinator in the library threads work along, is nearly all of them). The blackboard argument: if the first parser’s success on s leaves rest, witnessed by s = prefix1 ++ rest, and the next step’s success on rest leaves s', witnessed by rest = prefix2 ++ s', then substituting the second equation into the first and reassociating ++ gives s = (prefix1 ++ prefix2) ++ s' — a single witness for the combined parser:

s=prefix1++rest,rest=prefix2++s′⇒s=(prefix1++prefix2)++s′.s = prefix_1 \mathbin{+\!\!+} rest, \quad rest = prefix_2 \mathbin{+\!\!+} s' \;\Longrightarrow\; s = (prefix_1 \mathbin{+\!\!+} prefix_2) \mathbin{+\!\!+} s'.

Lemma bind_ParserSound : forall T U (p : Parser string T) (f : T -> Parser string U),
  ParserSound p -> (forall x, ParserSound (f x)) -> ParserSound (bind p f).
Proof.
  intros T U p f Hp Hf s v s' H.
  unfold bind, Parser_HasBind in H.
  destruct (p s) as [[x rest] |] eqn:Hps; [ | discriminate].
  destruct (Hp s x rest Hps) as [prefix1 Heq1].
  destruct (Hf x rest v s' H) as [prefix2 Heq2].
  exists (prefix1 ++ prefix2).
  rewrite Heq1, Heq2, string_append_assoc.
  reflexivity.
Qed.

Walking it: unfolding sequencing (bind) and destructing the first parser’s result rules out its failure case (discriminate) and leaves the success case, where the first parser’s own soundness (Hp) hands back prefix1 and the continuation’s soundness (Hf, applied at the value and leftover the first parser actually produced) hands back prefix2. The witness for the whole thing is just their concatenation, and string_append_assoc — the one associativity fact about ++ this argument needs, proved earlier in the file by a routine induction — turns the two separate equations into the single one the goal asks for. Choice between two parsers gets a companion proof by the same shape, case-splitting on whether the first branch matched:

Lemma alt_ParserSound : forall T (p q : Parser string T),
  ParserSound p -> ParserSound q -> ParserSound (alt p q).
Proof.
  intros T p q Hp Hq s v s' H.
  unfold alt, Parser_HasAlt in H.
  remember (p s) as ps eqn:Hps.
  destruct ps as [[v0 s0] |].
  - simpl in H.
    injection H as Heqv Heqs'.
    rewrite <- Heqs'.
    exact (Hp s v0 s0 (eq_sym Hps)).
  - simpl in H.
    exact (Hq s v s' H).
Qed.

Walking the choice proof: alt p q runs p first, so the case split remember (p s) ... destruct mirrors the definition — if p succeeded, the whole result is p’s result and p’s own soundness witness serves unchanged; if p failed, the result is q’s and q’s witness serves. In symbols, whichever branch actually ran supplies the prefix: s=𝑝𝑟𝑒𝑓𝑖𝑥++s′s = \mathit{prefix} \mathbin{+\!\!+} s' either way.

With sequencing and choice both proved sound, every combinator built out of them inherits soundness essentially for free — no new induction is needed, only composition. Running two parsers and keeping the first result is defined as two nested sequencing steps, seqLpq=bindp(λx.bindq(λ_.retx))\mathrm{seqL}\;p\;q = \mathrm{bind}\;p\;(\lambda x.\; \mathrm{bind}\;q\;(\lambda \_.\; \mathrm{ret}\;x)), and its soundness proof is exactly that composition spelled out:

Lemma seqL_ParserSound : forall T U (p : Parser string T) (q : Parser string U),
  ParserSound p -> ParserSound q -> ParserSound (seqL p q).
Proof.
  intros T U p q Hp Hq. unfold seqL. apply bind_ParserSound.
  - exact Hp.
  - intros x. apply bind_ParserSound.
    + exact Hq.
    + intros _. apply ret_ParserSound.
Qed.

Every step is applying a lemma already proved; nothing here re-examines what the two sub-parsers actually do. Even repetition — the combinator that repeats a parser until it stops matching — inherits soundness by induction on the counter that bounds its recursion, using the two lemmas above at each step rather than reasoning about strings directly. On the blackboard the induction has exactly two cases:

manyp0=ret[]andmanyp(n+1)=alt(p then manypn)(ret[]),\mathrm{many}\;p\;0 = \mathrm{ret}\;[\,] \qquad\text{and}\qquad \mathrm{many}\;p\;(n+1) = \mathrm{alt}\;(\,p \text{ then } \mathrm{many}\;p\;n\,)\;(\mathrm{ret}\;[\,]),

the first sound because returning consumes nothing, the second sound by the alt and bind lemmas applied to pp’s soundness and the induction hypothesis.

Lemma many_ParserSound : forall T (p : Parser string T) fuel,
  ParserSound p -> ParserSound (many p fuel).
Proof.
  intros T p fuel Hp.
  induction fuel as [| c fuel' IH]; simpl.
  - apply ret_ParserSound.
  - apply alt_ParserSound.
    + apply bind_ParserSound.
      * exact Hp.
      * intros x. apply bind_ParserSound.
        -- exact IH.
        -- intros xs. apply ret_ParserSound.
    + apply ret_ParserSound.
Qed.

The induction hypothesis IH — “repetition is sound at the shorter counter” — is exactly what the recursive call needs, so the base case (returning immediately, which consumes nothing) and the recursive case (composed entirely from the two composition lemmas above) close the whole thing without a single new fact about strings. This is the sense in which soundness holds “once for the combinators, inherited by everything built from them”: a grammar with fifty rules built from sequencing, choice, and repetition gets this guarantee automatically, with no per-grammar proof obligation at all.

The completeness half: does the repetition counter ever cut things off early?

Repetition is written to recurse on a copy of the input, used purely as a bound on the iteration count, because Coq’s termination checker needs to see something shrink structurally at every call and the actual amount of input the sub-parser will consume isn’t known in advance. This raises exactly the completeness question from the first section, in a new form: does that artificial counter ever cut the loop off early — stop matching before the real input runs out, for no reason but that the counter reached zero first?

The first fact rules out one kind of problem directly: repetition never returns failure, no matter what the sub-parser does — for every counter nn and input ss there is some result rr with manypns=Some(r)\mathrm{many}\;p\;n\;s = \mathrm{Some}(r).

Lemma many_never_fails : forall T (p : Parser string T) fuel input,
  exists r, many p fuel input = Some r.
Proof.
  intros T p fuel input.
  destruct fuel as [| c fuel'].
  - simpl. apply ret_always_succeeds.
  - simpl. apply alt_always_succeeds_of_r. apply ret_always_succeeds.
Qed.

This is a case split on the counter alone: at zero, the definition unconditionally returns an empty list, and at a nonzero counter the outer combinator is a choice whose fallback branch also returns an empty list — and returning cannot fail, whatever branch runs. That rules out outright failure, but it does not yet say the count of repetitions matched is the right one. For that, the file introduces a stronger property that the library’s actual character-matching parsers all share: a success consumes exactly the one character at the head of the string it was run on, no more and no less.

(* `HeadConsuming`: the property `satisfy`/`any`/`exact`/`char` all
   share, that this codebase's uses of `many`/`many1` actually feed
   them — a success consumes exactly the head character of whatever
   string it is run on. Strictly stronger than `ParserSound` (which
   only asks for *some* consumed prefix): it pins down the leftover
   exactly, which is what lets the fuel/actual-input split collapse
   below. *)
Definition HeadConsuming {T : Type} (p : Parser string T) : Prop :=
  forall s v s', p s = Some (v, s') -> exists c, s = String c s'.

For a sub-parser with that property, calling repetition the way this codebase actually calls it — with the counter argument set to a literal copy of the real input — unfolds cleanly into exactly one step of matching, with no leftover bookkeeping about the counter at all:

𝚖𝚊𝚗𝚢pss={𝚂𝚘𝚖𝚎([],s)if ps=𝙽𝚘𝚗𝚎𝚂𝚘𝚖𝚎(v::xs,rest) where 𝚖𝚊𝚗𝚢ps′s′=𝚂𝚘𝚖𝚎(xs,rest)if ps=𝚂𝚘𝚖𝚎(v,s′)\texttt{many}\ p\ s\ s \;=\; \begin{cases} \texttt{Some}([\,],\,s) & \text{if } p\ s = \texttt{None} \\ \texttt{Some}(v :: xs,\,rest) \text{ where } \texttt{many}\ p\ s'\ s' = \texttt{Some}(xs,\,rest) & \text{if } p\ s = \texttt{Some}(v,\,s') \end{cases}

(* Single-step unfolding of the canonical (fuel = actual input) call:
   match `p` once against `s`, and on success the continuation is
   *exactly* `many p s' s'` at the leftover `s'` — no fuel bookkeeping
   left over, because `HeadConsuming` pins the leftover down to be
   literally the tail of `s`, which is also the fuel's own tail. *)
Theorem many_canonical_unfold : forall T (p : Parser string T), HeadConsuming p ->
  forall s, many p s s =
    match p s with
    | None => Some ([], s)
    | Some (v, s') =>
      match many p s' s' with
      | Some (xs, rest) => Some (v :: xs, rest)
      | None => None
      end
    end.
Proof.
  intros T p Hhc s.
  destruct s as [| c s0].
  - destruct (p "") as [[v s'] |] eqn:Hp.
    + destruct (Hhc _ _ _ Hp) as [c Heq]. discriminate Heq.
    + reflexivity.
  - destruct (p (String c s0)) as [[v s1] |] eqn:Hp.
    + destruct (Hhc _ _ _ Hp) as [c' Heq].
      injection Heq as _ Hs01.
      rewrite <- Hs01 in Hp.
      rewrite <- Hs01.
      destruct (many_step_some T p c s0 (String c s0) v s0 Hp) as (xs & rest & Hmfuel & Heq').
      rewrite Hmfuel. exact Heq'.
    + exact (many_step_none T p c s0 (String c s0) Hp).
Qed.

Walking it: split on whether s is empty. If it is, the sub-parser cannot possibly succeed on the empty string while consuming exactly one head character (a success there would need to exhibit a head character of the empty string, which discriminate rejects), so both sides reduce to “stop, empty list” and agree by reflexivity. If s is String c s0, split on whether the sub-parser matches: on failure, an earlier helper lemma (many_step_none, proved a few lines above by unfolding one step of choice and sequencing) already shows both sides equal Some ([], s). On success, consuming exactly one head character means the leftover s1 must literally equal s0 — the counter’s own tail — so a second helper (many_step_some) lines the recursive call up exactly with repetition applied again at that same leftover. Nowhere does the counter run short of what the real input needs: because each successful match consumes one real character and peels one character off the counter in lock step, the two never drift apart, and the recursion bottoms out only when the real input does. That answers the completeness question posed above: the counter is not an approximation of the real stopping point, it tracks it exactly.

What this proof does not cover

Two boundaries are worth stating outright. First, the combinator that parses a separator-delimited list — one item, then repeatedly a separator followed by another item — gets the same soundness proof as repetition, but not the same clean completeness argument. The reason is concrete: each repetition of that combinator consumes from two sub-parsers, the separator and the item, before the counter is decremented by only one character, so the “counter’s tail equals the real leftover” argument above, which relies on consuming exactly one character per step, does not carry over — a correct version of that argument would need to track both sub-parsers’ consumption together, and the file leaves that case unproved rather than stating a weaker claim under the same name. Second, and more fundamental: nothing here connects a combinator-built parser back to an independent description of “the language this grammar describes.” What is proved is about the building blocks — matching a character, sequencing, choice, repetition — not about the claim that a particular grammar, run through the interpreter in Part 2, accepts exactly the strings a reader of that grammar would expect. That would be a separate, larger theorem, connecting the interpreter’s behavior to an independent specification of ABNF’s meaning, and it is not attempted here.

Part 2 — using a grammar to build a parser automatically

Can a grammar stand in for hand-written code?

Suppose a new text format needs to be read: a line in a configuration file, a header in a network request, a record in a log file. The ordinary way to get a parser for it is to write one — functions that inspect the text piece by piece, one function for each part of the format, glued together and tested by hand. There is another possibility. Describe the format’s syntax once, as a set of rules — this token, then that token, one of several alternatives, zero or more repetitions of something, one rule referring to another — and hand those rules to a single, completely generic program that reads them and builds a working parser on the spot, with none of the format’s own logic written down as code anywhere. Does that actually hold up for a format with real structure — several rules referring to each other, repetition, alternation — or only for a single rule matching one character?

From the grammar to a parser

An earlier post built exactly the generic program the question above asks for: give it a grammar (already parsed into data) and the name of its starting rule, and it returns a working parser, having written no code specific to that grammar anywhere along the way — make_parser does the building, and parse_with_abnf wraps the whole thing so the grammar can be handed over as ordinary text rather than pre-parsed data. That was demonstrated there on a grammar with a single rule. Here is a grammar with the structure real formats actually have: an HTTP-style header line, the kind of thing that appears one per line in a network request, combining alternation, repetition, concatenation, and rules that refer to each other. Written the way a standards document would present it:

header = name ":" SP value CRLF
name   = 1*ALPHA
value  = 1*(ALPHA / DIGIT / SP)

One sentence per rule: a header is a name, then a literal colon, then a single space, then a value, then a line ending; a name is one or more alphabetic characters; a value is one or more characters that are each either a letter, a digit, or a space. ALPHA, DIGIT, SP, and CRLF are not defined anywhere in this grammar — they are three of RFC 5234’s built-in core rules (letters, digits, the space character, and a line ending respectively), which the interpreter recognizes by name without the grammar author having to spell them out.

Here is that same grammar as it actually appears in the file, encoded as a Coq string. ABNF requires each rule on its own line, so the grammar text is built by concatenating each rule with a line-feed character (crlf_s, defined once and reused) between them; a literal double-quote character inside a Coq string is written as two consecutive double quotes, so "":"" inside the source below is just the two-character ABNF token ":":

(* Test: a small realistic message format, not a single-feature toy --
   an HTTP-style header line ("Host: example com"), combining
   alternation, repetition, concatenation, rule references, and three
   different core rules (ALPHA, SP, CRLF) in one grammar. *)
Definition test_grammar_header :=
  ("header = name "":"" SP value CRLF" ++ crlf_s ++
   "name = 1*ALPHA" ++ crlf_s ++
   "value = 1*(ALPHA / DIGIT / SP)" ++ crlf_s)%string.

Example test_interp_header :
  parse_with_abnf test_grammar_header "header" ("Host: example com" ++ crlf_s) 30 =
    Some (Node "header"
      (Node "name" (Leaf "H" :: Leaf "o" :: Leaf "s" :: Leaf "t" :: nil)
       :: Leaf ":"
          :: Leaf " "
             :: Node "value"
                  (Leaf "e" :: Leaf "x" :: Leaf "a" :: Leaf "m" :: Leaf "p" ::
                   Leaf "l" :: Leaf "e" :: Leaf " " :: Leaf "c" :: Leaf "o" ::
                   Leaf "m" :: nil)
                :: Leaf crlf_s :: nil)).
Proof. reflexivity. Qed.

parse_with_abnf is the two-stage pipeline described above: it first parses the grammar text itself into structured data, then hands that data to make_parser to build an actual parser function on the fly, and runs that parser on the input string — here, "Host: example com" followed by a line feed, standing in for one line of an HTTP request. No parse_header function was written by hand anywhere in this codebase; the grammar text above is the entire specification, and the proof that it produces the stated tree is reflexivity — Coq runs the whole pipeline (parse the grammar, build the parser, run it on the input) and checks the result against the literal value written down.

Read the output tree, and its shape says something worth noticing. name and value are rules the grammar author actually wrote, and each becomes its own labelled Node, wrapping the individual characters it matched (Node "name" [Leaf "H"; Leaf "o"; Leaf "s"; Leaf "t"] for "Host", and similarly for value matching "example com", all eleven characters — including the interior space — kept as individual leaves, because value’s rule allows a space anywhere a letter or digit would go). The colon and the space between name and SP appear as bare Leafs spliced directly into header’s children, with no wrapping node — a quoted literal and a core rule are not user-written rules, so the interpreter has no name of the grammar author’s choosing to label them with. And the trailing Leaf crlf_s rewards a second look: the test input ends in a single line-feed byte, not a full carriage-return-then-line-feed pair, and it still matches CRLF — RFC 5234’s core rule for line endings accepts any of the three common conventions (CR LF, bare LF, bare CR), and that acceptance happens automatically because CRLF is one of the core rules built into the interpreter, not something the grammar author had to reimplement.

That answers the question this part of the post opened with. Writing the grammar as text, in the same style a standards document would use, naming the rules that are needed and relying on the core rules RFC 5234 already supplies for common building blocks (letters, digits, whitespace, line endings), and handing the text to parse_with_abnf together with the input to read, is enough — the parser itself is never written by hand, only built, once per call, by walking the grammar’s own structure.
