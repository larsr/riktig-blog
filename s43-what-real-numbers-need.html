What real numbers need: least-upper-bound completeness

A field is a set with +, *, a zero, a one, negation, and inverses of everything except zero, obeying the ordinary arithmetic laws — the rationals Q are the standard example. An ordered field adds a total order <= that is compatible with the arithmetic: adding the same thing to both sides preserves the order, and the product of two nonnegative elements is nonnegative. The rationals are an ordered field too. So what separates the real numbers from the rationals is not any of the algebra and not the order — it is one further property, which this post is about.

The property Q is missing

Take the set of rationals whose square is at most two:

P={q∣q⋅q≤2}.P = \{\, q \mid q \cdot q \le 2 \,\}.

This set is inhabited (it contains 1) and it is bounded above (by 2, say — nothing in it exceeds 2). Ask for its least upper bound, the smallest rational that sits above every element of P. There isn’t one. Any rational upper bound can be nudged down a little and still be an upper bound, closing in on 2\sqrt 2 from above without ever reaching it, because 2\sqrt 2 is not rational. The candidate boundary falls outside the number system. That single failure — a bounded, inhabited set with no least upper bound — is the one thing that makes Q “gappy” and the reals “complete”.

Least-upper-bound completeness is the axiom that closes the gap: every inhabited, bounded-above set does have a least upper bound (a supremum) inside the system. That, plus the ordered-field structure, is the whole specification of the real numbers.

The specification, in full

Here is the entire class, verbatim from Interface/Reals.v. It takes a type R that is already a Field and asks for a compatible total order and the completeness axiom:

Class Reals (R : Type) `{Field R} := {
  (* Total order, compatible with the setoid equality. *)
  r_le         : Rel R;
  r_le_proper  : Proper (equ ==> equ ==> iff) r_le;
  r_le_refl    : forall x : R, r_le x x;
  r_le_trans   : forall x y z : R, r_le x y -> r_le y z -> r_le x z;
  r_le_antisym : forall x y : R, r_le x y -> r_le y x -> x == y;
  r_le_total   : forall x y : R, r_le x y \/ r_le y x;

  (* Ordered-field compatibility. *)
  r_le_add_compat : forall x y z : R, r_le x y -> r_le (x + z) (y + z);
  r_le_mul_compat : forall x y : R, r_le 0 x -> r_le 0 y -> r_le 0 (x * y);

  (* Least-upper-bound completeness: every inhabited set bounded above
     has a supremum.  This is the axiom Q cannot satisfy — the set
     { q | q * q <= 2 } is inhabited and bounded but has no rational
     supremum — and the one existence theorems (IVT, MVT) consume. *)
  r_complete : forall P : R -> Prop,
    (exists x : R, P x) ->
    (exists b : R, forall x : R, P x -> r_le x b) ->
    exists s : R,
      (forall x : R, P x -> r_le x s) /\
      (forall b : R, (forall x : R, P x -> r_le x b) -> r_le s b)
}.

Read it in three groups. The first six fields introduce the order r_le and its laws: it respects the field’s own equality == (r_le_proper), it is reflexive, transitive, antisymmetric, and total — that is exactly what makes <= a total order rather than a partial one. The next two fields, r_le_add_compat and r_le_mul_compat, are the compatibility conditions that turn “a field with an order” into “an ordered field”: you may add the same z to both sides of an inequality, and a product of two nonnegative elements stays nonnegative.

The last field, r_complete, is the real content. Read it as a sentence about an arbitrary set P : R -> Prop. If P is inhabited (exists x, P x) and bounded above (exists b above every element), then there exists an s that is (i) an upper bound — forall x, P x -> r_le x s — and (ii) the least one: any other upper bound b satisfies r_le s b, so s sits below every competitor. Those two conjuncts are the definition of “least upper bound”, written out with no appeal to anything outside the class.

No instance — on purpose

One thing must be said plainly. Interface/Reals.v is the file quoted above and nothing else: it states the class and closes with one line,

#[global] Hint Mode Reals ! - : typeclass_instances.

which only tells Coq’s instance search to wait until it knows the carrier type R before trying to solve a Reals goal. There is no Instance anywhere in the file. Nowhere does the file exhibit a concrete type and prove it satisfies Reals. The specification says what a real-number system must satisfy and stops there.

That is deliberate, and it is worth being clear about why it is a reasonable choice rather than an unfinished one.

Building an actual model of Reals means constructing the reals — as Dedekind cuts, or as equivalence classes of Cauchy sequences of rationals, or by importing Coq’s standard-library R — and then discharging every field of the class for that construction. Each of these is a substantial, self-contained development in its own right, and they have essentially nothing to do with the things one proves using completeness. The intermediate value theorem, the mean value theorem, the existence of suprema in any argument that needs them — these are statements about any type satisfying Reals, and their proofs quote r_complete and the order laws by name, never the internal machinery of cuts or sequences. The file comment names this split directly: “Impl/ Rstd.v (stdlib R, classical) and later a constructive Cauchy build are the intended models; everything downstream programs against this class only.”

The reward for the split is concrete. Anything proved generically against the Reals class — with a Context {R : Type} `{Reals R} at the top, the way Theory/Exp.v proves things against Exp — holds automatically for every model the class will ever have. Prove the intermediate value theorem once from r_complete, and it is simultaneously a theorem about the classical stdlib reals and about the constructive Cauchy reals, the moment either instance is supplied. The construction is a plug fitted later into a socket whose shape is already fixed; the theorems are wired to the socket, not the plug.

So this short file has exactly one thing to show, and it has shown it: the eight order-and-compatibility laws and the one completeness axiom that, together, are the definition of a real-number system — with the construction left as the separate task it genuinely is.
