I have three more tasks i want to add.

  1. for the lambda calculus i want to define a function that converts a lambda term, to one where all function calls are made with let-bound variables as arguments, i.e defined in let bindings higher up. this is done with a set of conversions like defunctionalisation, making functions take all argumentes, i.e not have any bound variables inside them, and instead have a context argument, etc. the conversion should be done as recursive rewrite on the nodes of the lambda term tree. finally the term should be a 1-to-1 mapping into another type that enforces that all function calls are in the form we just said - variable for function and variables for the arguments. then it is easy to convert the program to machine code. note, there are a few functions that will actually need to exist, like reading /writinga word/byte from an address, add words/bytes, etc. but all of the lambdas defined in the original program should disappear by this transform.

  2. I read a paper by Stephanie Weirich where she made a very efficient modification to substitution. instead of doing the full substitution everytime, one could replace the recursive call with essentially a thunk that remembered what should be substituted, and it was only expanded when needed. that means that two consecutive substitutions could sometimes make them cancel out if the second one changed the first one. And one could also have a more efficient data structure to see if a variable matched any of the perhaps many substitutions that were applied with just one lookup. This idea of delaying the work further down in the tree and having an algebra for combining and reducing the work task feels like it should be possible to express in a more abstract way, like withsome kind of advanced fold and monoidal rule … please investigate and implement and write this particular usecase as an example of how to use that. also start with a simpler example first to explain the idea. I think this is a very useful pattern to make functional code not having to traverse the same datastructure again and again. its a bit like the map f . map g = map (f . g), right? but for a Traversable ?

  3. I looked at the formalization of Geometric Algebra and it didn’t quite look like i expected. When i have read about geometric algebra it has shown properties about the wedge and dot product and how they can define the geometric product. Perhaps one can define it in more ways, but at least we should be able to show that they are equivalent. And then there has been proofs that multiplying the wedge product of different base vectors become multi-vectors or blades or what’s it called. and how the geometric algebra vectors are a 2^d dimensional vector field constructed on an underlying d dimensional vector space. And how the geometric 2d space is isomorphic to the complex numbers. And then the inner product was tied into the norm and the rules about how to rotate vectors by rotors. Please create a blog post of these things, first in regular notation, and then a corresponding coq implementation of the same calculations. And then we can write a program that takes a rigid body of straight beams and connections that can either rotate or fold the joint 360 deg (without gimble lock problems), and with geometric algebra it becomes easy to chain them together to model a robot so we can compute the position of its limbs as a function of the steering angles.
