1. Equivalences catalog — docs/EQUIVALENCES.md

A single file listing every proven equation/rewrite rule in the library — every Theorem/Lemma whose statement is an == (or = for the Channel/Lambda tracks, which don’t use the Ops setoid) or an iff — grouped by theory area (algebra foundations, linear algebra, category theory, lambda calculus, analysis, channels). Status: done, docs/EQUIVALENCES.md, ~250 entries.

3. Architecture audit — Interface/Theory/Impl trichotomy

Requested 2026-07-08 alongside a suspicion of “slide away from the spec/theory/impl separation.” Investigated by reading all 78 files under Interface/, Theory/, Impl/ plus the topic-based Channel/Lambda tracks against docs/DESIGN.md’s stated rule. Verdict: the core algebra tower (Monoid through VectorSpace/InnerProduct/Order/Norm) holds the trichotomy cleanly; real drift is concentrated in the faster-moving linear-algebra wave (Basis/Bilinear/Integral/LinearMap hosting undisclosed Theory content in Interface/; no Interface/Determinant.v, forcing Impl/Det.v and Impl/Elim.v to carry general theorems; Theory/Hom.v and Theory/VSIso.v importing Impl, reversing the stated dependency direction). Channel/ and Lambda/’s topic-based organization is confirmed a deliberate, sound, different discipline — not drift. Full itemized findings and fixes: docs/LINALG-ROADMAP.md, Wave L8. Status: done (investigation); the fixes themselves are queued, non-blocking quality debt.

2. “Getting started” blog post — hands-on examples per domain

A blog post teaching how to actually use the library from a fresh import — the real pain point: with no Require Export tower (a deliberate house-style choice, unlike the original memprog-era development), it isn’t obvious what to import to get going. The post should: - explain the Import-vs-Export tradeoff explicitly: targeted Require Import avoids name clashes (enabled by the per-file prefix convention — PW_, LV_, C_, Hom_, …) at the cost of the reader needing to know what to ask for; give a short cheat sheet of “if you want X, import Y” for the common cases; - give one worked example per major domain (algebra/linear algebra, category theory, lambda calculus, channels — pick 1-2 more if a clean one exists), each following the same shape: state something in the abstract layer (an interface/typeclass), then prove a concrete instance satisfies it — the “descend to implementation only when needed” discipline the whole project is built on. - every example must actually compile; verify before writing prose around it.

Status: done, blog #19 (0019-getting-started-five-imports.md). Found and fixed a real bug along the way: Theory/Category.v’s Arguments iso_to {x y}. etc. were section-local, not Global, so they silently reverted to fully-explicit on export — now fixed.
