Analysis roadmap — from instances to flervariabelanalys

Piece-by-piece order for growing the theories toward multivariable calculus (Jacobian, div, curl, line and surface integrals) and space-time. Each stage is small, independently landable, and only depends on the stages before it. Nothing here is started until the category/lambda track is done; this document exists so every piece is introduced in its logical place.

Design bets already placed (why this will go smoothly)

  • Setoid equality everywhere. Constructive real numbers are equivalence classes of Cauchy sequences — their == is not Coq’s eq. Our whole hierarchy is already stated over EqRel, so reals are just another instance, not a rework.
  • InnerProduct has no positivity axiom (deliberate, noted in the interface). A symmetric bilinear form of signature (+,−,−,−) is therefore already an InnerProduct — space-time needs no new structure, only an instance. Positivity becomes an add-on mixin when order arrives (stage 2).
  • GeometricAlgebra is in place. Div and curl are the two halves of one GA product (∇F = ∇·F + ∇∧F), and the spacetime algebra is GA over Minkowski. The vector-calculus zoo collapses into structures we already have.

Stage 1 — Instances and finite sums (no new analysis)

Instantiate what exists; meet the first genuinely new tool.

  • Impl/ListMonoid.v — lists with ++: first non-commutative monoid (shows why Monoid ≠ AbGroup).
  • Impl/Qf.v — stdlib Q as the first Field instance (decidable, axiom-free).
  • Theory/Pointwise.v — the workhorse: if G is a Monoid/Group/Ring/VectorSpace, then so is X -> G, pointwise. One general construction, many structures for free.
  • n-dimensional vectors as Fin n -> F — a special case of Theory/Pointwise.v; VectorSpace F (Fin n -> F) costs nothing.
  • Theory/BigOp.v — finite indexed sums over a monoid (fold over Fin n). Prerequisite for: inner product on Fin n -> F (⟨u,v⟩ = Σ uᵢvᵢ), Basis, matrices, the Jacobian.

Stage 1½ — Representation isomorphisms (swap data layouts by rewriting)

The bridge between the algebra and category tracks, and the foundation for every later “same computation, different layout” proof.

  • Theory/UnitVS.v — unit is the zero space ({0}); categorically a zero object of Vect: both Initial and Terminal (state it with the category interfaces — Vect differs from Set here).
  • Theory/ProdVS.v — A × B componentwise; the categorical product.
  • Interface/LinearMap.v + VSIso — structure-preserving maps and isos; an iso yields rewrite equations (vsi_fromto, vsi_tofrom, Proper instances) so representations are swapped by setoid rewriting, mid-proof.
  • Theory/VSIso.v — the monoidal kit: vsiso_unitor (A × unit) ≅ A, vsiso_braiding (A × B) ≅ (B × A), vsiso_associator ((A×B)×C) ≅ (A×(B×C)); transport of VectorSpace structure along a VSIso.
  • Normalization by computation: a Fixpoint mapping any pair-tree of spaces to its right-nested normal form, composing the three isos on the way — so ((A,B),(C,(D,E))) ≅ (A,(B,(C,(D,(E,()))))) is computed, not proved per shape (executable coherence).
  • Impl/ListVS.v — the showpiece: list R with == = “equal up to trailing zeros” is the finitely-supported sequences ⊕_ℕ R — a real vector space (raw lists under eq are NOT one; the setoid design is what makes this work). Prove list R ≅ colimit of (Fin n -> R) informally via: fixed-length Fin n -> R ≅ n-fold nested pairs (by normalization) and zero-padding embeddings.
  • Same enc/dec shape as the Codec layer, but two-sided: representation isos are the lossless special case of the channel-stack prisms — computations conjugate across them freely.
  • The motivating application: AoS ↔︎ SoA. A table of records is a record of columns: (Fin n -> A × B) ≅ (Fin n -> A) × (Fin n -> B) — the product distributing over the container, proved once as one more iso in this family (vsiso_soa, with soa_split/soa_merge as its to/from). A program written against the abstract container interface transports across the layout change with its proofs; changing list-of-structs to separate column arrays becomes a rewrite, not a rewrite-everything. (Raw list (A × B) needs the equal-length side condition — length indexing or the setoid handles it, as with ListVS.)
  • Feeds stage 6 directly: reshape/tiling/flash-attention equivalences are rewrites along these isos (for ⊗ as well as ×); AoS↔︎SoA is the same move applied to memory layout, which is where the memprog README’s “change the data representation without fear” is cashed in.

Stage 2 — Order

  • Interface/Order.v — HasLe operational class, Poset, total order.
  • Interface/OrderedRing.v, OrderedField.v — compatibility laws (a <= b -> a + c <= b + c; products of nonnegatives).
  • Theory/Abs.v — absolute value from order + neg; triangle inequality. This is the metric raw material.
  • PositiveDefinite mixin for InnerProduct (needs order): the real inner product; Cauchy–Schwarz becomes provable.

Stage 3 — Norms and linear maps

  • Interface/Norm.v — HasNorm + laws (definiteness, homogeneity ‖a *: u‖ == |a| * ‖u‖, triangle). Axiomatized separately rather than derived from norm2 (deriving needs square roots, i.e. completeness — don’t couple the stages).
  • Interface/LinearMap.v — structure-preserving maps between vector spaces (the categorical arrows of the linear world; also what a total derivative is in stage 5).

Stage 4 — Reals (the one real decision point)

State the interface first: CompleteOrderedField (ordered field + completeness — least-upper-bound, or Cauchy-completeness; pick when we get there, they’re interderivable classically).

Two implementations, in order:

  1. Impl/Rstd.v — Coq stdlib R. Pragmatic, classical. The stdlib’s axioms stay confined to this Impl file; interfaces and theories remain axiom-free, so the “no Axiom” rule holds where it matters — anything proved against the interface doesn’t care which model.
  2. (later, optional) constructive Cauchy reals over Q — our setoid hierarchy is already shaped for it.

Complex numbers follow immediately: Impl/Cx.v as pairs over any ordered field with i² = −1 — Field instance + 2-dim real vector space. Hermitian inner products want a HasConj operational class with involution laws (a small *-ring structure — one interface file). GA note for the blog: ℂ is the even subalgebra of GA(2).

Stage 5 — Limits, continuity, derivative

ε-δ over normed spaces (filters are overkill at our scale):

  • limit_at f a L := forall eps, 0 < eps -> exists delta, ... in any normed space over an ordered field. No completeness needed to define limits — only to prove existence theorems. So this stage only depends on stages 2–3.
  • Theory/Limits.v — algebra of limits (sum, scalar, product), the rewrite-friendly workhorse lemmas.
  • Continuity := limit equals value; composition of continuous maps.
  • Derivative, twice:
    1. one variable: f' a as limit of the difference quotient (Field division — that’s why Field is in the tower);
    2. total derivative as a LinearMap (Fréchet): f'(a) : V →ₗ W the best linear approximation. The Jacobian is not a new concept — it is this linear map written in a basis (stage 1 BigOp + Basis pay off here).
  • Existence theorems (IVT, MVT) — the first consumers of completeness.

Stage 6 — Flervariabelanalys

  • Partial derivatives = total derivative applied to basis vectors.
  • Jacobian = matrix of the total derivative (Basis + BigOp). This is where the matmul-repo material migrates in.
  • div, curl via GA: the geometric derivative ∇F splits as ∇·F + ∇∧F; div and curl are its grades. One definition, both operators, and curl (grad f) = 0 / div (curl F) = 0 become grade-bookkeeping lemmas instead of coordinate computations.
  • Integrals, interface-first (house style): specify Integral by its laws — linearity, interval additivity, FTC as the spec — program and prove against that, then construct Riemann sums as the Impl (completeness needed only in the construction). Line integrals = 1-D integral of a pulled-back field along a path; surface integrals and Stokes are the far end of this road.
  • Change of variables, in two layers matching the two derivatives: 1-D substitution ∫ f(g(t)) g'(t) dt == ∫ f (FTC + chain rule, both already on this list); multivariable substitution with the Jacobian determinant |det Dg| as the volume factor — det comes from linalg wave L3, so the statement is assemblable before the (hard) proof; state it as the spec early, prove the affine case first, full version rides on the integral construction.

Stage 6½ — Backprop and computation-graph reorganization

Added 2026-07-08: the motivating application is reasoning formally about automatic differentiation and different ways of organizing a transformer’s computation (flash attention, KV-cache reshaping). Theory/Jacobian.v already proved the load-bearing fact — the Jacobian chain rule IS mat_of_comp — so this stage is mostly composing existing pieces, not new machinery.

  • Gradients via Riesz representation. For scalar-valued f (codomain = F), the gradient is the unique g with Df(a)(h) == ⟨g,h⟩ for all h — connects Theory/Adjoint.v’s inner-product machinery to Theory/Derivative.v’s derivative; existence needs finite dimension (a basis), uniqueness should be immediate from PosInner’s definiteness.
  • Concrete differentiable layers as instances: matmul, softmax, ReLU, layernorm, each as an is_deriv witness with its derivative computed — this is where the abstraction touches an actual transformer. None exist yet.
  • Multi-argument functions should already work for free — a layer depending on both input and weights is a function on a product space (Theory/ProdVS.v), and Theory/Derivative.v is generic over any VectorSpace — worth a concrete worked example to confirm rather than assume.
  • VJPs as an efficient Impl, not full Jacobian matrix formation — real backprop composes linear-map applications directly without materializing the matrix; a legitimate “descend to implementation for efficiency” instance once the abstract theory above exists.

Stage 7 — Space-time

  • Impl/Minkowski.v — Fin 4 -> R with the (+,−,−,−) bilinear form: an InnerProduct instance today, thanks to positivity-free design. Could even land early (after stage 1) as a showcase — no analysis required.
  • GA over Minkowski = the spacetime algebra: boosts as rotors, electromagnetic field as a bivector. Blog material of the highest order.

Stage 8 — The Lagrangian, and numerics by refinement

The capstone that ties analysis back to the project’s thesis. Goal: define the Lagrangian against the real-number interface, instantiate a program that computes it, then refine that program to executable arithmetic — rationals for approximation, intervals for guaranteed enclosures — with the refinement proved sound.

  • Expression language for real formulas (program-as-data, again): an Expr AST whose exact denotation lands in the CompleteOrderedField interface. Same move as memprog’s Expr/eval/compile pipeline, now for numerics.
  • Rational evaluator — computes in Q, carries an error budget. Ball arithmetic (center c : Q, radius r : Q) is the proof-friendly formulation.
  • Interval evaluator — [a, b] enclosures. NOTE the honesty point: intervals do NOT form a field (x − x has positive width; * only sub-distributes), so this is not an Impl of the field interface. The correct statement is a soundness theorem: real_denotation e ∈ interval_eval e — a containment refinement, the same shape as every other refinement link in the project.
  • Lagrangian showcase: L(q, q̇) = T − V for a concrete system (harmonic oscillator, pendulum) — definable as soon as stages 1–3 exist (it’s polynomial in q, q̇; derivatives only needed later for Euler–Lagrange). Then: discrete action, and the refinement chain exact-L → rational-L-with-error → interval-L.
  • Connection to ~/mechanics: the symplectic-integration work is the eventual consumer — a verified integrator step whose interval version provably encloses the true trajectory is the long-range target. Euler–Lagrange / calculus of variations is far-future and deliberately not on this list.

Explicitly deferred

Topology-first treatment (open sets, filters), measure theory / Lebesgue, differential forms as a separate formalism (GA covers the vector-calculus needs first). Each can be added as its own interface later without disturbing the tower.
