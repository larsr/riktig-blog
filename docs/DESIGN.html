Design & style

How code and proofs are written in this repo (riktig). What we aim for is in VISION.md; this document is the style contract.

Program style

  1. Top level reads in the domain’s language. A program is a pipeline of named stages (fetch url >>= decodeJson >>= traverse_ process) readable as a description of what it does.

  2. Program-as-data. Arg parsing produces one value — a Plan — that defines what the program does; a small interpreter executes it. No behavior decided by scattered if-statements mid-flow. (The free-monad DSLs are the formalized version of this same pattern.)

  3. Errors are values; policy floats to the top. Lower layers never fudge (no "" for a missing file). They return explicit failures; what to do about them (skip / retry / abort / default) is a field of the Plan, set at the top. Parse, don’t validate.

  4. No layer leakage. Lower layers are named in their own vocabulary and stay parametrically polymorphic: a fetcher maps URL → bytes and cannot know upper datatypes — parametricity as architectural enforcement (the same trick that stops MemProg programs from forging addresses).

  5. Short and clear; find the abstraction line. Every component is the most general program of its kind — a cache knows nothing about auth, an auth fetcher nothing about LLMs; expiry is part of what “cache” means, not a special case above it. Use-case knowledge exists only at the composition point, which then reads as one sentence (“get the cookie from the cache at xyz or else request-and-cache it; invoke the llm”). Boilerplate is a smell that a concept wasn’t factored out — refactor the concept, don’t elaborate the handler.

  6. Recursive Coq programs: code once, as a plain lambda term; obligations after. (Adopted 2026-07-15; the model is theories/Programs/Merge.v.) The governing principle: do not stray from lambda calculus. A program is a plain CIC term — fun, match, Fix, application — that you could print and recognize as what you wrote; machinery that elaborates the source into something else (Program Fixpoint’s sigma-packed _func wrappers and subset types, the Equations plugin’s clause compilation) is out, however convenient. Concretely: a recursive program that is not structurally recursive is one refine whose body is a single ordinary ML-looking term: uncurry ∘ Fix (well_founded_ltof _ <measure>) turns the curried arguments into the one tuple the measure needs, the tuple is matched ONCE with flat, first-match clauses under a single p = x convoy, conditions are written if dec (..) then .. else .., and every termination hole is discharged by one combined tactic line after the term ((subst p; unfold ltof; simpl; abstract lia)) — abstract, because lia’s generated proof terms get very large, and in a Defined (transparent) definition they would otherwise sit inside the program term itself and be re-typechecked at every unfolding; abstract hides them behind an opaque constant and keeps the lambda term small. The clause equations are then derived from the definition via Fix_eq, in the model file’s four-line shape:

    rewrite Fix_eq.
- destruct dec; f_equal.
- intros [[|x x'] [|y y']] f g H; auto.
  destruct dec; rewrite !H; f_equal.

    — the equation branch is destruct dec; f_equal, and the extensionality premise closes by the pointwise rewrite !H, so no functional_extensionality: the model file’s Print Assumptions is closed under the global context. The dec-vs-boolean bridge is a general lemma (lem1 there), not per-proof gymnastics. Reasoning about the program goes through the derived equations, never by unfolding the fixpoint — the same rule as 3’s “never unfold interp”. (Program Fixpoint achieves code-once too but its generated obligations capture the recursive function inside subset proofs, making the derived equations need a le_unique detour — the Fix form above is both shorter and cleaner.)

    Two further rules, made explicit 2026-07-16. No auxiliary function that is the function: defining f_aux that differs from f only in argument order (to put the decreasing argument where Fix wants it) and then wrapping f a b := f_aux b a states the program twice and doubles the derived equations; the tupled uncurry ∘ Fix form with the right measure (snd for gcd) writes the program once in its conventional argument order. A worker with a genuinely different signature (a reparameterization the recursion needs, not a shuffle) is not covered by this rule, but prefer recursing on the caller’s own arguments with a measure when possible. No fuel: termination is always by a well-founded measure, never by a fuel : nat argument threaded through the recursion — fuel changes the function’s type, pollutes every equation with “enough fuel” side conditions, and proves nothing about the algorithm. The older refine (Fix ...) files (BinarySearch.v, Quicksort.v, MergeSort.v) predate the tupled form and stand as written until their scheduled rewrite; Gcd.v was converted 2026-07-16 (aux eliminated, and with it the file’s functional_extensionality import — the flat motive closes Fix_eq’s premise pointwise).

Proof style

(The worked-out model is enter-docker/math; the discipline mirrors lean-test/DESIGN.md.)

  1. Four layers. Ops.v — operational typeclasses, one per symbol, no laws. Props.v — single-law property classes (Assoc, Idl, …). Interface/ — structure classes bundling ops and laws, axioms only, stated as readable foralls, with Instance bindings connecting their axioms to the property classes. Theory/ — lemmas derived generically via Section/Context, including the abstract structure-transfer constructions (a function space, product, Hom-space, or subspace inherits structure — instances with no data structure of their own). Impl/ — data-structure realizations of the abstract structures, so one can compute with them: Z, Q, matrices as Fin m -> Fin n -> F, coefficient lists, the complex record — the mathematical analogue of Programs/, and usable by programs. A file does exactly one of these jobs. (Refined 2026-07-16: the transfer constructions Pointwise, FieldVS, HomVS, ProdVS, UnitVS, Subspace moved from Impl/ to Theory/ under this criterion — they are abstract theory, not representations; the test for Impl/ is “does this file give you something to compute with”.)

    Fourth pattern, blessed 2026-07-08: a Theory/ file may define the canonical generic algorithm for the construction it studies — bigop/bigopF in Theory/BigOp.v, Gram-Schmidt’s gs_aux/gs in Theory/GramSchmidt.v — when the algorithm has no separate abstract spec worth writing (there is no “Summation” or “Orthogonalization” Interface for it to satisfy) and is itself generic over an existing Interface (Monoid, PosInner), not tied to one concrete carrier. The algorithm is the theory being studied. New operational vocabulary still belongs in Interface/; concrete data-structure representations still belong in Impl/. (The 2026-07-08 wording also excluded instance-minting from Theory/; the 2026-07-16 refinement above supersedes that — abstract structure-transfer instances are Theory content, and only computable representations are Impl content.)

    Fifth pattern, blessed 2026-07-08, narrowed 2026-07-16: a Theory/ file may import a specific Impl/ file when its content is theory about that named representation itself. The original motivating cases (Theory/Hom.v over HomVS/FieldVS, Theory/VSIso.v over ProdVS/UnitVS/Pointwise) dissolved when those constructions moved to Theory/ — they are now ordinary Theory-imports-Theory. What remains under this pattern: theory about a genuine representation, e.g. Theory/MatRep.v, Theory/Trace.v, Theory/MatMulBilinear.v, and Theory/Determinant.v importing Impl/Matrix.v (their theorems are ABOUT the matrix representation), and Theory/Merge.v importing Programs/Merge.v (properties of a named program). The test is unchanged: could the file’s theorems be stated without naming that specific representation? Rule 6 still forbids the OTHER direction and the accidental case — a generic algorithm over an abstract Interface must still not import a concrete representation’s lemmas it doesn’t need to name.

    Standing exception, recorded 2026-07-13: three directories sit deliberately outside the four-layer split altogether — theories/Channel/, theories/Lambda/, and theories/Programs/. Each is a self-contained track about concrete objects-of-study (traces and protocols; a calculus; executable example programs) rather than abstract structures-with-many-instances, so the Ops/Props/Interface/Theory/Impl decomposition would add layers without adding instances; internally each file keeps spec, theory and implementation content in clearly ordered, usually explicitly numbered sections. The Channel/ and Lambda/ exemption was examined and recorded in docs/LINALG-ROADMAP.md’s Wave L8 section (the 2026-07-08 audit); Programs/’s is stated in theories/Interface/Decidable.v’s header (“plain Coq = and stdlib sumbool; nothing from the algebra tower”). Decidable itself is a full Interface/Theory/Impl family as of 2026-07-16: Interface/Decidable.v (the Decidable/EqDec classes and the dec vocabulary), Theory/Decidable.v (closure under the connectives, equality transported to lists and pairs, dec_if_bool), Impl/Decidable.v (nat and bool). theories/Examples/ is a fourth such track — compiled usage examples pinned from the blog, depending on everything, depended on by nothing. Two clarifications recorded 2026-07-16: theories/Programs/ holds the implementations of algorithms and their derived clause equations; properties proved ABOUT a program (permutation, sortedness, correctness) go in theories/Theory/ next to the rest of the proved mathematics (Theory/Merge.v about Programs/Merge.v is the model). And theories/Lib/ holds plain Coq conveniences that are part of no mathematical structure — the calc chains (Lib/Calc.v) and measure induction (Lib/MeasureInd.v, a placeholder until a proper theory of induction replaces it); they may be used by any track and depend on nothing.

  2. Operational typeclasses carry notation. HasAdd/Zero/EqRel give +, 0, == for any instance; proofs read like textbook derivations (rewrite <- assoc; rewrite invr; rewrite idr; reflexivity).

  3. Setoid discipline. Work over an equivalence == with Equivalence and Proper instances so rewrite works everywhere — including under binds, once program equivalence === exists for the memory DSL. Never prove a program property by unfold interp; simpl; if you are unfolding an interpreter, the missing piece is an equivalence lemma.

  4. Each lemma is the most general statement of its fact (rule 5 above, applied to proofs). Arithmetic side-conditions get factored into named reusable lemmas rather than inline lia everywhere.

  5. Honest incompleteness. No Axiom. An Admitted must carry a comment with a proof sketch — a roadmap, not a shrug. Specs owe an equivalence proof to a canonical formulation (a structural Sorted must be proven equivalent to pairwise Ordered).

  6. Import discipline. Dependencies point one direction: implementations import theories, theories import structures, never the reverse; an algorithm written against an abstract interface must not import a concrete instance’s lemmas.

  7. Interface record fields: #[global] locality, explicit carriers (decided 2026-07-03). Substructure and operational fields of a structure class are declared #[global] (#[global] grp_monoid :: Monoid G), not with the default export locality — precisely so that rule 6’s fine-grained Require Import works: importing Interface.Ring alone brings the whole instance chain, no Require Export towers. And law fields annotate their carriers explicitly (forall a b : G, not bare forall a b), because record elaboration does not infer them.
