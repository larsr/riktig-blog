Blog style — the actual rules, with real examples

This supersedes the looser five-criteria version from the 2026-07-13 audit. Read this whole document before writing or rewriting a single post. It exists because Lars has had to ask for this — his words — “at least five times.” The failure mode is specific: each new writing session gets a decent brief, drifts a little from it, and the next session’s brief is written from the drifted result instead of from first principles. This document is the fix: it is the first-principles version, held constant, quoted from directly in every future brief instead of re-derived.

The one-sentence test

A reader who has never heard of this project, dropped onto this one post with no other context, must come away understanding one idea, correctly, with no sense that they were being sold something.

If a sentence would not survive that reader’s first read — because it assumes they know what “riktig” is, or because it name-drops a destination three topics away, or because it sounds like marketing copy — cut it.

This also means: never assume the reader already knows what the post is about to define. Do not open with “we already know X” or lean on a name the reader hasn’t been given yet. See Rule 8 for the required shape this forces on every post.

Rule 1 — no project framing in tool-building posts

Do not write “riktig,” “the project,” “verified software,” “modifiability,” or any sentence whose actual content is why this whole endeavor matters. That framing belongs in exactly one place: a capstone post written after the tool-building series is complete, titled something like “putting the tools to work,” which is the ONLY post allowed to say why any of this was worth building. Every other post is a plain exposition of one mathematical or computational idea, full stop.

Bad (real, from post #1’s opening, currently in the repo):

This is the first post in a series documenting the riktig project.

What we are doing, and why. We are building software whose correctness is proved, layer by layer, in the Coq proof assistant — sorting algorithms down to machine code, a small operating-system kernel, network protocols generated from their grammars. The goal is not proof for its own sake: it is modifiability.

A reader who opened this post to learn what a monoid is has been told, before a single mathematical word, that they are reading marketing for an operating-system kernel that doesn’t appear in the post at all.

Good (what the same post should open with instead):

Programmers already know half of what a mathematical structure is. An interface says which operations exist: this thing can be compared, that thing can be concatenated. What an interface does not say is what the operations mean. A sort function that returns its input unsorted satisfies the interface list → list perfectly. A mathematical structure is the missing half — this post is about that missing half, and about a handful of structures built from it, each one a little richer than the last.

Same information content, zero throat-clearing, states the actual destination in the first paragraph.

Rule 2 — self-contained means self-contained

A post may assume the reader has read the earlier posts in the syllabus (docs/BLOG-SYLLABUS.md) — that’s what a series is — but it must recap what it needs in its own words, in two or three sentences, not by reference (“as post #3 explained…”). Never assume anything from a LATER post. Never mix two unrelated ideas at equal weight in one post — if a proof leans on a fact from a different area, state the fact plainly (one sentence, no proof) and move on; don’t make the reader detour.

Bad (real, from the old audit’s findings — post #17 mixed):

Channel liveness and matrix determinants in one post, no connection between them, no signal to the reader that they are now reading about something unrelated.

Good: one post, one idea. If two results genuinely belong together (they share a proof technique, say), the post’s title and opening paragraph say so explicitly, in one sentence, before either result is presented.

Rule 3 — show the real proof, and show how to use the thing

Every post’s biggest single failure mode has been telling instead of showing. Lars’s exact words on post #1, the best post so far: “it showed too little of proofs and how to use the code.” Concretely:

  • Every structure/definition introduced needs at least one theorem proved in the post, with the real Coq source pasted in (copied from the file, never reconstructed from memory — check it compiles by reading the actual file with the Read tool right before pasting it, not by recalling an earlier draft).
  • Show the blackboard version of the proof next to the Coq version (post #1’s cancel_l treatment, and the newer calc-chain posts, are the right model — see Rule 5).
  • Show at least one concrete usage: instantiate the structure at a real type (Z, Q, a small example type), run a computation, show the output. If the codebase has an Example demonstrating this, quote it; if not, and the post needs one to make the idea concrete, say so honestly rather than inventing prose that sounds like code.

Bad: “the file proves the whole inclusion lattice as small lemmas — each a short induction, each doing exactly what its name says.” (This describes proofs without showing one. A reader has no way to check the claim or learn the technique.)

Good: paste the actual lemma statement and the actual proof, then walk it in one or two sentences.

Rule 4 — titles must match content, and say something specific

A title is a promise. “Introducing channels” promises an overview and delivers one — fine. But a title like “Full generality” or “Any adjacent pair” (real titles from the current series) promises nothing checkable — a reader cannot tell from the title whether they want to read the post, and cannot tell after reading it whether the title was earned. Every title must name the actual mathematical content: what structure, what theorem, what technique. If you can’t compress the post’s content into a title that would let a stranger predict roughly what proof appears inside, the post’s scope is wrong — fix the scope, not the title.

Rule 5 — the proof-walking format that works

This format, from post #1’s cancel_l and the later calc-chain posts, is the house style — use it for every proof shown:

  1. State the theorem in one sentence of English.
  2. Show the blackboard-style version in real LaTeX math — inline $...$ or display $$...$$ (the build renders both via pandoc’s --mathml; never fake math with ASCII art, unicode approximations, or a code block). For an equational proof this is the displayed chain of equalities, each step annotated with the law that justifies it.
  3. Show the real Coq Theorem/Proof/Qed block, pasted verbatim.
  4. Walk it: one short sentence per proof step, connecting the Coq tactic to the blackboard step it corresponds to.

Both halves are required every time: a Coq proof displayed without its LaTeX-math counterpart fails this rule, and so does a math chain whose Coq version is only described.

Do not paraphrase a theorem statement. Do not write ... inside a displayed Coq block to elide something — either show the whole thing or don’t show it. A garbled or partial quote is worse than none (the 2026-07-13 audit’s worst finding: a displayed statement that didn’t even parse, naming a function that doesn’t exist).

Rule 6 — no internal jargon, no unexplained acronyms, no tone-deaf voice

Define every piece of jargon the first time it’s used, in the same sentence, in plain words. Never use words from how the project talks about itself internally — “the dispatch,” “the wave,” “the audit,” agent/session/orchestration language of any kind. The voice is a patient teacher explaining something they find genuinely elegant, not someone performing confidence. Concretely, avoid:

  • Declarative one-liners that sound clever but assert nothing checkable (“this is the whole business model of algebra” is borderline-fine once, in moderation — a whole post full of sentences like it reads as bluster, not exposition).
  • Unexplained forward references (“this pays off later” — say what pays off, or cut the sentence).
  • Foreshadowing more than one sentence at the very end (“next post: X” is fine; a running commentary about the shape of the whole series is not).

Also apply the existing vocabulary rule from the project’s own CLAUDE.md (no American sports/military/startup-hype idiom — “unlock,” “move the needle,” “battle-tested,” etc.) — that rule is unaffected and still in force, it is a narrower special case of “plain, precise language” that this document generalizes.

A second, stricter register applies to blog prose specifically, on top of CLAUDE.md’s general list: avoid inflated-sounding engineering phrasing that asserts authority without saying anything checkable — “load-bearing,” “honest” used as a hedge word (“an honest hypothesis,” “the honest thing to do,” “honestly required”), “the real story,” “the key insight,” and similar. (CLAUDE.md allows “load-bearing” as ordinary engineering vocabulary elsewhere in the project; for blog prose it still reads as jargon and should be avoided.) Say the actual reason instead: not “this hypothesis is honestly required” but “the theorem needs this hypothesis because, without it, exp need not be surjective, and log has nothing to invert.” Use plain computer science and mathematics vocabulary — the words a textbook or a patient colleague would use — not the words a project’s own internal notes would use about itself.

Rule 6b — concise, concrete, never clever (added 2026-07-19)

Lars, on the calc post: it “started good but soon became very verbose and did not get to the point”; and on the sentence “a typeclass is a named shape of fact”: “What the heck is a shape of a fact??? That is just the kind of jargon i wanted to avoid.” Two binding rules fall out:

  1. No metaphorical definitions. When a post defines a technical term, the definition must be concrete and operational, not a coined metaphor. “A typeclass is a named shape of fact” is banned-by-example; write instead what the thing does: “a typeclass declares a family of facts indexed by a type, and an instance registers one such fact in a database Coq searches when a proof needs one.” If a metaphor genuinely helps, it comes AFTER the concrete definition, never instead of it. The test: a reader who has never seen the term must be able to predict what the Coq code looks like from the definition alone.
  2. Length discipline. Say each thing once. A post that restates its point in different words per section, or walks three examples where one suffices, is too long — cut it. When a post exists mainly to demonstrate one mechanism (as the calc post does), the mechanism should be reached within the first quarter of the post, and the whole post should be as short as the content allows. When revising for length, prefer deleting whole redundant passages over compressing sentences into fragments (the harness rule: selectivity, not compression).

Rule 7 — never write about code that doesn’t exist

If a topic in docs/BLOG-SYLLABUS.md is marked “needs implementation,” do not write the post. Say so and stop, or write the Coq first, then the post from the real result. This has bitten the project before (fabricated theorem statements, invented lemma names) and the fix is procedural, not a matter of trying harder: read the actual current .v file with the Read tool immediately before quoting anything from it, every time, no exceptions, even if you (the writing agent) believe you already know its contents from earlier in the same session.

Rule 8 — problem, then formalization, then proof, then usage

Every post moves through four movements, in this order, and does not skip or reorder them:

  1. The problem being modeled. State the real mathematical or computational situation the post is about, in plain language, with NO reference to Coq, no class/record/type name, no implementation detail of any kind. A reader must understand the underlying question — what is being asked, and why it isn’t already obvious — before seeing a single line of code or a single structure name.
  2. The formalization. Only now introduce the actual Coq structure or definition, and connect it explicitly back to part 1: which field or law captures which piece of the problem just stated.
  3. The proof. Walk the actual theorem(s), per Rule 5’s format.
  4. How to use it. Concrete usage — instantiate at a real example, run it, show the output, or show how a caller invokes the construction.

Bad (starts from the implementation): “A Determinant is a class with three fields: det_mf, a multilinear form; det_alt, a proof it is alternating; and det_basis_1, a normalization law.” A reader has been handed field names before being told what problem those fields solve.

Good (starts from the problem): “Given a linear map from a space to itself, is there a single number that says whether it stretches volume, shrinks it, or flips orientation — and can that number be pinned down by its own defining properties, without ever writing down a matrix or a formula for computing it? … Coq calls this collection of properties a Determinant class: …”

This rule governs the whole post’s shape, not just its opening paragraph — a post that states the problem well in one sentence and then digresses back into implementation details before the formalization section still fails this rule.

Grading

Before a rewritten post is considered done, grade it against these eight rules, A/B/C each, and report the grades. A post below B on any rule is not finished. The previous audit’s grading scale still applies; this document adds Rules 1, 2 (tightened), 4, and 7 to it.
