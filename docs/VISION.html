riktig — correct, real, propersource

Vision, state, and roadmap

Written 2026-07-03, after a survey of coq-memprog and its sibling repos; now the working document of riktig, the consolidated monorepo those repos migrate into (§8). The README states the philosophy; this file records where everything actually stands and how it fits together.

1. The aim

Correct code that one can modify and be sure it still works — without having to redo the complete program. Changes should only require satisfying local obligations; the layering guarantees the rest. That is the whole vision. Everything else in this document is instantiations of it, at different levels:

  • an algorithm — quicksort, refined from spec to machine code;
  • a kernel — the minikernel as a library-like base;
  • programs on the kernel — a tcp stack, tty, ssh, a web server, web sockets;
  • protocols as showcases — protobufs, JSON, BSON, ASN.1, …, each generated from its grammar, never hand-reimplemented.

The method: a short specification at the top, real executables at the bottom, a chain of small provable design decisions in between (data structure, algorithm, arithmetic representation, calling convention, instruction set, binary format). Proof obligations stay local and compose. Programming-language agnostic: the specifications and refinement stacks are the long-lived artifacts, not any implementation language.

Two directions of the same idea:

  • Vertical refinement: refine how a program computes — memory-monad DSL → expression compilation → stack frames → machine code → x86 bytes → loadable binary.
  • Horizontal refinement (channels): refine where a computation happens in a stack of communication layers — as long as each layer satisfies its codec law, computations and their proofs transport across layers.

How we write code and proofs is a separate contract: see DESIGN.md.

4. The proof style (from enter-docker/math; now ported into theories/)

The math/ library’s style is the answer to why SwapProof got stuck:

  • Operational typeclasses carry notation: HasAdd/Zero/EqRel in core.v give +, 0, == for any instance.
  • Property classes (Assoc, Idl, Invl, …) name reusable facts; structure classes state axioms as readable foralls, with Instance bindings connecting them to the property classes.
  • Setoid discipline: everything over == with Equivalence + Proper, so rewrite works everywhere; proofs are textbook chains (rewrite <- assoc; rewrite invr; rewrite idr; reflexivity).
  • Three directories: structure/ (axioms only), theory/ (generic lemmas via Section/Context), impl/ (instances).

Applied here — the keystone piece to build next:

structure/  program equivalence:  p === q  on Mem/MemT (observational,
            via interp from equal machines);  Refines relation
theory/     Proper for bindMem (rewrite under binds); memory-law algebra
            (read-after-write, write-write-commute at distinct addrs,
            bind associativity); Refines composes transitively
impl/       swap_correct as first client (a few rewrites, no unfold);
            then the chain links: compile e === retT (eval e)  (i.e.
            qsort1 ≡ qsort0), then qsort2, qsortS, qsortMC

5. Integration points between the pieces

minikernel as the bottom of the chain

The chain currently bottoms out at Linux ELF — a huge unspecified trusted base. The minikernel replaces it with ~3100 lines we wrote:

  • Its PROG format (magic, code_size, reloc_count, entry; flat blob
    • u32 reloc offsets) is far simpler than ELF. A 32-bit variant of MemProgX86.hs emitting PROG instead of ELF lets Coq-refined programs run on our own kernel: registers → EAX..EDI, drop REX prefixes; entry ABI is (mem, size, kctx*) on the stack, so the data-base register is initialized from the first argument.
  • Its kctx table (~25 function pointers: print, proc_create_on_cpu, fs_*, mem_alloc, nic_send/recv, …) is the entire syscall surface — small enough to specify as SysCall effects in the Mem DSL.
  • Kernel-side verification targets, smallest first: proc_switch (13 asm instructions), the CAS lock protocol, kmalloc no-overlap / no-double-free, one-CPU-per-slot scheduler invariant.

Target chain: Coq spec → Mem DSL → … → MC Instr → x86-32 → PROG → minikernel → QEMU/hardware — every layer ours.

Protocols: grammars as the single source of truth (LangSec)

Motivation: most security bugs are incorrect parsing of hostile input, and every language re-implements every format (“JSON by some guy”). All data exchange should be specified by a grammar, mechanically derived from the normative spec. The make test-abnf discipline (RFC text verbatim in Spec.v) is the strongest form of this: the spec text is the artifact.

Architecture decision (resolves the “ABNF can’t express <count n><n bytes>” problem):

  • ParseExpr is the core — it is a free monad with Bind, so data-dependent formats are already expressible (do! n <- int; take n). ABNF is one frontend (Generator.v) compiling into the core; add frontends per formalism: .proto, ASN.1 modules, a packet-layout DSL for prose-diagram specs (TCP headers).
  • Restrict dependence for analyzability: raw Bind makes grammars opaque (no unambiguity analysis, no serializer inversion). Keep Bind in the core but define a tame layer (Take/Repeat à la Narcissus/EverParse/Daedalus) and prove the strong theorems there.
  • Two round-trip laws per format: faithful (flatten (parse s) = s, wire side — exists) and parse (serialize x) = x (value side — future, requires serializer generation; Narcissus shows it feasible in Coq). The pair is what makes independent implementations agree.

Prior art to consult: LangSec (Sassaman/Bratus/Patterson), Narcissus (Coq, parser+serializer derivation), EverParse/3D (F*, deployed in Hyper-V), Daedalus (Galois), RecordFlux, Kaitai Struct.

Channels: composable layered stacks

Programs talk over stacks like tcp → tls → http → json → openapi, any layer swappable as long as decode (encode x) = Some x.

  • Codec typeclass in the enter-docker style (enc/dec/law, notation A ⇌ B), with a composition instance proving the law composes once — this is the README’s “proofs compose” requirement made concrete.
  • Layers are prisms, not isos: dec ∘ enc = id total; enc ∘ dec = id only on canonical wire forms (JSON whitespace…). faithful is the wire-side direction; grammar unambiguity links the two. The asymmetry is where interop/smuggling bugs live — state both laws explicitly per layer.
  • Transports (TCP/TLS) are stateful processes, not pure codecs: their contract is trace identity (recv_trace = send_trace); endpoint state machines are verified with the same Refines machinery as the qsort chain.
  • Moving computations across layers = conjugation enc ∘ f ∘ dec; the codec law gives the commuting square, so correctness transports.
  • Smallest artifact: Codec + compose + three instances we already own (string_to_bytes, Tree/flatten via faithful, Tree → Json) — the first file where the parser library meets the algebra style.

tty / ssh: haskell-tty

~/haskell-tty (~950 LOC) implements a POSIX line discipline as a pure state machine:

step :: Side -> Termios -> LineState -> ByteString -> (LineState, [Event])
-- Event = Forward bytes | Echo bytes | Signal sig | Eof | WindowChange …

Canonical editing is feature-complete (ERASE/KILL/WERASE/REPRINT/ LNEXT/EOF), ISIG (^C  Z), input flags, OPOST. Two wirings, which is itself a two-implementations-one-spec equivalence in the wild: Session (kernel’s ldisc, cooked pty) vs PureSession (our ldisc in the data path, kernel pty raw — exactly the SSH server architecture). End-to-end tests already run keystrokes over TCP → own ldisc → raw pty → bash, including ^C and erase. Termios.hs already maps RFC 4254 SSH opcodes.

Fit: - The ~200-line pure core (Canonical.hs + Output.hs) transliterates to Coq almost line-by-line — a realistic, useful formalization target with an existing Haskell oracle and test suite. - The tty is another layer in the channel stack (Forward down, Echo up, Signal out-of-band), with statable laws. - Distance to “own ssh with tty forwarding”: SSH channel framing (RFC 4254) around the existing TCP server + winsize wiring; crypto later (or lean on the verified-channel work). Architecture already matches. - minikernel has serial but no pty/process groups: either port the ldisc in, or run it host-side over minikernel’s serial as a dumb pipe. - Dogfood target eventually: minikernel’s tcp.c (796 lines of hand-rolled packet parsing — the canonical shotgun parser) replaced by a grammar-generated parser refined through the pipeline.

7. Development environment: nix devcontainer + persistent cache

Goal: anyone can attach VS Code to a well-defined, reproducible environment; Haskell/Coq toolchains are cached so container rebuilds don’t recompile the world. Six variants exist across the repos in different stages; enter-docker’s git history is the laboratory where the idea evolved (.envrc→docker run → exec-into-running-container → devcontainer detection → plain devcontainer.json).

The mature endpoint is ~/hterms/.devcontainer/ — use it as the template:

  • Base mcr.microsoft.com/devcontainers/base:debian (FHS userland, vscode user with passwordless sudo — no root, no UID pain).
  • Determinate Nix installer (--init none), multi-user daemon started by postStartCommand with waitFor so terminals open ready.
  • Cache: named docker volume PROJECT-nix-store mounted at /nix — survives container rebuilds. (Named volume beats a bind-mounted dir: no UID mapping issues; per-project volumes avoid nixpkgs-pin collisions. A bind mount works too if host-visibility is wanted, but then run as host UID.)
  • Optional team cache: S3 substituter in flake.nix nixConfig + read-only ~/.aws bind mount forwarded to the daemon — the only piece that makes the multi-GiB GHC/HLS closure fast on a fresh machine. Droppable for solo use; the volume alone covers local reuse.
  • direnv: .envrc = use flake; pre-trust /workspaces prefix and pre-warm the shell in onCreateCommand. See hterms/docs/nix-cache.md.

Superseded / droppable elsewhere: the .envrc→docker-exec orchestration and container-reuse detection from enter-docker’s history (devcontainer.json owns the lifecycle now); the third-party xtruder/nix-devcontainer image (own Dockerfile is more transparent); root-user containers and postCreate nix.conf copying (coq-memprog, AndreasQQ — coq-memprog’s devcontainer also lacks a /nix volume, so it re-downloads toolchains on rebuild: add the named volume). The whole good setup is ~50 lines: Dockerfile (base+nix+direnv), devcontainer.json (volume mount + lifecycle hooks), start-nix-daemon.sh, .envrc.

Cleanup action: extract the hterms pattern (minus AWS bits) into a template, apply it here (riktig has no devcontainer yet) and to coq-memprog, then converge the other repos.

8. Consolidation: one repo, proper subdirs (this repo)

Why sharing had been hard: the projects lived as branches of one repo (coq-math, coq-matmul, coq-memprog are branches of enter-docker), pushed through a chain of local remotes (coq-memprog → matmul → enter-docker → git.ri.se). There was no single URL with a clean history per project. This monorepo fixes that: one main branch, one remote, one flake, one devcontainer (§7), one CI.

The open decisions are made. The repo is riktig (RI as in RI/RISE, Swedish for “correct, real, proper”, and Rikard), Coq namespace Riktig; the kernel program will be named KLAS (for Klas — “Kernel as a Library, Applications in Sync”).

The subdirectories foreseen for the migrated material keep their plan: under theories/ — Category/ (categories, functors; seed: Interleave.v’s classes), Linear/ (vectors, linear maps, tensors; reordering/tiling equivalences — matmul micro-kernel proofs; future: flash-attention ≡ transformer, KV-cache ≡ recompute — same theorem family: blocked/streamed evaluation equals the spec), Sched/ (interleavings, schedules, process fusion), Mem/ (the memory DSL: Effect/ Theory/ Algorithm/ Bridge/, with QsortProof.v → Bridge/), Parser/ (grammars, ABNF, generated parsers, Equiv proofs), Channel/ (Codec class, layer laws, transports-as-traces); under programs/ — klas/ (minikernel: library-like kernel offering workers/channels; programs may link it, replace it, or FUSE with it — spec via kctx effects in Riktig.Mem), tty/ (haskell-tty: pure ldisc + Session/PureSession wiring), ssh/ (own ssh: tty forwarding over Channel stacks, RFC 4254 framing; uses programs/tty as a library); plus extracted/ for Coq→Haskell extraction output.

Using the theories from another repo (Coq namespacing). Inside the monorepo it’s one _CoqProject: -Q theories Riktig maps the dir tree onto module paths, so files say From Riktig Require Import Interface.Group. — exactly how mathcomp does it (-Q . mathcomp). For consumption from outside, the flake exports a package that installs the compiled .vo files into $out/lib/coq/<ver>/user-contrib/Riktig/; a consumer repo adds this repo as a flake input, puts the package in its devShell, and Coq finds Riktig.* via COQPATH automatically. (An opam package is the wider- distribution variant of the same thing; nix flake input is enough for our own repos.)

Endgame (the reason for the typed-interface discipline): a computer where every interface to the surrounding world is typed and specified, so that when something breaks, the blame is provably on the other side of the interface — “well-typed programs can’t be blamed” (Wadler & Findler) as a systems-engineering principle. Each spec at a boundary (kctx, wire grammars, channel laws, termios/ldisc) is one plank of that.
