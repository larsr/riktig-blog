Memprog roadmap — refinement as rewriting, from a sorting spec to a running ELF

Written 2026-07-18 after an interview with Lars, porting the ideas of ~/coq-memprog into this tree. This is the reference document for the whole track: read it before touching any Memprog file. Decisions recorded here were made explicitly; do not silently revisit them.

1. What coq-memprog is, and what it lacks

~/coq-memprog demonstrates a refinement chain for quicksort: seven executable levels from a C-like program in a free-monad memory DSL down to a Linux ELF binary that really sorts. The levels run and agree on test inputs, but the steps between them were done manually — no proof links any level to the next. Its own VISION.md diagnoses why the one attempted proof stalled: unfold interp; simpl blew up, because the repo lacks an equivalence-relation layer with Proper instances — exactly the setoid machinery this tree uses everywhere.

The port, in one sentence: redo the memprog chain so that every transformation step — between levels and within a level — is a rewrite with respect to a program equivalence, in the same style as the algebra tower’s calc-chain proofs.

What carries over from coq-memprog (as material, always rewritten in this tree’s style, never pasted): the level datatypes, the interpreters, the quicksort/swap/sum programs, the x86-64 instruction encodings and the minimal-ELF layout from MemProgX86.hs.

2. Decisions from the interview (2026-07-18)

  1. Programs: quicksort is the driving example through every level; swap and sum are micro-programs kept alive at every level where they make sense (swap especially — it is the smallest program that touches memory, and the blog walkthrough). Bubblesort is dropped.
  2. Everything in Coq: the x86-64 encoder and the ELF builder are Coq functions; the final ELF image is a list Z (byte values) computed by vm_compute. Only the last centimeter is external: a python script dumps the computed bytes to a binary file and executes it. No Haskell, no extraction.
  3. Layout: strict Programs/Theory split (the house file-layout rule). Syntax, interpreters, encoders, concrete programs → theories/Programs/Memprog/. Views, equivalences, laws, proofs → theories/Theory/Memprog/.
  4. Blog: one post (syllabus entry, Part 12 or a new Part 13) after the executable chain and its tests are green — honest about which links are proved and which are Admitted obligations.
  5. Termination: no fuel in program syntax. Loops are a While constructor of the syntax; recursion is meta-level Fix over an explicit bound with a sufficiency lemma. Fuel exists only inside interpreters (unavoidable for arbitrary machine code), and equivalence statements handle it existentially (§5).
  6. Frame property: a separate predicate (touches_only), stated and proved (or Admitted with roadmap) per program, NOT baked into the equivalence relation. The wider OS/allocator story stays a header-comment sketch (§7).
  7. x86 semantics: the port includes a Coq emulator for the x86-64 subset the encoder emits, so that every adjacent pair of levels has a statable equivalence from day one. (In memprog the bytes had no semantics at all — the last link was covered only by running the ELF.)

3. The levels

Each level is a separate datatype (a parse tree) with its own emulator, and each is testable by vm_compute on the shared test inputs. L1–L3 share the Mem syntax; they differ in which sublanguage the program uses, and the downward step is a compilation function.

# Level Datatype Emulator New design decision captured
L0 Specification list Z → list Z (a Coq function) + the spec predicates — (it is math) what “sorted permutation, rest untouched” means
L1 Memory DSL Mem addr val a free monad, with While as syntax interp over Machine (assoc-list store, sp, hp) data lives at addresses; reads can fail
L2 Expression level L1 whose arithmetic is Expr trees, first with eval_expr then compiled by compile_expr same interp arithmetic becomes data (the first “program as data” step)
L3 Stack frames L1 restricted to a calling convention: withFrame, fp-relative slots, push-args same interp locals and arguments become memory at negative offsets
L4 Machine code list (Instr Z) — the RISC instruction set mc_step/mc_run over MachState (registers, pc, sp, flags) + the same memory control flow becomes numbers (branch offsets)
L5 x86-64 bytes list Z (byte values 0..255) x86_step/x86_run over a byte-addressed machine (the emitted subset only) instruction lengths vary; branches become byte displacements
L6 ELF image list Z — header ∥ stub ∥ code ∥ data the host CPU (via the dump script); the x86 emulator covers the code segment the OS contract: entry point, one RWX PT_LOAD segment, exit code ABI

The compilation functions between levels (all total Coq functions): compile_expr (L2), the calling-convention transform is written by hand per program in memprog — here L3 programs are still hand-written against the frame combinators, and the obligation is their equivalence to the L2 versions; codegen : L3 → list (Instr Z) (L4, hand-written per program in memprog — same treatment, plus a real compiler is a stretch goal); assemble : list (Instr Z) → list Z (L5, the two-pass branch-displacement resolution from MemProgX86.hs); elf_image : list Z → list Z → list (Reg*Z) → list Z (L6).

5. The equivalence design (the actual point)

Within a level. For programs p q : MemT addr val a at the same level, observational equivalence relative to fuel and machine:

p === q  :=  for every machine m there exists F such that for all
             fuel ≥ F,  obs (interp fuel p m) = obs (interp fuel q m)

where obs projects the result and the store (not sp/hp bookkeeping, which intermediate levels are allowed to use differently — the exact projection is Equiv.v’s first design task; start with result + whole store and weaken only if a law needs it). This is an Equivalence, registered with the tree’s EqRel/Setoid conventions so calc chains work; the essential instances are

Proper (equ ==> pointwise equ ==> equ) bindT
Proper (equ ==> equ ==> equ) seqT

— rewriting under binds is what makes “optimization = rewrite” true, and its absence is precisely why coq-memprog’s SwapProof died. While needs its own congruence (equal conditions and equal bodies give equal loops).

MemLaws.v then proves the equational algebra every later proof runs on, e.g.

read_after_write   : writeM a v >> readM a  ===  writeM a v >> retT v
write_write_comm   : a ≠ b → writeM a v >> writeM b w === writeM b w >> writeM a v
write_shadow       : writeM a v >> writeM a w === writeM a w
bind_assoc, bind_ret_l, bind_ret_r
while_unfold       : While c b === ifM c (b >> While c b) (retT tt)   (up to obs)

Each is one Hint Rewrite-able law; program transformations inside a level (instruction scheduling at L1, expression simplification at L2) are calc chains over these laws — the same working style as Theory/Group.v, which is the whole thesis of the track.

Between levels. For adjacent levels with different state types, a refinement relation with an explicit state correspondence, e.g. for L4 vs L1:

refines_mc (code : list (Instr Z)) (p : MemT Z Z unit) (R : region) :=
  for every machine m loaded with code and matching setup,
  mc_run reaches HALT and the two final stores agree on the view
  of R (and the frame predicate holds outside R)

The correspondences: L1≈L2 and L2≈L3 share the machine type (plain === on programs, no simulation needed — this is why the first links are provable early); L3→L4 relates assoc-store cells + explicit stack to MachState registers/pc/flags; L4→L5 relates instruction indices to byte offsets (the assembler’s offsets table IS the simulation map); L5→L6 is layout only (the code bytes appear at the entry point; stated against the x86 machine with the ELF’s memory image). Refines composes transitively (one lemma in Refines.v), which is the “proofs compose” requirement of the original README made formal.

The view. mem_view m base n reads n consecutive cells at base — the “an address corresponds to the contents of a list” idea. The end-to-end statement for each sort, at every level, is one shape:

view-in is xs  →  run  →  view-out is a sorted permutation of xs,
and touches_only (stack region ∪ array region) held throughout.

L0’s qsort_correct in coq-memprog (sorted + permutation + outside-range-unchanged, proved, zero Admitted) is the model for the spec side; this tree already owns sortedness/permutation vocabulary (Theory/Sorted.v, Theory/Quicksort.v) — reuse it, do not redefine.

6. Termination without fuel in the syntax

The fuel rule (Lars, 2026-07-18, binding for the whole tree): never use fuel for a computation we know terminates. Fuel is relevant only when running something that might loop forever — an arbitrary machine-code program, an untrusted input — where we want to run for a while and see whether we get an answer or an error. Terminating recursion (sorting lists, manipulating trees, structural compilation) never takes fuel: it takes Fix over a genuine measure, and the measure is part of understanding the algorithm.

Applied to this track:

  • Loops: While : MemT addr val bool → MemT addr val unit → Mem … is a syntax constructor. (Precisely: condition and body are sub-syntax; the constructor keeps the tree finite — iteration happens in the interpreter.) This also makes loop bodies rewritable subterms, which fuel-unrolled loops are not — the rewriting story needs While as data anyway.
  • Recursion (quicksort’s): the L1–L3 program builders take lo hi : Z as meta-level arguments, so the recursion is plain Fix over the measure Z.to_nat (hi − lo + 1) — exactly Programs/Merge.v’s pattern, no bound parameter, no sufficiency side-condition. The generated tree is finite because the algorithm terminates, and the termination argument is visible in the measure.
  • Interpreters: interp, mc_run, x86_run take fuel — this is the one legitimate use: they execute arbitrary programs of their language, including ones that loop (While (retT true) …, a B 0 instruction), so they are genuinely partial. Equivalences handle fuel existentially (“∃F, ∀fuel≥F”, §5), and per-program termination lemmas (interp_enough : F(n) suffices for input size n) discharge the existentials concretely — the lemma that says “this particular program terminates” is exactly what earns it the right to be run under fuel without caveats.

7. The frame property and the allocator sketch

touches_only R p: interpreting p from any machine changes the store only at addresses in R (sp/hp may move; cells outside R compare equal afterwards). Stated per program at L1–L4; at L5/L6 the region becomes “the PT_LOAD segment minus the code bytes” and is an obligation against the x86 machine.

The surrounding-world story — “the OS/allocator handed this program a chunk, and correctness is relative to that grant” — is deliberately only a sketch in this port. Header comment in Theory/Memprog/Frame.v: the eventual shape is a hypothesis-style contract (the machine’s store outside the granted region is arbitrary and must be preserved; the granted region’s bounds come in as parameters, exactly how exp_surj/ftc enter as disclosed hypotheses elsewhere in this tree), and the ELF loader’s PT_LOAD grant is its concrete instance. Not formalized now; the predicate’s shape is chosen so it can grow into that without restating programs.

8. Phases, deliverables, done-criteria

Phase 1 — the executable port (Programs/Memprog/* complete): all seven levels compile; Tests.v’s vm_compute Examples pin every level’s output on the shared inputs (swap, sum, quicksort on the 9-element list, plus one degenerate input each: empty, singleton, duplicates); make memprog-elf builds and runs the ELFs and checks exit codes. No Admitted anywhere in Phase 1 (it contains no theorems beyond computational Examples).

DONE 2026-07-18. Files: Syntax.v Machine.v Expr.v Frame.v MC.v (core), X86.v Elf.v (byte level with a real emulator + the ELF image as computed data), Swap.v Sum.v Quicksort.v Tests.v (programs + battery), ElfGen.v + scripts/elf-run.py + make memprog-elf (the live run). While is a syntax constructor; Mem’s result type became an inductive index so While can hold subprograms at fixed types. Quicksort recurses by Fix over Z.to_nat (hi-lo+1), no fuel, the pivot’s recursive calls guarded by an in-range decision (out-of-range faults instead of looping; Phase 2 proves it never fires). x86 bytes gained a decoder + subset emulator (memprog had none), so every level pair is emulator-testable; the same bytes that pass the emulator check are the bytes that run as native ELF. Exit codes verified: swap 40, sum 15, quicksort min 1 / max 9. Zero Admitted, zero Axiom across the whole track so far.

Phase 2 — the obligations, stated honestly: View.v, Equiv.v (setoid + Proper — these must be proved, not admitted, or nothing else works), Frame.v and Obligations.v with the chain links as Admitted-with-roadmap:

swap_L1_L3, swap_L3_L4, swap_L4_L5          (smallest program first)
qsort_L1_L2eval, qsort_L2eval_L2comp,
qsort_L2comp_L3, qsort_L3_L4, qsort_L4_L5
elf_layout_L5_L6
touches_only_{swap,sum,qsort} at L1, L4

Rule for this tree: Obligations.v is the ONLY file allowed Admitted, each carries a proof-sketch comment, and docs/TODO.md lists them. Everything else keeps the tree’s zero-Admitted status.

Phase 3 — the rewriting rules: MemLaws.v proved (the within-level algebra, each law a Qed); Refines.v proved (composition); then convert obligations to Qed smallest-first: the L1≈L2 links are pure === rewriting (compile_expr correctness is a structural induction + MemLaws — the “smallest link that exercises all the machinery” from coq-memprog’s own roadmap), swap before sum before quicksort, simulation links (L3→L4, L4→L5) last and possibly spanning several sessions. Each converted link should be a calc chain readable as the transformation it justifies.

Phase 4 — the blog post: syllabus entry 57 (added when Phase 1 completes, marker [has code]); one post: the problem (why manual compilation steps are where correctness dies), the levels as datatypes with the swap program shown at every level including its actual ELF bytes, one real within-level rewrite as a calc chain, and a plain statement of which links are proved vs open obligations.

9. Conventions that bind this track

Everything general in this tree applies: Hint Mode discipline (annotate binders; explicit fields; one Field per Context), abstract lia in transparent terms, calc chains for equational proofs, no Program Fixpoint/Equations (plain Fix over measures — Programs/Merge.v is the model), CLAUDE.md vocabulary in comments, Interface/Theory/Programs layout. Specific to this track: coq-memprog is reference material, never a source to copy text from — every ported definition is rewritten in this tree’s idiom; Z is the value/address type at every concrete level (as in memprog); registers stay the 8-register RISC file with the same x86-64 mapping (R0→RAX … R7→R9, R10 = data base) so MemProgX86.hs remains usable as an encoding oracle when writing X86.v.
