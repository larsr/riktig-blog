riktig #19 — Getting started: five imports, five domains

Nineteenth post documenting the riktig project. Not a new theorem — a practical guide to actually using* the library, written after discovering the hard way that “no Require Export” has a real cost for a newcomer.*

What we are doing, and why. Verified software in Coq, built for modifiability. Eighteen posts have shown proofs; none have shown what it’s like to sit down and write your own. This one does — five short, working examples, one per major area of the library, each following the same shape the library itself is built on: state something abstractly, against an interface, then prove a concrete instance satisfies it. The examples are checked in at theories/Examples/GettingStarted.v and rebuilt with the tree, so they keep compiling as the tree moves — if a library change breaks anything shown here, the build breaks there.

Why this needed writing

Early development of this project’s predecessor used Require Export liberally: import one file, transitively get everything it depends on, start proving. Convenient — and a name-clash time bomb, since two unrelated files can easily define a lemma called assoc or refl. riktig deliberately does the opposite: every file states exactly what it needs with Require Import, nothing more, and gets away with it because of a naming convention — each Impl/ file prefixes its instance names (PW_ for Pointwise, LV_ for ListVS, C_ for Complex, Hom_ for HomVS…). No clashes, ever.

The cost lands on the reader. Require Import is not transitive — importing a Theory/ file does not hand you the Ops notations it used internally; you must ask for Ops yourself, every time, in every file. That’s a real, if small, tax on every new proof you write, and it isn’t obvious in advance which files you need. So: a cheat sheet, proven by actually writing the code.

1. Algebra

Abstract statement, over any abelian group:

From Riktig Require Import Ops Props Interface.AbGroup Theory.AbGroup.

Section Generic.
  Context {G : Type} `{AbGroup G}.

  Theorem swap_cancel : forall a b c : G, (a + b) + (c + - a) == b + c.
  Proof.
    intros a b c.
    rewrite (comm a b), assoc, (add_swap_l a c (- a)), invr, idr.
    reflexivity.
  Qed.
End Generic.

Concrete instance, over ℤ:

From Coq Require Import ZArith.
From Riktig Require Import Impl.Zr.

Example z_swap_cancel : (3 + 5 + (7 + - 3))%Z = (5 + 7)%Z.
Proof. exact (swap_cancel (G := Z) 3%Z 5%Z 7%Z). Qed.

The non-obvious import: Props. Nothing in the type of swap_cancel mentions Props — but comm, assoc, invr, idr are lemma names that live there, not in Interface.AbGroup, which only instantiates those properties without re-exporting their names. Skip Props and you get The variable comm was not found — correct, but not obviously pointing at the fix.

2. Linear algebra

Abstract statement, over any linear map:

From Coq Require Import Morphisms QArith.
From Coq.Vectors Require Fin.
From Riktig Require Import Ops Interface.Field Interface.VectorSpace
  Interface.LinearMap Impl.Qf Impl.FinVec.

Section Generic.
  Context {F U V : Type} {FF : Field F}
          {VU : VectorSpace F U} {VV : VectorSpace F V}.

  Theorem lm_combo : forall (f : LinearMap F U V) (a b : F) (u v : U),
    f (a *: u + b *: v) == a *: f u + b *: f v.
  Proof.
    intros f a b u v.
    rewrite (lm_add f), (lm_scale f), (lm_scale f).
    reflexivity.
  Qed.
End Generic.

Concrete instance: a scaling map on ℚ2\mathbb{Q}^2 (Fin.t 2 -> Q), built from the field’s own scale laws:

Definition scale_map {F V} {FF : Field F} {VV : VectorSpace F V} (c : F)
  : LinearMap F V V := (* ... three law proofs, then Build_LinearMap *) .

Definition triple : LinearMap Q (Fin.t 2 -> Q) (Fin.t 2 -> Q) :=
  scale_map (3#1)%Q.

Example fin2_combo : forall u v : Fin.t 2 -> Q,
  triple (2 *: u + 5 *: v) == 2 *: triple u + 5 *: triple v.
Proof. intros u v. exact (lm_combo triple (2#1)%Q (5#1)%Q u v). Qed.

The non-obvious pitfall: import order, not import contents. Writing Impl.Qf/Impl.FinVec after the generic Section (rather than at the top of the file) made scalar multiplication resolve to the field’s self-scaling instance before the vector type was pinned down — a flexible-carrier typeclass search picking the wrong instance silently, producing a baffling type mismatch on a line that looked completely correct. Moving every import above every Section fixed it outright. Rule of thumb: all imports first, always — this codebase’s Hint Mode discipline (documented in project memory as the fix for a much worse historical bug — a compiler-crashing search divergence) depends on the carrier being syntactically pinned down before instance search fires, and import order can undermine that even when every individual import is correct.

3. Category theory

Abstract statement, over any category, about isomorphisms:

From Riktig Require Import Ops Interface.Category Theory.Category.

Section Generic.
  Context {O : Type} `{Category O}.

  Theorem iso_absorb : forall (x y : O) (f : Iso x y),
    (iso_from f ∘ iso_to f) ∘ iso_from f == iso_from f.
  Proof.
    intros x y f.
    rewrite (iso_fromto f), cat_idl.
    reflexivity.
  Qed.
End Generic.

Concrete instance: bool is isomorphic to itself via negb (types-as-objects, Impl/CoqCat.v):

From Coq Require Import Bool.
From Riktig Require Import Impl.CoqCat.

Definition negb_negb_iso : @Iso Type CoqCat bool bool := {|
  iso_to := negb; iso_from := negb;
  iso_fromto := negb_involutive; iso_tofrom := negb_involutive
|}.

Example negb_absorb : forall b : bool, negb (negb (negb b)) = negb b.
Proof. exact (iso_absorb _ _ negb_negb_iso). Qed.

The sharpest gotcha of this whole post, and a real bug this exercise found and fixed: Theory/Category.v declared Arguments iso_to {x y}. inside a Section, without the Global keyword. A plain, non-Global Arguments directive is section-local — it silently reverts the moment the section closes, so anyone who just Require Imports the file gets back the fully-explicit signature, no warning given. The symptom isn’t a clean “wrong number of arguments” — it’s a bizarre type mismatch buried in typeclass elaboration, because Coq inserts fresh implicits for the class’s own parameters and then feeds your argument into the wrong slot. It took Print Implicit iso_from to even see what had happened. Now fixed — Global Arguments, as it should have been from the start — but it’s a sharp enough trap to name here for the general lesson: if you write Arguments {x y}. inside a Section and mean for it to survive export, say Global Arguments.

4. Lambda calculus

Abstract fact (post #5’s if_of_true-style branch law), instantiated at two concrete closed terms:

From Riktig Require Import Ops Lambda.Syntax Lambda.Conv Lambda.Church.

Example ifb_true_idt_k : IFB TRUE idt k == idt.
Proof.
  apply IF_TRUE.
  - apply value_abs.
  - apply value_abs.
Qed.

(* Cross-check: the same fact, by running the evaluator on both sides
   and comparing normal forms. *)
Example ifb_true_idt_k_by_eval : IFB TRUE idt k == idt.
Proof. apply (conv_by_eval 20 5). reflexivity. Qed.

Two independent proofs of the same statement — one by the algebraic law, one by brute-force evaluation — agreeing, is itself a small sanity check on the whole conv setoid from post #5. No import surprises here: Lambda/Church.v carries its own comment naming exactly what it needs from Ops.

5. Channels

A concrete fact about a specific loss pattern, then the same fact via a general lattice lemma:

From Coq Require Import List.
Import ListNotations.
From Riktig Require Import Channel.Trace.

Example drop_middle : lossy [1; 2; 3] [1; 3].
Proof. apply subseq_keep, subseq_drop, subseq_keep, subseq_nil. Qed.

Example perfect_is_lossy_123 : lossy [1; 2; 3] [1; 2; 3].
Proof. apply perfect_lossy. reflexivity. Qed.

The non-obvious import, in the other direction from example 2: Channel/Trace.v itself does Import ListNotations. at its own top — but a plain Import (unlike Export) does not propagate to a client file. You have to re-Import ListNotations yourself, even though the library file you’re building on visibly already did. Small, but it’s exactly the shape of surprise the whole “no Require Export” policy produces, working as designed: nothing is handed to you implicitly, which is safe, and occasionally means repeating one line.

The cheat sheet

I want to… Import
use ==, +, *:, - notations at all Ops
name a law like comm/assoc/invr directly Props
state something over an abstract Monoid/Group/AbGroup/Ring/Field Interface.<Name>
use derived facts (cancel_l, mul_0_l, …) Theory.<Name>
work with vectors/linear maps abstractly Interface.VectorSpace, Interface.LinearMap
a concrete field to compute in Impl.Qf (ℚ), Impl.Zr (ℤ), Impl.Complex (ℂ)
a concrete finite-dimensional space Impl.FinVec (Fin n -> F with its dot product)
categories, functors, isomorphisms Interface.Category, Theory.Category
Coq’s own types as a category Impl.CoqCat
the lambda calculus, runnable Lambda.Syntax (eval), Lambda.Conv (the == setoid), Lambda.Church (booleans/pairs)
channel fault models Channel.Trace

And the two rules that generalize past this list: put every import before every Section (order can silently redirect instance search), and Require Import never re-exports transitively — if a file you depend on used a notation or a name, and you want to use it too, ask for it yourself.
